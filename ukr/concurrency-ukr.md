## Розкажіть про модель пам'яті Java
Ментальна модель: __модель пам'яті Java (Java Memory Model, JMM)__ — це «контракт» між програмістом і середовищем виконання про те, які записи одного потоку гарантовано побачить інший потік і в якому порядку. Без цього контракту компілятор, процесор і кеші вільні переупорядковувати операції та кешувати значення так, що один потік може просто не побачити змін, зроблених іншим.

Формально модель пам'яті визначає набір дій міжпотокової взаємодії (зокрема, читання і запис змінної, захоплення і звільнення монітора, читання і запис `volatile` змінної, запуск нового потоку), а також визначає відношення між цими діями — _happens-before_. Це абстракція, що означає: якщо операція _X_ пов'язана відношенням happens-before з операцією _Y_, то весь код після операції _Y_, що виконується в одному потоці, бачить усі зміни, зроблені іншим потоком до операції _X_.

Існує кілька основних правил для відношення happens-before:

+ У межах одного потоку будь-яка операція happens-before будь-якою операцією, що йде за нею у вихідному коді;
+ Звільнення монітора (unlock) happens-before захоплення того самого монітора (lock);
+ Вихід із `synchronized` блоку/методу happens-before вхід у `synchronized` блок/метод на тому самому моніторі;
+ Запис `volatile` поля happens-before читання того самого `volatile` поля;
+ Завершення методу `run()` екземпляра класу `Thread` happens-before вихід із методу `join()` або повернення `false` методом `isAlive()` екземпляром того самого потоку;
+ Виклик методу `start()` екземпляра класу `Thread` happens-before початок методу `run()` екземпляра того самого потоку;
+ Виклик методу `interrupt()` на потоці happens-before виявлення потоком факту, що цей метод був викликаний, — або шляхом викидання винятку `InterruptedException`, або за допомогою методів `isInterrupted()` чи `interrupted()`;
+ Зв'язок happens-before транзитивний, тобто якщо _X_ happens-before _Y_, а _Y_ happens-before _Z_, то _X_ happens-before _Z_;
+ Звільнення/захоплення монітора і запис/читання у `volatile` змінну пов'язані відношенням happens-before лише тоді, коли операції проводяться над одним і тим самим екземпляром об'єкта;
+ У відношенні happens-before беруть участь лише два потоки; про поведінку решти потоків нічого сказати не можна, доки в кожному з них не настане відношення happens-before з іншим потоком.

> Раніше тут згадувалося правило «завершення конструктора happens-before початок методу `finalize()`». Метод `finalize()` оголошений застарілим (deprecated) у Java 9 і призначений до видалення, тож на сучасних версіях на нього покладатися не варто — для звільнення ресурсів використовуйте `try-with-resources` або `java.lang.ref.Cleaner`.

Можна виділити кілька основних областей, що стосуються моделі пам'яті:

__Видимість (visibility).__ Один потік може на якийсь час тимчасово зберегти значення деяких полів не в основну пам'ять, а в регістри або локальний кеш процесора, тож другий потік, що виконується на іншому процесорі, читаючи з основної пам'яті, може не побачити останніх змін поля. І навпаки: якщо потік протягом якогось часу працює з регістрами та локальними кешами, читаючи дані звідти, він може одразу не побачити змін, зроблених іншим потоком в основну пам'ять.

До питання видимості мають стосунок такі ключові слова мови Java: `synchronized`, `volatile`, `final`.

З точки зору Java всі змінні (за винятком локальних змінних, оголошених усередині методу) зберігаються в головній пам'яті, доступній усім потокам; крім цього, кожен потік має локальну — робочу — пам'ять, де він зберігає копії змінних, з якими працює, і під час виконання програми потік працює лише з цими копіями. Слід зазначити, що це не вимога до реалізації, а лише модель, що пояснює поведінку програми: так, у ролі локальної пам'яті не обов'язково виступає кеш-пам'ять — це можуть бути регістри процесора, або потоки можуть взагалі не мати локальної пам'яті.

При вході в `synchronized` метод або блок потік оновлює вміст локальної пам'яті, а при виході із `synchronized` методу або блоку потік записує зміни, зроблені в локальній пам'яті, у головну. Така поведінка `synchronized` методів і блоків випливає з правил для відношення happens-before: оскільки всі операції з пам'яттю happens-before звільнення монітора, а звільнення монітора happens-before захоплення монітора, то всі операції з пам'яттю, зроблені потоком до виходу із `synchronized` блоку, мають бути видимі будь-якому потоку, що входить у `synchronized` блок для того самого монітора. Дуже важливо, що це правило працює лише в тому разі, якщо потоки синхронізуються, використовуючи один і той самий монітор!

Що стосується `volatile` змінних, то запис таких змінних здійснюється в основну пам'ять, оминаючи локальну, і читання `volatile` змінної також здійснюється з основної пам'яті, тобто значення змінної не може зберігатися в регістрах чи локальній пам'яті потоку, а операція читання цієї змінної гарантовано поверне останнє записане в неї значення.

Також модель пам'яті визначає додаткову семантику ключового слова `final`, що стосується видимості: після того як об'єкт був коректно створений, будь-який потік може бачити значення його `final` полів без додаткової синхронізації. «Коректно створений» означає, що посилання на створюваний об'єкт не повинне використовуватися доти, доки не завершився конструктор об'єкта (тобто не повинно «витекти» з конструктора — наприклад, через `this` у listener). Наявність такої семантики для `final` дозволяє створювати незмінні (immutable) об'єкти, що містять лише `final` поля; такі об'єкти можуть вільно передаватися між потоками без забезпечення синхронізації при передачі.

Є одна проблема, пов'язана з `final` полями: реалізація дозволяє змінювати значення таких полів після створення об'єкта (це можна зробити, наприклад, через reflection). Якщо значення `final` поля — константа, чиє значення відоме на момент компіляції, зміна такого поля може не мати ефекту, бо звернення до цієї змінної могли бути замінені компілятором на константу. Також специфікація дозволяє інші оптимізації, пов'язані з `final` полями. Тому рекомендується змінювати `final` поля об'єкта лише всередині конструктора — інакше поведінка не специфікована.

__Reordering (переупорядкування).__ Для збільшення продуктивності процесор/компілятор можуть переставляти місцями деякі інструкції/операції. Точніше, з точки зору потоку, що спостерігає за виконанням операцій в іншому потоці, операції можуть бути виконані не в тому порядку, в якому вони йдуть у вихідному коді. Той самий ефект може спостерігатися, коли один потік кладе результати першої операції в регістр чи локальний кеш, а результат другої операції потрапляє безпосередньо в основну пам'ять. Тоді другий потік, звертаючись до основної пам'яті, може спершу побачити результат другої операції, і лише потім — першої, коли всі регістри чи кеші синхронізуються з основною пам'яттю. Ще одна причина reordering — процесор може змінити порядок виконання операцій, якщо, наприклад, вважатиме, що така послідовність виконається швидше.

Питання reordering також регулюється набором правил для відношення happens-before, і ці правила мають наслідок, що застосовується на практиці: операції читання і запису `volatile` змінних не можуть бути переупорядковані з операціями читання і запису інших `volatile` і не-`volatile` змінних. Цей наслідок робить можливим використання `volatile` змінної як прапорця, що сигналізує про завершення якоїсь дії. В інших випадках правила, що стосуються порядку виконання операцій, гарантують упорядкованість лише для конкретного набору ситуацій (як-от захоплення і звільнення монітора), у всіх інших залишаючи компілятору і процесору повну свободу для оптимізацій.

[до змісту](#Багатопотоковість)

## Що таке «потокобезпека»?
Потокобезпека (thread-safety) — властивість об'єкта або коду, що гарантує: при виконанні чи використанні кількома потоками код поводитиметься так, як передбачено. Наприклад, потокобезпечний лічильник не пропустить жодного підрахунку, навіть якщо один і той самий його екземпляр використовуватиметься кількома потоками.

[до змісту](#Багатопотоковість)

## У чому різниця між _«конкурентністю»_ і _«паралелізмом»_?
Коротко: __конкурентність — це про структуру__ (як упоратися з багатьма задачами одночасно), __паралелізм — це про виконання__ (як швидше виконати одну задачу, розклавши її на частини, що йдуть фізично одночасно). Конкурентність можлива і на одному ядрі (потоки чергуються), паралелізм без кількох ядер неможливий.

Конкурентність (concurrency) — це спосіб одночасного розв'язання множини задач.

Ознаки:

+ Наявність кількох потоків керування (наприклад, _Thread_ у Java, _корутина_ в Kotlin); якщо потік керування один, конкурентного виконання бути не може;
+ Недетермінований результат виконання. Результат залежить від випадкових подій, реалізації та того, як проведено синхронізацію. Навіть якщо кожен потік повністю детермінований, підсумковий результат буде недетермінованим.

Паралелізм (parallelism) — це спосіб виконання різних частин однієї задачі.

Ознаки:

+ Не обов'язково має кілька потоків керування;
+ Може давати детермінований результат: наприклад, результат множення кожного елемента масиву на число не зміниться, якщо множити масив частинами паралельно.

[до змісту](#Багатопотоковість)

## Що таке _«кооперативна багатозадачність»_? Який тип багатозадачності використовує Java? Чим зумовлений цей вибір?

__Кооперативна багатозадачність__ — це спосіб поділу процесорного часу між потоками, за якого кожен потік зобов'язаний добровільно віддавати керування наступному.

Переваги такого підходу — простота реалізації, менші накладні витрати на перемикання контексту.

Недоліки — якщо один потік завис або поводиться некоректно, то зависає вся система цілком, і інші потоки ніколи не отримають керування.

Java використовує __витісняльну (preemptive) багатозадачність__, за якої рішення про перемикання між потоками процесу приймає операційна система.

На відміну від кооперативної багатозадачності, керування операційній системі передається незалежно від стану працюючих застосунків, завдяки чому окремі завислі потоки процесу, як правило, не «підвішують» усю систему. За рахунок регулярного перемикання між задачами також поліпшується чутливість застосунку та підвищується оперативність звільнення ресурсів, що більше не використовуються.

У реалізації витісняльна багатозадачність відрізняється від кооперативної, зокрема тим, що потребує обробки системного переривання від апаратного таймера.

[до змісту](#Багатопотоковість)

## Що таке _ordering_, _as-if-serial semantics_, _sequential consistency_, _visibility_, _atomicity_, _happens-before_, _mutual exclusion_, _safe publication_?
__ordering__ — механізм, що визначає, коли один потік може побачити _out-of-order_ (переупорядкований) порядок виконання інструкцій іншого потоку. CPU задля підвищення продуктивності може переупорядковувати процесорні інструкції та виконувати їх у довільному порядку доти, доки для потоку всередині не буде видно жодних відмінностей. Гарантія, що надається цим механізмом, називається __as-if-serial semantics__.

__sequential consistency__ — те саме, що й _as-if-serial semantics_: гарантія того, що в межах одного потоку побічні ефекти від усіх операцій будуть такими, ніби всі операції виконуються послідовно.

__visibility__ — визначає, коли дії в одному потоці стають видимі з іншого потоку.

__happens-before__ — логічне обмеження на порядок виконання інструкцій програми. Якщо зазначено, що запис у змінну й подальше її читання пов'язані через цю залежність, то хоч би як переупорядковувалися інструкції під час виконання, у момент читання всі пов'язані з процесом записи результатів уже зафіксовані й видимі.

__atomicity__ — атомарність операцій. Атомарна операція виглядає як єдина й неподільна команда процесора, що може бути або вже виконаною, або ще не виконаною — проміжного стану не існує.

__mutual exclusion__ (взаємовиключна блокування) — механізм, що гарантує потоку ексклюзивний доступ до ресурсу. Використовується для запобігання одночасному доступу до спільного ресурсу. У кожен момент часу таким ресурсом може володіти лише один потік. Найпростіший приклад: `synchronized (obj) { … }`.

__safe publication__ — безпечне «оприлюднення» об'єктів іншим потокам із поточного, не порушуючи обмежень _visibility_. Способи такої публікації в Java:

+ `static {}` ініціалізатор;
+ `volatile` змінні;
+ `atomic` змінні;
+ збереження у спільну змінну, коректно захищену за допомогою `synchronized()`, синхронізаторів чи інших конструкцій, що створюють _read/write memory barrier_;
+ `final` змінні в спільному об'єкті, що був коректно проініціалізований.

## Чим відрізняється процес від потоку?
__Процес__ — екземпляр програми під час виконання, незалежний об'єкт, якому виділено системні ресурси (наприклад, процесорний час і пам'ять). Кожен процес виконується в окремому адресному просторі: один процес не може отримати доступ до змінних і структур даних іншого. Якщо процес хоче отримати доступ до чужих ресурсів, потрібно використовувати міжпроцесну взаємодію (IPC) — це можуть бути конвеєри (pipes), файли, сокети, канали зв'язку між комп'ютерами тощо.

Для кожного процесу ОС створює так званий «віртуальний адресний простір», до якого процес має прямий доступ. Цей простір належить процесу, містить лише його дані й перебуває в повному його розпорядженні. Операційна система ж відповідає за те, як віртуальний простір процесу проєктується на фізичну пам'ять.

__Потік (thread)__ — певний спосіб виконання процесу, що визначає послідовність виконання коду в процесі. Потоки завжди створюються в контексті якогось процесу, і все їхнє життя проходить лише в його межах. Потоки можуть виконувати один і той самий код і маніпулювати одними й тими самими даними, а також спільно використовувати дескриптори об'єктів ядра, оскільки таблиця дескрипторів створюється не в окремих потоках, а в процесах. Оскільки потоки витрачають істотно менше ресурсів, ніж процеси, у процесі роботи вигідніше створювати додаткові потоки й уникати створення нових процесів.

[до змісту](#Багатопотоковість)

## Що таке _«зелені потоки»_ і чи є вони в Java?
__Зелені (легковагі) потоки (green threads)__ — потоки, що емулюються віртуальною машиною або середовищем виконання. Створення зеленого потоку не передбачає створення реального потоку ОС.

Віртуальна машина бере на себе турботу про перемикання між різними green threads, а сама працює як один потік ОС. Це дає кілька переваг: потоки ОС відносно дорогі в більшості POSIX-систем, а крім того, перемикання між native threads набагато повільніше, ніж між green threads. Система може підтримувати набагато більшу кількість green threads, ніж потоків ОС. Наприклад, набагато практичніше запускати новий green thread для нового HTTP-з'єднання до вебсервера, ніж створювати новий native thread.

Однак є й недолік класичної моделі «один потік ОС»: у ній неможливо виконувати два потоки фізично одночасно навіть на багатоядерній машині, бо з точки зору планувальника ОС усе виглядає одним потоком.

Класичні green threads застосовувалися в ранніх версіях Java; починаючи з версії 1.2 Java за замовчуванням використовує native threads.

> Важливе сучасне доповнення: у __Java 21__ (фінально, після прев'ю в Java 19–20, JEP 444) з'явилися __віртуальні потоки (virtual threads)__ — це фактично нова, набагато потужніша інкарнація ідеї зелених потоків. Багато віртуальних потоків мультиплексуються (M:N) на невелику кількість потоків-носіїв (carrier threads) ОС, що дозволяє створювати мільйони потоків і писати простий блокуючий код, який масштабується. На відміну від старих green threads, віртуальні потоки повноцінно використовують усі ядра, бо carrier threads — це справжні native threads у пулі ForkJoinPool. Це сучасна відповідь на питання «чи є в Java зелені потоки»: так, у вигляді virtual threads.

## Якими способами можна створити потік?
+ Створити нащадка класу `Thread` і перевизначити його метод `run()`;
+ Створити об'єкт класу `Thread`, передавши йому в конструкторі екземпляр класу, що реалізує інтерфейс `Runnable`. Цей інтерфейс містить метод `run()`, який виконуватиметься в новому потоці. Потік завершить виконання, коли завершиться його метод `run()`;
+ Викликати метод `submit()` (або `execute()`) у екземпляра `ExecutorService`, передавши йому як параметр екземпляр класу, що реалізує інтерфейс `Runnable` або `Callable` (містить метод `call()`, у якому описана логика виконання). На практиці це найкращий спосіб, бо пул потоків керує життєвим циклом потоків за вас;
+ Починаючи з __Java 21__ — створити віртуальний потік: `Thread.startVirtualThread(runnable)` або через `Executors.newVirtualThreadPerTaskExecutor()`.

[до змісту](#Багатопотоковість)

## Чим різняться `Thread` і `Runnable`?
`Thread` — це клас, певна надбудова над фізичним потоком.

`Runnable` — це інтерфейс, що являє собою абстракцію над виконуваною задачею.

Окрім того, що `Runnable` допомагає розв'язати проблему множинного успадкування (бо успадкувавшись від `Thread`, ви вже не зможете успадкуватися ні від чого іншого), безсумнівний плюс від його використання полягає в тому, що він дозволяє логічно відокремити логіку виконання задачі від безпосереднього керування потоком. Тому передавати `Runnable` в `Thread` чи в пул потоків — кращий підхід, ніж успадковуватися від `Thread`.

[до змісту](#Багатопотоковість)

## У чому полягає різниця між методами `start()` і `run()`?
Попри те, що `start()` викликає метод `run()` усередині себе, це не те саме, що просто виклик `run()`. Якщо `run()` викликати як звичайний метод, то він виконається в тому самому потоці й жодного нового потоку не запуститься — на відміну від випадку, коли ви викликаєте метод `start()`, який створює новий потік і вже в ньому виконує `run()`.

## Як примусово запустити потік?
Ніяк. У Java не існує жодного способу примусово запустити потік у конкретний момент. Це контролюється планувальником JVM/ОС, і Java не надає жодного API для керування цим процесом. Ви можете лише попросити запуск (`start()`), але коли саме потік фактично почне виконуватися — вирішує планувальник.

## Що таке _«монітор»_ в Java?
__Монітор__ (його часто ототожнюють із м'ютексом, mutex) — це засіб контролю за доступом до ресурсу. У монітора може бути щонайбільше один власник у кожен момент часу. Отже, якщо хтось використовує ресурс і захопив монітор для забезпечення одноосібного доступу, то інший, хто хоче використати той самий ресурс, мусить дочекатися звільнення монітора, захопити його й лише потім почати використовувати ресурс.

Зручно уявляти монітор як ідентифікатор потоку, що його захопив: якщо він порожній — ресурс вільний; якщо ні — ресурс зайнятий, і можна стати в чергу й чекати звільнення.

У Java кожен екземпляр об'єкта має монітор, що контролюється безпосередньо віртуальною машиною. Використовується він так: будь-який нестатичний `synchronized`-метод при своєму виклику передусім намагається захопити монітор того об'єкта, у якого він викликаний (на який він може послатися як на `this`). Якщо це вдалося — метод виконується. Якщо ні — потік зупиняється й чекає, поки монітор буде відпущено.

[до змісту](#Багатопотоковість)

## Дайте визначення поняттю «синхронізація».
Синхронізація — це механізм, що координує доступ кількох потоків до спільних даних так, щоб запобігти пошкодженню стану та забезпечити видимість змін між потоками.

У Java кожен об'єкт має одне вбудоване блокування (монітор), завдяки якому лише один потік одночасно може отримати доступ до критичної секції коду, захищеної цим монітором. Якщо потік отримав блокування, жоден інший потік не може увійти в синхронізований код на тому самому моніторі, поки блокування не буде знято. Коли потік, що володіє блокуванням, виходить із синхронізованого коду, блокування знімається. Тоді інший потік може отримати блокування й виконати синхронізований код. Якщо потік намагається отримати блокування, коли ним володіє інший потік, він переходить у стан очікування (Blocked), доки блокування не зніметься.

[до змісту](#Багатопотоковість)

## Які існують способи синхронізації в Java?
+ __Системна синхронізація з використанням `wait()`/`notify()`__. Потік, що чекає на виконання якихось умов, викликає у спільного об'єкта метод `wait()`, попередньо захопивши його монітор. На цьому його робота призупиняється. Інший потік може викликати на цьому самому об'єкті метод `notify()` (знову ж таки, попередньо захопивши монітор об'єкта), внаслідок чого потік, що чекає на об'єкті, «прокидається» й продовжує виконання. В обох випадках монітор треба захоплювати в явному вигляді, через `synchronized`-блок, бо методи `wait()`/`notify()` самі по собі не синхронізовані.

+ __Системна синхронізація з використанням `join()`__. Метод `join()`, викликаний у екземпляра класу `Thread`, дозволяє поточному потоку зупинитися до того моменту, як потік, пов'язаний із цим екземпляром, завершить роботу.

+ __Використання класів із пакета `java.util.concurrent`__, що надає набір класів для організації міжпотокової взаємодії. Приклади таких класів — `Lock`, `Semaphore`, `CountDownLatch` тощо. Концепція цього підходу полягає у використанні високорівневих синхронізаторів і атомарних операцій замість низькорівневих `wait()`/`notify()`.

## У яких станах може перебувати потік?
В Java стан потоку формалізований у переліку `Thread.State` (метод `getState()`). Фактичні значення такі:

+ __`NEW`__. Потік створено (`new Thread(...)`), але метод `start()` ще не викликано. У цьому стані потік не вважається «живим».
+ __`RUNNABLE`__. Потік або виконується, або готовий до виконання й чекає на процесорний час від планувальника. У `Thread.State` немає окремого стану «Running» — і виконання, і готовність об'єднані в `RUNNABLE`.
+ __`BLOCKED`__. Потік заблокований в очікуванні захоплення монітора, щоб увійти в `synchronized` блок/метод (або повторно в нього зайти після `wait()`).
+ __`WAITING`__. Потік чекає невизначений час: викликано `wait()` без таймауту, `join()` без таймауту або `LockSupport.park()`. Вивести з цього стану може `notify()`/`notifyAll()`, завершення відповідного потоку тощо.
+ __`TIMED_WAITING`__. Потік чекає обмежений час: `sleep(ms)`, `wait(ms)`, `join(ms)`, `LockSupport.parkNanos()` тощо.
+ __`TERMINATED`__. Метод `run()` повністю виконано (або потік завершився через виняток). Завершений потік не може перейти в жоден інший стан — повторний виклик `start()` спричинить `IllegalThreadStateException`.

[до змісту](#Багатопотоковість)

## Чи можна створювати нові екземпляри класу, поки виконується `static synchronized` метод?
Так, можна. `static synchronized` метод захоплює монітор об'єкта `Class` (наприклад, `MyClass.class`), а не монітори майбутніх екземплярів. Створення нового екземпляра (виклик конструктора через `new`) не потребує захоплення монітора класу, тож воно ніяк не блокується виконанням статичного синхронізованого методу.

## Навіщо може бути потрібен `private` м'ютекс?
Об'єкт для синхронізації роблять `private`, щоб сторонній код не міг на нього синхронізуватися й випадково (або зловмисно) спричинити взаємне блокування або змагання за блокування. Класичний антипатерн — синхронізація на `this` чи на публічному полі: тоді будь-який зовнішній код, що має посилання на ваш об'єкт, може захопити той самий монітор і вплинути на вашу синхронізацію. Виділений `private final Object lock = new Object();` ізолює блокування й робить його деталлю реалізації.

## Як працюють методи `wait()` і `notify()`/`notifyAll()`?
Ці методи визначені в класі `Object` і призначені для взаємодії потоків між собою при міжпотоковій синхронізації.

+ `wait()`: звільняє монітор і переводить викликаючий потік у стан очікування доти, доки інший потік не викличе метод `notify()`/`notifyAll()`;
+ `notify()`: сигналізує одному з потоків, що чекають на цьому моніторі, щоб він прокинувся;
+ `notifyAll()`: сигналізує всім потокам, що чекають на цьому моніторі.

Коли викликано метод `wait()`, потік звільняє блокування на об'єкті й переходить у стан очікування (`WAITING`). Метод `notify()` подає сигнал одному з потоків, що чекають на об'єкті, аби той перейшов у стан готовності. При цьому неможливо передбачити, який саме з потоків стане готовим. Метод `notifyAll()` змушує всі потоки, що чекають на об'єкті, повернутися в стан готовності. Якщо жоден потік не чекає на `wait()`, то виклик `notify()`/`notifyAll()` нічого не робить.

Потік може викликати `wait()` чи `notify()` для певного об'єкта, лише якщо він наразі утримує монітор цього об'єкта. Тому `wait()`, `notify()` і `notifyAll()` мають викликатися тільки із синхронізованого коду на тому самому моніторі.

Важливий практичний нюанс: `wait()` завжди слід викликати в циклі, перевіряючи умову (`while (!condition) lock.wait();`), а не в `if`. Це захищає від так званих _spurious wakeups_ (хибних пробуджень) і від ситуації, коли умова вже не виконується на момент, коли потік знову захопив монітор.

[до змісту](#Багатопотоковість)

## У чому різниця між `notify()` і `notifyAll()`?
На `wait()` одного монітора можуть «висіти» одразу кілька потоків. При виклику `notify()` лише один із них виходить із `wait()`, намагається захопити монітор і потім продовжує роботу з оператора, наступного за `wait()`. Який саме з них прокинеться — заздалегідь невідомо. А при виклику `notifyAll()` усі потоки, що висять на `wait()`, виходять із нього й усі намагаються захопити монітор. Зрозуміло, що в будь-який момент часу монітор може утримувати лише один потік, а решта чекають своєї черги. Порядок черги визначається планувальником потоків.

На практиці майже завжди безпечніше використовувати `notifyAll()`: `notify()` може «розбудити не той потік» (наприклад, якщо на одному моніторі чекають потоки з різними умовами), що призведе до зависання. `notify()` варто застосовувати лише тоді, коли ви впевнені, що всі потоки, які чекають, еквівалентні.

[до змісту](#Багатопотоковість)

## Чому методи `wait()` і `notify()` викликаються тільки в синхронізованому блоці?
Бо потік повинен утримувати монітор об'єкта, на якому він викликає `wait()`/`notify()` — інакше буде кинуто `IllegalMonitorStateException`. Глибша причина: `wait()` має атомарно звільнити монітор і заснути, а `notify()` має сигналізувати під захистом того самого монітора. Якби ці виклики дозволялися без утримання монітора, виникла б класична гонка «lost wakeup»: потік перевірив умову, але не встиг заснути, як інший потік уже зробив `notify()` — і сигнал було б втрачено. Утримання монітора на час перевірки умови та виклику `wait()` усуває цю гонку.

## Чим відрізняється робота методу `wait()` з параметром і без параметра?
`wait()`

+ __без параметрів__ звільняє монітор і переводить викликаючий потік у стан очікування доти, доки інший потік не викличе `notify()`/`notifyAll()` (стан `WAITING`);
+ __з параметрами__ (`wait(long millis)` / `wait(long millis, int nanos)`) змусить потік чекати задану кількість часу або до виклику `notify()`/`notifyAll()` — що настане раніше (стан `TIMED_WAITING`). Після спливання таймауту потік прокинеться сам.

[до змісту](#Багатопотоковість)

## Чим відрізняються методи `Thread.sleep()` і `Thread.yield()`?
Метод `yield()` — це підказка планувальнику, що поточний потік готовий поступитися процесором іншим потокам тієї самої або вищої пріоритетності. Потік переходить зі стану виконання у стан готовності (`RUNNABLE`), але монітори при цьому не звільняє. Планувальник може цю підказку проігнорувати, і наступним виконуватися може й той самий потік. На практиці `yield()` використовується рідко й здебільшого для тонкого тюнінгу/тестування.

Метод `sleep()` присипляє поточний потік на заданий час (стан `TIMED_WAITING`), не звільняючи при цьому утримуваних моніторів. Це ключова відмінність від `wait()`: `sleep()` не відпускає блокування, а `wait()` відпускає.

[до змісту](#Багатопотоковість)

## Як працює метод `Thread.join()`?
Коли потік викликає `join()` на іншому потоці, поточний потік чекає, доки той інший потік не завершиться:

```java
void join()
void join(long millis)
void join(long millis, int nanos)
```

Технічно `join()` усередині реалізований через `wait()` на об'єкті `Thread`: коли потік завершує `run()`, JVM викликає `notifyAll()` на ньому, чим пробуджує всіх, хто на ньому «join-иться». Існування happens-before між завершенням `run()` і поверненням із `join()` гарантує, що потік, який чекав, побачить усі зміни, зроблені завершеним потоком.

## Що таке _deadlock_?
__Взаємне блокування (deadlock)__ — ситуація, коли двоє чи більше потоків заблоковані назавжди, бо кожен чекає на ресурс, який утримує інший. Виникає за одночасного дотримання чотирьох умов (умови Коффмана):

1. __Взаємне виключення:__ принаймні один ресурс утримується в неподільному режимі — лише один потік може використовувати ресурс у будь-який момент часу.
2. __Утримання й очікування:__ потік утримує щонайменше один ресурс і запитує додаткові ресурси, які утримуються іншими потоками.
3. __Відсутність примусового витіснення:__ ресурс не можна відібрати в потоку примусово — він має звільнити його сам.
4. __Циклічне очікування:__ існує замкнений ланцюг потоків, де кожен чекає звільнення ресурсу, який утримує наступний у ланцюзі.

Найпростіший спосіб уникнути взаємного блокування — не допускати циклічного очікування. Цього можна досягти, захоплюючи монітори спільних ресурсів завжди в одному визначеному порядку (lock ordering) і звільняючи їх у зворотному.

[до змісту](#Багатопотоковість)

## Що таке _livelock_?
_livelock_ — тип взаємного блокування, за якого кілька потоків не заблоковані, а навпаки — активно виконують роботу, але постійно реагують на дії одне одного й через це не просуваються вперед. Фактичної помилки не виникає, потоки не зависають у `BLOCKED`, але корисний прогрес дорівнює нулю, а CPU марно завантажений. Часто виникає як результат невдалих спроб уникнути deadlock (потоки «ввічливо» відступають одночасно й нескінченно).

> Реальний приклад livelock — коли двоє людей зустрічаються у вузькому коридорі й кожен, намагаючись бути ввічливим, відступає вбік; так вони нескінченно рухаються з боку в бік, абсолютно не просуваючись у потрібному напрямку.

[до змісту](#Багатопотоковість)

Статичний метод `Thread.holdsLock(lock)` повертає `true`, коли поточний потік утримує монітор зазначеного об'єкта.

## На якому об'єкті відбувається синхронізація при виклику `static synchronized` методу?
Синхронізований статичний метод не має доступу до `this`, але має доступ до об'єкта класу `Class`, що існує в єдиному екземплярі, — саме він і виступає монітором для синхронізації статичних методів. Таким чином, така конструкція:

```java
public class SomeClass {

    public static synchronized void someMethod() {
        // code
    }
}
```

еквівалентна такій:

```java
public class SomeClass {

    public static void someMethod() {
        synchronized (SomeClass.class) {
            // code
        }
    }
}
```

## Для чого використовується ключове слово `volatile`, `synchronized`, `transient`, `native`?
__`volatile`__ — змушує читати/писати значення змінної безпосередньо з/в основну пам'ять, забороняючи кешувати її в регістрах чи локальному кеші потоку, а також обмежує переупорядкування навколо доступів до неї (гарантує видимість і happens-before). Якщо змінна примітивного типу — цього достатньо для забезпечення видимості. Якщо ж змінна є посиланням на об'єкт — синхронізовано (видиме) буде лише значення самого посилання; дані всередині об'єкта при цьому не стають потокобезпечними. Важливо: `volatile` не забезпечує атомарності складених операцій (наприклад, `count++`).

__`synchronized`__ — зарезервоване слово, що дозволяє досягати взаємного виключення й видимості в позначених ним методах або блоках коду через захоплення монітора.

Ключові слова `transient` і `native` до багатопотоковості жодного стосунку не мають: перше вказує поля класу, які не потрібно серіалізувати, а друге сигналізує, що метод реалізований у платформозалежному (зазвичай C/C++) коді через JNI.

[до змісту](#Багатопотоковість)

## У чому відмінності між `volatile` і _Atomic_ змінними?
`volatile` гарантує видимість значення єдиної змінної між потоками, але не гарантує атомарності складених операцій. Наприклад, операція `count++` не стане атомарною лише тому, що `count` оголошено `volatile`, — це насправді три дії (читання, інкремент, запис), між якими можливе втручання іншого потоку.

З іншого боку, клас `AtomicInteger` (та інші `Atomic*`) надає атомарні методи для виконання таких складених операцій, наприклад `getAndIncrement()` — атомарна заміна оператора інкремента. Усередині `Atomic*` побудовані на інструкції CAS (compare-and-swap), що дає атомарність без блокувань. Аналогічні атомарні версії існують і для інших типів даних (`AtomicLong`, `AtomicReference` тощо).

| Властивість | `volatile` | `Atomic*` |
|---|---|---|
| Видимість | так | так |
| Атомарність читання/запису | так | так |
| Атомарність складених операцій (`++`, `compareAndSet`) | ні | так |
| Реалізація | бар'єри пам'яті | CAS + бар'єри пам'яті |

## У чому полягають відмінності між `compareAndSet()` і `weakCompareAndSetPlain()` у класах `Atomic*`?
> Примітка: історична назва `weakCompareAndSwap()` у сучасному API (з Java 9) — це `weakCompareAndSetPlain()`; «звичайний» варіант — `compareAndSet()`.

+ `compareAndSet()` має семантику `volatile`: створює бар'єри пам'яті й дає гарантії happens-before; за рівності очікуваного значення він завжди виконує заміну (не може «безпідставно» провалитися);
+ «слабка» версія (`weakCompareAndSetPlain()` та інші weak-варіанти) може повертати `false` без видимої причини (spurious failure), навіть якщо значення збігається, тож її майже завжди використовують у циклі;
+ слабкі варіанти можуть мати слабші гарантії впорядкування пам'яті й сильно залежать від кеша/архітектури CPU; вони потенційно дешевші, але виграш реальний лише в дуже гарячих циклах і не на всіх архітектурах. На практиці у переважній більшості випадків слід використовувати звичайний `compareAndSet()`.

## Що означає _«пріоритет потоку»_?
Пріоритети потоків використовуються планувальником для ухвалення рішень про те, коли якому потоку дозволити працювати. Теоретично високопріоритетні потоки отримують більше процесорного часу, ніж низькопріоритетні. Практично ж обсяг часу процесора, який отримує потік, залежить від багатьох чинників, окрім пріоритету, і поведінка сильно залежить від ОС — на пріоритети потоків покладатися не варто.

Щоб установити пріоритет потоку, використовується метод класу `Thread`: `final void setPriority(int level)`. Значення `level` змінюється в межах від `Thread.MIN_PRIORITY = 1` до `Thread.MAX_PRIORITY = 10`. Пріоритет за замовчуванням — `Thread.NORM_PRIORITY = 5`.

Отримати поточне значення пріоритету можна викликом `final int getPriority()` у екземпляра `Thread`.

[до змісту](#Багатопотоковість)

## Що таке _«потоки-демони»_?
Потоки-демони працюють у фоновому режимі разом із програмою, але не є невід'ємною частиною програми. Якщо процес може виконуватися у фоні роботи основних потоків і його діяльність полягає в обслуговуванні основних потоків застосунку, то такий процес можна запустити як потік-демон за допомогою методу `setDaemon(boolean value)`, викликаного у потоку до його запуску (`start()`). Метод `boolean isDaemon()` дозволяє визначити, чи є потік демоном. Базова властивість потоків-демонів: JVM завершується (а разом із нею й усі демони), коли завершуються всі недемонські потоки — на відміну від звичайних потоків, JVM не чекає завершення демонів. Класичний приклад потоку-демона — потік збирача сміття.

[до змісту](#Багатопотоковість)

## Чи можна зробити основний потік програми демоном?
Ні. Прапорець `daemon` потоку можна змінювати лише до його запуску, а основний потік (`main`) на момент виконання вашого коду вже запущений — спроба викликати `setDaemon()` на ньому призведе до `IllegalThreadStateException`. До того ж семантично це не має сенсу: потоки-демони існують лише для обслуговування основних потоків і не можуть існувати без них.

[до змісту](#Багатопотоковість)

## Що означає _«приспати»_ потік?
Це означає призупинити його на певний проміжок часу, викликавши під час його виконання статичний метод `Thread.sleep()`, передавши потрібну кількість часу в мілісекундах. До спливання цього часу потік може бути виведений зі стану сну викликом `interrupt()`, що спричинить викидання `InterruptedException`. Важливо: `sleep()` не звільняє утримувані монітори.

[до змісту](#Багатопотоковість)

## Чим відрізняються два інтерфейси `Runnable` і `Callable`?
+ Інтерфейс `Runnable` з'явився в Java 1.0, а інтерфейс `Callable` був уведений у Java 5 у складі `java.util.concurrent`;
+ Класи, що реалізують `Runnable`, мають реалізувати метод `run()`; класи, що реалізують `Callable`, — метод `call()`;
+ Метод `Runnable.run()` не повертає значення; `Callable.call()` повертає результат (типу `V`), який можна отримати через об'єкт `Future`;
+ Метод `run()` не може кидати перевірювані (checked) винятки, тоді як метод `call()` — може.

[до змісту](#Багатопотоковість)

## Що таке `FutureTask`?
`FutureTask` — це скасовуване асинхронне обчислення. Цей клас надає базову реалізацію `Future` з методами для запуску й скасування обчислення, запиту його стану та отримання результату. Результат можна отримати лише коли обчислення завершено; метод `get()` блокується, якщо обчислення ще триває. Об'єкти `FutureTask` можуть обгортати об'єкти `Callable` і `Runnable`. Оскільки `FutureTask` реалізує `Runnable`, його можна передати в `Executor` на виконання.

[до змісту](#Багатопотоковість)

## У чому полягають відмінності між `CyclicBarrier` і `CountDownLatch`?
`CountDownLatch` (засув зі зворотним відліком) дає змогу будь-якій кількості потоків чекати, доки не завершиться певна кількість операцій, що виконуються в інших потоках, перш ніж вони будуть «відпущені» для продовження роботи. У конструктор `CountDownLatch(int count)` обов'язково передається кількість операцій, яку треба виконати (метод `countDown()`), щоб засув «відпустив» заблоковані на `await()` потоки.

> Прикладом `CountDownLatch` із життя може бути збір екскурсійної групи: поки не набереться певна кількість людей, екскурсія не почнеться.

`CyclicBarrier` реалізує патерн синхронізації «Бар'єр». Циклічний бар'єр — це точка синхронізації, у якій зустрічається й блокується задана кількість паралельних потоків. Щойно всі потоки прибули, виконується опційна дія (або не виконується, якщо бар'єр ініціалізовано без неї), і після цього бар'єр «ламається», а потоки, що очікували, звільняються. У конструктори `CyclicBarrier(int parties)` і `CyclicBarrier(int parties, Runnable barrierAction)` обов'язково передається кількість сторін, що мають «зустрітися», і опційно — дія, яка має статися, коли сторони зустрілися, але перед тим, як їх буде відпущено.

> `CyclicBarrier` є альтернативою методу `join()`, який «збирає» потоки тільки після того, як вони повністю виконалися.

Головна відмінність: «засув» (`CountDownLatch`) можна використати лише один раз — після того як його лічильник досягне нуля, він уже не скидається; а «бар'єр» (`CyclicBarrier`) можна використовувати багаторазово (циклічно), навіть після того як він «зламається».

[до змісту](#Багатопотоковість)

## Що таке _race condition_?
__Стан гонки (race condition)__ — помилка проєктування багатопотокової системи, за якої коректність результату напряму залежить від відносного порядку (таймінгу) виконання потоків, що ним керувати неможливо. Класичний приклад — два потоки одночасно роблять `count++` без синхронізації: оскільки інкремент не атомарний, одне зі збільшень може загубитися. Стан гонки спричиняє недетерміновані, важко відтворювані помилки.

[до змісту](#Багатопотоковість)

## Чи існує спосіб розв'язання проблеми _race condition_?
Поширені способи розв'язання:

+ __Використання локальної копії / `volatile`__ — для випадку єдиної змінної, коли копіювання або оновлення атомарне (одна машинна команда). `volatile` забезпечує видимість, але не рятує від складених неатомарних операцій.
+ __Синхронізація__ — операції над спільним ресурсом виконуються в синхронізованому блоці (`synchronized`), `Lock`-ах або через атомарні класи (`Atomic*`).
+ __Незмінність (immutability)__ — якщо спільні дані ніколи не змінюються після створення, гонки за ними неможливі в принципі.
+ __Комбінування методів__ — наприклад, копіювати «небезпечні» змінні всередині синхронізованого блоку, щоб скоротити час утримання блокування.

Очевидних автоматичних способів виявлення й виправлення станів гонки не існує. Найкращий спосіб позбутися гонок — правильне проєктування багатозадачної системи з мінімізацією спільного змінюваного стану.

[до змісту](#Багатопотоковість)

## Як зупинити потік?
Наразі в Java прийнято повідомний (cooperative) порядок зупинки потоку. Старі керівні методи `stop()`, `suspend()` і `resume()` (з JDK 1.0) давно позначені як `deprecated` через потенційні взаємні блокування та пошкодження стану (а `stop()`/`suspend()` у нових версіях ще й видаляються).

Для коректної зупинки потоку використовують метод `Thread.interrupt()`. Він виставляє внутрішній прапорець-статус переривання. Стан цього прапорця можна перевірити методами `isInterrupted()` або `Thread.interrupted()` (для поточного потоку). Метод `interrupt()` також здатний вивести потік зі стану очікування або сну: якщо у потоку були викликані `sleep()` або `wait()`, поточний стан перерветься й буде кинуто `InterruptedException` (при цьому прапорець скидається).

Схема дій така:

+ Реалізувати потік;
+ У потоці періодично перевіряти статус переривання через `isInterrupted()`;
+ Якщо прапорець змінився або під час очікування/сну кинуто `InterruptedException`, отже, потік намагаються зупинити ззовні;
+ Ухвалити рішення — продовжити роботу (якщо з якихось причин зупинитися неможливо) або звільнити заблоковані ресурси й завершити виконання.

Можлива проблема цього підходу — блокування на потоковому введенні-виведенні. Якщо потік заблокований на читанні даних, виклик `interrupt()` із цього стану його не виведе. Розв'язання залежить від типу джерела даних: для файлу довготривале блокування малоймовірне, тож можна просто дочекатися виходу з `read()`; якщо ж читання пов'язане з мережею — варто використовувати неблокуюче введення-виведення з Java NIO (або закрити канал, що перерве блокування).

Другий варіант — зробити власний аналог `interrupt()`: оголосити в класі потоку прапорець зупинки й виставляти його ззовні. Недоліки: по-перше, потоки в стані очікування таким способом не «оживити»; по-друге, прапорець обов'язково має бути `volatile`, інакше через кешування інший потік може не побачити його зміну вчасно.

[до змісту](#Багатопотоковість)

## Чому не рекомендується використовувати метод `Thread.stop()`?
`stop()` примусово зупиняє потік у недетермінованому місці виконання, через що стає зовсім незрозуміло, що робити з ресурсами, які він утримує. Потік міг відкрити мережеве з'єднання — що тоді робити з даними, які ще не зчитані? Якщо потік утримував блокування спільного ресурсу — `stop()` негайно знімає всі його монітори, лишаючи спільні дані в неузгодженому (пошкодженому) стані, який бачитимуть інші потоки. Те саме стосується з'єднання з базою даних: якщо потік зупинять посеред транзакції, її ніхто не закриє. Саме через цю фундаментальну небезпеку `stop()` оголошено застарілим, а коректну зупинку слід робити кооперативно через `interrupt()`.

[до змісту](#Багатопотоковість)

## Що відбувається, коли в потоці викидається виняток?
+ Якщо виняток не спіймано, потік «помирає» (переходить у стан `TERMINATED`). Важливо: непійманий виняток в одному потоці не зачіпає інші потоки — вони продовжують працювати.
+ Перед смертю потоку JVM передає виняток його обробникові непійманих винятків. `Thread.UncaughtExceptionHandler` — інтерфейс із методом `uncaughtException(Thread t, Throwable e)`. Коли потік збирається завершитися через непійманий виняток, JVM шукає обробник через `Thread.getUncaughtExceptionHandler()` (спершу персональний обробник потоку, потім обробник групи потоків, потім дефолтний, заданий через `Thread.setDefaultUncaughtExceptionHandler()`) і викликає його, передавши потік та виняток.

[до змісту](#Багатопотоковість)

## У чому різниця між `interrupted()` і `isInterrupted()`?
Механізм переривання потоку в Java реалізований через внутрішній прапорець — статус переривання. Виклик `Thread.interrupt()` встановлює цей прапорець. Перевірити його дозволяють два методи:

+ Статичний `Thread.interrupted()` перевіряє статус переривання __поточного__ потоку і __скидає__ прапорець (повертає його в `false`). Тобто це деструктивна перевірка.
+ Нестатичний `isInterrupted()` перевіряє статус переривання конкретного потоку (на якому викликаний) і __не змінює__ прапорець. Часто використовується одним потоком для перевірки стану іншого.

[до змісту](#Багатопотоковість)

## Що таке _«пул потоків»_?
Створення потоку — затратна за часом і ресурсами операція, а кількість потоків у межах одного процесу обмежена. Щоб уникнути цих проблем і ефективніше керувати множиною потоків, у Java реалізовано механізм пулу потоків (thread pool): набір заздалегідь створених потоків, які повторно використовуються для виконання задач. Це дозволяє не платити щоразу за створення потоку, обмежити їхню кількість і збалансувати навантаження.

Починаючи з Java 5, Java API надає фреймворк `Executor`, що дозволяє створювати різні типи пулів потоків:

+ `Executor` — спрощений інтерфейс, містить один метод `execute(Runnable)` для передачі задачі на виконання;
+ `ExecutorService` — розширений інтерфейс пулу з можливістю завершення всіх потоків (`shutdown()`), подання `Callable`, отримання `Future` тощо;
+ `AbstractExecutorService` — базовий клас, що реалізує `ExecutorService`;
+ `Executors` — фабрика об'єктів, пов'язаних із пулом потоків, що дозволяє створити основні типи пулів;
+ `ThreadPoolExecutor` — пул із гнучким налаштуванням, може служити базовим класом для нестандартних пулів;
+ `ForkJoinPool` — пул для виконання задач типу `ForkJoinTask`;
+ ... та інші.

Методи `Executors` для створення пулів:

+ `newCachedThreadPool()` — якщо є вільний потік, задача виконується в ньому, інакше до пулу додається новий потік. Потоки, що не використовуються понад хвилину, завершуються й видаляються з пулу. Розмір пулу необмежений. Призначений для виконання багатьох невеликих асинхронних задач;
+ `newFixedThreadPool(int nThreads)` — створює пул із фіксованою кількістю потоків. Якщо нові задачі додаються, коли всі потоки активні, вони зберігаються в черзі для виконання пізніше. Якщо потік завершився через помилку, на його місце буде запущено новий. Потоки живуть, доки пул не буде явно закрито методом `shutdown()`;
+ `newSingleThreadExecutor()` — однопотоковий пул (гарантує послідовне виконання задач);
+ `newScheduledThreadPool(int corePoolSize)` — пул для виконання задач через заданий час або періодично;
+ `newSingleThreadScheduledExecutor()` — однопотоковий планувальний пул;
+ `newWorkStealingPool()` (з Java 8) — пул на основі `ForkJoinPool` з алгоритмом work-stealing;
+ `newVirtualThreadPerTaskExecutor()` (з Java 21) — створює новий віртуальний потік на кожну задачу;
+ варіанти з `ThreadFactory` — аналогічно попереднім, але з власною фабрикою потоків (зручно для іменування потоків);
+ `unconfigurableExecutorService(ExecutorService executor)` — обгортка над пулом, що забороняє змінювати його конфігурацію.

[до змісту](#Багатопотоковість)

## Якого розміру має бути пул потоків?
Налаштовуючи розмір пулу, важливо уникнути двох помилок: занадто мало потоків (черга на виконання зростатиме, споживаючи багато пам'яті, а CPU простоюватиме) або занадто багато потоків (уповільнення всієї системи через часті перемикання контексту й конкуренцію за ресурси).

Оптимальний розмір залежить від кількості доступних процесорів і природи задач:

+ Для __CPU-bound__ задач (обмежених швидкістю обчислень) на N-процесорній системі максимального використання CPU досягають пулом приблизно з `N` або `N + 1` потоків.
+ Для __I/O-bound__ задач (що чекають на введення-виведення — наприклад, читання HTTP-запиту із сокета) може знадобитися більше потоків, ніж процесорів, бо не всі потоки працюють увесь час. Оцінивши профілюванням відношення часу очікування (`WT`) до часу обробки (`ST`), для N-процесорної системи потрібно приблизно `N * (1 + WT/ST)` потоків для повного завантаження процесорів.

Використання процесора — не єдиний фактор. Зі зростанням пулу можна впертися в обмеження планувальника, доступної пам'яті чи інших системних ресурсів (кількість сокетів, дескрипторів файлів, з'єднань із базою даних).

[до змісту](#Багатопотоковість)

## Що буде, якщо черга пулу потоків уже заповнена, але подається нове завдання?
Якщо черга пулу заповнилася (і всі потоки зайняті), нова задача буде «відхилена» згідно з політикою відхилення (`RejectedExecutionHandler`). За замовчуванням у `ThreadPoolExecutor` це політика `AbortPolicy`, яка кидає `RejectedExecutionException`. Інші вбудовані політики: `CallerRunsPolicy` (задачу виконує сам потік, що її подав), `DiscardPolicy` (задача мовчки відкидається), `DiscardOldestPolicy` (відкидається найстаріша задача в черзі, а нова додається).

[до змісту](#Багатопотоковість)

## У чому полягає різниця між методами `submit()` і `execute()` у пулу потоків?
Обидва методи подають задачу в пул, але між ними є різниця.

`execute(Runnable command)` визначений в інтерфейсі `Executor`, виконує подану задачу й нічого не повертає. Якщо задача кине непійманий виняток, він піде в `UncaughtExceptionHandler` потоку.

`submit()` — перевантажений метод інтерфейсу `ExecutorService`. Він приймає задачі типів `Runnable` і `Callable` й повертає об'єкт `Future`, через який можна контролювати виконання, отримати результат і — що важливо — дізнатися про виняток: якщо задача кинула виняток, він буде загорнутий у `ExecutionException` і виплине при виклику `future.get()`. Тому при використанні `submit()` важливо завжди викликати `get()`, інакше винятки «проковтуються» мовчки.

[до змісту](#Багатопотоковість)

## У чому полягають відмінності між стеком (stack) і купою (heap) з точки зору багатопотоковості?
__Стек__ — ділянка пам'яті, тісно пов'язана з потоком. Кожен потік має власний стек, що зберігає локальні змінні, параметри методів і стек викликів (frames). Змінна, що лежить у стеку одного потоку, не видима іншому — тому локальні змінні за визначенням потокобезпечні (якщо вони не «витікають» назовні).

__Купа__ — спільна ділянка пам'яті, що поділяється між усіма потоками. Усі об'єкти (екземпляри класів) створюються в купі. Саме спільна купа й породжує проблеми багатопотоковості: коли кілька потоків читають/пишуть один об'єкт. Задля продуктивності потік може кешувати значення з купи у своєму стеку/регістрах; щоб змусити потік читати/писати значення безпосередньо в купу й бачити чужі зміни, використовують `volatile` (для одного поля) чи синхронізацію.

[до змісту](#Багатопотоковість)

## Як поділитися даними між двома потоками?
Даними між потоками можна ділитися через спільний об'єкт або через паралельні (потокобезпечні) структури даних, наприклад `BlockingQueue` (класичний патерн «виробник-споживач») чи `ConcurrentHashMap`.

Окремо варто згадати клас-синхронізатор `Exchanger`: він призначений для обміну даними між рівно двома потоками. Він параметризується типом даних, якими потоки обмінюються. Обмін здійснюється єдиним методом `exchange()`: кожен потік передає свій об'єкт і блокується, доки інший потік не викличе `exchange()` зі свого боку, після чого обидва отримують дані одне одного.

[до змісту](#Багатопотоковість)

## Який параметр запуску JVM використовується для контролю розміру стеку потоку?
`-Xss` (наприклад, `-Xss512k`). Він задає розмір стеку для кожного потоку. Замалий стек може спричинити `StackOverflowError` на глибокій рекурсії, завеликий — обмежити максимальну кількість потоків, які можна створити.

[до змісту](#Багатопотоковість)

## Як отримати дамп потоків?
> Примітка: тут ідеться саме про _thread dump_ (стан і стеки всіх потоків), хоча в деяких місцях нижче згадуються й heap-дампи — це різні речі. Для діагностики зависань/deadlock потрібен саме thread dump.

Найпоширеніші способи отримати дамп потоків:

+ __`jstack <pid>`__ — утиліта з JDK, друкує стек-трейси всіх потоків процесу; найзручніший спосіб для діагностики зависань і deadlock-ів (jstack автоматично виявляє deadlock-и);
+ __`jcmd <pid> Thread.print`__ — сучасна альтернатива jstack;
+ __Сигнал `Ctrl+Break` (Windows) / `Ctrl+\` чи `kill -3 <pid>` (SIGQUIT, Unix)__ — JVM виведе дамп потоків у стандартний вивід застосунку;
+ __JVisualVM / JConsole / JDK Mission Control__ — графічні інструменти, що показують стан потоків у реальному часі.

Для __heap-дампу__ (зовсім інша задача — аналіз пам'яті) використовують `jmap -dump:...`, `jcmd <pid> GC.heap_dump`, або опцію `-XX:+HeapDumpOnOutOfMemoryError`, що автоматично створює дамп купи при `OutOfMemoryError` (ідеально для production).

## Що таке _ThreadLocal-змінна_?
`ThreadLocal` — клас, що дозволяє, маючи одну змінну, мати окреме її значення для кожного потоку. Це спосіб уникнути спільного стану взагалі: замість синхронізації доступу до спільної змінної кожен потік працює зі своєю ізольованою копією.

У кожного потоку (екземпляра `Thread`) є асоційована з ним таблиця ThreadLocal-значень. Ключами таблиці є посилання на об'єкти `ThreadLocal`, а значеннями — об'єкти, «захоплені» цими ThreadLocal-змінними. Доступ до значення отримують через методи `get()` / `set()`.

Наприклад, якщо оголосити `ThreadLocal<Object> locals = new ThreadLocal<>();`, а потім у потоці зробити `locals.set(myObject)`, то для цього потоку `locals.get()` поверне `myObject`, тоді як інший потік може покласти в той самий `locals` інше значення — і вони не заважатимуть одне одному.

Важливі застереження:

+ `ThreadLocal` ізолює саме посилання, а не самі об'єкти. Якщо ізольовані в різних потоках посилання вказують на той самий об'єкт, потокобезпеки не буде.
+ Ініціалізувати значення (`set()`) треба в тому самому потоці, що його використовуватиме. Помилка — викликати `set()` у головному потоці, очікуючи побачити значення в дочірньому: дочірній потік побачить `null` (для цього існує окремий `InheritableThreadLocal`).
+ У пулах потоків `ThreadLocal` потребує особливої обережності: оскільки потоки повторно використовуються, значення може «протекти» в наступну задачу. Завжди прибирайте значення через `remove()` після використання, інакше можливі витоки пам'яті та логічні помилки.

[до змісту](#Багатопотоковість)

## Назвіть відмінності між `synchronized` і `ReentrantLock`
У Java 5 з'явився інтерфейс `Lock`, що надає можливості ефективнішого й тоншого контролю блокування ресурсів. `ReentrantLock` — найпоширеніша реалізація `Lock`. Він дає таку саму базову семантику, як `synchronized` (зокрема, реентерабельність — той самий потік може повторно захопити блокування, яке вже утримує), але з розширеними можливостями.

Що означає реентерабельність (reentrant)? З блокуванням пов'язаний лічильник захоплень: якщо потік, що вже утримує блокування, захоплює його знову, лічильник збільшується, і для повного звільнення потрібно зняти блокування стільки ж разів. Це аналогічно семантиці `synchronized`: якщо потік входить у вкладений `synchronized` блок на тому самому моніторі, він проходить, а блокування знімається лише при виході із зовнішнього блоку.

```java
Lock lock = new ReentrantLock();

lock.lock();
try {
    // update object state
} finally {
    lock.unlock();
}
```

Основні відмінності:

| Властивість | `synchronized` | `ReentrantLock` |
|---|---|---|
| Звільнення блокування | автоматичне (при виході з блоку/методу) | вручну, обов'язково у `finally` |
| Спроба захопити без блокування (`tryLock()`) | немає | є |
| Захоплення з таймаутом | немає | є (`tryLock(time, unit)`) |
| Перериване очікування блокування | немає | є (`lockInterruptibly()`) |
| Справедливість (fairness) | немає | можна задати в конструкторі |
| Кілька умов (`Condition`) | один неявний wait-set | багато (`newCondition()`) |
| Масштабованість за сильної конкуренції | гірша | зазвичай краща |

Резюме: коли конкуренції за блокування немає або вона мінімальна, `synchronized` зазвичай простіший і не повільніший (а сучасні JVM його добре оптимізують). Якщо ж є помітна конкуренція або потрібні розширені можливості (таймаут, перериваність, `tryLock`, кілька умов) — обирайте `ReentrantLock`. Головний ризик `ReentrantLock` — забути `unlock()` у `finally`.

## Що таке `ReadWriteLock`?
`ReadWriteLock` — інтерфейс, що оперує парою пов'язаних блокувань: одне для читання, друге для запису. Ідея: читання не конфліктує з читанням, тож блокування читання можуть одночасно утримувати багато потоків-читачів — доти, доки не з'явиться потік-письменник. Блокування запису ексклюзивне (його утримує лише один потік, і поки воно утримується, читачів немає). Це підвищує пропускну здатність у сценаріях, де читань значно більше, ніж записів.

Реалізація — `ReentrantReadWriteLock`.

```java
ReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock rLock = rwLock.readLock();
Lock wLock = rwLock.writeLock();

wLock.lock();
try {
    // exclusive write
} finally {
    wLock.unlock();
}

rLock.lock();
try {
    // shared reading
} finally {
    rLock.unlock();
}
```

> Сучасна альтернатива (з Java 8) — `StampedLock`, що додає режим оптимістичного читання (`tryOptimisticRead()`), який зовсім не блокує письменників і може дати ще кращу продуктивність на read-важких сценаріях. Але `StampedLock` не реентерабельний і складніший у використанні.

[до змісту](#Багатопотоковість)

## Що таке _«блокуючий метод»_?
__Блокуючий метод__ — метод, що не повертає керування викликачу, доки завдання не виконається. Наприклад, `accept()` у `ServerSocket` блокується в очікуванні підключення клієнта, а `take()` у `BlockingQueue` блокується, поки в черзі не з'явиться елемент. Протилежність — асинхронні/неблокуючі методи, що повертають керування одразу (часто разом із `Future`/`CompletableFuture`), не чекаючи завершення завдання.

[до змісту](#Багатопотоковість)

## Що таке _«фреймворк Fork/Join»_?
Фреймворк Fork/Join, представлений у JDK 7, — це набір класів та інтерфейсів, що дозволяють використовувати переваги багатопроцесорної архітектури. Він розроблений для задач, які можна рекурсивно розбити на менші підзадачі, розв'язувані паралельно (патерн «divide and conquer»).

+ Етап __Fork__: велика задача ділиться на менші підзадачі, що у свою чергу теж діляться, — доти, доки задача не стане достатньо тривіальною, щоб розв'язати її послідовно.
+ Етап __Join__: далі (опційно) йде «згортка» — розв'язки підзадач об'єднуються, доки не вийде розв'язок усієї задачі.

Розв'язання підзадач (зокрема й саме розбиття) відбувається паралельно.

> Для деяких задач етап Join не потрібен. Наприклад, для паралельного QuickSort масив рекурсивно ділиться на дедалі менші діапазони, доки не вироджується в тривіальний випадок з 1 елемента.

Ключова перевага — алгоритм __work-stealing__: потоки, що завершили власні підзадачі, можуть «вкрасти» підзадачі з черг інших потоків, які ще зайняті, чим вирівнюється навантаження. На Fork/Join побудовані `ForkJoinPool` і паралельні стріми (`Collection.parallelStream()`).

[до змісту](#Багатопотоковість)

## Що таке `Semaphore`?
`Semaphore` — синхронізатор-лічильник, що обмежує кількість потоків, які одночасно мають доступ до ресурсу. Початкове значення лічильника (кількість «дозволів», permits) задається в конструкторі. Коли потік хоче зайти в захищену секцію, він викликає `acquire()` (лічильник зменшується на одиницю); виходячи — `release()` (лічильник збільшується). Якщо лічильник дорівнює нулю, наступний `acquire()` блокується, доки хтось не звільнить дозвіл. `Semaphore` використовують для захисту дорогих ресурсів, доступних в обмеженій кількості, — наприклад, обмеження кількості одночасних з'єднань із базою даних у пулі. Семафор з одним дозволом (`new Semaphore(1)`) поводиться як м'ютекс (але, на відміну від монітора, його може звільнити інший потік, ніж той, що захопив).

[до змісту](#Багатопотоковість)

## Що таке _double checked locking Singleton_?
__Double-checked locking Singleton__ — спосіб створення потокобезпечного Singleton, що намагається оптимізувати продуктивність, захоплюючи блокування лише тоді, коли екземпляр створюється вперше. Після того як екземпляр створено, наступні виклики `getInstance()` повертають його взагалі без синхронізації.

```java
class DoubleCheckedLockingSingleton {
    private static volatile DoubleCheckedLockingSingleton instance;

    static DoubleCheckedLockingSingleton getInstance() {
        DoubleCheckedLockingSingleton current = instance;
        if (current == null) {
            synchronized (DoubleCheckedLockingSingleton.class) {
                current = instance;
                if (current == null) {
                    instance = current = new DoubleCheckedLockingSingleton();
                }
            }
        }
        return current;
    }
}
```

Ключовий момент: `volatile` тут обов'язковий. Проблема double-checked locking криється в моделі пам'яті Java, точніше — у порядку створення об'єкта. Запис посилання `instance` міг би стати видимим іншому потоку __раніше__, ніж завершилася ініціалізація об'єкта (через reordering), і той потік, побачивши ненульове посилання, почав би використовувати недобудований об'єкт. До Java 5 ця ідіома була зламана навіть із `volatile`; починаючи з Java 5 семантику `volatile` посилили (вона створює потрібний бар'єр пам'яті), і ця ідіома працює — але лише з `volatile`. На практиці частіше обирають простіші й надійніші підходи (enum або holder idiom, див. наступне питання).

[до змісту](#Багатопотоковість)

## Як створити потокобезпечний Singleton?
+ __Static field__ (eager-ініціалізація). Просто й потокобезпечно, бо JVM гарантує безпечну ініціалізацію статичних полів, але екземпляр створюється завжди, навіть якщо не знадобиться.

```java
public class Singleton {
    public static final Singleton INSTANCE = new Singleton();
}
```

+ __Enum__ — найнадійніший спосіб (рекомендований Джошуа Блохом). Потокобезпечний «з коробки», стійкий до проблем серіалізації та reflection, які можуть зламати інші реалізації.

```java
public enum Singleton {
    INSTANCE;
}
```

+ __Synchronized Accessor__ — ліниво, потокобезпечно, але синхронізація на кожному виклику `getInstance()` уповільнює доступ.

```java
public class Singleton {
    private static Singleton instance;

    public static synchronized Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

+ __Double Checked Locking & `volatile`__ — ліниво, синхронізація лише при першому створенні (див. попереднє питання). `volatile` обов'язковий.

```java
public class Singleton {
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        Singleton localInstance = instance;
        if (localInstance == null) {
            synchronized (Singleton.class) {
                localInstance = instance;
                if (localInstance == null) {
                    instance = localInstance = new Singleton();
                }
            }
        }
        return localInstance;
    }
}
```

+ __On Demand Holder Idiom__ (Initialization-on-demand holder) — ліниво й потокобезпечно без жодної синхронізації. Вкладений клас завантажується (і створює екземпляр) лише при першому зверненні до `getInstance()`, а потокобезпеку гарантує сама JVM при ініціалізації класу.

```java
public class Singleton {

    private static class SingletonHolder {
        private static final Singleton HOLDER_INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return SingletonHolder.HOLDER_INSTANCE;
    }
}
```

На практиці для більшості випадків достатньо __enum__ або __holder idiom__.

[до змісту](#Багатопотоковість)

## Чим корисні незмінні об'єкти?
Незмінність (immutability) кардинально спрощує багатопотоковий код: незмінний об'єкт можна вільно передавати між потоками й читати без жодної синхронізації, бо його стан ніколи не змінюється — а отже, немає що псувати й немає станів гонки. Безпечно опублікований незмінний об'єкт (з `final` полями) гарантовано видимий усім потокам.

У Java немає анотації, що автоматично робить об'єкт незмінним, тож розробник створює такий клас сам, дотримуючись правил:

+ клас оголосити `final` (щоб не можна було перевизначити поведінку через спадкоємця);
+ усі поля — `private final`, ініціалізувати лише в конструкторі;
+ не надавати сетерів і будь-яких методів, що змінюють стан;
+ при роботі зі змінюваними об'єктами всередині — робити захисні копії (defensive copy) і в конструкторі, і в гетерах, щоб посилання не «витекло».

> З Java 16 для незмінних носіїв даних зручно використовувати `record` — він автоматично робить поля `final` і генерує конструктор, гетери, `equals`/`hashCode`.

[до змісту](#Багатопотоковість)

## Що таке _busy spin_?
__busy spin__ (busy waiting, активне очікування) — техніка, за якої потік замість того, щоб заснути, виконує порожній цикл, постійно перевіряючи умову. На відміну від `wait()`, `sleep()` чи `yield()`, що віддають процесор, busy spin утримує ядро зайнятим. Навіщо це робити? Щоб уникнути перемикання контексту й зберегти «теплим» кеш процесора: у багатоядерних системах присплений потік може прокинутися вже на іншому ядрі, що спричинить дороге перебудовування кешу. Тому за дуже короткого очікування й вимог до наднизької латентності (наприклад, у високочастотній торгівлі) busy spin може бути швидшим. Але це коштує марного спалювання CPU, тож застосовувати його варто лише в дуже специфічних низьколатентних сценаріях. У Java 9+ є підказка `Thread.onSpinWait()`, що дозволяє процесору оптимізувати такий цикл очікування.

## Перелічіть принципи, яких ви дотримуєтесь у багатопотоковому програмуванні
+ __Надавайте потокам осмислені імена.__ Налагодження багатопотокового коду — складна задача; `OrderProcessor`, `QuoteProcessor` набагато інформативніші за `Thread-1`, `Thread-2`. Ім'я має відображати задачу потоку.
+ __Уникайте або мінімізуйте блокування.__ Блокування затратне, а перемикання контексту — ще більше. Тримайте критичну секцію якомога меншою; синхронізований блок зазвичай кращий за синхронізований метод, бо дає точніший контроль над масштабом блокування.
+ __Надавайте перевагу незмінності й локальному стану__ перед спільним змінюваним станом — найкращий спосіб уникнути гонок — не мати що ділити.
+ __Обробляйте переривання уважно.__ Не «ковтайте» `InterruptedException` мовчки — або обробіть його, або відновіть прапорець (`Thread.currentThread().interrupt()`) і прокиньте далі. Немає нічого гіршого за ресурс, що лишився заблокованим, або систему в неузгодженому стані через незавершену транзакцію.
+ __Пам'ятайте про обробку винятків у пулах.__ Пул потоків часто «ковтає» винятки: при `submit()` обов'язково викликайте `future.get()`, щоб виняток виплив; при `execute()` подбайте про `UncaughtExceptionHandler`.
+ __Надавайте перевагу високорівневим синхронізаторам над `wait()`/`notify()`.__ `CountDownLatch`, `Semaphore`, `CyclicBarrier`, `CompletableFuture` спрощують код, написані експертами й добре оптимізовані. Реалізувати складний потік керування на голих `wait()`/`notify()` важко й помилконебезпечно.
+ __Надавайте перевагу concurrent-колекціям над synchronized-колекціями.__ `ConcurrentHashMap`, `CopyOnWriteArrayList` тощо сучасніші й масштабованіші, ніж обгортки `Collections.synchronizedXxx()`.

## Яке з наведених тверджень про потоки хибне?
1. Якщо метод `start()` викликається двічі для одного й того самого об'єкта `Thread`, під час виконання генерується виняток.
2. Порядок, у якому запускалися потоки, може не збігатися з порядком їх фактичного виконання.
3. Якщо метод `run()` викликається напряму для об'єкта `Thread`, під час виконання генерується виняток.
4. Якщо метод `sleep()` викликається для потоку під час виконання синхронізованого коду, блокування не знімається.

Правильна відповідь: __3__. Якщо метод `run()` викликати напряму для об'єкта `Thread`, виняток __не__ генерується. Однак код у `run()` виконається поточним, а не новим потоком. Правильний спосіб запустити потік — викликати `start()`, що призводить до виконання `run()` у новому потоці.

Решта тверджень правильні: виклик `start()` двічі спричинить `IllegalThreadStateException` (твердження 1); порядок виконання визначає планувальник, а не порядок запуску (твердження 2); `sleep()` не звільняє утримувані блокування (твердження 4).

## Дано 3 потоки Т1, Т2 і Т3. Як реалізувати виконання в послідовності Т1, Т2, Т3?
Найпростіше — скористатися методом `join()`. Щоб гарантувати послідовність, кожен потік перед своєю роботою чекає завершення попереднього: Т3 у своєму `run()` спершу робить `Т2.join()`, а Т2 — `Т1.join()`. Так Т1 завершиться першим, потім Т2, потім Т3. Як альтернатива — можна послідовно запускати потоки й одразу робити `join()` у головному потоці (`t1.start(); t1.join(); t2.start(); t2.join(); t3.start(); t3.join();`), або скористатися `CountDownLatch`-ами для координації.

## Напишіть мінімальний неблокуючий стек (лише два методи — `push()` і `pop()`).
Неблокуючий (lock-free) стек на основі CAS-циклу над `AtomicReference` на голову списку (алгоритм Трейбера):

```java
class NonBlockingStack<T> {
    private final AtomicReference<Element<T>> head = new AtomicReference<>(null);

    NonBlockingStack<T> push(final T value) {
        final Element<T> current = new Element<>();
        current.value = value;
        Element<T> recent;
        do {
            recent = head.get();
            current.previous = recent;
        } while (!head.compareAndSet(recent, current));
        return this;
    }

    T pop() {
        Element<T> result;
        Element<T> previous;
        do {
            result = head.get();
            if (result == null) {
                return null;
            }
            previous = result.previous;
        } while (!head.compareAndSet(result, previous));
        return result.value;
    }

    private static class Element<E> {
        private E value;
        private Element<E> previous;
    }
}
```

## Напишіть мінімальний неблокуючий стек (лише два методи — `push()` і `pop()`) з використанням `Semaphore`.
> Зауваження щодо термінології: тут «неблокуючий» вжито в сенсі «потокобезпечний без явного `synchronized`/`Lock`». Семафор із одним дозволом фактично є блокуванням, тож суворо кажучи цей варіант блокуючий — на відміну від попереднього lock-free стека.

```java
class SemaphoreStack<T> {
    private final Semaphore semaphore = new Semaphore(1);
    private Node<T> head = null;

    SemaphoreStack<T> push(T value) {
        semaphore.acquireUninterruptibly();
        try {
            head = new Node<>(value, head);
        } finally {
            semaphore.release();
        }
        return this;
    }

    T pop() {
        semaphore.acquireUninterruptibly();
        try {
            Node<T> current = head;
            if (current != null) {
                head = head.next;
                return current.value;
            }
            return null;
        } finally {
            semaphore.release();
        }
    }

    private static class Node<E> {
        private final E value;
        private final Node<E> next;

        private Node(E value, Node<E> next) {
            this.value = value;
            this.next = next;
        }
    }
}
```

## Напишіть мінімальний неблокуючий ArrayList (лише чотири методи — `add()`, `get()`, `remove()`, `size()`).
Підхід copy-on-write: при кожній модифікації масив копіюється, а `volatile`-посилання атомарно перемикається на нову копію. Читання (`get`/`size`) працюють без блокувань.

```java
class NonBlockingArrayList<T> {
    private volatile Object[] content = new Object[0];

    NonBlockingArrayList<T> add(T item) {
        return add(content.length, item);
    }

    NonBlockingArrayList<T> add(int index, T item) {
        if (index < 0) {
            throw new IllegalArgumentException();
        }
        boolean needsModification = index > content.length - 1;
        if (!needsModification) {
            if (item == null) {
                needsModification = content[index] != null;
            } else {
                needsModification = !item.equals(content[index]);
            }
        }
        if (needsModification) {
            final Object[] renewed = Arrays.copyOf(content, Math.max(content.length, index + 1));
            renewed[index] = item;
            content = renewed;
        }
        return this;
    }

    NonBlockingArrayList<T> remove(int index) {
        if (index < 0 || index >= content.length) {
            throw new IllegalArgumentException();
        }
        int size = content.length - 1;
        final Object[] renewed = new Object[size];
        System.arraycopy(content, 0, renewed, 0, index);
        if (index < size) {
            System.arraycopy(content, index + 1, renewed, index, size - index);
        }
        content = renewed;
        return this;
    }

    @SuppressWarnings("unchecked")
    T get(int index) {
        return (T) content[index];
    }

    int size() {
        return content.length;
    }
}
```

> Виправлено: в оригіналі копіювання «хвоста» в `remove()` мало умову `index + 1 < size`, через що при видаленні передостаннього елемента останній елемент губився; коректна умова — `index < size`.

## Напишіть потокобезпечну реалізацію класу з неблокуючим методом `BigInteger next()`, який повертає елементи послідовності: `[1, 2, 4, 8, 16, ...]`.
CAS-цикл над `AtomicReference<BigInteger>`; кожен наступний елемент — попередній, зсунутий вліво на 1 біт (тобто помножений на 2):

```java
class PowerOfTwo {
    private final AtomicReference<BigInteger> current = new AtomicReference<>(null);

    BigInteger next() {
        BigInteger recent, next;
        do {
            recent = current.get();
            next = (recent == null) ? BigInteger.ONE : recent.shiftLeft(1);
        } while (!current.compareAndSet(recent, next));
        return next;
    }
}
```

## Напишіть найпростіший багатопотоковий обмежений буфер з використанням `synchronized`.
Кільцевий буфер (ring buffer) фіксованої місткості з координацією через `wait()`/`notify()` (патерн «виробник-споживач»):

```java
class QueueSynchronized<T> {
    private final Object[] content;
    private final int capacity;

    private int out;
    private int in;
    private int size;

    QueueSynchronized(final int capacity) {
        this.capacity = capacity;
        this.content = new Object[capacity];
        this.out = 0;
        this.in = 0;
        this.size = 0;
    }

    private int cycleInc(int index) {
        return (++index == capacity) ? 0 : index;
    }

    @SuppressWarnings("unchecked")
    synchronized T get() throws InterruptedException {
        while (size == 0) {
            wait();
        }
        final Object value = content[out];
        content[out] = null;
        out = cycleInc(out);
        size--;
        notifyAll();
        return (T) value;
    }

    synchronized QueueSynchronized<T> put(T value) throws InterruptedException {
        while (size == capacity) {
            wait();
        }
        content[in] = value;
        in = cycleInc(in);
        size++;
        notifyAll();
        return this;
    }
}
```

> Реалізацію переписано порівняно з оригіналом. В оригіналі синхронізація розбивалася на три різні монітори (`this`, `isEmpty`, `isFull`), а перевірка `size` робилася поза синхронізованим блоком і умовно не зсувала індекси `in`/`out` — це повноцінні стани гонки й логічні помилки (індекси могли розійтися із `size`, перший елемент перезаписувався). Тут усе зведено до одного монітора (`this`), перевірка умов у `while`, а пробудження через `notifyAll()` — це канонічний коректний варіант.

## Напишіть найпростіший багатопотоковий обмежений буфер з використанням `ReentrantLock`.
Той самий кільцевий буфер, але з `ReentrantLock` і двома окремими `Condition` (одна для «не порожньо», друга для «не повно») — це чистіше за один wait-set, бо `signal()` будить саме потрібну категорію потоків:

```java
class QueueReentrantLock<T> {
    private final Object[] content;
    private final int capacity;

    private int out;
    private int in;
    private int size;

    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notEmpty = lock.newCondition();
    private final Condition notFull = lock.newCondition();

    QueueReentrantLock(int capacity) {
        this.capacity = capacity;
        this.content = new Object[capacity];
        this.out = 0;
        this.in = 0;
        this.size = 0;
    }

    private int cycleInc(int index) {
        return (++index == capacity) ? 0 : index;
    }

    @SuppressWarnings("unchecked")
    T get() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (size == 0) {
                notEmpty.await();
            }
            final Object value = content[out];
            content[out] = null;
            out = cycleInc(out);
            size--;
            notFull.signal();
            return (T) value;
        } finally {
            lock.unlock();
        }
    }

    QueueReentrantLock<T> put(T value) throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (size == capacity) {
                notFull.await();
            }
            content[in] = value;
            in = cycleInc(in);
            size++;
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
        return this;
    }
}
```

> Реалізацію виправлено порівняно з оригіналом: прибрано безглузде блокування в конструкторі (об'єкт ще не опублікований, блокувати нема від кого, до того ж `lock()` там був поза `try`); умовний зсув індексів зведено до безумовного, що усуває розсинхронізацію `in`/`out` із `size`; `Condition`-и названо за їхньою семантикою (`notEmpty`/`notFull`). `lockInterruptibly()` залишено, щоб потік міг реагувати на переривання, чекаючи на блокування.
