## Що таке _JDBC_?
__JDBC, Java DataBase Connectivity (з'єднання з базами даних у Java)__ — промисловий стандарт взаємодії Java-застосунків із різними СУБД. Реалізований у вигляді пакета `java.sql`, що входить до складу Java SE.

Ментальна модель: JDBC — це універсальний «перехідник» між вашим Java-кодом і конкретною СУБД. Ви пишете код проти єдиного набору інтерфейсів (`Connection`, `Statement`, `ResultSet`), а драйвер конкретної бази «перекладає» ці виклики на її рідний протокол.

JDBC ґрунтується на концепції драйверів, які дають змогу отримати з'єднання з базою даних за спеціально описаним URL. Драйвери, що лежать у classpath, виявляються автоматично через механізм `ServiceLoader` (файл `META-INF/services/java.sql.Driver` у JAR драйвера) — починаючи з JDBC 4.0 (Java 6) ручна реєстрація через `Class.forName(...)` більше не потрібна. Коли програма запитує з'єднання за певним URL, `DriverManager` обирає драйвер, який відповідає за відповідний протокол.

## У чому полягають переваги використання JDBC?
Перевагами JDBC вважають:

+ Легкість розробки: розробник може не знати специфіки бази даних, з якою працює;
+ Код практично не змінюється, якщо компанія переходить на іншу базу даних (кількість змін залежить виключно від відмінностей між діалектами SQL);
+ Не потрібно додатково встановлювати клієнтську програму;
+ До будь-якої бази даних можна під'єднатися через зручно описуваний URL.

## Що собою являє JDBC URL?
__JDBC URL__ складається з:

+ `<protocol>:` (протокол) — завжди `jdbc:`.
+ `<subprotocol>:` (підпротокол) — це ім'я драйвера або ім'я механізму з'єднання з базою даних. Підпротокол може підтримуватися одним або кількома драйверами. Типові приклади підпротоколів — `mysql`, `postgresql`, `oracle:thin`. (Історичний приклад `odbc` стосувався мосту JDBC-ODBC, який було вилучено в Java 8, тож на практиці він уже не використовується.)
+ `<subname>` (підім'я) — це ідентифікатор бази даних. Значення підімені може змінюватися залежно від підпротоколу й також може мати під-підім'я із синтаксисом, що визначається розробником драйвера. Призначення підімені — надати всю інформацію, потрібну для пошуку бази даних. Наприклад, якщо база даних розташована в мережі, до складу підімені JDBC URL має бути включена мережева адреса за такою угодою: `//<hostname>:<port>/<subsubname>`.

Приклад JDBC URL для підключення до бази даних MySQL «Test», що розташована за адресою localhost і очікує з'єднань на порту 3306: `jdbc:mysql://localhost:3306/Test`

## З яких частин складається JDBC?
JDBC складається з двох частин:

+ __JDBC API__ — містить набір класів та інтерфейсів, що визначають доступ до баз даних. Ці класи та інтерфейси оголошені у двох пакетах — `java.sql` і `javax.sql`;
+ __JDBC-драйвер__ — компонент, специфічний для кожної бази даних.

JDBC перетворює виклики рівня API на «рідні» команди того чи іншого сервера баз даних.

## Перелічіть основні класи та інтерфейси JDBC.
+ `java.sql.DriverManager` — дає змогу завантажити та зареєструвати потрібний JDBC-драйвер, а потім отримати з'єднання з базою даних.

+ `javax.sql.DataSource` — розв'язує ті самі задачі, що й _DriverManager_, але зручнішим і універсальнішим способом (підтримує пули з'єднань, конфігурацію через JNDI). На практиці в сучасних застосунках з'єднання отримують саме через `DataSource`, а не напряму через `DriverManager`. Існують також `javax.sql.ConnectionPoolDataSource` і `javax.sql.XADataSource`, призначені для пулів з'єднань та розподілених (XA) транзакцій.

+ `java.sql.Connection` — забезпечує формування запитів до джерела даних і керування транзакціями. Також передбачені інтерфейси `javax.sql.PooledConnection` і `javax.sql.XAConnection`.

+ `java.sql.Statement`, `java.sql.PreparedStatement` і `java.sql.CallableStatement` — ці інтерфейси дають змогу надіслати запит до джерела даних.

+ `java.sql.ResultSet` — оголошує методи, що дають змогу переміщатися набором даних і зчитувати значення окремих полів у поточному записі.

+ `java.sql.ResultSetMetaData` — дає змогу отримати інформацію про структуру набору даних.

+ `java.sql.DatabaseMetaData` — дає змогу отримати інформацію про структуру джерела даних.

## Як зареєструвати драйвер JDBC?
Починаючи з JDBC 4.0 (Java 6) реєстрація драйвера відбувається __автоматично__: будь-який драйвер у classpath виявляється через механізм `ServiceLoader` (запис у `META-INF/services/java.sql.Driver`). Тож зазвичай нічого реєструвати вручну не потрібно — достатньо одразу викликати `DriverManager.getConnection(...)`.

Історично (для драйверів до JDBC 4.0) реєстрацію робили одним із способів:

+ `Class.forName("повне ім'я класу драйвера")` — статичний ініціалізатор класу драйвера сам викликав `DriverManager.registerDriver(...)`. Це найпоширеніший застарілий спосіб.

+ `DriverManager.registerDriver(new SomeDriver())` — пряма реєстрація екземпляра драйвера.

> __NB!__ Варіант `Class.forName(...).newInstance()` зі старих джерел зайвий і застарілий: достатньо самого `Class.forName(...)`, бо реєстрація відбувається у статичному блоці класу драйвера, а не в конструкторі.

## Як установити з'єднання з базою даних?
Для встановлення з'єднання з базою даних використовується статичний виклик `java.sql.DriverManager.getConnection(...)`.

Як параметр може передаватися:

+ URL бази даних
```java
static Connection getConnection(String url)
```

+ URL бази даних і набір властивостей для ініціалізації
```java
static Connection getConnection(String url, Properties info)
```

+ URL бази даних, ім'я користувача та пароль
```java
static Connection getConnection(String url, String user, String password)
```

У результаті виклику буде встановлено з'єднання з базою даних і створено об'єкт класу `java.sql.Connection` — своєрідну «сесію», у контексті якої й відбуватиметься подальша робота з базою даних.

> __NB!__ У продакшн-коді з'єднання зазвичай отримують не напряму через `DriverManager`, а з пулу через `DataSource` (наприклад, HikariCP). Відкриття фізичного з'єднання — дорога операція, тому пул перевикористовує вже відкриті з'єднання.

## Які рівні ізоляції транзакцій підтримуються в JDBC?
__Рівень ізоляції транзакцій__ — значення, що визначає, наскільки одна транзакція ізольована від ефектів інших паралельних транзакцій. Вищий рівень ізоляції підвищує узгодженість даних, але може знижувати кількість паралельно виконуваних транзакцій. І навпаки, нижчий рівень ізоляції дає змогу виконувати більше паралельних транзакцій, але послаблює гарантії узгодженості даних.

Під час використання транзакцій для забезпечення цілісності даних СУБД застосовує блокування (lock), щоб обмежити доступ інших звернень до даних, що беруть участь у транзакції. Такі блокування потрібні, щоб запобігти таким аномаліям:

+ _«брудне» читання (dirty read)_ — читання даних, доданих або змінених транзакцією, яка згодом не підтвердиться (відкотиться);

+ _неповторюване читання (non-repeatable read)_ — під час повторного читання в межах однієї транзакції раніше прочитаний рядок виявляється зміненим або видаленим;

+ _фантомне читання (phantom read)_ — ситуація, коли під час повторного виконання тієї самої вибірки в межах однієї транзакції з'являються нові рядки, яких не було раніше.

Рівні ізоляції транзакцій визначені як константи інтерфейсу `java.sql.Connection`:

+ `TRANSACTION_NONE` — драйвер не підтримує транзакції;

+ `TRANSACTION_READ_UNCOMMITTED` — дозволяє транзакціям бачити незафіксовані зміни даних: припускає брудне, неповторюване та фантомне читання;

+ `TRANSACTION_READ_COMMITTED` — будь-яка зміна, зроблена в транзакції, не видна ззовні, доки її не зафіксовано: запобігає брудному читанню, але припускає неповторюване та фантомне;

+ `TRANSACTION_REPEATABLE_READ` — забороняє брудне та неповторюване читання, фантомне читання дозволене;

+ `TRANSACTION_SERIALIZABLE` — брудне, неповторюване та фантомне читання заборонені.

> __NB!__ Сервер бази даних може підтримувати не всі рівні ізоляції, а назва рівня ще не гарантує однакову поведінку в різних СУБД. Наприклад, у PostgreSQL `READ UNCOMMITTED` фактично працює як `READ COMMITTED` (брудного читання немає взагалі), а в MySQL/InnoDB рівнем за замовчуванням є `REPEATABLE READ`. Інтерфейс `java.sql.DatabaseMetaData` надає інформацію про рівні ізоляції, які підтримує конкретна СУБД.

Рівень ізоляції транзакції можна задати методом `setTransactionIsolation()` об'єкта `java.sql.Connection`. Отримати інформацію про чинний рівень ізоляції допоможе метод `getTransactionIsolation()`.

## За допомогою чого формуються запити до бази даних?

Для виконання запитів до бази даних у Java використовуються три інтерфейси:

+ `java.sql.Statement` — для операторів SQL без параметрів;
+ `java.sql.PreparedStatement` — для операторів SQL з параметрами та часто виконуваних операторів;
+ `java.sql.CallableStatement` — для виконання збережених у базі процедур.

Об'єкти, що реалізують ці інтерфейси, створюються за допомогою методів об'єкта `java.sql.Connection`:

+ `java.sql.Connection.createStatement()` повертає об'єкт _Statement_;
+ `java.sql.Connection.prepareStatement()` повертає об'єкт _PreparedStatement_;
+ `java.sql.Connection.prepareCall()` повертає об'єкт _CallableStatement_;

## Чим відрізняється Statement від PreparedStatement?
+ __Statement__: використовується для простих випадків запиту без параметрів.
+ __PreparedStatement__: попередньо компілює запит, який може містити вхідні параметри та виконуватися кілька разів з різним набором цих параметрів.

Перед виконанням СУБД розбирає кожен запит, оптимізує його та створює «план» (query plan) його виконання. Якщо той самий запит виконується кілька разів, СУБД може кешувати план його виконання й не повторювати етапи розбору та оптимізації. Завдяки цьому запит виконується швидше.

| Критерій | `Statement` | `PreparedStatement` |
|---|---|---|
| Параметри | немає, SQL збирається конкатенацією рядків | `?`-плейсхолдери, значення задаються через `setXxx()` |
| Прекомпіляція / кеш плану | ні | так, вигідно при повторному виконанні |
| Захист від SQL-ін'єкцій | немає (значення вклеюються в текст SQL) | є (значення передаються окремо від тексту запиту) |
| Коли застосовувати | разові DDL/прості запити без вхідних даних | переважна більшість випадків, особливо з даними користувача |

Підсумовуючи: _PreparedStatement_ вигідно відрізняється від _Statement_ тим, що при повторному використанні з одним або кількома наборами параметрів дає переваги заздалегідь прекомпільованого та кешованого запиту, водночас допомагаючи уникнути SQL-ін'єкцій. Що сказати на співбесіді: у продакшн-коді за замовчуванням слід використовувати `PreparedStatement` саме через захист від ін'єкцій, а виграш у продуктивності — приємний бонус.

## Як виконується запит до бази даних і обробка результатів?
Виконання запитів здійснюється викликом методів об'єкта, що реалізує інтерфейс `java.sql.Statement`:

+ __`executeQuery()`__ — для запитів, результатом яких є один набір значень, наприклад запитів `SELECT`. Результатом виконання є об'єкт класу `java.sql.ResultSet`;

+ __`executeUpdate()`__ — для виконання операторів `INSERT`, `UPDATE` або `DELETE`, а також операторів _DDL (Data Definition Language)_. Метод повертає ціле число, що показує, скільки записів було модифіковано;

+ __`execute()`__ — виконує SQL-команди, які можуть повертати різні результати. Повертає `true`, якщо перший результат містить _ResultSet_, і `false`, якщо перший результат — це кількість модифікованих записів або результат відсутній. Щоб отримати перший результат, потрібно викликати метод `getResultSet()` або `getUpdateCount()`. Решта результатів доступні через виклик `getMoreResults()`, який за потреби можна виконувати багаторазово.

Об'єкт з інтерфейсом `java.sql.ResultSet` зберігає в собі результат запиту до бази даних — певний набір даних, усередині якого є курсор, що вказує на один з елементів набору — поточний запис.

Використовуючи курсор, можна переміщатися набором даних за допомогою методу `next()`.

> __NB!__ Одразу після отримання набору даних його курсор перебуває перед першим записом, тож щоб зробити перший запис поточним, потрібно викликати метод `next()`. Метод `next()` повертає `false`, коли записи закінчилися, тому типовий обхід має вигляд `while (rs.next()) { ... }`.

Вміст полів поточного запису доступний через виклики методів `getInt()`, `getFloat()`, `getString()`, `getDate()` і подібних.

Приклад повного циклу «запит → обробка результатів» із автоматичним закриттям ресурсів (try-with-resources):

```java
String sql = "SELECT id, name FROM account WHERE balance > ?";
try (PreparedStatement ps = connection.prepareStatement(sql)) {
    ps.setBigDecimal(1, new BigDecimal("1000"));
    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) {
            long id = rs.getLong("id");
            String name = rs.getString("name");
            System.out.println(id + ": " + name);
        }
    }
}
```

## Як викликати збережену процедуру?
__Збережені процедури__ — це іменований набір операторів SQL, що зберігається на сервері. Таку процедуру можна викликати з Java-коду за допомогою об'єкта, що реалізує інтерфейс `java.sql.CallableStatement` (він є нащадком `Statement` та `PreparedStatement`).

Вибір об'єкта залежить від характеристик збереженої процедури:

+ без параметрів → достатньо `Statement`;
+ із вхідними параметрами → `PreparedStatement`;
+ із вхідними та/або вихідними параметрами → `CallableStatement`.

> Якщо невідомо, як було визначено збережену процедуру, для отримання інформації про неї (наприклад, імен і типів параметрів) можна використати методи `java.sql.DatabaseMetaData`, що дають змогу отримати інформацію про структуру джерела даних.

Приклад виклику збереженої процедури з вхідними та вихідними параметрами:

```java
public void runStoredProcedure(final Connection connection) throws Exception {
    // описуємо виклик збереженої процедури
    String procedure = "{ call procedureExample(?, ?, ?) }";

    // підготовляємо виклик (try-with-resources закриє CallableStatement)
    try (CallableStatement cs = connection.prepareCall(procedure)) {
        // встановлюємо вхідні параметри
        cs.setString(1, "abcd");
        cs.setInt(2, 10);

        // реєструємо вихідний параметр
        cs.registerOutParameter(3, java.sql.Types.INTEGER);

        // запускаємо виконання збереженої процедури
        cs.execute();

        // отримуємо результат вихідного параметра
        int result = cs.getInt(3);
        System.out.println(result);
    }
}
```

> __NB!__ В оригіналі цього прикладу було кілька помилок: друкарська помилка `vois` замість `void`; той самий параметр (1 і 2) одночасно встановлювався як вхідний (`setString`/`setBoolean`) і реєструвався як вихідний (`registerOutParameter`) — на більшості СУБД такий параметр має бути типу `INOUT`, інакше це некоректно. Тут приклад виправлено: окремі вхідні параметри (`1`, `2`) і окремий вихідний параметр (`3`).

## Як закрити з'єднання з базою даних?
З'єднання з базою даних закривається викликом методу `close()` у відповідного об'єкта `java.sql.Connection` або, що краще, за допомогою механізму try-with-resources (з'явився в Java 7), який гарантовано закриє ресурс навіть у разі винятку.

> __NB!__ Закривати слід усі JDBC-ресурси (`ResultSet`, `Statement`, `Connection`). Якщо ви оголошуєте їх у try-with-resources, вони закриваються автоматично у зворотному порядку, тож вручну стежити за порядком закриття не потрібно.

```java
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement("SELECT 1");
     ResultSet rs = ps.executeQuery()) {
    // робота з rs
} // rs, ps, conn закриються автоматично
```

> __NB!__ Якщо з'єднання отримане з пулу (`DataSource`), виклик `close()` не закриває фізичне з'єднання, а повертає його назад до пулу.
