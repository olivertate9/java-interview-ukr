## Що таке _ООП_?
__Об'єктно-орієнтоване програмування (ООП)__ — методологія програмування, що ґрунтується на поданні програми у вигляді сукупності об'єктів, кожен з яких є екземпляром певного класу, а класи утворюють ієрархію успадкування.

+ об'єктно-орієнтоване програмування використовує як основні логічні конструктивні елементи об'єкти, а не алгоритми;
+ кожен об'єкт є екземпляром певного класу;
+ класи утворюють ієрархії.

Програму вважають об'єктно-орієнтованою, лише якщо виконані всі три зазначені вимоги. Зокрема, програмування, що не використовує успадкування, називають не об'єктно-орієнтованим, а програмуванням за допомогою абстрактних типів даних.

Згідно з парадигмою ООП програма складається з об'єктів, що обмінюються повідомленнями. Об'єкти можуть мати стан, і єдиний спосіб змінити стан об'єкта — надіслати йому повідомлення, у відповідь на яке об'єкт може змінити власний стан.

## Назвіть основні принципи _ООП_.
+ _Інкапсуляція_ — приховування реалізації.
+ _Успадкування_ — створення нової сутності на базі вже наявної.
+ _Поліморфізм_ — можливість мати різні форми для однієї й тієї самої сутності.
+ _Абстракція_ — набір спільних характеристик.
+ _Надсилання повідомлень_ — форма зв'язку, взаємодії між сутностями.
+ _Повторне використання_ — усе перелічене вище працює на повторне використання коду.

Зауваження: класичним «трьома стовпами» ООП найчастіше називають **інкапсуляцію, успадкування й поліморфізм**, а абстракцію додають як четвертий принцип. Наведений вище порядок зручний для запам'ятовування, оскільки кожен наступний принцип спирається на попередні, але це радше методичне впорядкування, ніж жорстко зафіксований стандарт.

## Що таке _«інкапсуляція»_?
__Інкапсуляція__ — це об'єднання даних і методів, що з ними працюють, в одному класі та приховування внутрішньої реалізації за публічним інтерфейсом. Назовні видно лише те, *що* об'єкт уміє, а не те, *як* він це робить.

Головна думка, заради якої все й існує: інкапсуляція дозволяє **вільно змінювати реалізацію класу, не ламаючи код, який цим класом користується**. Чим менше клас обіцяє назовні, тим більше можна міняти всередині. («Користувач» тут — це не людина за програмою, а інший програміст, чий код викликає твій клас.)

### Як реалізовано в Java
+ **Модифікатори доступу** `private`, (package-private — без модифікатора), `protected`, `public` — керують тим, звідки видно поле чи метод. Базовий прийом: поля роблять `private`, а доступ до них — через `public`-методи.
+ **Гетери/сетери** — методи для контрольованого читання та зміни стану. Сетер може валідувати значення, логувати, кидати виняток; гетер може повертати копію чи обчислене значення.
+ **Розбиття на пакети** + `package-private`-видимість дозволяє приховати допоміжні класи всередині пакета.

### Порушення інкапсуляції
+ **Публічні (або package-private) змінювані поля** — `public int balance;`. Будь-хто може записати невалідне значення в обхід перевірок.
+ **Гетер/сетер на кожне поле «за звичкою».** Якщо сетер просто присвоює значення без жодної логіки, поле фактично лишається відкритим — інкапсуляції тут стільки ж, скільки в `public`-полі.
+ **Витік посилання на змінюваний внутрішній об'єкт** — гетер повертає сам внутрішній `List`/масив/`Date`, і зовнішній код міняє його напряму, минаючи клас. Те саме на вході: сетер зберігає передане посилання, не роблячи копії.
+ **Зайве розширення видимості заради тестів** (`private` → `protected`/`public`, щоб «дотягнутися» з тесту) — тест прив'язується до реалізації.

### Приклад
```java
class Account {
    private int balance; // ззовні напряму не дістати

    public int getBalance() {
        return balance;
    }

    public void withdraw(int amount) {
        if (amount <= 0 || amount > balance) {
            throw new IllegalArgumentException("Некоректна сума");
        }
        balance -= amount; // усі зміни проходять через одні «двері», які контролює сам клас
    }
}
```
Якби `balance` був `public`, зовнішній код міг би зробити `account.balance = -500` і клас цьому б ніяк не завадив. З приватним полем усі зміни проходять через метод, який валідує суму. Більше того — зовнішній код знає лише `getBalance()`, тож усередині можна будь-коли змінити тип `balance`, рахувати його зі списку транзакцій чи додати кеш, і жоден рядок коду-користувача не доведеться міняти.

## Що таке _«успадкування»_?
__Успадкування__ — це механізм, що дозволяє побудувати новий клас на основі вже наявного, успадкувавши його поля й методи. Нащадок отримує всю доступну йому поведінку базового класу й може її **доповнювати** (додавати нові члени) та **перевизначати** (змінювати реалізацію успадкованих методів).

Клас, від якого успадковують, називають _базовим_, _батьківським_ або _суперкласом_; новий клас — _похідним_, _нащадком_ або _підкласом_.

Головна думка для співбесіди: успадкування моделює відношення **«є» (is-a)** — підклас є окремим випадком суперкласу (`Dog` є `Animal`). Завдяки цьому об'єкт підкласу можна використовувати скрізь, де очікується суперклас (принцип підстановки Лісков), що є основою поліморфізму.

### Що дає успадкування
+ **Повторне використання коду** — спільну поведінку виносять у базовий клас, а не дублюють у кожному нащадку.
+ **Поліморфізм** — з об'єктами підкласів можна працювати через тип суперкласу, не знаючи конкретного типу.
+ **Розширюваність** — нову поведінку додають через новий підклас, не чіпаючи наявний код.

### Особливості успадкування в Java
+ Java підтримує лише **одиничне успадкування класів**: клас може мати лише один безпосередній суперклас (`extends` приймає один клас). Це зроблено навмисно, щоб уникнути «проблеми ромба» множинного успадкування реалізації.
+ Якщо клас не вказує `extends` явно, він неявно успадковує `java.lang.Object` — тому всі класи в Java мають спільного предка.
+ **Конструктори не успадковуються.** Конструктор підкласу першим ділом викликає конструктор суперкласу — явно через `super(...)` або неявно (`super()`), щоб правильно ініціалізувати успадковану частину об'єкта.
+ Успадковуються лише члени, доступні підкласу за модифікатором доступу. `private`-члени фізично присутні в об'єкті, але напряму недоступні; `final`-метод не можна перевизначити; `final`-клас не можна успадкувати взагалі.
+ До членів суперкласу звертаються через `super` (наприклад, `super.method()` усередині перевизначеного методу).

### Успадкування проти композиції
Успадкуванням не варто зловживати: воно створює жорсткий зв'язок між класами й порушує інкапсуляцію (підклас залежить від деталей суперкласу). Поширене правило — **«віддавай перевагу композиції перед успадкуванням»**: якщо відношення не «є» (is-a), а «має» (has-a), краще тримати інший об'єкт полем, а не успадковуватися. Успадкування виправдане лише за справжнього відношення is-a.

### Приклад
```java
class Account {
    protected double balance;

    public void deposit(double amount) {
        balance += amount;
    }

    public double getBalance() {
        return balance;
    }
}

// SavingsAccount є Account — успадковує deposit()/getBalance()
// і додає власну поведінку.
class SavingsAccount extends Account {
    private final double interestRate;

    public SavingsAccount(double interestRate) {
        this.interestRate = interestRate;
    }

    public void applyInterest() {       // нова поведінка
        deposit(balance * interestRate); // повторне використання успадкованого deposit()
    }
}
```

## Успадкування та інтерфейси в Java
Окрім успадкування класів (`extends`), у Java є другий, незалежний механізм — **реалізація інтерфейсів** (`implements`). Інтерфейс описує *контракт* (набір методів), а не реалізацію.

Ключова відмінність від успадкування класів — **множинність**: клас може реалізувати **скільки завгодно інтерфейсів** одночасно, хоча успадковувати може лише один клас.

```java
interface Flyable {
    void fly();
}

interface Swimmable {
    void swim();
}

// Один клас, кілька контрактів одночасно.
class Duck extends Animal implements Flyable, Swimmable {
    @Override
    public void fly() { /* ... */ }

    @Override
    public void swim() { /* ... */ }
}
```

Тому в Java розрізняють:
+ **успадкування типу (type inheritance)** — через інтерфейси; клас зобов'язується надати методи контракту. Множинне — дозволене.
+ **успадкування реалізації (implementation inheritance)** — через `extends` від класу; нащадок отримує готовий код. Лише одиничне.

Корисні деталі про інтерфаси на співбесіді:
+ Інтерфейси теж утворюють ієрархію: один інтерфейс може **розширювати** інші через `extends` (і навіть кілька одразу: `interface A extends B, C`).
+ З **Java 8** інтерфейси можуть містити `default`-методи (з реалізацією) та `static`-методи; з **Java 9** — `private`-методи (допоміжні для `default`). `default`-методи частково повертають у Java множинне успадкування *поведінки*, але не стану — полів-екземплярів в інтерфейсах немає (лише `public static final` константи).
+ Якщо клас успадковує два `default`-методи з однаковою сигнатурою від різних інтерфейсів — виникає конфлікт, і клас зобов'язаний перевизначити метод явно (це і є контрольоване розв'язання «проблеми ромба»).

## Що таке _«поліморфізм»_?
__Поліморфізм__ — це здатність працювати з об'єктами різних типів через єдиний спільний інтерфейс, не знаючи їхнього конкретного типу. Один і той самий виклик методу дає різну поведінку залежно від того, який об'єкт за ним стоїть.

Головна вигода: код, що використовує об'єкти, залежить від **абстракції** (інтерфейс або базовий клас), а не від конкретних реалізацій. Завдяки цьому можна додавати нові типи, не змінюючи вже наявний код.

### Види поліморфізму
+ **Поліморфізм підтипів (динамічний, runtime)** — те, що зазвичай мають на увазі в ООП. Реалізується через **перевизначення методів (overriding)**: посилання має тип базового класу/інтерфейсу, а викликається метод фактичного об'єкта. Конкретна реалізація обирається під час виконання — це називають _пізнім (динамічним) зв'язуванням_.
+ **Ad hoc поліморфізм** — реалізується через **перевантаження методів (overloading)**: кілька методів з однаковим іменем, але різними параметрами; який саме викликати, вирішує компілятор за статичними типами аргументів (раннє зв'язування).
+ **Параметричний поліморфізм** — у Java це **дженерики (generics)**: один і той самий код працює з різними типами, наприклад `List<T>` однаково зберігає елементи будь-якого типу.

На співбесіді під «поліморфізмом» майже завжди мають на увазі **перший вид** — динамічну диспетчеризацію через перевизначення.

### Приклад: колекції
Класичний приклад поліморфізму в Java — інтерфейс `List` і його реалізації. Код працює зі змінною типу `List`, не знаючи й не цікавлячись, що саме за нею — `ArrayList`, `LinkedList` чи щось інше:

```java
List<String> names = new ArrayList<>(); // оголошений тип — List, фактичний — ArrayList
names.add("Андрій");
names.add("Олена");
```

Тут `names.add(...)` викликає реалізацію `add` саме з `ArrayList`. Якщо замінити лише перший рядок на `new LinkedList<>()`, увесь інший код працюватиме без жодної зміни — викликатиметься вже реалізація `add` з `LinkedList`. Це і є поліморфізм підтипів: виклик той самий, поведінка — від фактичного об'єкта.

Сила підходу видно, коли метод приймає **абстракцію**, а не конкретний клас:

```java
// Метод залежить від інтерфейсу List, а не від конкретної реалізації.
// Тому йому можна передати будь-яку реалізацію — наявну чи майбутню.
public static int totalLength(List<String> items) {
    int total = 0;
    for (String item : items) {
        total += item.length();
    }
    return total;
}

totalLength(new ArrayList<>(List.of("a", "bb")));  // працює
totalLength(new LinkedList<>(List.of("ccc")));     // теж працює — код методу незмінний
```

Тут же й практичний наслідок для дизайну: змінні та параметри варто оголошувати за **інтерфейсом** (`List`, `Map`, `Collection`), а не за конкретним класом (`ArrayList`, `HashMap`). Тоді реалізацію легко замінити, а код, що її використовує, не доведеться чіпати — це пряме застосування принципу «програмуй на рівні інтерфейсу, а не реалізації».

Ще один щоденний приклад поліморфізму — `toString()`, `equals()`, `hashCode()`: ви викликаєте їх через посилання типу `Object`, а виконується перевизначена версія конкретного класу.

### `@Override`
Перевизначення (overriding) — це коли підклас надає власну реалізацію методу базового класу, **зберігаючи його сигнатуру**. Анотація `@Override` не обов'язкова, але бажана: вона змушує компілятор перевірити, що метод дійсно перевизначає метод суперкласу. Якщо ви випадково змінили сигнатуру (наприклад, інший тип параметра), отримаєте помилку компіляції, а не мовчазне перевантаження (overloading) замість перевизначення.

## Що таке _«абстракція»_?
__Абстракція__ — це виділення в об'єкті лише тих характеристик, які важливі **залежно від потреб конкретної задачі**, і відкидання всіх інших як несуттєвих. Ключове тут саме «залежно від потреб»: не існує єдиного «правильного» набору полів для сутності — він визначається тим, для чого ця сутність потрібна в конкретному контексті.

### Один клас — різні моделі залежно від потреб
Найкраще це видно на прикладі, коли в різних системах є клас з однаковою назвою, але зовсім різним наповненням, бо різні потреби.

Візьмемо сутність **`Person`** (людина).

У системі **аеропорту** нас цікавить ідентифікація пасажира та політ:
```java
class Person {
    private String passportNumber;
    private String fullName;
    private LocalDate dateOfBirth;
    private String nationality;
    private String visaStatus;
    // тут немає ваги, зросту, групи крові — для реєстрації на рейс це неважливо
}
```

У **медичній** системі та сама людина моделюється зовсім інакше — тут важливе здоров'я, а не громадянство чи віза:
```java
class Person {
    private String medicalCardNumber;
    private BloodType bloodType;
    private double weightKg;
    private double heightCm;
    private List<String> allergies;
    private List<Diagnosis> diagnoses;
    // тут немає номера паспорта чи візи — для лікування це неважливо
}
```

Клас називається однаково — `Person` — і моделює один і той самий об'єкт реального світу (людину), але **набори полів не перетинаються**, бо потреби систем різні. Лікарню не цікавить віза, а аеропорт не зберігає групу крові. Це і є абстракція: кожна система виділяє рівно ті характеристики, які потрібні їй, і ігнорує решту.

Висновок для співбесіди: абстракція — це не «прибрати деталі взагалі», а «прибрати деталі, **неважливі для цієї задачі**». Той самий об'єкт може мати скільки завгодно різних коректних абстракцій.

### Абстракція в коді: `abstract`-клас та `interface`
У Java абстракцію як механізм підтримують **абстрактні класи** та **інтерфейси** — вони дозволяють описати *що* сутність уміє (набір методів), не фіксуючи *як* саме це реалізовано:

```java
abstract class Shape {
    abstract double area(); // що вміє будь-яка фігура — порахувати площу; як саме — визначає підклас

    void printArea() {      // спільна поведінка для всіх фігур
        System.out.println("Площа: " + area());
    }
}

class Circle extends Shape {
    private final double radius;

    Circle(double radius) {
        this.radius = radius;
    }

    @Override
    double area() {
        return Math.PI * radius * radius;
    }
}
```
Код, що працює зі `Shape`, оперує абстракцією «фігура вміє порахувати площу», не знаючи, коло це, прямокутник чи трикутник.

## Що являє собою _«обмін повідомленнями»_?
Об'єкти взаємодіють, надсилаючи й отримуючи повідомлення. Повідомлення — це запит на виконання дії, доповнений набором аргументів, які можуть знадобитися під час виконання дії. В ООП надсилання повідомлення (виклик методу) — це єдиний спосіб передати керування об'єкту. Якщо об'єкт має «відповідати» на це повідомлення, то в нього повинен бути відповідний цьому повідомленню метод. Так само об'єкти, використовуючи свої методи, можуть і самі надсилати повідомлення іншим об'єктам. Обмін повідомленнями реалізується за допомогою динамічних викликів, що в деяких мовах (наприклад, Smalltalk) призводить до надзвичайно пізнього зв'язування (extreme late binding).

>Нехай потрібно створити фізичну модель, що описує кулі різних розмірів, які стикаються. Традиційний підхід до розв'язання цього завдання приблизно такий: визначається набір даних, що описують кожну кулю (наприклад, її координати, маса та прискорення); кожній кулі присвоюється унікальний ідентифікатор (наприклад, організовується масив, значення індексу якого відповідає номеру кулі), що дозволить відрізняти кожну з куль від усіх інших. Нарешті, пишеться підпрограма з назвою, скажімо, `bounce`; ця процедура має на основі номера кулі та її початкових параметрів відповідним чином змінювати дані, що описують кулю. На відміну від традиційного підходу, об'єктно-орієнтована версія програми моделює кожну з куль за допомогою об'єкта. При цьому об'єкт, що відповідає конкретній кулі, містить не лише її параметри, а й увесь код, що описує поведінку кулі при різних взаємодіях. Так, кожна куля матиме власний метод `bounce()`. Замість того, щоб викликати підпрограму `bounce` з аргументом, що визначає, скажімо, кулю №3, потрібно буде передати об'єкту «куля №3» повідомлення, що приписує йому виконати зіткнення.

## Розкажіть про основні поняття ООП: _«клас»_, _«об'єкт»_, _«інтерфейс»_.
__Клас__ — це спосіб опису сутності, що визначає стан і поведінку, яка залежить від цього стану, а також правила для взаємодії з цією сутністю (контракт).

З точки зору програмування клас можна розглядати як набір даних (полів, атрибутів, членів класу) і функцій для роботи з ними (методів).

З точки зору структури програми клас є складним типом даних.

__Об'єкт (екземпляр)__ — це окремий представник класу, що має конкретний стан і поведінку, які повністю визначаються класом. Кожен об'єкт має конкретні значення атрибутів і методи, що працюють із цими значеннями на основі правил, заданих у класі.

__Інтерфейс__ — це набір методів класу, доступних для використання. Інтерфейсом класу буде набір усіх його публічних методів у сукупності з набором публічних атрибутів. По суті, інтерфейс специфікує клас, чітко визначаючи всі можливі дії над ним.

## У чому полягають переваги та недоліки об'єктно-орієнтованого підходу в програмуванні?
Переваги:

+ Об'єктна модель цілком природна, оскільки насамперед орієнтована на людське сприйняття світу, а не на комп'ютерну реалізацію.
+ Класи дозволяють вести конструювання з корисних компонентів, що мають прості інструменти, що дозволяє абстрагуватися від деталей реалізації.
+ Дані та операції над ними утворюють певну сутність і не розносяться по всій програмі, як нерідко буває у випадку процедурного програмування, а описуються разом. Локалізація коду й даних покращує наочність і зручність супроводу програмного забезпечення.
+ Інкапсуляція дозволяє привнести властивість модульності, що полегшує розподіл виконання завдання між кількома виконавцями та оновлення версій окремих компонентів.
+ Можливість створювати розширювані системи.
+ Використання поліморфізму виявляється корисним при:
    + Обробці різнорідних структур даних. Програми можуть працювати, не розрізняючи виду об'єктів, що суттєво спрощує код. Нові види можуть бути додані будь-якої миті.
    + Зміні поведінки під час виконання. На етапі виконання один об'єкт може бути замінений іншим, що дозволяє легко, без зміни коду, адаптувати алгоритм залежно від того, який використовується об'єкт.
    + Реалізації роботи з нащадками. Алгоритми можна узагальнити настільки, що вони вже зможуть працювати більш ніж з одним видом об'єктів.
    + Можливості описати незалежні від застосунку частини предметної області у вигляді набору універсальних класів, або фреймворку, який надалі буде розширено за рахунок додавання частин, специфічних для конкретного застосунку.
+ Повторне використання коду:
    + Скорочується час на розробку, який може бути віддано іншим завданням.
    + Компоненти багаторазового використання зазвичай містять набагато менше помилок, ніж щойно розроблені, адже вони вже не раз перевірялися.
    + Коли певний компонент використовується одразу кількома клієнтами, поліпшення, що вносяться в його код, одночасно справляють позитивний вплив і на безліч програм, які з ним працюють.
    + Якщо програма спирається на стандартні компоненти, її структура та інтерфейс користувача стають більш уніфікованими, що полегшує її розуміння та спрощує використання.

Недоліки:

+ У складних ієрархіях класів поля та методи зазвичай успадковуються з різних рівнів. І не завжди легко визначити, які поля та методи фактично належать до цього класу.
+ Код для обробки повідомлення іноді «розмазаний» по багатьох методах (інакше кажучи, обробка повідомлення вимагає не одного, а багатьох методів, які можуть бути описані в різних класах).
+ Документування класів — завдання складніше, ніж це було у випадку процедур і модулів. Оскільки будь-який метод може бути перевизначено, у документації має йтися не лише про те, що робить цей метод, а й про те, у якому контексті він викликається.
+ Неефективність і неощадливий розподіл пам'яті на етапі виконання (через витрати на динамічне зв'язування та перевірки типів на етапі виконання).
+ Надмірна універсальність. Часто міститься більше методів, ніж це реально необхідно поточній програмі. А оскільки зайві методи не можуть бути видалені, вони стають мертвим вантажем.

## Що мають на увазі в плані принципів ООП вирази _«є»_ та _«має»_?
__«є»__ (is-a) передбачає успадкування.
__«має»__ (has-a) передбачає асоціацію (агрегацію або композицію).

## У чому різниця між _композицією_ та _агрегацією_?
Асоціація позначає зв'язок між об'єктами. Композиція й агрегація — окремі випадки асоціації «частина-ціле».

_Композиція_ — вкладений об'єкт не може існувати без власника (власник створює та знищує його):

```java
class Engine {
    private final int horsepower;

    Engine(int horsepower) {
        this.horsepower = horsepower;
    }
}

class Car {
    private final Engine engine; // Engine живе й помирає разом із Car

    Car(int horsepower) {
        this.engine = new Engine(horsepower); // Car сам створює Engine
    }
}
```

_Агрегація_ — вкладений об'єкт існує незалежно, передається ззовні:

```java
class Driver {
    private final String name;

    Driver(String name) {
        this.name = name;
    }
}

class Car {
    private Driver driver; // Driver може існувати без Car

    Car(Driver driver) {
        this.driver = driver; // Car отримує Driver ззовні
    }
}
```

Коротко: якщо `new` усередині класу — композиція; якщо об'єкт приходить ззовні — агрегація.

Краще використовувати агрегацію та впроваджувати залежності через конструктор. Це автоматично робить код тестовним і відповідає принципу інверсії залежностей (Dependency Inversion) із SOLID. Композицію варто застосовувати, коли є чітка причина «сховати» створення об'єкта всередину класу.

## Що таке _статичне_ та _динамічне зв'язування_?
Зіставлення виклику методу з тілом методу називають зв'язуванням. Якщо зв'язування проводиться компілятором (компонувальником) перед запуском програми, то воно називається _статичним_, або _раннім зв'язуванням (early binding)_.

Своєю чергою, _пізнє зв'язування (late binding)_ — це зв'язування, що проводиться безпосередньо під час виконання програми, залежно від типу об'єкта. Пізнє зв'язування також називають _динамічним (dynamic)_, або _зв'язуванням на стадії виконання (runtime binding)_. У мовах, що реалізують пізнє зв'язування, повинен існувати механізм визначення фактичного типу об'єкта під час роботи програми для виклику відповідного методу. Інакше кажучи, компілятор не знає типу об'єкта, але механізм виклику методів визначає його та викликає відповідне тіло методу. Механізм пізнього зв'язування залежить від конкретної мови, але неважко припустити, що для його реалізації в об'єкти має включатися якась додаткова інформація.

Для більшості методів Java використовується механізм пізнього (динамічного) зв'язування, якщо тільки метод не був оголошений як `final`, `static` або `private` (приватні методи є `final` за замовчуванням, оскільки не успадковуються й не можуть бути перевизначені). Зауважте, що `static`-методи зв'язуються за статичним (оголошеним) типом і не беруть участі в поліморфізмі.
