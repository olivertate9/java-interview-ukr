## Що таке _«сервлет»_?
__Сервлет__ — це Java-компонент серверного боку, який розширює функціональні можливості сервера. Сервлет взаємодіє з клієнтами за принципом «запит-відповідь». Хоча сервлети можуть обслуговувати будь-які запити, зазвичай вони використовуються для розширення вебсерверів.

Технічно `Servlet` — це інтерфейс; на практиці сервлети успадковують від `HttpServlet`, який надає готову реалізацію для HTTP.

Більшість потрібних для створення сервлетів класів та інтерфейсів містяться в пакетах `javax.servlet` та `javax.servlet.http` (а починаючи з Jakarta EE 9 — у `jakarta.servlet` та `jakarta.servlet.http`, оскільки після передачі Java EE до Eclipse Foundation простір імен було перейменовано з `javax` на `jakarta`).

Основні методи сервлета:

+ `public void init(ServletConfig config) throws ServletException` — запускається одразу після завантаження сервлета в пам'ять;
+ `public ServletConfig getServletConfig()` — повертає посилання на об'єкт, що надає доступ до інформації про конфігурацію сервлета;
+ `public String getServletInfo()` — повертає рядок із інформацією про сервлет, наприклад: автор і версія сервлета;
+ `public void service(ServletRequest request, ServletResponse response) throws ServletException, java.io.IOException` — викликається для обробки кожного запиту;
+ `public void destroy()` — виконується перед вивантаженням сервлета з пам'яті.

Специфікація сервлетів пройшла довгий шлях: Servlet 3.1 (JSR-340, 2013), Servlet 4.0 (JSR-369, 2017, із підтримкою HTTP/2 та server push), а далі специфікація перейшла під егіду Jakarta EE: Jakarta Servlet 5.0 (зміна `javax` → `jakarta`), 6.0 та новіші.

## У чому полягають переваги технології сервлетів над CGI (Common Gateway Interface)?
+ Сервлети забезпечують кращу продуктивність обробки запитів і ефективніше використання пам'яті за рахунок багатопотоковості: на кожен запит створюється новий потік (thread), що дешевше за породження окремого процесу під кожен запит, як це відбувається в CGI.
+ Сервлети є платформо- та системно-незалежними. Веб-застосунок, написаний із використанням сервлетів, може бути запущений у будь-якому контейнері сервлетів, що реалізує стандарт, і в будь-якій операційній системі.
+ Використання сервлетів підвищує надійність програми, оскільки контейнер сервлетів самостійно дбає про життєвий цикл сервлетів (а отже й про витоки пам'яті), безпеку та збирання сміття.
+ Сервлети відносно прості у вивченні та підтримці, тож розробник може зосередитися лише на бізнес-логіці застосунку, а не на внутрішній реалізації веб-технологій.

## Яка структура веб-проєкту?
(структура наведена за стандартним розташуванням каталогів Maven)

`src/main/java` — вихідний код застосунку/бібліотеки

`src/main/resources` — ресурсні файли застосунку/бібліотеки

`src/main/filters` — файли фільтрів ресурсів (resource filters)

`src/main/webapp` — вихідні файли веб-застосунку (JSP, статичні ресурси, `WEB-INF`)

`src/test/java` — вихідний код тестів

`src/test/resources` — ресурсні файли тестів

## Що таке _«контейнер сервлетів»_?
__Контейнер сервлетів__ — програма (сервер), яка займається системною підтримкою сервлетів та забезпечує їхній життєвий цикл відповідно до правил, визначених у специфікаціях. Може працювати як повноцінний самостійний вебсервер, бути постачальником сторінок для іншого вебсервера або інтегруватися в Jakarta EE (Java EE) сервер застосунків.

Контейнер сервлетів забезпечує обмін даними між сервлетом і клієнтами, бере на себе виконання таких функцій, як створення програмного середовища для сервлета, ідентифікацію та авторизацію клієнтів, організацію сесії для кожного з них.

Найвідоміші реалізації контейнерів сервлетів:

+ Apache Tomcat
+ Jetty
+ Undertow
+ WildFly (раніше JBoss AS)
+ GlassFish / Eclipse GlassFish
+ IBM WebSphere
+ Oracle WebLogic

## Навіщо потрібні сервери застосунків, якщо є контейнери сервлетів?
Контейнер сервлетів (як-от Tomcat чи Jetty) реалізує лише веб-рівень. Сервер застосунків (WildFly, WebSphere, WebLogic) — це повний стек Jakarta EE поверх веб-контейнера, що «з коробки» додає корпоративні можливості:

+ __Пули з'єднань із БД__
    + Періодична перевірка доступності СУБД та оновлення з'єднання при відновленні після збоїв
    + Підміна прав доступу при підключенні
    + Балансування навантаження між кількома СУБД, визначення доступності або недоступності того чи іншого вузла
    + Захист пулу з'єднань від некоректного коду в застосунку, який через недогляд не повертає з'єднання, — пул просто відбирає його після певного тайм-ауту
+ __JMS__
    + Доступність сервера черг повідомлень «з коробки»
    + Можливість кластеризації черг, тобто побудови розподілених черг, розташованих одразу на кількох серверах, що суттєво підвищує масштабованість і доступність застосунку
    + Можливість міграції черг — у разі падіння одного із серверів його черги автоматично переміщуються на інший, зберігаючи необроблені повідомлення
    + У деяких серверах застосунків підтримується _Unit-of-Order_ — гарантований порядок обробки повідомлень, що відповідають певним критеріям
+ __JTA__ — вбудована підтримка розподілених транзакцій для забезпечення узгодженості даних у різних СУБД або чергах
+ __Безпека__
    + Наявність багатьох провайдерів безпеки та автентифікації:
        + у вбудованому або зовнішньому _LDAP-сервері_
        + у базі даних
        + у різних _Internet-directory_ (спеціалізованих застосунках для керування правами доступу)
    + Доступність _Single-Sign-On_ (можливості спільного використання сесії користувача між застосунками) за допомогою _Security Assertion Markup Language (SAML) 1/2_ або _Simple and Protected Negotiate (SPNEGO)_ та _Kerberos_: один із серверів виступає базою для зберігання користувачів, усі інші сервери при автентифікації звертаються до цієї бази
    + Можливість авторизації за протоколом _eXtensible Access Control Markup Language (XACML)_, що дозволяє описувати доволі складні політики (наприклад, застосунок доступний користувачу лише в робочий час)
    + Кластеризація всього переліченого вище
+ __Масштабованість і висока доступність__. Для контейнера сервлетів зазвичай також можна налаштувати кластеризацію, але вона буде доволі примітивною через такі обмеження:
    + Складність передавання сесії користувача з одного _центру обробки даних (ЦОД)_ до іншого через Інтернет
    + Відсутність можливості ефективно налаштувати реплікацію сесій на великому (40–50 екземплярів серверів) кластері
    + Неможливість забезпечити міграцію екземплярів застосунку на інший сервер
    + Недоступність механізмів автоматичного моніторингу та реакції на помилки
+ __Керованість__
    + Наявність єдиного центру керування — т. зв. _AdminServer_ та аналога _NodeManager_, що забезпечує:
        + Можливість одночасного запуску кількох екземплярів сервера
        + Перегляд стану запущених екземплярів сервера, обробників тієї чи іншої черги, на тому чи іншому сервері, кількості з'єднань із тією чи іншою БД
+ __Адміністративний канал і розгортання в промисловому режимі__. Деякі сервери застосунків дозволяють увімкнути так званий «адміністративний канал» — окремий порт, запити по якому мають пріоритет:
    + Перегляд стану (виконуваних транзакцій, потоків, черг) у разі недоступності («зависання») сервера
    + Оновлення застосунків «на льоту», без простою:
        + додавання на сервер нової версії застосунку в «закритому» режимі, поки користувачі продовжують працювати з попередньою
        + тестування коректності розгортання нової версії
        + «прихований» перехід усіх користувачів на нову версію

## Як контейнер сервлетів керує життєвим циклом сервлета, коли і які методи викликаються?
Контейнер сервлетів керує чотирма фазами життєвого циклу сервлета:

+ __Завантаження класу сервлета__ — коли контейнер отримує запит для сервлета (або при старті, якщо вказано `load-on-startup`), відбувається завантаження класу сервлета в пам'ять і виклик його конструктора без параметрів.
+ __Ініціалізація сервлета__ — після завантаження класу контейнер створює об'єкт `ServletConfig` для цього сервлета і передає його через метод `init()`. Саме тут звичайний клас «оживає» як сервлет.
+ __Обробка запитів__ — після ініціалізації сервлет готовий обробляти запити. Для кожного запиту клієнта контейнер виділяє потік (зазвичай із пулу) і викликає метод `service()`, передаючи посилання на об'єкти запиту та відповіді.
+ __Знищення__ — коли контейнер зупиняється або зупиняється застосунок, контейнер знищує сервлет викликом методу `destroy()`.

Таким чином, сервлет створюється в одному екземплярі й живе протягом усього часу роботи застосунку (на відміну від звичайних об'єктів, які знищуються збирачем сміття щойно перестають використовуватися). Увесь життєвий цикл сервлета можна описати як послідовність викликів методів:

+ `public void init(ServletConfig config)` — використовується контейнером для ініціалізації сервлета. Викликається один раз за час життя сервлета.
+ `public void service(ServletRequest request, ServletResponse response)` — викликається для кожного запиту. Метод не може бути викликаний раніше за виконання методу `init()`.
+ `public void destroy()` — викликається для знищення сервлета (один раз за час життя сервлета).

> Важливо: оскільки екземпляр сервлета один, а потоків багато, метод `service()` (і `doGet()`/`doPost()`) виконується конкурентно — поля сервлета не є потокобезпечними. Про це детальніше — у питанні про багатопотокову безпеку.

## Що таке _«дескриптор розгортання»_?
Дескриптор розгортання — це конфігураційний файл артефакта, який буде розгорнутий у контейнері сервлетів. У специфікації Jakarta EE (раніше Java EE) дескриптор розгортання описує те, як компонент, модуль або застосунок (наприклад, веб-застосунок або корпоративний застосунок) має бути розгорнутий.

Цей файл задає параметри розгортання для модуля чи застосунку з певними налаштуваннями, параметри безпеки та описує конкретні вимоги до конфігурації. Синтаксис файлів дескриптора розгортання — XML.

> Примітка: починаючи зі Servlet 3.0 (Java EE 6) дескриптор розгортання `web.xml` здебільшого необов'язковий — більшість конфігурації задається анотаціями (`@WebServlet`, `@WebFilter`, `@WebListener`). `web.xml` досі корисний, коли потрібно перевизначити анотації, налаштувати параметри централізовано або задати елементи, для яких анотацій немає.

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<web-app xmlns="http://xmlns.jcp.org/xml/ns/javaee"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://xmlns.jcp.org/xml/ns/javaee
        http://xmlns.jcp.org/xml/ns/javaee/web-app_3_1.xsd"
    version="3.1">

    <display-name>Display name.</display-name>
    <description>Description text.</description>

    <servlet>
        <servlet-name>ExampleServlet</servlet-name>
        <servlet-class>xyz.company.ExampleServlet</servlet-class>
        <load-on-startup>1</load-on-startup>
        <init-param>
            <param-name>configuration</param-name>
            <param-value>default</param-value>
        </init-param>
    </servlet>

    <servlet-mapping>
        <servlet-name>ExampleServlet</servlet-name>
        <url-pattern>/example</url-pattern>
    </servlet-mapping>

    <servlet>
        <servlet-name>ExampleJSP</servlet-name>
        <jsp-file>/sample/Example.jsp</jsp-file>
    </servlet>

    <context-param>
        <param-name>myParam</param-name>
        <param-value>the value</param-value>
    </context-param>
</web-app>
```

Для веб-застосунків дескриптор розгортання має називатися `web.xml` і розташовуватися в каталозі `WEB-INF` у корені веб-застосунку. Це стандартний дескриптор, визначений у специфікації. Існують і інші типи дескрипторів, зокрема `glassfish-web.xml` (раніше `sun-web.xml`) із даними розгортання, специфічними для GlassFish, або `application.xml` у каталозі `META-INF` для корпоративних (EAR) застосунків.

## Які дії необхідно виконати при створенні сервлетів?
Найпростіший спосіб (Servlet 3.0+) — оголосити сервлет анотацією, без `web.xml`:

```java
@WebServlet(urlPatterns = "/example",
    initParams = { @WebInitParam(name = "config", value = "default") })
public class ExampleServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws ServletException, IOException {
        // логіка обробки запиту
    }
}
```

Альтернативно сервлет можна описати в дескрипторі розгортання `web.xml`:

```xml
<servlet>
    <servlet-name>ExampleServlet</servlet-name>
    <servlet-class>xyz.company.ExampleServlet</servlet-class>
    <init-param>
        <param-name>config</param-name>
        <param-value>default</param-value>
    </init-param>
</servlet>
<servlet-mapping>
    <servlet-name>ExampleServlet</servlet-name>
    <url-pattern>/example</url-pattern>
</servlet-mapping>
```

> У початковому матеріалі порядок елементів `<servlet-mapping>` і `<servlet>` був переставлений; за схемою `web.xml` зазвичай спершу йде `<servlet>`, потім `<servlet-mapping>`.

Далі — створити клас, успадкувавши його від `HttpServlet`, і реалізувати логіку в методі `service()` або в методах `doGet()`/`doPost()`.

## У якому випадку потрібно перевизначати метод `service()`?
Метод `service()` перевизначають, коли потрібно, щоб сервлет обробляв усі запити (і `GET`, і `POST`, і решту) в одному методі.

Коли контейнер сервлетів отримує запит клієнта, він викликає метод `service()`, який залежно від HTTP-методу запиту делегує виклик до `doGet()`, `doPost()`, `doPut()` тощо. Перевизначаючи `service()` напряму, ви втрачаєте цю автоматичну диспетчеризацію, тому на практиці частіше перевизначають конкретні `doXxx()`-методи.

[до змісту](#servlets-jsp-jstl)

## Чи є сенс визначати для сервлета конструктор? Яким чином краще ініціалізувати дані?
Особливого сенсу визначати для сервлета конструктор немає: на момент роботи конструктора сервлет ще не «оживлений» контейнером — у нього немає `ServletConfig`, а отже й доступу до параметрів ініціалізації та `ServletContext`. Тому дані краще ініціалізувати не в конструкторі, а перевизначивши метод `init()`, у якому вже є доступ до параметрів ініціалізації через об'єкт `ServletConfig`.

[до змісту](#servlets-jsp-jstl)

## Чому необхідно перевизначати лише метод `init()` без аргументів?
Метод `init()` перевизначають, якщо потрібно ініціалізувати якісь дані до того, як сервлет почне обробляти запити.

`GenericServlet` надає дві форми: `init(ServletConfig config)` і зручний `init()` без параметрів. Метод `init(ServletConfig config)` зберігає переданий `config` у внутрішнє поле й наприкінці викликає `init()` без параметрів. Якщо ви перевизначите `init(ServletConfig config)` і забудете викликати `super.init(config)`, контейнер не збереже `config`, і подальші виклики `getServletConfig()`, `getServletContext()` повернуть `null` — типова помилка.

Щоб цього уникнути, рекомендують перевизначати саме `init()` без параметрів: він викликається після того, як базовий клас уже коректно зберіг конфігурацію, тож ризику «забути `super()`» немає.

## Які найпоширеніші завдання виконуються в контейнері сервлетів?
+ __Підтримка обміну даними.__ Контейнер сервлетів надає простий спосіб обміну даними між вебклієнтом (браузером) і сервлетом. Завдяки контейнеру немає потреби створювати слухача сокета на сервері, розбирати «сирий» запит і вручну формувати відповідь — усі ці важливі й складні завдання вирішує контейнер, а розробник зосереджується на бізнес-логіці.
+ __Керування життєвим циклом сервлетів і ресурсів.__ Від завантаження сервлета в пам'ять, ініціалізації — до знищення. Контейнер також надає допоміжні утиліти, наприклад JNDI, для керування пулом ресурсів.
+ __Підтримка багатопотоковості.__ Контейнер самостійно виділяє потік (зазвичай із пулу) для кожного запиту й передає йому об'єкти запиту та відповіді. Завдяки цьому сервлет не ініціалізується заново на кожен запит, що економить пам'ять і зменшує час до початку обробки.
+ __Підтримка JSP.__ JSP-сторінки не схожі на звичайні Java-класи, але контейнер перетворює кожну JSP на сервлет і далі керує нею як звичайним сервлетом.
+ __Інші завдання.__ Контейнер керує пулом ресурсів, пам'яттю застосунку, взаємодіє зі збирачем сміття, надає можливості налаштування безпеки тощо.

## Що ви знаєте про _сервлетні фільтри_?
__Сервлетний фільтр__ — це придатний для повторного використання Java-код, що дозволяє перетворювати вміст HTTP-запитів, HTTP-відповідей та інформацію в їхніх заголовках. Фільтр виконує попередню обробку запиту до того, як той потрапить у сервлет, і/або подальшу обробку відповіді, що виходить із сервлета.

Ментальна модель: фільтр — це «прошарок» у конвеєрі обробки запиту. Класичні застосування — автентифікація/авторизація, логування, стиснення (gzip), шифрування, заміна кодування, CORS.

Сервлетні фільтри можуть:

+ перехоплювати запит до того, як буде викликаний сервлет;
+ досліджувати вміст запиту до виклику сервлета;
+ модифікувати заголовки та дані запиту, обгортаючи вхідний запит;
+ модифікувати заголовки та дані відповіді, обгортаючи вихідну відповідь;
+ перехоплювати обробку після звернення до сервлета.

Фільтр можна налаштувати на роботу з одним сервлетом або групою сервлетів. Основою для фільтрів служить інтерфейс `javax.servlet.Filter` (нині `jakarta.servlet.Filter`), що оголошує методи:

+ `default void init(FilterConfig config) throws ServletException`;
+ `void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException`;
+ `default void destroy()`.

> У сучасних версіях специфікації `init()` і `destroy()` мають реалізацію за замовчуванням (`default`), тож обов'язковим до реалізації лишається тільки `doFilter()`.

Метод `init()` викликається один раз, перш ніж фільтр почне працювати, і налаштовує конфігурацію фільтра. Метод `doFilter()` виконує безпосередньо роботу фільтра й викликається на кожен запит, що проходить через цей фільтр. Після завершення роботи фільтра викликається `destroy()`.

Інтерфейс `FilterConfig` містить методи для отримання імені фільтра, його параметрів ініціалізації та `ServletContext`. У `doFilter()` фільтр отримує поточні `request`, `response` та `FilterChain` — ланцюжок фільтрів, призначених для обробки. У `doFilter()` фільтр може робити із запитом і відповіддю будь-що — збирати дані або обгортати об'єкти для нової поведінки. Потім фільтр викликає `chain.doFilter(request, response)`, щоб передати керування наступному фільтру (а останній — сервлету). Після повернення з цього виклику фільтр може виконати додаткову роботу над відповіддю, наприклад зберегти реєстраційну інформацію.

Після компіляції клас-фільтр потрібно оголосити та _«прив'язати» (map)_ до одного чи кількох сервлетів. Оголошення й підключення фільтра відображається в дескрипторі розгортання `web.xml` усередині елементів `<filter>` і `<filter-mapping>` (або, починаючи зі Servlet 3.0, анотацією `@WebFilter`). Для підключення фільтра до сервлета використовують вкладені елементи `<filter-name>` і `<servlet-name>`.

> Оголошення класу-фільтра `FilterConnect` з ім'ям `FilterName`:

```xml
<filter>
    <filter-name>FilterName</filter-name>
    <filter-class>FilterConnect</filter-class>
    <init-param>
        <!-- фільтр має параметр ініціалізації `active` зі значенням `true` -->
        <param-name>active</param-name>
        <param-value>true</param-value>
    </init-param>
</filter>
```

> У вихідному коді тут була друкарська помилка: закривальний тег `</param-true>` замість `</param-value>`. Виправлено.

> Підключення фільтра `FilterName` до сервлета `ServletName`:

```xml
<filter-mapping>
    <filter-name>FilterName</filter-name>
    <servlet-name>ServletName</servlet-name>
</filter-mapping>
```

Щоб зв'язати фільтр зі сторінками HTML або групою сервлетів, використовують тег `<url-pattern>`:

> Підключення фільтра `FilterName` до всіх викликів `.html`-сторінок:

```xml
<filter-mapping>
    <filter-name>FilterName</filter-name>
    <url-pattern>*.html</url-pattern>
</filter-mapping>
```

Порядок, у якому контейнер будує ланцюжок фільтрів для запиту, визначається так:

+ спершу застосовуються фільтри, прив'язані через `<url-pattern>`, у тому порядку, в якому їхні `<filter-mapping>` зустрічаються в `web.xml`;
+ потім — фільтри, прив'язані через `<servlet-name>`, також у порядку появи відповідних елементів у дескрипторі.

## Навіщо в сервлетах використовуються різні _listener_?
__Listener (слухач)__ працює як тригер: виконує певні дії при настанні якоїсь події в життєвому циклі веб-застосунку.

Слухачі, поділені за областю видимості (scope):

+ _Request_:
    + `ServletRequestListener` — ловить момент створення та знищення запиту;
    + `ServletRequestAttributeListener` — прослуховує події з атрибутами запиту.
+ _Context_:
    + `ServletContextListener` — ловить момент ініціалізації або знищення контексту (типове місце для ініціалізації пулів з'єднань, прогріву кешів тощо);
    + `ServletContextAttributeListener` — прослуховує події з атрибутами в контексті.
+ _Session_:
    + `HttpSessionListener` — ловить момент створення та знищення сесії;
    + `HttpSessionAttributeListener` — прослуховує події з атрибутами в сесії;
    + `HttpSessionActivationListener` — спрацьовує під час міграції сесії між різними JVM у розподілених застосунках;
    + `HttpSessionBindingListener` — також прослуховує події з атрибутами в сесії. Різниця між `HttpSessionAttributeListener` і `HttpSessionBindingListener`: перший оголошується в `web.xml`/анотацією, його екземпляр створює контейнер в однині й застосовує до всіх сесій; другий — це сам об'єкт-атрибут, який реалізує інтерфейс і отримує сповіщення, коли його кладуть у сесію чи виймають із неї; екземпляр і його прив'язку до сесії розробник створює «вручну».

Підключення слухачів через `web.xml`:

```xml
<web-app>
    ...
    <listener>
        <listener-class>xyz.company.ExampleListener</listener-class>
    </listener>
    ...
</web-app>
```

> Або, починаючи зі Servlet 3.0, анотацією `@WebListener` над класом слухача — без правок `web.xml`.

`HttpSessionBindingListener` підключається інакше — як атрибут безпосередньо в сесію, тобто щоб його задіяти, потрібно:

+ створити екземпляр класу, що реалізує цей інтерфейс;
+ покласти створений екземпляр у сесію за допомогою `setAttribute(String, Object)`.

## Коли варто використовувати фільтри сервлетів, а коли слухачів?
Фільтри варто застосовувати, коли потрібно обробляти вхідні чи вихідні дані запиту/відповіді (автентифікація, перетворення формату, стиснення, шифрування тощо). Слухачів — коли потрібно реагувати на події життєвого циклу (створення/знищення контексту, сесії, запиту, зміну атрибутів).

## Як реалізувати запуск сервлета одночасно із запуском застосунку?
За замовчуванням контейнер сервлетів завантажує сервлет за першим запитом клієнта (lazy loading).

Якщо потрібно завантажити сервлет одразу на старті застосунку (наприклад, ініціалізація сервлета триває довго), слід використати елемент `<load-on-startup>` у дескрипторі або атрибут `loadOnStartup` анотації `@WebServlet`.

> У вихідному матеріалі згадувалася неіснуюча анотація `@loadOnStartup`. Насправді це атрибут анотації `@WebServlet`, наприклад: `@WebServlet(urlPatterns = "/example", loadOnStartup = 1)`.

Якщо значення параметра від'ємне, сервлет завантажиться за запитом клієнта. Інакше — на старті застосунку, причому чим менше число, тим раніше сервлет потрапить у чергу завантаження.

```xml
<servlet>
    <servlet-name>ExampleServlet</servlet-name>
    <servlet-class>xyz.company.ExampleServlet</servlet-class>
    <load-on-startup>1</load-on-startup>
</servlet>
```

## Як обробити в застосунку винятки, кинуті іншим сервлетом?
Коли застосунок кидає виняток, контейнер сервлетів обробляє його та формує стандартну відповідь (зазвичай HTML-сторінку помилки). Так само поводиться контейнер і для кодів помилок на кшталт 404, 403 тощо.

Окрім цього, можна написати власні сервлети (або JSP-сторінки) для обробки винятків і помилок, указавши їх у дескрипторі розгортання:

```xml
<error-page>
    <error-code>404</error-code>
    <location>/AppExceptionHandler</location>
</error-page>

<error-page>
    <exception-type>javax.servlet.ServletException</exception-type>
    <location>/AppExceptionHandler</location>
</error-page>
```

Основне завдання таких обробників — обробити помилку/виняток і сформувати зрозумілу відповідь користувачу: наприклад, надати посилання на головну сторінку чи опис помилки.

## Що являє собою `ServletConfig`?
Інтерфейс `javax.servlet.ServletConfig` (нині `jakarta.servlet.ServletConfig`) використовується для передавання сервлету конфігураційної інформації. Кожен сервлет має власний екземпляр `ServletConfig`, що його створює контейнер.

Параметри конфігурації задають елементами `init-param` у `web.xml`:

```xml
<servlet>
    <servlet-name>ExampleServlet</servlet-name>
    <servlet-class>xyz.company.ExampleServlet</servlet-class>
    <init-param>
        <param-name>exampleParameter</param-name>
        <param-value>parameterValue</param-value>
    </init-param>
</servlet>
```

або анотацією `@WebInitParam`:

```java
@WebServlet(
    urlPatterns = "/example",
    initParams = {
        @WebInitParam(name = "exampleParameter", value = "parameterValue")
    }
)
public class ExampleServlet extends HttpServlet {
    //...
}
```

Отримати `ServletConfig` сервлета можна методом `getServletConfig()`.

[до змісту](#servlets-jsp-jstl)

## Що являє собою `ServletContext`?
Унікальний (у межах веб-застосунку) об'єкт `ServletContext` реалізує інтерфейс `javax.servlet.ServletContext` (нині `jakarta.servlet.ServletContext`) і надає сервлетам доступ до параметрів усього веб-застосунку. Спільні параметри задають елементом `<context-param>` у `web.xml`:

```xml
<web-app>
    ...
    <context-param>
        <param-name>exampleParameter</param-name>
        <param-value>parameterValue</param-value>
    </context-param>
    ...
</web-app>
```

Об'єкт `ServletContext` можна отримати методом `getServletContext()` (наявний у `ServletConfig` та `GenericServlet`). Починаючи зі Servlet 3.0, `ServletContext` був розширений і тепер дозволяє програмно додавати сервлети, фільтри та слухачів (`addServlet()`, `addFilter()`, `addListener()`). Інтерфейс має й багато інших корисних методів: `getServerInfo()`, `getMimeType()`, `getResourceAsStream()`, `getRealPath()` тощо.

[до змісту](#servlets-jsp-jstl)


## У чому відмінності `ServletContext` і `ServletConfig`?

| | `ServletConfig` | `ServletContext` |
|---|---|---|
| Область | один конкретний сервлет | увесь веб-застосунок (спільний для всіх сервлетів) |
| Призначення | параметри ініціалізації окремого сервлета | параметри й ресурси, спільні для всіх сервлетів |
| Конфігурація | `<init-param>` у `<servlet>` / `@WebInitParam` | `<context-param>` у `<web-app>` |
| Атрибути | змінювати атрибути не можна (тільки read-only параметри) | атрибути можна додавати/змінювати/видаляти у час виконання |

## Для чого потрібен інтерфейс `ServletResponse`?
Інтерфейс `ServletResponse` використовується для надсилання даних клієнту. Усі його методи служать саме цій меті:

+ `String getCharacterEncoding()` — повертає кодування символів (наприклад, `UTF-8`), у якому видаватиметься відповідь;
+ `void setLocale(Locale locale)`/`Locale getLocale()` — задають/повертають локаль, що використовується у відповіді;
+ `ServletOutputStream getOutputStream()`/`PrintWriter getWriter()` — повертають потоки виведення даних (байтовий і символьний відповідно);
+ `void setContentLength(int len)` — встановлює значення HTTP-заголовка _Content-Length_;
+ `void setContentType(String type)` — встановлює значення HTTP-заголовка _Content-Type_;
+ `void reset()` — скидає буфер і заголовки до значень за замовчуванням, якщо відповідь ще не була відправлена клієнту;
+ та інші.

## Що таке `Request Dispatcher`?
Інтерфейс `RequestDispatcher` використовується для передавання запиту іншому ресурсу, при цьому можна додати дані, отримані з цього ресурсу, до власної відповіді сервлета. Також цей інтерфейс служить для внутрішньої комунікації між сервлетами в межах одного контексту.

В інтерфейсі оголошено два методи:

+ `void forward(ServletRequest request, ServletResponse response)` — повністю передає обробку запиту іншому ресурсу (сервлету, JSP чи HTML-файлу) на сервері.
+ `void include(ServletRequest request, ServletResponse response)` — включає вміст ресурсу (сервлета, JSP чи HTML-сторінки) у поточну відповідь.

Отримати `RequestDispatcher` можна методом `ServletContext.getRequestDispatcher(String path)`, де шлях, що починається з `/`, інтерпретується відносно кореня поточного контексту (також доступний `ServletRequest.getRequestDispatcher(String path)`, який дозволяє відносні шляхи).

[до змісту](#servlets-jsp-jstl)

## Як з одного сервлета викликати інший сервлет?
Для виклику сервлета з того самого застосунку використовують механізм внутрішньої комунікації сервлетів (_inter-servlet communication_) через методи `RequestDispatcher`:

+ `forward()` — передає обробку запиту іншому сервлету;
+ `include()` — дозволяє включити результат роботи іншого сервлета у відповідь, що повертається.

Якщо потрібно викликати сервлет з іншого застосунку, `RequestDispatcher` уже не підійде — він визначений лише в межах поточного контексту. Для цього використовують метод `HttpServletResponse.sendRedirect()`, якому передають повний URL іншого сервлета. Для передавання даних між сервлетами можна використати `cookies` або параметри запиту.

[до змісту](#servlets-jsp-jstl)

## Чим відрізняється `sendRedirect()` від `forward()`?
__`forward()`__:

+ Виконується на боці сервера;
+ Запит перенаправляється на інший ресурс у межах того самого сервера (контексту);
+ Не залежить від протоколу клієнтського запиту, оскільки забезпечується контейнером сервлетів;
+ Не можна застосовувати для переходу в інший контекст;
+ Клієнт не знає про фактично оброблюваний ресурс, URL у рядку браузера залишається попереднім;
+ Виконується швидше за `sendRedirect()` (один HTTP-запит);
+ Визначений в інтерфейсі `RequestDispatcher`.

__`sendRedirect()`__:

+ Виконується на боці клієнта;
+ Клієнту повертається відповідь `302 (redirect)`, після чого браузер робить новий запит на вказаний URL (зокрема й на інший сервер);
+ Може використовуватися лише з HTTP-клієнтами;
+ Дозволяє перенаправляти й в інший контекст або на зовнішній ресурс;
+ URL у рядку браузера змінюється на адресу нового ресурсу;
+ Повільніший за `forward()`, бо потребує додаткового кругообігу «запит-відповідь»;
+ Визначений в інтерфейсі `HttpServletResponse`.

[до змісту](#servlets-jsp-jstl)

## Для чого використовуються атрибути сервлетів і як з ними працювати?
Атрибути використовуються для передавання об'єктів між компонентами веб-застосунку (сервлетами, JSP, фільтрами).

Працювати з атрибутами можна методами `setAttribute()`, `getAttribute()`, `removeAttribute()`, `getAttributeNames()`, які надають інтерфейси `ServletRequest`, `HttpSession` і `ServletContext` — для областей видимості _request_, _session_ і _application (context)_ відповідно. Вибір області визначає час життя та видимість атрибута: запит → один запит; сесія → поки живе сесія користувача; контекст → увесь час роботи застосунку, для всіх користувачів.

[до змісту](#servlets-jsp-jstl)

## Яким чином можна допустити в сервлеті deadlock?
Взаємне блокування (deadlock) можна отримати, наприклад, допустивши циклічні виклики: метод `doGet()` викликає `doPost()`, а `doPost()` — `doGet()`, із захопленням блокувань у різному порядку. Узагалі deadlock виникає так само, як у будь-якому багатопотоковому коді: два потоки захоплюють однакові блокування у протилежному порядку.

[до змісту](#servlets-jsp-jstl)

## Як отримати реальне розташування сервлета на сервері?
Реальний (фізичний) шлях до ресурсу на сервері можна отримати з об'єкта `ServletContext`:

`getServletContext().getRealPath(request.getServletPath())`.

> Зауваження: у середовищах, де застосунок розгорнуто не в розпакованому вигляді (наприклад, запущено просто з WAR-архіву), `getRealPath()` може повернути `null` — фізичного файлу на диску немає.

## Як отримати інформацію про сервер із сервлета?
Інформацію про сервер можна отримати з об'єкта `ServletContext`:

`getServletContext().getServerInfo()`.

[до змісту](#servlets-jsp-jstl)

## Як отримати IP-адресу клієнта на сервері?
IP-адресу клієнта можна отримати викликом `request.getRemoteAddr()`.

> На практиці за проксі/балансувальником `getRemoteAddr()` поверне адресу проксі, а не реального клієнта. Реальну адресу тоді шукають у заголовку `X-Forwarded-For` (`request.getHeader("X-Forwarded-For")`).

## Які класи-обгортки для сервлетів ви знаєте?
Власну обробку `ServletRequest` і `ServletResponse` можна реалізувати, додавши нові або перевизначивши наявні методи класів-обгорток `ServletRequestWrapper` (`HttpServletRequestWrapper`) та `ServletResponseWrapper` (`HttpServletResponseWrapper`).

> У вихідному тексті обидві обгортки відповіді були помилково названі `HttpServletRequestWrapper`. Правильна обгортка для відповіді — `HttpServletResponseWrapper`. Це класичний паттерн «декоратор»: наприклад, обгортка відповіді, що перехоплює потік виведення для gzip-стиснення.

[до змісту](#servlets-jsp-jstl)

## У чому відмінності `GenericServlet` і `HttpServlet`?
Абстрактний клас `GenericServlet` — незалежна від протоколу реалізація інтерфейсу `Servlet`. Абстрактний клас `HttpServlet` своєю чергою розширює `GenericServlet` для протоколу HTTP, додаючи метод `service(HttpServletRequest, HttpServletResponse)` та `doGet()`/`doPost()`/... для диспетчеризації за HTTP-методом.

[до змісту](#servlets-jsp-jstl)

## Чому клас `HttpServlet` оголошений як абстрактний?
Клас `HttpServlet` надає лише загальну реалізацію сервлета для HTTP-протоколу. Реалізацію ключових методів `doGet()`, `doPost()` тощо, що містять бізнес-логіку, перекладено на розробника: за замовчуванням вони повертають помилку `HTTP 405 Method Not Allowed`. Сам по собі `HttpServlet` корисної роботи не виконує, тому його оголошено абстрактним, щоб змусити успадкувати й перевизначити потрібні методи.

[до змісту](#servlets-jsp-jstl)

## Які основні методи присутні в класі `HttpServlet`?
+ `doGet()` — для обробки HTTP-запитів `GET`;
+ `doPost()` — для обробки HTTP-запитів `POST`;
+ `doPut()` — для обробки HTTP-запитів `PUT`;
+ `doDelete()` — для обробки HTTP-запитів `DELETE`;
+ `doHead()` — для обробки HTTP-запитів `HEAD`;
+ `doOptions()` — для обробки HTTP-запитів `OPTIONS`;
+ `doTrace()` — для обробки HTTP-запитів `TRACE`.

[до змісту](#servlets-jsp-jstl)

## Чи варто хвилюватися про багатопотокову безпеку, працюючи із сервлетами?
Методи `init()` і `destroy()` викликаються один раз за життєвий цикл сервлета — щодо них турбуватися не варто.

А ось методи `doGet()`, `doPost()`, `service()` викликаються на кожен запит клієнта, і оскільки контейнер обробляє запити багатопотоково (один екземпляр сервлета, багато потоків), про потокобезпеку думати обов'язково. Правила ті самі, що й у звичайному багатопотоковому коді: локальні змінні методів створюються окремо для кожного потоку й безпечні, а при роботі зі спільним станом — полями сервлета чи зовнішніми ресурсами — потрібна синхронізація або інші прийоми багатопотокового програмування. Загальна порада: тримайте сервлети stateless, не зберігайте стан запиту в полях екземпляра.

[до змісту](#servlets-jsp-jstl)

## Який метод HTTP не є ідемпотентним?
HTTP-метод називається __ідемпотентним__, якщо багаторазове повторення однакового запиту дає той самий ефект на сервері, що й одноразове (а не «повертає однаковий результат» — це різні поняття; ідемпотентність стосується ефекту, а не тіла відповіді). Ідемпотентними є `GET`, `PUT`, `DELETE`, `HEAD`, `OPTIONS`, `TRACE`. Неідемпотентний метод — `POST`: кожен його виклик може створювати новий ресурс або змінювати стан щоразу.

> Окремо варто розрізняти _безпечні_ (safe) методи, які взагалі не змінюють стан сервера: `GET`, `HEAD`, `OPTIONS`, `TRACE`. У вихідному тексті використовувався неточний термін «незмінюваний»; коректний термін — «ідемпотентний».

Наприклад, для доступу до статичної HTML-сторінки використовують `GET` (читання, без побічних ефектів). Коли потрібно зберегти нову інформацію (скажімо, у базі даних) — використовують `POST`.

[до змісту](#servlets-jsp-jstl)

## Які є методи відправлення даних із клієнта на сервер?
+ `GET` — використовується для запиту вмісту вказаного ресурсу (зображення, гіпертекстового документа тощо). Разом із запитом можуть передаватися параметри як частина URI; значення можуть братися з полів форми або задаватися напряму в URL. Запити можуть кешуватися й мають обмеження на розмір. Це основний метод взаємодії браузера й вебсервера.
+ `POST` — використовується для передавання даних користувача в тілі HTTP-запиту. Дані упаковані в тіло запиту згідно з полем заголовка `Content-Type`. При використанні `POST` під URI розуміється ресурс, який оброблятиме запит.

> Існують і інші методи передавання даних на сервер — `PUT` (створити/замінити ресурс) і `PATCH` (часткове оновлення), які активно використовуються в REST API.

[до змісту](#servlets-jsp-jstl)

## У чому різниця між методами `GET` і `POST`?

| | `GET` | `POST` |
|---|---|---|
| Де передаються дані | у рядку URL (query string) | у тілі HTTP-запиту |
| Обмеження на обсяг | так, через ліміт довжини URL | практично немає (обмежується налаштуваннями сервера) |
| Видимість даних | видно в URL, історії, логах | у тілі, не видно в URL |
| Кешування / закладки | так | ні |
| Ідемпотентність | ідемпотентний і безпечний | неідемпотентний |
| Призначення | читання даних | зміна стану, надсилання даних |

> Уточнення щодо вихідного тексту: твердження про «ліміт URL у 1024 символи» і «ліміт POST ~2 Mb» — некоректні. Стандарт HTTP не задає жорсткого ліміту довжини URL; на практиці браузери та сервери обмежують її приблизно 8 КБ (часто ~2000 символів сумарно). Максимальний розмір тіла `POST` цілком залежить від налаштувань сервера/застосунку.

«Безпечність» `POST` відносна: дані не показуються в адресному рядку, але без HTTPS вони так само передаються відкритим текстом. Справжню конфіденційність дає лише TLS (HTTPS), а не вибір методу.

[до змісту](#servlets-jsp-jstl)

## У чому різниця між `PrintWriter` і `ServletOutputStream`?
`PrintWriter` — клас для роботи із символьним потоком (текст); його екземпляр отримують методом `ServletResponse.getWriter()`.

`ServletOutputStream` — клас для роботи з байтовим потоком (бінарні дані: зображення, PDF тощо); його екземпляр отримують методом `ServletResponse.getOutputStream()`.

[до змісту](#servlets-jsp-jstl)

## Чи можна одночасно використовувати в сервлеті `PrintWriter` і `ServletOutputStream`?
Ні. При спробі викликати в межах однієї відповіді і `getWriter()`, і `getOutputStream()` буде кинуто `java.lang.IllegalStateException` із повідомленням, що інший метод уже був викликаний.

[до змісту](#servlets-jsp-jstl)

## Розкажіть про інтерфейс `SingleThreadModel`.
`SingleThreadModel` — маркерний інтерфейс (не оголошує жодного методу). Якщо сервлет його реалізує, контейнер гарантує, що метод `service()` цього сервлета не виконуватиметься одночасно у двох потоках: контейнер або синхронізує доступ до єдиного екземпляра, або підтримує пул екземплярів і перенаправляє запит вільному.

Інакше кажучи, контейнер усуває конфлікти при одночасному доступі до полів та методів екземпляра. Проте інші спільні ресурси (статичні поля, об'єкти в сесії та контексті, зовнішні з'єднання) лишаються доступними іншим потокам, тож реальної потокобезпеки інтерфейс не дає. Користі від нього мало, тому ще у Servlet 2.4 його оголошено `deprecated`, а у Servlet 6.0 (Jakarta EE 10) — узагалі вилучено. Використовувати його не слід.

[до змісту](#servlets-jsp-jstl)

## Що означає _URL encoding_? Як це здійснити в Java?
__URL Encoding__ — процес перетворення даних у форму, придатну для передавання в URL: символи, які не можна вживати безпосередньо (пробіли, нелатинські символи, спецсимволи), замінюються escape-послідовностями виду `%XX`. У Java для кодування рядка використовують `java.net.URLEncoder.encode(String s, String charset)`, для зворотної операції — `java.net.URLDecoder.decode(String s, String charset)`. Як кодування слід передавати `"UTF-8"`.

> `Hello мир!` перетворюється на `Hello+%D0%BC%D0%B8%D1%80%21` (пробіл `URLEncoder` кодує як `+`; для кодування шляху, де пробіл має бути `%20`, поведінка відрізняється).

[до змісту](#servlets-jsp-jstl)

## Які різні методи керування сесією в сервлетах ви знаєте?
HTTP — протокол без стану (stateless): сам по собі він не зберігає контекст між запитами одного клієнта. __Сесія (сеанс)__ — це логічне з'єднання між клієнтом і сервером на певний час, протягом якого клієнт може зробити скільки завгодно запитів, а сервер «пам'ятає», що це той самий клієнт. Сесії потрібні, щоб зберігати дані між кількома запитами (кошик в інтернет-магазині, проходження кількох кроків форми, відповіді на тести тощо).

Способи забезпечити унікальний ідентифікатор сесії:

+ __User Authentication__ — користувач надає облікові дані під час автентифікації, які далі використовуються для підтримання сеансу. Не працює коректно, якщо користувач увійшов одночасно з кількох місць.
+ __HTML Hidden Field__ — приховане поле HTML-форми з унікальним значенням. Не годиться для звичайних посилань (потребує надсилання форми щоразу) і небезпечне — ідентифікатор легко підмінити.
+ __URL Rewriting__ — додавання ідентифікатора сеансу як параметра URL. Доволі марудно, бо потребує відстеження ідентифікатора в кожному запиті/відповіді.
+ __Cookies__ — невеликі фрагменти даних, що їх надсилає сервер і зберігає браузер. Не працює, якщо клієнт вимкнув cookies.
+ __Session Management API__ — спеціальний API контейнера, побудований поверх перелічених механізмів і такий, що розв'язує їхні часткові проблеми:
    + крім самого відстеження сесії, часто потрібно зберігати додаткові дані про неї для наступних запитів — це багато ручної роботи;
    + жоден із попередніх механізмів не універсальний, для кожного знайдеться сценарій, де він не працює.

> На практиці майже завжди використовують саме Session Management API контейнера (`HttpSession`), а той «під капотом» спирається на cookie з `JSESSIONID`, із запасним варіантом URL rewriting.

[до змісту](#servlets-jsp-jstl)

## Що таке _cookies_?
__Cookies («куки»)__ — невеликий фрагмент даних, який надсилає вебсервер і зберігає браузер на пристрої користувача. Щоразу при зверненні до сайту браузер надсилає відповідні цьому сайту cookies серверу у складі HTTP-запиту. Cookies застосовують для зберігання даних на боці клієнта, зокрема для:

+ автентифікації користувача (ідентифікатор сесії);
+ зберігання персональних уподобань і налаштувань;
+ відстеження стану сеансу;
+ ведення статистики.

> Для безпеки сесійні cookies варто позначати прапорцями `HttpOnly` (недоступні з JavaScript), `Secure` (лише через HTTPS) і `SameSite` (захист від CSRF).

[до змісту](#servlets-jsp-jstl)

## Які методи для роботи з cookies передбачені в сервлетах?
Servlet API підтримує cookies через клас `javax.servlet.http.Cookie` (нині `jakarta.servlet.http.Cookie`):

+ Щоб отримати масив cookies із запиту, використовують `HttpServletRequest.getCookies()`. Методу для додавання cookies у `HttpServletRequest` не передбачено.
+ Щоб додати cookie до відповіді, використовують `HttpServletResponse.addCookie(Cookie c)`. Методу для отримання cookies з `HttpServletResponse` немає.

[до змісту](#servlets-jsp-jstl)

## Що таке _URL Rewriting_?
__URL Rewriting__ — додавання ідентифікатора сесії (`jsessionid`) безпосередньо до URL. Цей механізм може використовуватися для керування сесією, коли в клієнта вимкнено _cookies_.

[до змісту](#servlets-jsp-jstl)

## Навіщо потрібні та чим відрізняються методи `encodeURL()` і `encodeRedirectURL()`?
Обидва методи `HttpServletResponse` додають до URL ідентифікатор сесії (`jsessionid`), якщо клієнт не підтримує cookies — це і є реалізація URL rewriting.

`encodeURL()` використовують для звичайних URL усередині сторінки (гіперпосилання, форми).

`encodeRedirectURL()` використовують для URL, який передаватиметься в `sendRedirect()` — він застосовує дещо інші правила кодування, специфічні для перенаправлення.

Отже: для посилань на сторінці — `encodeURL()`, для перенаправлень — `encodeRedirectURL()`.

> Уточнення щодо вихідного тексту: ці методи додають `jsessionid`, але _не_ виконують перекодування спецсимволів і пробілів на кшталт `java.net.URLEncoder.encode()` — це різні речі. Якщо клієнт підтримує cookies, методи повертають URL без змін.

[до змісту](#servlets-jsp-jstl)

## Що таке _«сесія»_?
__Сесія__ — це сеанс зв'язку між клієнтом і сервером, що встановлюється на певний час. Сеанс установлюється безпосередньо між клієнтом і вебсервером у момент отримання першого запиту до веб-застосунку (точніше — при першому виклику `request.getSession()`). Кожен клієнт має власний сеанс, який зберігається до завершення роботи із застосунком (або до тайм-ауту/інвалідації).

[до змісту](#servlets-jsp-jstl)

## Як сповістити об'єкт у сесії, що сесія недійсна або завершилася?
Щоб об'єкт отримував сповіщення про прив'язку до сесії та відв'язку від неї, він має реалізувати інтерфейс `javax.servlet.http.HttpSessionBindingListener` (нині `jakarta.servlet.http.HttpSessionBindingListener`). Два його методи — `valueBound()` і `valueUnbound()` — викликаються відповідно при додаванні об'єкта як атрибута до сесії та при його видаленні (зокрема при інвалідації чи завершенні сесії).

[до змісту](#servlets-jsp-jstl)

## Який існує ефективний спосіб упевнитися, що всі сервлети доступні лише для користувача з дійсною сесією?
Фільтри сервлетів перехоплюють усі запити між контейнером і сервлетом, тому логічно реалізувати один фільтр, що перевіряє наявність і валідність сесії в кожному запиті, і за потреби перенаправляє неавтентифікованого користувача на сторінку входу. Це централізоване рішення, що не потребує дублювання перевірки в кожному сервлеті.

[до змісту](#servlets-jsp-jstl)

## Як ми можемо забезпечити _transport layer security_ для нашого веб-застосунку?
Щоб забезпечити _transport layer security_, потрібно налаштувати підтримку TLS/SSL (тобто HTTPS) на боці контейнера сервлетів. Конкретні кроки залежать від реалізації контейнера. На практиці TLS часто термінують на рівні зворотного проксі чи балансувальника (nginx, ALB) перед контейнером.

[до змісту](#servlets-jsp-jstl)

## Як організувати підключення до бази даних, забезпечити журналювання в сервлеті?
При роботі з великою кількістю підключень до бази даних їх рекомендують ініціалізувати в _ServletContextListener_ (на старті застосунку) і зберегти як атрибут контексту, щоб ними могли користуватися інші сервлети. На практиці краще не керувати з'єднаннями вручну, а використати пул з'єднань (HikariCP, або `DataSource` із JNDI, наданий контейнером).

Журналювання підключають стандартним для логера способом (наприклад, для SLF4J/Logback чи Log4j 2 — конфігураційний файл), а ініціалізацію за потреби виконують у тому самому _context listener_.

[до змісту](#servlets-jsp-jstl)

## Які основні особливості з'явилися у специфікації _Servlet 3_?
+ __Анотації сервлетів.__ До Servlet 3 уся конфігурація містилася у `web.xml`, що було незручно й помилконебезпечно. З'явилися анотації `@WebServlet`, `@WebInitParam`, `@WebFilter`, `@WebListener`, які зробили `web.xml` необов'язковим.
+ __Web Fragments.__ Веб-застосунок може складатися з багатьох модулів: кожен модуль описує свій фрагмент конфігурації у `web-fragment.xml` у каталозі `META-INF/` свого JAR. Це дозволяє розбивати застосунок на окремі модулі, що постачаються як `.jar`-файли в каталозі `WEB-INF/lib/`.
+ __Динамічне додавання вебкомпонентів.__ З'явилася можливість програмно реєструвати сервлети, фільтри та слухачів через `ServletContext` (`addServlet()`, `addFilter()`, `addListener()`). Це дозволяє будувати динамічні системи, де компонент створюється й реєструється лише за потреби.
+ __Асинхронна обробка.__ Підтримка асинхронної обробки (`AsyncContext`, `request.startAsync()`) дозволяє звільнити потік-обробник запиту, передавши тривалу роботу іншому потоку, не утримуючи зайнятим сервер.

[до змісту](#servlets-jsp-jstl)

## Які способи автентифікації доступні сервлету?
Специфікація сервлетів визначає чотири типи перевірки автентичності:

+ __HTTP Basic Authentication__ — `BASIC`. При доступі до закритих ресурсів браузер показує вікно з проханням ввести облікові дані. Дані передаються лише закодованими в Base64 (не зашифрованими!), тому безпечні тільки поверх HTTPS.
+ __Form Based Login__ — `FORM`. Використовується власна HTML-форма входу (приклад нижче).
+ __HTTP Digest Authentication__ — `DIGEST`. Облікові дані передаються у вигляді хешу, а не відкритим текстом; на практиці майже не вживається.
+ __HTTPS Client Authentication__ — `CLIENT-CERT`. Автентифікація за клієнтським X.509-сертифікатом.

```xml
<login-config>
    <auth-method>FORM</auth-method>
    <form-login-config>
        <form-login-page>/login.html</form-login-page>
        <form-error-page>/error.html</form-error-page>
    </form-login-config>
</login-config>
```

## Що таке _Java Server Pages (JSP)_?
__JSP (JavaServer Pages)__ — платформо-незалежна, переносна й легко розширювана технологія розробки веб-застосунків, що дозволяє створювати вміст зі статичними та динамічними компонентами. JSP-сторінка містить текст двох типів: статичні дані (HTML, SVG, WML або XML) та _JSP-елементи_, що формують динамічний вміст. Додатково можуть використовуватися _бібліотеки JSP-тегів_ і _EL (Expression Language)_.

Код JSP-сторінки транслюється у Java-код сервлета (у Tomcat — компілятором _Jasper_), а потім компілюється у байт-код JVM.

JSP-сторінки виконуються на сервері під керуванням веб-застосунку Jakarta EE (Java EE). Зазвичай вони упаковані в архіви `.war` або `.ear`.

> Контекст: JSP — це класична, але вже застаріла технологія представлення. У сучасних проєктах її здебільшого витіснили шаблонізатори на кшталт Thymeleaf, а також SPA-фронтенди (React/Angular/Vue) поверх REST/GraphQL API. Знати JSP корисно для співбесід і підтримки legacy-систем.

[до змісту](#servlets-jsp-jstl)

## Навіщо потрібен JSP?
JSP розширює технологію сервлетів, даючи змогу створювати динамічні сторінки із HTML-подібним синтаксисом.

Хоча генерувати представлення можна й у сервлеті, більша частина веб-сторінки статична, тож код сервлета, що вручну «склеює» HTML-рядки, виходить перевантаженим, нечитабельним і помилконебезпечним. JSP дозволяє писати переважно звичайний HTML, вкраплюючи динаміку лише там, де вона потрібна.

Ще одна перевага JSP — гаряче розгортання: можна замінити одну сторінку іншою прямо в контейнері без перекомпіляції всього проєкту чи перезапуску сервера.

Утім, серйозну бізнес-логіку в JSP писати не варто — сторінка має слугувати лише представленням (view).

[до змісту](#servlets-jsp-jstl)

## Опишіть, як обробляються JSP-сторінки, починаючи від запиту до сервера й закінчуючи відповіддю користувачу.
Коли користувач переходить за посиланням на сторінку `page.jsp`, браузер надсилає HTTP-запит `GET /page.jsp`. На основі цього запиту й тексту самої сторінки сервер (точніше — контейнер) під час першого звернення генерує Java-клас сервлета, компілює його та виконує отриманий сервлет, який формує HTML-відповідь. Цю відповідь сервер і надсилає назад користувачу. За наступних запитів уже скомпонований сервлет використовується повторно (без повторної трансляції/компіляції), якщо сторінка не змінювалася.

## Розкажіть про етапи (фази) життєвого циклу JSP.
Якщо подивитися на код усередині JSP-сторінки, він виглядає як HTML, а не як Java-клас. Перетворенням JSP на Java-сервлет (а потім — на HTML-відповідь) займається контейнер.

Життєвий цикл JSP складається з кількох фаз, якими керує JSP-контейнер:

+ __Translation__ — перевірка й парсинг коду JSP-сторінки для генерації коду сервлета.
+ __Compilation__ — компіляція вихідного коду сервлета у байт-код.
+ __Class Loading__ — завантаження скомпільованого класу в пам'ять.
+ __Instantiation__ — виклик конструктора без параметрів завантаженого класу для створення екземпляра.
+ __Initialization__ — виклик методу `jspInit()` об'єкта JSP-класу та ініціалізація конфігурації параметрами з дескриптора розгортання (`web.xml`). Після цієї фази JSP здатна обробляти запити клієнтів. Зазвичай ці фази відбуваються після першого запиту (тобто лінива ініціалізація), але JSP можна налаштувати на завантаження й ініціалізацію на старті застосунку — за аналогією із сервлетами.
+ __Request Processing__ — основна фаза обробки запитів клієнтів JSP-сторінкою. Обробка багатопотокова й аналогічна сервлетам: на кожен запит виділяється потік, створюються об'єкти запиту й відповіді, виконується метод `_jspService()`.
+ __Destroy__ — остання фаза, на якій клас JSP видаляється з пам'яті (виклик `jspDestroy()`). Зазвичай це відбувається при зупинці сервера чи вивантаженні застосунку.

[до змісту](#servlets-jsp-jstl)

## Розкажіть про методи життєвого циклу JSP.
Контейнер сервлетів (наприклад, Tomcat, GlassFish) генерує з JSP-сторінки клас сервлета, що успадковує `javax.servlet.jsp.HttpJspBase` (нині `jakarta.servlet.jsp.HttpJspBase`) і має такі методи:

+ `jspInit()` — викликається один раз за життєвий цикл JSP для ініціалізації конфігураційних параметрів із дескриптора розгортання. Метод можна перевизначити в _JSP declaration_ (`<%! ... %>`);
+ `_jspService()` — метод, що генерується контейнером автоматично й відповідає безпосередньо коду JSP-сторінки. Оголошений в інтерфейсі `HttpJspPage`; його ім'я починається з підкреслення, і він відрізняється від інших методів життєвого циклу тим, що його НЕ можна перевизначати;
+ `jspDestroy()` — викликається контейнером для видалення об'єкта з пам'яті (на фазі Destroy). Викликається один раз і доступний для перевизначення, даючи змогу звільнити ресурси, створені в `jspInit()`.

[до змісту](#servlets-jsp-jstl)

## Які методи життєвого циклу JSP можна перевизначити?
Перевизначити можна лише `jspInit()` і `jspDestroy()`. Метод `_jspService()` перевизначати не можна — його повністю генерує контейнер.

[до змісту](#servlets-jsp-jstl)

## Як можна запобігти прямому доступу до JSP-сторінки з браузера?
До каталогу `/WEB-INF/` немає прямого доступу ззовні. Тому JSP-сторінки можна розмістити всередині нього й тим самим заборонити прямий доступ із браузера. Доступ до них тоді можливий лише через сервлет (контролер), що робить `forward`. Щоб сторінку у `WEB-INF` можна було адресувати, її потрібно описати в дескрипторі розгортання:

```xml
<servlet>
    <servlet-name>Example</servlet-name>
    <jsp-file>/WEB-INF/example.jsp</jsp-file>
    <init-param>
        <param-name>exampleParameter</param-name>
        <param-value>parameterValue</param-value>
    </init-param>
</servlet>

<servlet-mapping>
    <servlet-name>Example</servlet-name>
    <url-pattern>/example.jsp</url-pattern>
</servlet-mapping>
```

## Яка різниця між _динамічним_ і _статичним_ вмістом JSP?
Статичний вміст JSP (HTML, CSS, JavaScript, зображення тощо) не змінюється під час роботи застосунку — він однаковий для всіх користувачів і запитів.

Динамічний вміст формується залежно від запиту й дій користувача. Зазвичай його створюють за допомогою виразів EL (Expression Language), бібліотек JSP-тегів (JSTL) тощо.

## Як закоментувати код у JSP?
+ `<!-- HTML-коментар -->` — потрапляє у згенерований HTML і видимий клієнту в коді сторінки.
+ `<%-- JSP-коментар --%>` — обробляється на фазі трансляції й НЕ потрапляє у відповідь клієнту. Саме такий тип слід використовувати для будь-яких коментарів у коді та налагоджувальної інформації.

[до змісту](#servlets-jsp-jstl)

## Які існують основні типи тегів JSP?
+ _Вираз JSP (Expression)_: `<%= expression %>` — вираз, результат якого виводиться у відповідь;
+ _Скриплет JSP (Scriptlet)_: `<% code %>` — код, що додається в метод `_jspService()`;
+ _Оголошення JSP (Declaration)_: `<%! code %>` — код (поля, методи), що додається в тіло класу сервлета поза методом `_jspService()`;
+ _Директива page_: `<%@ page att="value" %>` — директиви для контейнера з інформацією про параметри сторінки;
+ _Директива include_: `<%@ include file="url" %>` — файл, що підключається на фазі трансляції JSP у сервлет (статичне включення);
+ _Коментар JSP_: `<%-- comment --%>` — коментар, що ігнорується під час трансляції.

[до змісту](#servlets-jsp-jstl)

## Що ви знаєте про дії JSP (_Action tag_ і _JSP Action Elements_)?
__Action tag__ та __JSP Action Elements__ надають способи роботи з JavaBeans, підключення ресурсів, перенаправлення запитів і створення динамічних XML-елементів. Такі елементи завжди починаються з `jsp:` і використовуються безпосередньо на JSP-сторінці без підключення сторонніх бібліотек чи додаткових налаштувань.

Найчастіше вживані JSP Action Elements:

+ `jsp:include`: `<jsp:include page="відносний URL" flush="true"/>` — підключити файл під час обробки запиту (динамічне включення). Якщо файл потрібно підключити на фазі трансляції — використовують директиву `<%@ include %>`;
+ `jsp:useBean`: `<jsp:useBean .../>` або `<jsp:useBean ...>...</jsp:useBean>` — знайти або створити JavaBean;
+ `jsp:setProperty`: `<jsp:setProperty .../>` — встановити властивості JavaBean (явно або зі значення параметра запиту);
+ `jsp:forward`: `<jsp:forward page="відносний URL"/>` — передати запит іншій сторінці;
+ `jsp:plugin`: `<jsp:plugin ...>...</jsp:plugin>` — згенерувати теги `OBJECT`/`EMBED` для аплета на технології Java Plugin. (Застаріле: Java-аплети й Java Plugin вилучено з браузерів та JDK ще близько 2017–2019 рр.; на практиці не використовується.)

[до змісту](#servlets-jsp-jstl)

+ _Model_ — класи даних і бізнес-логіки;
+ _View_ — JSP-сторінки;
+ _Controller_ — сервлети.

## Які області видимості змінних існують у JSP?
Область видимості об'єкта визначається контекстом, у який його поміщено; від неї ж залежить і час життя об'єкта.

У JSP передбачено такі області видимості (scope):

+ `page` — об'єкт доступний лише на тій сторінці, де його визначено. На включених (`jsp:include`) і переадресованих (`jsp:forward`) сторінках він уже недоступний. Це область за замовчуванням;
+ `request` — об'єкт доступний на поточній сторінці, а також на сторінці, якій передано запит через `jsp:forward`, або на включеній через `jsp:include`;
+ `session` — об'єкт прив'язується до сесії користувача, доступний на всіх JSP-сторінках і живе, поки існує сесія (або поки його примусово не видалять);
+ `application` — об'єкт доступний усім користувачам на всіх сторінках і живе протягом усієї роботи застосунку (або до примусового видалення з контексту).

Отже, щоб об'єкт був доступний усім сторінкам, обирають `application` (для всіх користувачів) або `session` (лише для поточного).

Область задають атрибутом `scope` тега `jsp:useBean`:

```
<jsp:useBean id="myBean" class="com.example.MyBean" scope="session"/>
```

Якщо атрибут `scope` не вказано, за замовчуванням використовується `page`.

## Які неявні, внутрішні об'єкти та методи є на JSP-сторінці?
__JSP implicit objects (неявні об'єкти)__ створюються контейнером під час трансляції JSP у сервлет, щоб полегшити роботу розробникам. Їх можна використовувати напряму у скриплетах для передавання інформації в сервісні методи, але не можна — у JSP Declaration (`<%! %>`), бо той код виноситься на рівень класу.

Існує 9 неявних об'єктів, доступних прямо на JSP-сторінці. Сім із них оголошені як локальні змінні на початку методу `_jspService()`, а два (`request`, `response`) є його аргументами.

+ `out` (`JspWriter`):

```
<strong>Current Time is</strong>: <% out.print(new Date()); %><br>
```

+ `request` (`HttpServletRequest`):

```
<strong>Request User-Agent</strong>: <%= request.getHeader("User-Agent") %><br>
```

+ `response` (`HttpServletResponse`):

```
<strong>Response</strong>: <% response.addCookie(new Cookie("Test","Value")); %>
```

+ `config` (`ServletConfig`):

```
<strong>User init param value</strong>: <%= config.getInitParameter("User") %><br>
```

+ `application` (`ServletContext`):

```
<strong>User context param value</strong>: <%= application.getInitParameter("User") %><br>
```

+ `session` (`HttpSession`):

```
<strong>User Session ID</strong>: <%= session.getId() %><br>
```

+ `pageContext` (`PageContext`):

```
<% pageContext.setAttribute("Test", "Test Value"); %>
<strong>PageContext attribute</strong>: {Name="Test",Value="<%= pageContext.getAttribute("Test") %>"}<br>
```

+ `page` (`Object`, посилання на сам екземпляр сервлета сторінки):

```
<strong>Generated Servlet Name</strong>: <%= page.getClass().getName() %>
```

+ `exception` (`Throwable`, доступний лише на сторінках помилок):

```
<strong>Exception occured</strong>: <%= exception %><br>
```

## Які неявні об'єкти не доступні у звичайній JSP-сторінці?
Неявний об'єкт `exception` недоступний у звичайних JSP-сторінках. Він доступний лише на сторінках помилок (тих, де директива `page` має атрибут `isErrorPage="true"`) і служить для перехоплення винятку, кинутого іншою JSP-сторінкою, щоб показати клієнту корисну інформацію.

## Що ви знаєте про `PageContext` і які переваги його використання?
Неявний об'єкт `pageContext` — екземпляр класу `javax.servlet.jsp.PageContext` (нині `jakarta.servlet.jsp.PageContext`) — надає доступ до всіх просторів імен (областей видимості), асоційованих із JSP-сторінкою, та до її різних атрибутів. Решта неявних об'єктів додаються до `pageContext` автоматично.

`PageContext` — абстрактний клас; його екземпляр контейнер отримує через `JspFactory.getPageContext()` і звільняє через `JspFactory.releasePageContext()`.

Можливості `PageContext`:

+ єдиний API для обслуговування просторів імен різних областей видимості;
+ зручні методи для доступу до різних `public`-об'єктів;
+ механізм отримання `JspWriter` для виведення;
+ керування використанням сесії сторінкою;
+ експонування атрибутів директиви `page` середовищу скриптингу;
+ механізми перенаправлення або включення поточного запиту в інші компоненти застосунку;
+ механізм обробки винятків на сторінках помилок.

## Як сконфігурувати параметри ініціалізації для JSP?
Параметри ініціалізації для JSP задають у `web.xml` так само, як для сервлетів — елементами `<servlet>` і `<servlet-mapping>`. Єдина відмінність — замість `<servlet-class>` указують розташування JSP-сторінки через `<jsp-file>`:

```xml
<servlet>
    <servlet-name>Example</servlet-name>
    <jsp-file>/WEB-INF/example.jsp</jsp-file>
    <init-param>
        <param-name>exampleParameter</param-name>
        <param-value>parameterValue</param-value>
    </init-param>
</servlet>
```

## Чому не рекомендується використовувати скриплети (скриптові елементи) у JSP?
JSP-сторінки призначені насамперед для відображення представлення (_view_), а бізнес-логіка (_controller_) і модель (_model_) мають жити в сервлетах чи класах-моделях. Завдання JSP — формувати HTML-відповідь із даних, переданих через атрибути. Більшість JSP — це HTML, і щоб верстальникам було легше з нею працювати, є елементи _action_, _JSP EL_ і _JSTL_. Саме їх і слід використовувати замість скриплетів як «міст» між HTML- та Java-частинами: скриплети змішують логіку з розміткою, ускладнюють тестування й підтримку.

[до змісту](#servlets-jsp-jstl)

## Чи можна визначити клас усередині JSP-сторінки?
Визначити клас усередині JSP-сторінки можна, але це вважається поганою практикою:

```java
<%!
// у блоці оголошення (declaration) — стає вкладеним класом згенерованого сервлета
private static class ExampleOne {
    //...
}
%>

<%
// у скриплеті — стає локальним класом усередині методу _jspService()
class ExampleTwo {
    //...
}
%>
```

> У вихідному тексті локальний клас у скриплеті був помилково оголошений із модифікатором `private`. Локальні класи (всередині методу) не можуть мати модифікаторів доступу — це не скомпілюється. Виправлено.

[до змісту](#servlets-jsp-jstl)

## Що ви знаєте про Мову виразів JSP (JSP Expression Language – EL)?
__JSP Expression Language (EL)__ — скриптова мова виразів, що дає доступ до Java-компонентів (JavaBeans) із JSP. Починаючи з JSP 2.0, EL використовують усередині JSP-тегів, щоб відокремити Java-код від JSP, спростити доступ до компонентів і максимально зменшити (або зовсім прибрати) Java-код на сторінках.

EL розроблено, щоб дизайнерам із мінімальними знаннями Java було простіше: достатньо знати, як викликати потрібні методи. До появи EL у JSP були спеціальні теги (скриплети, вирази тощо), що дозволяли писати Java-код прямо на сторінці.

EL дозволяє:

+ доступ до властивостей бінів і елементів колекцій/мап;
+ обчислення арифметичних, логічних і реляційних виразів;
+ спрощене звернення до неявних об'єктів JSP;
+ виклик власних функцій (через бібліотеки тегів).

> Уточнення: EL — мова _виразів_, а не повноцінна мова програмування. Твердження вихідного тексту про «створення та зміну змінних» і «керування потоком виконання, ветвлення, ітерації» некоректне — цього в EL немає (цикли й розгалуження роблять тегами JSTL, як-от `<c:if>`, `<c:forEach>`). Виправлено.

Вирази EL записують у конструкції `${ ... }`. Починаючи з JSP 2.1, є також «відкладене» обчислення у формі `#{ ... }` (використовується разом із JSF). Конструкція може стояти окремо або в правій частині присвоєння атрибута тега.

[до змісту](#servlets-jsp-jstl)

## Які типи операторів EL ви знаєте?
Оператори EL покривають найпоширеніші маніпуляції даними:

+ Реляційні: `==` (або `eq`), `!=` (або `ne`), `<` (або `lt`), `>` (або `gt`), `<=` (або `le`), `>=` (або `ge`).
+ Арифметичні: `+`, `-`, `*`, `/` (або `div`), `%` (або `mod`).
+ Логічні: `&&` (або `and`), `||` (або `or`), `!` (або `not`).
+ Оператор `empty` — перевіряє значення на `null` або «порожнечу» залежно від типу: порожній рядок, порожня колекція/мапа чи масив.

> У вихідному тексті альтернативу до `!=` було вказано як `neq` — насправді в EL це `ne`. Виправлено.

[до змісту](#servlets-jsp-jstl)

## Назвіть неявні, внутрішні об'єкти JSP EL та їхні відмінності від об'єктів JSP.
EL надає власні неявні об'єкти для доступу до атрибутів у різних областях видимості та до параметрів запиту. Важливо: вони відрізняються від неявних об'єктів JSP-скриплетів. Єдиний спільний для EL і JSP-сторінки об'єкт — це `pageContext`.

Основні неявні об'єкти EL:

| Об'єкт | Призначення |
|---|---|
| `pageScope` | мапа атрибутів області `page` |
| `requestScope` | мапа атрибутів області `request` |
| `sessionScope` | мапа атрибутів області `session` |
| `applicationScope` | мапа атрибутів області `application` |
| `param` | значення параметрів запиту (перше значення) |
| `paramValues` | масиви значень параметрів запиту |
| `header` | значення HTTP-заголовків запиту |
| `headerValues` | масиви значень HTTP-заголовків |
| `cookie` | мапа cookies запиту |
| `initParam` | контекстні параметри ініціалізації (`context-param`) |
| `pageContext` | об'єкт `PageContext` поточної сторінки (єдиний спільний із JSP) |

> Примітка: у вихідному тексті була згадка про таблицю, але самої таблиці не наведено. Тут її додано.

[до змісту](#servlets-jsp-jstl)

## Як вимкнути можливість використання EL у JSP?
Вимкнути обробку EL на сторінці можна двома способами:

+ директивою на сторінці: `<%@ page isELIgnored="true" %>`;
+ через `web.xml` (зручніше для вимкнення EL одразу на групі сторінок):

```xml
<jsp-config>
    <jsp-property-group>
        <url-pattern>*.jsp</url-pattern>
        <el-ignored>true</el-ignored>
    </jsp-property-group>
</jsp-config>
```

> У вихідному тексті директива містила «українські лапки» (`«true»`). У реальному коді мають бути звичайні подвійні лапки: `isELIgnored="true"`. Виправлено.

## Що таке _JSTL (JSP Standard Tag Library)_?
__JSTL (JavaServer Pages Standard Tag Library, Стандартна бібліотека тегів JSP)__ — розширення специфікації JSP (результат _JSR 52_), що додає бібліотеку JSP-тегів для типових потреб: розгалуження й цикли, обробка винятків, робота з URL, розбір XML, форматування та інтернаціоналізація.

JSTL є альтернативою скриплетам (прямим вставкам Java-коду). Стандартизований набір тегів кращий, бо отриманий код легше підтримувати й простіше відділяти бізнес-логіку від логіки відображення.

Щоб скористатися тегами JSTL, потрібно:

+ підключити залежність, наприклад у `pom.xml`:

```xml
<!-- Для Jakarta EE 9+ (простір імен jakarta.*) використовуйте координати jakarta.servlet.jsp.jstl.
     Нижче — класичний варіант для javax.* (Java EE 8 і раніше): -->
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>jstl</artifactId>
    <version>1.2</version>
</dependency>
```

> Уточнення: артефакт `jstl:jstl:1.2` (а також окремий `taglibs:standard`) застарілий. Для сучасних проєктів (Tomcat 10+/Jakarta EE 9+) використовують реалізацію Jakarta: `org.glassfish.web:jakarta.servlet.jsp.jstl` разом з `jakarta.servlet.jsp.jstl:jakarta.servlet.jsp.jstl-api`, а URI просторів імен — `jakarta.tags.*`.

+ указати простір імен потрібних тегів JSTL на JSP-сторінці (URI для класичної javax-версії):

```
<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/fmt" prefix="fmt" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/sql" prefix="sql" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/xml" prefix="x" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/functions" prefix="fn" %>
```

[до змісту](#servlets-jsp-jstl)

## З яких груп тегів складається бібліотека _JSTL_?
Групи тегів JSTL за функціональністю:

+ _Core Tags_ (`c`) — ітерації, умови, обробка винятків, робота з URL, `forward`/`redirect` тощо.
+ _Formatting & Localization Tags_ (`fmt`) — форматування чисел і дат, підтримка _i18n_ та _resource bundles_.
+ _SQL Tags_ (`sql`) — робота з базами даних (на практиці не рекомендується — змішує доступ до даних із представленням).
+ _XML Tags_ (`x`) — робота з XML-документами: парсинг, перетворення, виконання виразів _XPath_.
+ _Functions Tags_ (`fn`) — набір функцій для операцій із рядками тощо (конкатенація, розбиття, перевірка довжини).

[до змісту](#servlets-jsp-jstl)

## Яка різниця між `<c:set>` і `<jsp:useBean>`?
Обидва теги створюють і поміщають значення в задану область видимості, але `<jsp:useBean>` лише створює (чи знаходить) екземпляр конкретного типу, а `<c:set>` дозволяє ще й присвоїти змінній обчислене значення EL — наприклад, узяте з параметрів запиту, властивості об'єкта, сесії тощо.

[до змісту](#servlets-jsp-jstl)

## Чим відрізняється `<c:import>` від `<jsp:include>` і директиви `<%@include %>`?
+ `<%@ include %>` (директива) — статичне включення на фазі _трансляції_: вміст файлу буквально вставляється в код сторінки до компіляції.
+ `<jsp:include>` (action) — динамічне включення під час _обробки запиту_: підключає ресурс у межах того самого контексту застосунку.
+ `<c:import>` (JSTL) — найгнучкіше включення: на відміну від двох попередніх, може підключати ресурси не лише з поточного застосунку, а й _зовнішні_ (за абсолютним URL, навіть з іншого сервера), і дозволяє зчитати результат у змінну.

[до змісту](#servlets-jsp-jstl)

## Наведіть приклад використання власних тегів.
Функціональність JSP розширюють створенням власних тегів (custom tags). Тег описують у файлі дескриптора бібліотеки тегів `.tld`, реалізують Java-класом і підключають на сторінці через директиву `taglib`.

> /WEB-INF/exampleTag.tld

```xml
<?xml version="1.0" encoding="UTF-8"?>
<taglib version="2.1" xmlns="http://java.sun.com/xml/ns/j2ee"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://java.sun.com/xml/ns/j2ee web-jsptaglibrary_2_1.xsd">
    <tlib-version>1.0</tlib-version>
    <short-name>example</short-name>
    <uri>/WEB-INF/exampleTag</uri>
    <tag>
        <name>exampleTag</name>
        <tag-class>xyz.company.ExampleTag</tag-class>
        <body-content>empty</body-content>
        <info>The example tag displays Hello World!</info>
    </tag>
</taglib>
```

> xyz/company/ExampleTag.java

```java
package xyz.company;

import java.io.IOException;

import javax.servlet.jsp.JspException;
import javax.servlet.jsp.tagext.TagSupport;

public class ExampleTag extends TagSupport {
    private static final long serialVersionUID = 1L;

    @Override
    public int doStartTag() throws JspException {
        try {
            pageContext.getOut().print("Hello World!");
        } catch (IOException ioException) {
            throw new JspException("Error: " + ioException.getMessage());
        }
        return SKIP_BODY;
    }
}
```

> exampleTag.jsp

```
<%@ taglib uri="/WEB-INF/exampleTag.tld" prefix="example" %>
<%@ page session="false" pageEncoding="UTF-8" %>
<html>
<head>
<title>Example Tag</title>
</head>
<body>
    <h1>Example Tag</h1>
    <p><example:exampleTag /></p>
</body>
</html>
```

> У вихідному тексті у класі тег виводив `Hello World!`, але на JSP-сторінці викликався як `<example:exampleTage />` (зайва літера `e`), а тег `<p>` не був закритий. Ім'я тега має збігатися з оголошеним у `.tld` (`exampleTag`). Виправлено.

Також у власних тегах можна передбачати вхідні атрибути. Наприклад, потрібно відформатувати дуже довге число за шаблоном:

`<mytags:formatNumber number="123456.789" format="#,###.00"/>`

За шаблоном число має відобразитися як `123,456.79`. Оскільки JSTL такої функції не надає, для цього створюють власний тег.

> Сучасніший підхід — теги-файли (`.tag`-файли у `WEB-INF/tags/`) на основі `SimpleTagSupport`: вони простіші за класичні теги на `TagSupport`.

## Як зробити перенесення рядка в HTML засобами JSP?
Можна скористатися тегом `<c:out>` з атрибутом `escapeXml="false"`, який вимикає екранування HTML. Тоді браузер отримає HTML-розмітку як є й обробить елемент `<br>` як перенесення рядка:

`<c:out value="<br> creates a new line in HTML" escapeXml="false"></c:out>`

> Увага: у вихідному тексті було вказано `escapeXml="true"`, що дало б _протилежний_ результат — символи `<br>` були б екрановані й показані як текст, а не як перенесення. Щоб HTML інтерпретувався, потрібно саме `escapeXml="false"`. Майте на увазі: вимкнення екранування для даних, що надходять від користувача, відкриває XSS-уразливість, тому застосовувати його варто лише до довіреного вмісту.

[до змісту](#servlets-jsp-jstl)

## Чому не потрібно конфігурувати стандартні JSP-теги в `web.xml`?
Стандартні теги JSP (JSTL) не потрібно конфігурувати в `web.xml`, бо `.tld`-файли вже лежать усередині каталогу `/META-INF/` у JAR-файлах JSTL.

Коли контейнер завантажує веб-застосунок і знаходить `.tld`-файли в `/META-INF/` JAR-залежності, він автоматично робить ці теги доступними для JSP-сторінок. Залишається лише вказати простір імен директивою `taglib` на сторінці.

[до змісту](#servlets-jsp-jstl)

## Як можна обробити помилки JSP-сторінок?
Щоб обробляти винятки, кинуті на JSP-сторінці, створюють окрему сторінку помилки й позначають її директивою `page` з атрибутом `isErrorPage="true"`. Тоді на цій сторінці стає доступним неявний об'єкт `exception`, і можна показати клієнту інформативніше повідомлення. Прив'язку сторінок помилок задають у дескрипторі розгортання:

```xml
<error-page>
    <error-code>404</error-code>
    <location>/error.jsp</location>
</error-page>

<error-page>
    <exception-type>java.lang.Throwable</exception-type>
    <location>/error.jsp</location>
</error-page>
```

Сторінка, що генерує помилки, своєю чергою може вказати свою сторінку помилки директивою `<%@ page errorPage="error.jsp" %>`.

[до змісту](#servlets-jsp-jstl)

## Як відбувається обробка помилок за допомогою JSTL?
Для перехоплення й обробки винятків використовують теги JSTL Core `<c:catch>` і `<c:if>`.

> `<c:catch>` перехоплює виняток і кладе його у змінну `exception`, доступну для перевірки в `<c:if>`:

```
<c:catch var="exception">
    <% int x = 42 / 0; %>
</c:catch>
<c:if test="${exception ne null}">
    <p>Exception is: ${exception} <br />
       Exception Message: ${exception.message}</p>
</c:if>
```

Зверніть увагу, що в `<c:if>` використовується мова виразів EL.

[до змісту](#servlets-jsp-jstl)

## Як конфігурується JSP у дескрипторі розгортання.
Для налаштування різних параметрів JSP-сторінок використовують елемент `<jsp-config>`, який відповідає за:

+ керування елементами скриплетів на сторінці;
+ керування обробкою мови виразів (EL);
+ визначення шаблону URL для кодування сторінки;
+ визначення розміру буфера для об'єктів сторінки;
+ позначення груп ресурсів (за шаблоном URL), які слід обробляти як XML-документ;
+ оголошення бібліотек тегів (`<taglib>`).

```xml
<jsp-config>
    <taglib>
        <taglib-uri>http://company.xyz/jsp/tlds/customtags</taglib-uri>
        <taglib-location>/WEB-INF/exampleTag.tld</taglib-location>
    </taglib>
</jsp-config>
```

## Чи завжди створюється об'єкт сесії на JSP-сторінці, чи можна вимкнути його створення?

## Яка різниця між `JspWriter` і сервлетним `PrintWriter`?
`PrintWriter` — це об'єкт, що безпосередньо записує вміст у відповідь. `JspWriter` (неявний об'єкт `out` на JSP-сторінці) надбудовується над `PrintWriter` і додає буферизацію: коли буфер заповнюється або примусово скидається (flush), `JspWriter` передає накопичений вміст у `PrintWriter`, який і пише його у відповідь. Завдяки буферизації JSP може, наприклад, перенаправити запит чи змінити заголовки, доки буфер ще не скинуто.

## Опишіть загальні практичні принципи роботи з JSP.
Гарні практики роботи з JSP:

+ Уникати скриплетів на сторінці. Якщо елементів _action_, _JSTL_, _JSP EL_ не вистачає — краще написати власний тег.
+ Використовувати JSP-коментарі (`<%-- --%>`) для коментарів і налагодження — вони не потрапляють до клієнта.
+ Не розміщувати бізнес-логіку в JSP — сторінки призначені лише для формування відповіді клієнту.
+ Вимикати створення сесії на сторінках, де вона не потрібна, — для кращої продуктивності.
+ Тримати директиви `taglib`, `page` на початку сторінки — для читабельності.
+ Правильно розрізняти статичне й динамічне включення: директива `<%@ include %>` — для статичних ресурсів, `<jsp:include>` — для динамічних ресурсів часу виконання.
+ Обробляти винятки через сторінки помилок.
+ Виносити CSS і JavaScript в окремі файли й підключати їх.
+ У більшості випадків JSTL має покривати всі потреби. Якщо ні — спершу перегляньте логіку застосунку й спробуйте перенести обчислення в сервлет, а на JSP-сторінці через атрибути використовувати лише готовий результат.
