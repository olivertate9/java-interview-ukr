## У чому різниця між IO та NIO?

**Ментальна модель:** класичний `java.io` — це «один потік виконання тримає одне з'єднання й чекає на нього». NIO — це «один потік виконання обслуговує сотні з'єднань, реагуючи лише на ті, де реально з'явилися дані».

+ Java IO (input-output) є **потокоорієнтованим** (stream-oriented), а Java NIO (non-blocking io, з'явився у Java 4) — **буфер-орієнтованим** (buffer-oriented). Потокоорієнтований ввід/вивід читає/записує з потоку (stream) один або кілька байтів за раз, послідовно. Дані ніде не кешуються, тому довільно рухатися потоком уперед чи назад не можна. У Java NIO дані спочатку зчитуються в буфер (`Buffer`), що дає більше гнучкості при обробці — можна, наприклад, повернутися й перечитати вже зчитані байти.
+ Потоки в Java IO є **блокувальними**: коли потік виконання викликає `read()` чи `write()` будь-якого класу з пакета `java.io`, він блокується, доки дані не будуть зчитані або записані, і в цей час не може робити нічого іншого. **Неблокувальний** режим Java NIO дозволяє запросити дані з каналу (channel) й отримати лише те, що доступно зараз, або взагалі нічого, якщо даних поки немає. Замість того щоб простоювати в очікуванні, потік виконання може зайнятися чимось іншим. Те саме справедливо й для неблокувального виводу.
+ У Java NIO є **селектори** (`Selector`), які дозволяють одному потоку виконання моніторити кілька каналів. Можна зареєструвати кілька каналів у селекторі, а потім одним потоком обслуговувати ті канали, що мають готові для обробки дані або готові до запису. Саме це робить NIO основою для масштабованих серверів.

> Варто згадати **NIO.2** (з Java 7, пакет `java.nio.file`): класи `Path`, `Files`, `FileSystem`, `WatchService`. На практиці саме `Files`/`Path` сьогодні є рекомендованим способом роботи з файлами замість старого `java.io.File`.

## Які особливості NIO ви знаєте?
+ **Канали та селектори**: NIO підтримує різні типи каналів. Канал (channel) — це абстракція над низькорівневими об'єктами файлової системи (наприклад, відображені в пам'ять файли та блокування файлів), що дозволяє передавати дані з вищою швидкістю. Канали можуть працювати в неблокувальному режимі, тому Java надає селектор (`Selector`), який дозволяє вибрати готовий канал для передачі даних.
+ **Буфери**: є буферизація для всіх примітивних типів (крім `boolean`) — `ByteBuffer`, `CharBuffer`, `IntBuffer` тощо. Існує абстрактний клас `Buffer`, який надає операції `clear()`, `flip()`, `mark()`, `rewind()` та інші, а його підкласи — методи для отримання й запису даних.
+ **Кодування**: з'явилися кодери та декодери (`CharsetEncoder`/`CharsetDecoder`) для перетворення між байтами та символами Unicode.

## Що таке _«канали»_?
Канали (channels) — це логічні (не фізичні) портали, абстракції над низькорівневими об'єктами файлової системи (наприклад, відображені в пам'ять файли та блокування файлів), через які здійснюється ввід/вивід даних, а буфери є джерелами або приймачами цих переданих даних. Під час виводу дані, які треба відправити, поміщаються в буфер, який потім передається в канал. Під час вводу дані з каналу поміщаються в заздалегідь наданий буфер.

Канали нагадують трубопроводи, якими ефективно транспортуються дані між байтовими буферами та сутностями по той бік каналів. Канали — це шлюзи, що дають доступ до сервісів вводу/виводу операційної системи з мінімальними накладними витратами, а буфери — внутрішні кінцеві точки цих шлюзів, які використовуються для передачі та приймання даних.

## Назвіть основні класи потоків вводу/виводу.
Розрізняють два види потоків вводу/виводу:

+ **байтові** — `java.io.InputStream`, `java.io.OutputStream` (працюють з «сирими» байтами);
+ **символьні** — `java.io.Reader`, `java.io.Writer` (працюють із символами з урахуванням кодування).

## Які підкласи класу `InputStream` ви знаєте, для чого вони призначені?
+ `InputStream` — абстрактний клас, що описує потік вводу;
+ `BufferedInputStream` — буферизований вхідний потік;
+ `ByteArrayInputStream` дозволяє використати буфер у пам'яті (масив байтів) як джерело даних для вхідного потоку;
+ `DataInputStream` — вхідний потік для байтових даних із методами для читання стандартних типів даних Java;
+ `FileInputStream` — вхідний потік для читання інформації з файлу;
+ `FilterInputStream` — абстрактний клас, що надає інтерфейс для класів-надбудов, які додають до наявних потоків корисні властивості;
+ `ObjectInputStream` — вхідний потік для об'єктів (десеріалізація);
+ `PipedInputStream` реалізує поняття вхідного каналу (для обміну даними між потоками виконання);
+ `PushbackInputStream` — різновид буферизації, що забезпечує читання байта з подальшим його поверненням у потік: дозволяє «зазирнути» у вхідний потік і побачити, що надійде наступним, не вилучаючи інформації;
+ `SequenceInputStream` використовується для злиття двох або більше потоків `InputStream` в один.

> Клас `StringBufferInputStream` був deprecated ще з Java 1.1, бо він некоректно конвертує символи в байти (втрачає старші 8 біт). Для читання `String` як потоку слід використовувати `StringReader` або `ByteArrayInputStream` із явним кодуванням, тому в актуальному коді його не застосовують.

## Для чого використовується `PushbackInputStream`?
Це різновид буферизації, що забезпечує читання байта з подальшим його поверненням у потік. Клас `PushbackInputStream` дає механізм «зазирнути» у вхідний потік і побачити, що надійде наступним, не вилучаючи інформації. Типове застосування — парсери, де треба прочитати символ, прийняти рішення й «віддати» його назад, якщо він належить до наступної лексеми.

У класу є додатковий метод `unread()`, який повертає байт (або масив байтів) назад у потік.

[до змісту](#Потоки-вводувиводу-в-Java)

## Для чого використовується `SequenceInputStream`?
Клас `SequenceInputStream` дозволяє об'єднувати кілька екземплярів `InputStream` у логічно єдиний потік. Конструктор приймає або пару об'єктів `InputStream`, або `Enumeration` з них.

Під час роботи клас виконує запити на читання з першого `InputStream` до його кінця, а потім перемикається на наступний. По досягненні кінця кожного потоку пов'язаний з ним потік закривається. Закриття потоку, створеного об'єктом `SequenceInputStream`, призводить до закриття всіх відкритих потоків.

[до змісту](#Потоки-вводувиводу-в-Java)

## Який клас дозволяє читати дані з вхідного байтового потоку у форматі примітивних типів даних?
Клас `DataInputStream` дозволяє читати з потоку дані примітивних типів (`int`, `double` тощо) у незалежному від платформи форматі. Для кожного примітивного типу визначено свій метод читання:

+ `boolean readBoolean()` — зчитує з потоку однобайтове булеве значення;
+ `byte readByte()` — зчитує з потоку 1 байт;
+ `char readChar()` — зчитує з потоку значення `char`;
+ `double readDouble()` — зчитує з потоку 8-байтове значення `double`;
+ `float readFloat()` — зчитує з потоку 4-байтове значення `float`;
+ `int readInt()` — зчитує з потоку цілочислове значення `int`;
+ `long readLong()` — зчитує з потоку значення `long`;
+ `short readShort()` — зчитує значення `short`;
+ `String readUTF()` — зчитує з потоку рядок у модифікованому форматі UTF-8.

> Симетричний клас для запису — `DataOutputStream`. Дані, записані одним, мають читатися іншим: формат бінарний і платформонезалежний.

## Які підкласи класу `OutputStream` ви знаєте, для чого вони призначені?
+ `OutputStream` — абстрактний клас, що визначає потоковий байтовий вивід;
+ `BufferedOutputStream` — буферизований вихідний потік;
+ `ByteArrayOutputStream` — усі дані, що надсилаються в цей потік, розміщуються в буфері в пам'яті, який зростає за потреби;
+ `DataOutputStream` — вихідний потік байтів із методами для запису стандартних типів даних Java;
+ `FileOutputStream` — запис даних у файл на фізичному носії;
+ `FilterOutputStream` — абстрактний клас, що надає інтерфейс для класів-надбудов;
+ `PrintStream` — вихідний потік із методами `print()` та `println()` (саме його екземпляром є `System.out`);
+ `ObjectOutputStream` — вихідний потік для запису об'єктів (серіалізація);
+ `PipedOutputStream` реалізує поняття вихідного каналу.

## Які підкласи класу `Reader` ви знаєте, для чого вони призначені?
+ `Reader` — абстрактний клас, що описує символьний ввід;
+ `BufferedReader` — буферизований вхідний символьний потік (має зручний метод `readLine()`);
+ `CharArrayReader` — вхідний потік, що читає із символьного масиву;
+ `FileReader` — вхідний потік, що читає файл;
+ `FilterReader` — абстрактний клас, що надає інтерфейс для класів-надбудов;
+ `InputStreamReader` — вхідний потік, що транслює байти в символи згідно з кодуванням;
+ `LineNumberReader` — вхідний потік, що підраховує рядки;
+ `PipedReader` — вхідний канал;
+ `PushbackReader` — вхідний потік, що дозволяє повертати символи назад у потік;
+ `StringReader` — вхідний потік, що читає з рядка.

## Які підкласи класу `Writer` ви знаєте, для чого вони призначені?
+ `Writer` — абстрактний клас, що описує символьний вивід;
+ `BufferedWriter` — буферизований вихідний символьний потік;
+ `CharArrayWriter` — вихідний потік, що пише в символьний масив;
+ `FileWriter` — вихідний потік, що пише у файл;
+ `FilterWriter` — абстрактний клас, що надає інтерфейс для класів-надбудов;
+ `OutputStreamWriter` — вихідний потік, що транслює символи в байти згідно з кодуванням;
+ `PipedWriter` — вихідний канал;
+ `PrintWriter` — вихідний потік символів із методами `print()` та `println()`;
+ `StringWriter` — вихідний потік, що пише в рядок.

## У чому відмінність класу `PrintWriter` від `PrintStream`?

| Критерій | `PrintStream` | `PrintWriter` |
|---|---|---|
| Орієнтація | байтовий (`OutputStream`) | символьний (`Writer`) |
| Кодування | історично прив'язаний до байтів, гірше працює з Unicode | коректно працює із символами Unicode та явним `Charset` |
| Автоскидання буфера | при кожному `print()`/`println()` (якщо увімкнено autoflush) | можна вимкнути автоскидання й робити `flush()` явно |
| Винятки | методи не кидають `IOException`, помилку перевіряють через `checkError()` | так само не кидають винятки, помилку перевіряють через `checkError()` |

Коротко: `PrintWriter` — це «правильний» символьний варіант для тексту; `PrintStream` лишається переважно через сумісність (`System.out`/`System.err` — це `PrintStream`).

## Чим відрізняються та що спільного у `InputStream`, `OutputStream`, `Reader`, `Writer`?

Спільне: усі чотири — абстрактні базові класи ієрархії вводу/виводу, усі реалізують `Closeable` (а отже працюють із try-with-resources), будуються за патерном «Декоратор» (потоки обгортають один одного).

Відмінності:

+ `InputStream` та його нащадки — отримання **байтових** даних із різних джерел;
+ `OutputStream` та його нащадки — потоковий **байтовий** вивід;
+ `Reader` та його нащадки — потоковий ввід **символів** Unicode (з урахуванням кодування);
+ `Writer` та його нащадки — потоковий вивід **символів** Unicode.

Тобто розподіл йде за двома осями: ввід/вивід та байти/символи.

[до змісту](#Потоки-вводувиводу-в-Java)

## Які класи дозволяють перетворити байтові потоки на символьні та навпаки?
+ `OutputStreamWriter` — «міст» від `Writer` до `OutputStream`: символи, записані в потік, перетворюються на байти згідно із заданим кодуванням.
+ `InputStreamReader` — аналог для читання: байти з `InputStream` зчитуються й перетворюються на символи.

> Бажано завжди вказувати кодування явно, наприклад `new InputStreamReader(in, StandardCharsets.UTF_8)`. Без нього використовується кодування платформи за замовчуванням, що призводить до «крякозябр» на іншій машині. (З Java 18 типовим кодуванням для `Charset.defaultCharset()` стало UTF-8, але покладатися на платформу все одно не варто.)

[до змісту](#Потоки-вводувиводу-в-Java)

## Які класи дозволяють пришвидшити читання/запис за рахунок використання буфера?
+ `BufferedInputStream(InputStream in)` / `BufferedInputStream(InputStream in, int size)`,
+ `BufferedOutputStream(OutputStream out)` / `BufferedOutputStream(OutputStream out, int size)`,
+ `BufferedReader(Reader in)` / `BufferedReader(Reader in, int sz)`,
+ `BufferedWriter(Writer out)` / `BufferedWriter(Writer out, int sz)`.

Сенс буферизації: замість багатьох дрібних системних викликів читання/запису (кожен дорогий) дані накопичуються в буфері в пам'яті й передаються великими блоками. Тому обгортання `FileInputStream` у `BufferedInputStream` може дати кратне пришвидшення.

## Який клас призначений для роботи з елементами файлової системи?
`java.io.File` працює безпосередньо з файлами та каталогами. Цей клас дозволяє створювати нові елементи й отримувати інформацію про наявні: розмір, права доступу, час і дату зміни, шлях до батьківського каталогу.

> У сучасному коді (з Java 7) для тих самих задач рекомендують `java.nio.file.Path` + `java.nio.file.Files`: вони підтримують символьні посилання, атрибути файлів, кидають інформативні винятки замість мовчазного повернення `false`, і мають зручні методи на кшталт `Files.readAllLines()`, `Files.copy()`, `Files.walk()`.

## Які методи класу `File` ви знаєте?
Найуживаніші методи класу `File`:

+ `boolean createNewFile()` — намагається створити новий файл;
+ `boolean delete()` — намагається видалити каталог або файл;
+ `boolean mkdir()` — намагається створити новий каталог;
+ `boolean renameTo(File dest)` — намагається перейменувати файл або каталог;
+ `boolean exists()` — перевіряє, чи існує файл або каталог;
+ `String getAbsolutePath()` — повертає абсолютний шлях;
+ `String getName()` — повертає коротке ім'я файлу або каталогу;
+ `String getParent()` — повертає ім'я батьківського каталогу;
+ `boolean isDirectory()` — повертає `true`, якщо за вказаним шляхом розташований каталог;
+ `boolean isFile()` — повертає `true`, якщо за вказаним шляхом розташований файл;
+ `boolean isHidden()` — повертає `true`, якщо каталог або файл прихований;
+ `long length()` — повертає розмір файлу в байтах;
+ `long lastModified()` — повертає час останньої зміни файлу або каталогу;
+ `String[] list()` — повертає масив імен файлів і підкаталогів у заданому каталозі;
+ `File[] listFiles()` — повертає масив об'єктів `File` для файлів і підкаталогів у заданому каталозі.

## Що ви знаєте про інтерфейс `FileFilter`?
Інтерфейс `FileFilter` застосовується для перевірки, чи задовольняє об'єкт `File` певну умову. Він містить єдиний метод `boolean accept(File pathname)`, який треба реалізувати. Оскільки інтерфейс функціональний, його зручно задавати лямбдою. Наприклад:

```java
// було: анонімний клас з помилкою — методу isExists() у File немає
FileFilter filter = file -> file.exists() && file.isDirectory();
```

> У вихідному файлі тут був метод `file.isExists()`, якого в класі `File` не існує (правильно — `exists()`). Виправлено.

[до змісту](#Потоки-вводувиводу-в-Java)

## Як вибрати всі елементи певного каталогу за критерієм (наприклад, з певним розширенням)?
Метод `File.listFiles()` повертає масив об'єктів `File`, що містяться в каталозі. Як параметр він може приймати об'єкт, що реалізує `FileFilter` (фільтрація за об'єктом `File`) або `FilenameFilter` (фільтрація за іменем). У результат потраплять лише ті елементи, для яких `accept(...)` повернув `true`. Наприклад, відбір `.txt`-файлів:

```java
File[] txtFiles = dir.listFiles((d, name) -> name.endsWith(".txt"));
```

> Альтернатива з NIO.2: `Files.newDirectoryStream(path, "*.txt")` або `Files.list(path).filter(...)`.

[до змісту](#Потоки-вводувиводу-в-Java)

## Що ви знаєте про `RandomAccessFile`?
Клас `java.io.RandomAccessFile` забезпечує читання й запис даних у довільному місці файлу. Він не входить до ієрархії `InputStream` чи `OutputStream` — це повністю окремий клас із власними (здебільшого _native_) методами. Причина в тому, що його поведінка принципово відрізняється від інших класів вводу/виводу: усередині файлу можна вільно переміщатися вперед і назад (random access), тоді як звичайні потоки послідовні.

`RandomAccessFile` має такі специфічні методи:

+ `getFilePointer()` — поточна позиція у файлі;
+ `seek()` — переміщення на нову позицію у файлі;
+ `length()` — розмір файлу;
+ `setLength()` — встановлення розміру файлу;
+ `skipBytes()` — спроба пропустити певне число байтів;
+ `getChannel()` — отримання файлового каналу (`FileChannel`), асоційованого з файлом;
+ методи звичайного й форматованого читання з файлу (`read()`, `readInt()`, `readLine()`, `readUTF()` тощо);
+ методи звичайного й форматованого запису у файл (`write()`, `writeBoolean()`, `writeByte()` тощо).

Також варто зазначити, що конструктори `RandomAccessFile` вимагають другий аргумент — режим доступу до файлу: тільки читання (`"r"`), читання й запис (`"rw"`) або їхні різновиди.

[до змісту](#Потоки-вводувиводу-в-Java)

## Які режими доступу до файлу є у `RandomAccessFile`?
+ `"r"` — відкриває файл лише для читання. Виклик будь-якого методу запису призведе до `IOException`.
+ `"rw"` — відкриває файл для читання й запису. Якщо файлу ще немає, робиться спроба його створити.
+ `"rws"` — відкриває файл для читання й запису подібно до `"rw"`, але вимагає від системи синхронно записувати на фізичний носій кожну зміну **вмісту файлу та його метаданих**.
+ `"rwd"` — подібно до `"rws"`, але синхронно записує на носій лише зміни **вмісту файлу**; для змін метаданих синхронний запис не вимагається.

[до змісту](#Потоки-вводувиводу-в-Java)

## Які класи підтримують читання та запис потоків у стиснутому форматі?
+ `DeflaterOutputStream` — стиснення даних у форматі deflate;
+ `Deflater` — стиснення даних у форматі ZLIB;
+ `ZipOutputStream` — нащадок `DeflaterOutputStream` для стиснення у формат ZIP;
+ `GZIPOutputStream` — нащадок `DeflaterOutputStream` для стиснення у формат GZIP;
+ `InflaterInputStream` — розпакування даних у форматі deflate;
+ `Inflater` — розпакування даних у форматі ZLIB;
+ `ZipInputStream` — нащадок `InflaterInputStream` для розпакування формату ZIP;
+ `GZIPInputStream` — нащадок `InflaterInputStream` для розпакування формату GZIP.

Усі ці класи розташовані в пакеті `java.util.zip`.

## Чи існує можливість перенаправити потоки стандартного вводу/виводу?
Так. Клас `System` дозволяє перенаправити стандартний ввід, вивід і потік помилок простим викликом статичного методу:

+ `setIn(InputStream)` — для вводу;
+ `setOut(PrintStream)` — для виводу;
+ `setErr(PrintStream)` — для виводу помилок.

Типове застосування — перенаправлення `System.out` у файл або в `ByteArrayOutputStream` для перехоплення виводу в тестах.

## Який символ є роздільником при вказанні шляху у файловій системі?
Для різних операційних систем роздільник різний: для Windows це `\`, для Linux/macOS — `/`.

У Java роздільник для поточної ОС можна отримати через статичне поле `File.separator` (рядок) або `File.separatorChar` (символ). Окремо є `File.pathSeparator` — роздільник елементів у списку шляхів (наприклад, у `CLASSPATH`): `;` у Windows і `:` у Unix.

## Що таке _«абсолютний шлях»_ та _«відносний шлях»_?
**Абсолютний (повний) шлях** — це шлях, що вказує на одне й те саме місце у файловій системі незалежно від поточного робочого каталогу чи інших обставин. Повний шлях завжди починається з кореневого каталогу (`/` в Unix або, наприклад, `C:\` у Windows).

**Відносний шлях** задається відносно поточного робочого каталогу користувача або активного застосунку.

[до змісту](#Потоки-вводувиводу-в-Java)

## Що таке _«символьне посилання»_?
**Символьне (символічне) посилання** (також «симлінк», symbolic link) — спеціальний файл у файловій системі, який замість користувацьких даних містить шлях до файлу, що має бути відкритий при зверненні до цього посилання. Ціллю посилання може бути будь-який об'єкт: інше посилання, файл, каталог або навіть неіснуючий файл (у такому разі при спробі відкрити його буде повідомлення про відсутність файлу).

Символьні посилання використовуються для зручнішої організації структури файлів, оскільки:

+ дозволяють одному файлу чи каталогу мати кілька імен і різних атрибутів;
+ вільні від деяких обмежень, властивих жорстким посиланням (hard links): останні діють лише в межах однієї файлової системи (одного розділу) і не можуть посилатися на каталоги.

> У Java працювати із символьними посиланнями вміє саме NIO.2: `Files.createSymbolicLink()`, `Files.isSymbolicLink()`, `Files.readSymbolicLink()`. Старий `java.io.File` їх повноцінно не розрізняє.
