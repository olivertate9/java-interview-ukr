## Що таке _«серіалізація»_?
__Серіалізація (serialization)__ — процес перетворення стану об'єкта (структури даних) у лінійну послідовність байтів для подальшої передачі мережею або збереження. Серіалізовані дані потім можна відновити назад в об'єкт — це зворотний процес, __десеріалізація (deserialization)__.

Ментальна модель: серіалізація — це «фотографія» графа об'єктів у вигляді байтів. Десеріалізація відтворює цей граф, не викликаючи конструктор самого класу.

У Java, згідно зі специфікацією Java Object Serialization, є два стандартних способи серіалізації: стандартна серіалізація через інтерфейс `java.io.Serializable` та «розширена» — `java.io.Externalizable`.

Серіалізація дозволяє в певних межах змінювати клас, зберігаючи сумісність з раніше серіалізованими даними. Найважливіші зміни, з якими специфікація справляється автоматично:

+ додавання в клас нових полів;
+ зміна полів зі статичних на нестатичні;
+ зміна полів з `transient` на не-`transient`.

Зворотні зміни (з нестатичних полів на статичні та з не-`transient` на `transient`) або видалення полів потребують додаткового опрацювання залежно від того, який ступінь зворотної сумісності необхідний.

> Зверніше увагу: стандартна Java-серіалізація сьогодні вважається застарілим і небезпечним механізмом. На співбесіді корисно знати, як вона працює, але в реальних проєктах для обміну даними частіше беруть JSON (Jackson/Gson), Protocol Buffers чи інші явні формати. Деталі — в останньому питанні нижче.

## Опишіть процес серіалізації/десеріалізації з використанням `Serializable`.
`Serializable` — це маркерний інтерфейс (без методів). Сам факт його реалізації вмикає стандартний алгоритм серіалізації, який за допомогою рефлексії (Reflection API) виконує:

+ запис у потік метаданих про клас, асоційований з об'єктом (ім'я класу, ідентифікатор `serialVersionUID`, описи полів класу);
+ рекурсивний запис у потік описів суперкласів аж до `java.lang.Object` (не включно);
+ запис значень полів серіалізованого екземпляра, починаючи з полів найвищого суперкласу;
+ рекурсивний запис об'єктів, які є полями серіалізованого об'єкта.

При цьому раніше серіалізовані об'єкти повторно не серіалізуються (вони замінюються посиланням на вже записаний екземпляр), що дозволяє алгоритму коректно працювати з циклічними посиланнями.

Для десеріалізації під об'єкт виділяється пам'ять, після чого його поля заповнюються значеннями з потоку. __Конструктор самого серіалізованого класу при цьому НЕ викликається.__ Однак буде викликано конструктор без параметрів першого несеріалізованого батьківського класу в ієрархії; якщо такого конструктора немає — десеріалізація завершиться помилкою (`InvalidClassException`).

## Як змінити стандартну поведінку серіалізації/десеріалізації?
+ Реалізувати інтерфейс `java.io.Externalizable`, який дозволяє повністю задати власну логіку серіалізації. Спосіб серіалізації та десеріалізації описується в методах `writeExternal()` і `readExternal()`. Під час десеріалізації спочатку викликається публічний конструктор без параметрів, а потім на створеному об'єкті — метод `readExternal()`.
+ Якщо в серіалізованому класі реалізовано один із наведених нижче методів, механізм серіалізації використає його замість стандартного:
    + `writeObject(ObjectOutputStream out)` — запис об'єкта в потік;
    + `readObject(ObjectInputStream in)` — читання об'єкта з потоку;
    + `writeReplace()` — дозволяє перед записом замінити себе екземпляром іншого класу;
    + `readResolve()` — дозволяє після читання замінити прочитаний об'єкт на інший.

Ці методи мають бути `private` (для `writeObject`/`readObject` це обов'язкова умова, інакше їх не «побачить» механізм серіалізації). Усередині них зазвичай викликають `out.defaultWriteObject()` / `in.defaultReadObject()`, щоб дописати стандартну логіку власною, а не переписувати її повністю.

## Як виключити поля із серіалізації?
Для керування серіалізацією при оголошенні полів можна використати ключове слово `transient` — так поле виключається із загального процесу серіалізації.

[до змісту](#Серіалізація)

## Що означає ключове слово `transient`?
Поля класу, позначені модифікатором `transient`, не серіалізуються. Після десеріалізації таке поле отримує значення за замовчуванням для свого типу (`null` для посилань, `0` для чисел, `false` для `boolean`).

Зазвичай у таких полях зберігають:
+ проміжний (кешований) стан, який простіше обчислити заново, ніж серіалізувати (наприклад, кешований хеш чи похідне значення);
+ посилання на об'єкт, який не потребує серіалізації або не може бути серіалізованим (наприклад, `Thread`, відкрите з'єднання, `Logger`);
+ чутливі дані (паролі, ключі), які не варто записувати в потік.

## Який вплив на серіалізованість мають модифікатори полів `static` і `final`?
При стандартній серіалізації поля з модифікатором `static` __не серіалізуються__, бо належать класу, а не екземпляру. Відповідно, після десеріалізації статичне поле зберігає те значення, яке має клас на цей момент у поточній JVM, а не те, що було під час серіалізації.

Поля з модифікатором `final` серіалізуються як звичайні. При стандартній серіалізації (`Serializable`) їх значення відновлюється коректно, бо JVM записує `final`-поле напряму, в обхід конструктора. А от при використанні `Externalizable` десеріалізувати `final`-поле неможливо: `final`-поля мають бути ініціалізовані в конструкторі, а в `readExternal()` змінити їх значення вже не вийде. Отже, якщо потрібно серіалізувати об'єкт із `final`-полем, слід використовувати лише стандартну серіалізацію.

## Як не допустити серіалізацію?
Якщо клас успадковує `Serializable` (наприклад, від батька) і потрібно заборонити його серіалізацію, можна перевизначити `private`-методи `writeObject()` / `readObject()` так, щоб вони кидали `NotSerializableException`:

```java
private void writeObject(ObjectOutputStream out) throws IOException {
    throw new NotSerializableException("Person is not serializable");
}

private void readObject(ObjectInputStream in) throws IOException {
    throw new NotSerializableException("Person is not serializable");
}
```

Будь-яка спроба записати або прочитати такий об'єкт тепер призведе до винятку.

## Як створити власний протокол серіалізації?
Щоб задати повністю власний протокол серіалізації, достатньо реалізувати інтерфейс `Externalizable`, який містить два методи:

```java
public void writeExternal(ObjectOutput out) throws IOException;
public void readExternal(ObjectInput in) throws IOException, ClassNotFoundException;
```

На відміну від `Serializable`, тут розробник сам повністю контролює, які дані й у якому порядку записуються та читаються. Важлива умова: клас повинен мати публічний конструктор без параметрів, бо під час десеріалізації спочатку викликається саме він, а вже потім `readExternal()`.

## Яка роль поля `serialVersionUID` у серіалізації?
`serialVersionUID` — це ідентифікатор версії серіалізованого класу. Під час десеріалізації JVM порівнює `serialVersionUID` зі збережених даних із `serialVersionUID` поточного класу; якщо вони не збігаються — кидається `InvalidClassException`.

Якщо ми не оголошуємо `serialVersionUID` явно, середовище виконання Java обчислює його автоматично на основі багатьох метаданих класу (ім'я, поля та їх типи, модифікатори доступу, реалізовані інтерфейси, сигнатури методів тощо). Проблема в тому, що це значення дуже крихке: навіть незначна зміна класу (або навіть інший компілятор) може його змінити, через що десеріалізація старих даних перестане працювати.

Тому рекомендується __оголошувати `serialVersionUID` явно__ — це дає контроль над сумісністю версій:

```java
private static final long serialVersionUID = 20161013L;
```

## Коли варто змінювати значення поля `serialVersionUID`?
`serialVersionUID` варто змінювати свідомо тоді, коли в клас внесено __несумісні__ зміни і ви хочете, щоб старі серіалізовані дані більше не приймалися (наприклад, видалено поле або кардинально змінено його семантику). Зміна `serialVersionUID` сигналізує: «це інша, несумісна версія класу».

І навпаки: поки зміни сумісні (додавання нових полів), значення `serialVersionUID` лишають незмінним, щоб старі дані й далі читалися.

## У чому проблема серіалізації Singleton?
Проблема в тому, що десеріалізація створює __новий__ екземпляр класу (вона не викликає конструктор і не зважає на приватність). Тобто серіалізація дає можливість «розмножити» Singleton, що порушує його контракт єдиного екземпляра. Уникнути цього можна так:

+ заборонити серіалізацію взагалі (див. вище);
+ визначити метод `readResolve()` із сигнатурою `(default/public/private/protected) Object readResolve() throws ObjectStreamException`, який повертає вже наявний єдиний екземпляр замість щойно десеріалізованого:

```java
private Object readResolve() throws ObjectStreamException {
    return INSTANCE; // повертаємо єдиний екземпляр
}
```

Найнадійніший спосіб реалізувати серіалізований Singleton у сучасній Java — це `enum` із одним елементом: JVM гарантує коректну серіалізацію enum-ів без дублювання екземплярів автоматично.

## Які існують способи контролю за значеннями десеріалізованого об'єкта?
Десеріалізація обходить конструктор, тому вона може створити об'єкт у некоректному стані (з даними, які ніколи не пройшли б валідацію конструктора). Щоб це контролювати, є кілька механізмів.

Можна реалізувати інтерфейс `ObjectInputValidation` з перевизначенням методу `validateObject()` і зареєструвати валідатор у `readObject()` через `ObjectInputStream.registerValidation()`:

```java
public class Person implements java.io.Serializable,
                               java.io.ObjectInputValidation {
    private int age;
    // ...

    private void readObject(ObjectInputStream in)
            throws IOException, ClassNotFoundException {
        in.registerValidation(this, 0); // реєструємо себе як валідатор
        in.defaultReadObject();
    }

    @Override
    public void validateObject() throws InvalidObjectException {
        if ((age < 39) || (age > 60)) {
            throw new InvalidObjectException("Invalid age");
        }
    }
}
```

Якщо після десеріалізації вік виходить за межі 39…60, буде кинуто `InvalidObjectException`.

Також існують способи підписування та шифрування, які дозволяють переконатися, що дані не змінювали:

+ опис власної логіки шифрування/перевірки в `writeObject()` і `readObject()`;
+ обгортання об'єкта в `javax.crypto.SealedObject` (шифрування) та/або `java.security.SignedObject` (цифровий підпис). Ці класи самі є серіалізованими, тож обгорнутий об'єкт отримує захищену «упаковку». `SealedObject` шифрує дані симетричним ключем, а `SignedObject` дозволяє перевірити їх цілісність за підписом; ключами в обох випадках потрібно керувати окремо.

## Чому стандартну серіалізацію Java вважають проблемною?
Це поширене питання на сучасних співбесідах, бо стандартна Java-серіалізація має суттєві вади, і команда Java відкрито називає її «жахливою помилкою» (Brian Goetz, проєкт Amber). Що сказати на співбесіді:

+ __Безпека.__ Десеріалізація недовіреного потоку — класичне джерело RCE-вразливостей (gadget chains). `readObject()` може виконати довільний код ще до того, як ви побачите дані. Це причина багатьох CVE.
+ __Крихкість.__ Сумісність зав'язана на `serialVersionUID` та внутрішню структуру класу; будь-яка необережна зміна ламає старі дані.
+ __Невидимий API.__ `Serializable` фактично робить приватні поля частиною публічного контракту, обходить конструктори та інкапсуляцію.
+ __Продуктивність.__ Формат громіздкий і повільний порівняно з сучасними бінарними протоколами.

Що використовувати натомість: явні формати з контрольованою схемою — JSON (Jackson, Gson), Protocol Buffers, Avro, Kotlinx Serialization тощо. Якщо стандартна серіалізація все ж потрібна, з Java 9 (`ObjectInputFilter`) є можливість фільтрувати класи, що десеріалізуються, обмежуючи поверхню атаки.
