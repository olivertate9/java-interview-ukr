> ⚠️ **Зауваження про актуальність.** Усі питання нижче стосуються **log4j 1.x** — версії, яка офіційно припинила підтримку 5 серпня 2015 року й більше не має оновлень безпеки. Цей матеріал лишається у списку як історичний і концептуальний (поняття logger / appender / layout / рівні логування є спільними для всіх фреймворків), але **у новому коді log4j 1.x використовувати не можна**. Сучасний стек: **SLF4J** як фасад (API, до якого пишеться код) + одна з реалізацій — **Logback** або **Log4j 2** (`log4j-core`). Майте на увазі також критичну вразливість **Log4Shell (CVE-2021-44228)** у Log4j 2 до версії 2.17.1 — на співбесіді про неї часто запитують. Класи на кшталт `org.apache.log4j.*` з прикладів нижче належать саме застарілій версії 1.x.

## Які існують типи логів?
Логи прийнято поділяти за призначенням події, яку вони фіксують:

+ **системні** (System) — події рівня інфраструктури та взаємодії компонентів;
+ **безпеки** (Security) — автентифікація, авторизація, доступ до ресурсів;
+ **застосунку** (Application, Business) — бізнес-події, дії користувача в межах логіки програми.

> Користувач входить у застосунок, перевіряється пароль — це подія **безпеки** (Security). Далі він запускає якийсь модуль — це подія рівня **застосунку** (Application). Модуль під час старту звертається до іншого модуля по додаткові дані, робить ще якісь виклики — це вже **системні** дії (System).

## З яких частин складається система логування log4j?
Система логування складається з трьох основних частин:

+ керівної логуванням — **logger**;
+ що додає запис до журналу — **appender**;
+ що визначає формат запису — **layout**.

## Що таке _Logger_ у log4j?
**Logger** — це об'єкт класу `org.apache.log4j.Logger`, який слугує керівним інтерфейсом для логування повідомлень із можливістю задавати рівень деталізації. Саме logger вирішує, чи потрібно обробляти повідомлення (порівнюючи рівень повідомлення з налаштованим рівнем): якщо логування необхідне — повідомлення передається в appender; якщо ні — обробку цього повідомлення завершено.

> У сучасному стеку (SLF4J) аналогом є інтерфейс `org.slf4j.Logger`, який отримують через `LoggerFactory.getLogger(MyClass.class)`. Код пишуть до фасаду SLF4J, а конкретна реалізація (Logback чи Log4j 2) підставляється під час виконання.

## Що таке _Appender_ у log4j?
**Appender** — це іменований об'єкт-приймач подій журналу, що реалізує інтерфейс `org.apache.log4j.Appender` і відповідає за фактичне додавання подій до журналу (тобто визначає, **куди** пишеться лог). Appender викликає різні допоміжні інструменти — layout (компонувальник), фільтр, обробник помилок (якщо вони визначені й потрібні). У ході цієї роботи остаточно встановлюється необхідність запису повідомлення, а самому повідомленню надаються остаточні зміст і форма.

У log4j журналом (приймачем) може бути:

+ консоль;
+ файл;
+ сокет;
+ об'єкт класу, що реалізує `java.io.Writer` або `java.io.OutputStream`;
+ сховище JDBC;
+ тема (topic) JMS;
+ NT Event Log;
+ SMTP;
+ Syslog;
+ Telnet.

Найчастіше вживані appender-и log4j:

+ `org.apache.log4j.ConsoleAppender` — вивід у консоль;
+ `org.apache.log4j.FileAppender` — додавання у файл;
+ `org.apache.log4j.DailyRollingFileAppender` — додавання у файл із ротацією через заданий проміжок часу;
+ `org.apache.log4j.RollingFileAppender` — додавання у файл із ротацією після досягнення певного розміру;
+ `org.apache.log4j.varia.ExternallyRolledFileAppender` — розширення _RollingFileAppender_, що виконує ротацію файлу за командою, отриманою на заданому порту;
+ `org.apache.log4j.net.SMTPAppender` — надсилання повідомлення електронною поштою через SMTP;
+ `org.apache.log4j.AsyncAppender` — дає змогу, використовуючи окремий потік, організувати асинхронну роботу, коли повідомлення передаються далі лише після досягнення певного рівня заповненості проміжного буфера;
+ `org.apache.log4j.nt.NTEventLogAppender` — додавання в NT Event Log;
+ `org.apache.log4j.net.SyslogAppender` — додавання в Syslog;
+ `org.apache.log4j.jdbc.JDBCAppender` — запис у сховище через JDBC;
+ `org.apache.log4j.lf5.LF5Appender` — повідомлення передається у спеціальний графічний інтерфейс LogFactor5;
+ `org.apache.log4j.net.SocketAppender` — трансляція повідомлення на вказані адресу та порт;
+ `org.apache.log4j.net.SocketHubAppender` — розсилання повідомлення одразу кільком віддаленим серверам, з'єднаним за заданим портом;
+ `org.apache.log4j.net.TelnetAppender` — надсилання повідомлення за протоколом Telnet;
+ `org.apache.log4j.net.JMSAppender` — додавання повідомлення в JMS.

## Що таке _Layout_ у log4j?
**Layout** — нащадок класу `org.apache.log4j.Layout`, що надає можливість форматувати повідомлення перед додаванням до журналу (тобто визначає, **як виглядатиме** рядок логу).

У log4j є такі типи layout-ів:

+ `org.apache.log4j.SimpleLayout` — на виході отримуємо рядок, що містить лише рівень логування та саме повідомлення;
+ `org.apache.log4j.HTMLLayout` — форматує повідомлення як елемент HTML-таблиці;
+ `org.apache.log4j.xml.XMLLayout` — компонує повідомлення у форматі XML;
+ `org.apache.log4j.TTCCLayout` — на виході повідомлення доповнюється інформацією про час, потік, ім'я логера та вкладений діагностичний контекст (NDC);
+ `org.apache.log4j.PatternLayout` / `org.apache.log4j.EnhancedPatternLayout` — налаштування форматування повідомлення за допомогою шаблону, заданого користувачем (найгнучкіший і найуживаніший варіант на практиці).

## Перелічіть рівні логування у log4j. Назвіть порядок їх пріоритетності.
Стандартні рівні (від найвищого пріоритету до найнижчого):

+ **OFF** — логування вимкнено;
+ **FATAL** — фатальна помилка;
+ **ERROR** — помилка;
+ **WARN** — попередження;
+ **INFO** — інформаційне повідомлення;
+ **DEBUG** — детальна інформація для налагодження;
+ **TRACE** — найдетальніше трасування (з'явився у log4j 1.2.12).

Повний порядок пріоритетів між рівнями такий:

`ALL < TRACE < DEBUG < INFO < WARN < ERROR < FATAL < OFF`

Як це працює: коли для логера встановлено рівень, обробляються лише повідомлення цього рівня **та вищих**. Наприклад, за рівня `INFO` пройдуть `INFO`, `WARN`, `ERROR`, `FATAL`, а `DEBUG` і `TRACE` буде відфільтровано. `ALL` пропускає все, `OFF` — нічого.

> У сучасних SLF4J/Logback набір рівнів простіший: `TRACE < DEBUG < INFO < WARN < ERROR` (без окремих `FATAL`, `ALL`, `OFF` — фатальні помилки логують як `ERROR`).

## Які існують способи конфігурування log4j?
Щоб log4j почав працювати, йому потрібно надати конфігурацію. Це можна зробити кількома шляхами:

+ **Програмно** — у коді отримати logger, задати рівень логування, прикріпити appender і визначити спосіб форматування (layout).
+ **Через системну властивість** — указати файл або URL як аргумент під час запуску JVM: `-Dlog4j.configuration=шлях/до/файлу/конфігурації`, а потім прочитати його у програмі за допомогою `PropertyConfigurator.configure(...)` чи `DOMConfigurator.configure(...)` — для формату `.properties` або `XML` відповідно.
+ **Автоматичним пошуком у classpath** — log4j сам шукає файл конфігурації у classpath: спочатку `log4j.xml`, а якщо його не знайдено — `log4j.properties`.

> У Log4j 2 механізм інший: за замовчуванням шукаються файли `log4j2.xml`/`log4j2.properties` (а також підтримуються YAML та JSON), і конфігурація може автоматично перечитуватися без перезапуску застосунку. Logback використовує `logback.xml`.
