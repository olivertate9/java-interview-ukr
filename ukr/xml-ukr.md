## Що таке _XML_?
__XML, eXtensible Markup Language (розширювана мова розмітки)__ — мова з простим формальним синтаксисом, яка добре пристосована для створення й обробки документів програмами і водночас зручна для читання та створення документів людиною.

Ментальна модель: XML — це не формат із фіксованим набором тегів (як HTML), а *метамова* для опису власних форматів. Розробник сам визначає розмітку відповідно до потреб конкретної предметної області, обмежений лише синтаксичними правилами самої мови.

> На практиці у новій розробці XML здебільшого витіснений JSON для обміну даними через REST API (менш багатослівний, простіший для людини й машини). XML досі панує там, де потрібні строгі схеми, простори імен і трансформації: конфігурації (Spring, Maven `pom.xml`), документні стандарти (DOCX, SVG), SOAP-сервіси та обмін даними в enterprise/legacy-системах.

## Що таке _DTD_?
__DTD, Document Type Definition (визначення типу документа)__ — це наперед визначений набір правил, який задає допустимі зв'язки між елементами та атрибутами (тобто граматику документа).

> Наприклад, DTD для HTML стверджує, що тег `DIV` має бути всередині тега `BODY` і може траплятися багаторазово, `TITLE` — у `HEAD` і лише один раз, а `SCRIPT` — і там, і там скільки завгодно разів.

DTD зазвичай описується або безпосередньо в документі у вигляді конструкції, що починається з `<!DOCTYPE ... >`, або в окремому файлі.

> На співбесіді варто згадати: у сучасних проєктах DTD майже не використовують — його витіснив XSD (XML Schema), бо DTD не підтримує типи даних, простори імен і має власний синтаксис (не є XML). Крім того, обробка зовнішніх DTD — джерело відомої вразливості XXE (XML External Entity), тому в більшості парсерів її варто вимикати.

## Чим _well-formed XML_ відрізняється від _valid XML_?
Залежно від рівня відповідності стандартам документ може бути «well-formed» («правильно побудований») або «valid» («дійсний»). Це два різні рівні перевірки, і valid завжди передбачає well-formed.

Основні ознаки _well-formed XML_ випливають із формального опису стандарту:

+ Документ має рівно один кореневий елемент, у якому лежать усі інші. Тобто `<document>...</document><appendix>...</appendix>` — це не XML-документ.
+ Усі відкриті теги обов'язково мають бути закриті. HTML, наприклад, дозволяє не закривати багато тегів (`<p>`, `<li>`, `<td>` та інші). У XML так робити не можна.
+ Для одиночних (порожніх) тегів передбачено спеціальний запис — самозакривний тег: `<br/>`. Але можна написати й повністю: `<br></br>`.
+ Імена тегів чутливі до регістру. Якщо ви відкриваєте тег `<SiteDescription>`, то закривати його треба точно так само; `</sitedescription>` неприпустимо.
+ Теги не можуть порушувати вкладеність. Ось такого бути не повинно: `<em><b>...</em></b>`.
+ Усі значення атрибутів обов'язково беруться в лапки — подвійні (`"`) або одинарні (`'`).
+ Є три символи — `<`, `>` і `&`, — які мають бути екрановані за допомогою `&lt;`, `&gt;` і `&amp;`. Усередині значень атрибутів треба екранувати ще й лапки, у які цей атрибут узятий (`&quot;` для `"` або `&apos;` для `'`).
+ Усі символи в документі мають відповідати заявленому кодуванню.

Документ є _valid_, якщо він не лише well-formed, а ще й відповідає наперед заданій схемі (_DTD_ або _XSD_): правильні імена елементів, порядок і кількість вкладень, типи даних тощо.

__Коротко: _well-formed XML_ коректний синтаксично (його може розібрати парсер), а _valid XML_ коректний і синтаксично, і семантично (задовольняє правила наперед описаних словника й граматики — DTD/XSD).__

## Що таке «_простір імен_» в XML?
__Простір імен XML (XML namespace)__ — це колекція імен, ідентифікована за допомогою посилання URI, що використовується в XML-документах для позначення типів елементів та іменування атрибутів.

Навіщо він потрібен: один XML-документ може поєднувати елементи з кількох незалежних словників (наприклад, XHTML + SVG + власна розмітка). Простори імен розв'язують проблему колізій — коли в різних словниках є елементи з однаковими іменами (скажімо, `<table>` у HTML і `<table>` як «меблі»).

> Простори імен оголошуються за допомогою атрибута `xmlns`, значенням якого має бути _URI_, та префікса, що однозначно ідентифікує простір імен для елементів.

```xml
<root xmlns:h="http://www.w3.org/TR/html4/"
      xmlns:f="https://example.com/furniture">
    <h:table>
        <h:tr><h:td>Apples</h:td></h:tr>
    </h:table>
    <f:table>
        <f:width>80</f:width>
    </f:table>
</root>
```

Важливо: сам URI слугує лише унікальним ідентифікатором — за ним нічого не «завантажується», і він не зобов'язаний вести на реальний ресурс чи мати визначений словник.

## Що таке XSD? У чому його переваги над XML DTD?
__XSD, XML Schema Definition (XML-схема)__ — мова опису структури XML-документа. Зокрема, XML Schema описує:

+ _словник_ — імена елементів та атрибутів;
+ _модель вмісту_ — взаємозв'язки між елементами й атрибутами та структуру документа;
+ використовувані _типи даних_.

__Переваги XSD над DTD__ полягають у наступному:

+ XSD сам написаний у форматі XML і має звичний синтаксис, тоді як DTD — це окрема мова з власним синтаксисом. Завдяки цьому XSD можна обробляти тими самими інструментами, що й будь-який XML, і немає окремих проблем із кодуванням.

+ За використання XSD парсер може перевірити не лише синтаксис, а й структуру, модель вмісту та типи даних. У DTD існує фактично один тип — текст; якщо в числовому полі опиниться текст, документ усе одно пройде валідацію, бо DTD не вміє перевіряти типи.

+ XSD має багату систему типів (числа, дати, перелічення, обмеження діапазонів і шаблони через регулярні вирази тощо) і дозволяє визначати власні типи.

+ XSD підтримує простори імен, чого DTD не вміє.

+ До одного документа можна застосувати кілька схем (через простори імен), що дає змогу переюзовувати словники для типових задач.

## Які типи існують у XSD?
__Простий тип (simple type)__ — це визначення типу для значення, яке може бути вмістом елемента або значенням атрибута. Такий тип не може містити вкладені елементи чи мати атрибути.

```xsd
<xsd:element name="price" type="xsd:decimal"/>
...
<price>45.50</price>
```

__Складний тип (complex type)__ — це визначення типу для елементів, які можуть містити атрибути та/або інші вкладені елементи.

```xsd
<!-- Елемент <price> зі значенням-числом і атрибутом currency.
     Для розширення простого типу атрибутами використовується
     simpleContent + extension. -->
<xsd:element name="price">
    <xsd:complexType>
        <xsd:simpleContent>
            <xsd:extension base="xsd:decimal">
                <xsd:attribute name="currency" type="xsd:string"/>
            </xsd:extension>
        </xsd:simpleContent>
    </xsd:complexType>
</xsd:element>
...
<price currency="USD">45.50</price>
```

> Примітка: в оригінальному прикладі було `<xsd:complexType base='xsd:decimal'>` — це некоректний XSD (атрибут `base` у `complexType` не існує). Щоб елемент мав текстове значення *і* атрибут, використовують `simpleContent` з `extension`, як показано вище.

## Які ви знаєте методи читання XML? Опишіть сильні та слабкі сторони кожного методу.
__DOM (Document Object Model)__ — _об'єктний_ підхід — зчитує весь XML і відтворює його в пам'яті у вигляді деревоподібної об'єктної структури: документ — це набір вузлів (тегів), кожен з яких може мати необмежену кількість дочірніх вузлів. У підсумку виходить повне дерево документа.

> ➖ Низька швидкість.
> ➖ Витрачає багато пам'яті (весь документ тримається в пам'яті, тому великі файли можуть призвести до `OutOfMemoryError`).
> ➕ Простий у програмуванні.
> ➕ Дає довільний доступ до будь-якого вузла та дозволяє навігацію в обидва боки.
> ➕ Якщо в XML багато об'єктів із перехресними посиланнями, достатньо двічі пройтися деревом: спершу створити об'єкти й заповнити словник «назва-об'єкт», потім відновити посилання.
> ➕ Придатний і для читання, і для запису/модифікації.

__SAX (Simple API for XML)__ — _подієвий_ (push) підхід — читає XML, реагуючи на події (відкривний чи закривний тег, текст, атрибут) викликом наданих застосунком обробників (callback). На відміну від DOM, не тримає документ у пам'яті.

> ➕ Висока швидкість.
> ➕ Витрачає мало пам'яті — обробляє документ будь-якого розміру потоково.
> ➗ Доволі складний у програмуванні: стан доводиться відстежувати вручну.
> ➖ Немає довільного доступу — лише послідовний прохід уперед, без можливості повернутися назад.
> ➖ Якщо в XML багато перехресних посилань, доводиться вручну організовувати тимчасове зберігання, щоб потім зв'язати об'єкти.
> ➖ Придатний лише для читання.

__StAX (Streaming API for XML)__ — _потоковий_ (pull) підхід, доданий до стандартної бібліотеки в Java 6. Складається з двох рівнів API. Низькорівневий API на основі курсора (`XMLStreamReader`) дозволяє працювати з XML як із потоком лексем: застосунок сам запитує наступну подію й читає інформацію про неї. Високорівневий API на основі ітератора подій (`XMLEventReader`) подає XML як послідовність об'єктів-подій.

Ключова різниця між SAX і StAX: у SAX *парсер* керує процесом і «штовхає» події застосунку (push), а в StAX *застосунок* сам «витягує» наступну подію тоді, коли готовий (pull). Це робить StAX простішим у використанні й дозволяє легко зупинити читання посеред документа.

> ➗ Зберігає переваги SAX над DOM (швидкість, економію пам'яті).
> ➕ Не базується на callback-ах: застосунку не доводиться вручну тримати емульований стан парсера, як у SAX.
> ➕ Може як читати, так і записувати XML (через `XMLStreamWriter`), на відміну від «лише читання» в SAX.

## Коли слід використовувати _DOM_, а коли _SAX_, _StAX_ аналізатори?

DOM — природний вибір, коли предметом роботи є сама структура XML: коли потрібно знати її повністю, навігувати в довільному порядку, змінювати документ або багаторазово звертатися до різних його частин. Обмеження — розмір файлу має вміщатися в пам'ять.

SAX або StAX оптимальні для швидкого одноразового послідовного читання, особливо великих файлів, які не помістилися б у пам'ять цілком. Між ними зазвичай обирають StAX як простіший і гнучкіший (pull-модель), а SAX — коли потрібен максимальний контроль або робота з legacy-кодом.

## Які ви знаєте способи запису XML?

__Пряма (потокова) запис__ — пише XML тег за тегом, атрибут за атрибутом (наприклад, через `XMLStreamWriter` зі StAX).

> ➕ Висока швидкість.
> ➕ Економія пам'яті: не створюються проміжні об'єкти.
> ➖ Придатний лише для запису, послідовно вперед.

__Запис через DOM__ — спершу будує в пам'яті повну структуру XML, а потім серіалізує її.

> ➖ Нижча швидкість.
> ➖ Більше витрат пам'яті.
> ➕ Зручний, коли структуру треба формувати/змінювати в довільному порядку перед записом, і придатний як для запису, так і для читання.

## Що таке _JAXP_?
__JAXP, Java API for XML Processing (Java API для обробки XML)__ — набір стандартних API (входить до JDK), що уніфікує обробку XML у Java. Містить реалізації парсерів DOM, SAX і StAX, підтримує перетворення XSLT (пакет `javax.xml.transform`) та роботу зі схемами/DTD. Головна ідея JAXP — абстрагувати застосунок від конкретної реалізації парсера: код пишеться проти стандартного API, а конкретний парсер підключається через фабрики (наприклад, `DocumentBuilderFactory`).

[до змісту](#xml)

## Що таке _XSLT_?
__XSLT, eXtensible Stylesheet Language Transformations__ — мова перетворення XML-документів.

XSLT створювалася як частина _XSL (eXtensible Stylesheet Language)_ — мови стилів для XML. Під час перетворення XSLT-процесор зчитує вхідний XML-документ і таблицю(і) стилів XSLT і на основі правил із таблиці стилів формує новий результат — інший XML-документ, HTML або звичайний текст.

> Ментальна модель: XSLT — це декларативна мова трансформації «дерево XML → інше дерево/текст». Вона описує не покрокові інструкції, а *правила-шаблони* (`<xsl:template match="...">`), які зіставляються з вузлами вхідного документа. Вузли вибираються виразами мови _XPath_.
