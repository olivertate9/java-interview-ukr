## Які нововведення з'явилися в Java 8 та JDK 8?
Java 8 (березень 2014) — найбільший крок мови у бік функціонального стилю. Ментальна модель: код тепер можна передавати як дані (лямбди), а обробку колекцій описувати декларативно (Stream API), а не циклами.

Ключові нововведення:

+ методи інтерфейсів за замовчуванням (`default`-методи);
+ лямбда-вирази;
+ функціональні інтерфейси;
+ посилання на методи та конструктори (method/constructor references);
+ повторювані анотації (`@Repeatable`);
+ анотації на типи (type annotations);
+ рефлексія для параметрів методів (збереження імен параметрів);
+ _Stream API_ для роботи з колекціями;
+ паралельне сортування масивів (`Arrays.parallelSort()`);
+ нове API для роботи з датами й часом (`java.time`, JSR-310);
+ новий рушій JavaScript _Nashorn_ (зауваження: deprecated у Java 11, повністю прибраний у Java 15);
+ кілька нових класів для потокобезпечної роботи (зокрема `LongAdder`, `LongAccumulator`, `StampedLock`);
+ оновлене API для `Calendar` та `Locale`;
+ підтримка _Unicode 6.2.0_;
+ стандартний клас для роботи з _Base64_;
+ підтримка беззнакової арифметики (`Integer.parseUnsignedInt`, `Long.divideUnsigned` тощо);
+ покращена продуктивність конструктора `java.lang.String(byte[], *)` та методу `java.lang.String.getBytes()`;
+ нова реалізація `AccessController.doPrivileged`, що дозволяє встановлювати підмножину привілеїв без перевірки решти рівнів доступу;
+ алгоритми на основі пароля (_password-based_) стали стійкішими;
+ підтримка _SSL/TLS Server Name Indication (SNI)_ у _JSSE Server_; <!-- виправлено: в оригіналі помилково «NSI» -->
+ покращене сховище ключів (`KeyStore`);
+ доданий алгоритм _SHA-224_;
+ прибраний міст _JDBC – ODBC_;
+ прибраний _PermGen_, метадані класів тепер зберігаються у _Metaspace_ (нативна пам'ять, поза купою);
+ можливість створювати компактні профілі (compact profiles) платформи Java SE, що включають не всю платформу, а її частину;
+ інструментарій:
    + утиліта `jjs` для запуску _JavaScript Nashorn_;
    + команда `java` може запускати _JavaFX_-застосунки;
    + утиліта `jdeps` для аналізу _.class_-файлів та залежностей.

## Що таке _«лямбда»_? Яка структура та особливості використання лямбда-виразу?
__Лямбда__ — це набір інструкцій (фактично анонімна функція), який можна винести в окрему змінну й потім багаторазово викликати в різних місцях програми.

Основу лямбда-виразу складає _лямбда-оператор_ — стрілка `->`. Вона розділяє вираз на дві частини: ліворуч — список параметрів, праворуч — тіло лямбди, де виконуються дії.

Лямбда-вираз не виконується сам собою — він утворює реалізацію єдиного абстрактного методу _функціонального інтерфейсу_. Тобто функціональний інтерфейс повинен містити рівно один абстрактний метод (без реалізації).

```java
interface Operationable {
    int calculate(int x, int y);
}

public static void main(String[] args) {
    Operationable operation = (x, y) -> x + y;
    int result = operation.calculate(10, 20);
    System.out.println(result); //30
}
```

По суті лямбда-вирази — це стисліша форма анонімних внутрішніх класів, які застосовувалися для цього раніше. Важливе зауваження: лямбда — це НЕ просто синтаксичний цукор над анонімним класом. На відміну від анонімного класу, `this` усередині лямбди посилається на зовнішній екземпляр, а не на саму лямбду, і компілятор реалізує лямбди через `invokedynamic`, а не через генерацію окремого class-файлу.

+ _Відкладене виконання (deferred execution)_ — лямбда визначається один раз в одному місці, а викликається за потреби, будь-яку кількість разів і в довільному місці програми.

+ _Параметри лямбда-виразу_ мають відповідати за типом параметрам методу функціонального інтерфейсу:

```java
operation = (int x, int y) -> x + y;
//При написанні лямбди типи параметрів можна не вказувати — вони виводяться компілятором:
(x, y) -> x + y;
//Якщо метод не приймає параметрів, пишуться порожні дужки, наприклад:
() -> 30 + 20;
//Якщо метод приймає рівно один параметр, дужки можна опустити:
n -> n * n;
```

+ _Лямбди для методів, що нічого не повертають_, можуть не повертати жодного значення:

```java
interface Printable {
    void print(String s);
}

public static void main(String[] args) {
    Printable printer = s -> System.out.println(s);
    printer.print("Hello, world");
}
```

+ _Блочні лямбда-вирази_ обрамлюються фігурними дужками. У них можна використовувати вкладені блоки, цикли, конструкції `if`, `switch`, оголошувати змінні тощо. Якщо блочна лямбда має повертати значення, явно застосовується оператор `return`:

```java
Operationable operation = (int x, int y) -> {
    if (y == 0) {
        return 0;
    } else {
        return x / y;
    }
};
```

+ _Передача лямбда-виразу як параметра методу_:

```java
interface Condition {
    boolean isAppropriate(int n);
}

private static int sum(int[] numbers, Condition condition) {
    int result = 0;
    for (int i : numbers) {
        if (condition.isAppropriate(i)) {
            result += i;
        }
    }
    return result;
}

public static void main(String[] args) {
    System.out.println(sum(new int[] {0, 1, 0, 3, 0, 5, 0, 7, 0, 9}, (n) -> n != 0));
}
```

## До яких змінних має доступ лямбда-вираз?
Доступ до змінних зовнішньої області видимості з лямбди дуже схожий на доступ із анонімних класів. Можна посилатися на:

+ незмінні (_effectively final_ — не обов'язково помічені `final`, але такі, що після ініціалізації не змінюються) локальні змінні;
+ поля класу;
+ статичні змінні.

Спроба змінити захоплену локальну змінну з лямбди не компілюється — саме тому вона має бути effectively final. Поля класу та статичні змінні змінювати дозволено (їх захоплено через посилання на об'єкт/клас, а не за значенням).

## Як відсортувати список рядків за допомогою лямбда-виразу?
```java
public static List<String> sort(List<String> list) {
    list.sort((a, b) -> a.compareTo(b));
    return list;
}
```

На практиці краще скористатися готовим компаратором замість лямбди:

```java
list.sort(Comparator.naturalOrder());
// або з посиланням на метод:
list.sort(String::compareTo);
```

> Зауваження: `List.sort(Comparator)` з'явився в Java 8; раніше використовували `Collections.sort(list, comparator)` — він теж працює.

## Що таке «посилання на метод» (method reference)?
Якщо наявний метод уже робить усе потрібне, можна скористатися механізмом __method reference (посилання на метод)__ і передати цей метод напряму, не загортаючи його в лямбду. Посилання записується так:

+ `ім'я_класу::ім'я_статичного_методу` — для статичного методу;
+ `об'єкт::ім'я_методу` — для методу конкретного екземпляра;
+ `ім'я_класу::ім'я_методу` — для методу екземпляра, що передається першим аргументом (напр. `String::length`);
+ `ім'я_класу::new` — для конструктора.

Результат буде точно таким самим, як у випадку лямбди, що викликає цей метод.

```java
private interface Measurable {
    int length(String string);
}

public static void main(String[] args) {
    Measurable a = String::length;
    System.out.println(a.length("abc")); // 3
}
```

Посилання на методи зазвичай читабельніші за еквівалентні лямбди й дають компілятору точнішу інформацію про типи. Тож за можливості вибору між посиланням на наявний метод і лямбдою варто надавати перевагу посиланню на метод.

## Що таке «функціональні інтерфейси»?
__Функціональний інтерфейс__ — це інтерфейс, що визначає рівно один абстрактний метод. Саме тому його екземпляр можна створити лямбдою або посиланням на метод.

Щоб явно позначити інтерфейс як функціональний, додано анотацію `@FunctionalInterface`, що працює за принципом `@Override`: вона фіксує намір і змушує компілятор видати помилку, якщо в інтерфейсі з'явиться другий абстрактний метод.

Інтерфейс може містити скільки завгодно `default`- і `static`-методів та все одно лишатися функціональним, бо вони не абстрактні. Також не рахуються абстрактні методи, що повторюють публічні методи `java.lang.Object` (наприклад, `equals`).

## Для чого потрібні функціональні інтерфейси `Function<T,R>`, `DoubleFunction<R>`, `IntFunction<R>` та `LongFunction<R>`?
__`Function<T, R>`__ — інтерфейс, що реалізує функцію, яка отримує на вхід об'єкт типу `T` і повертає об'єкт типу `R` (єдиний абстрактний метод — `apply`).

```java

Function<String, String> backToString = toInteger.andThen(String::valueOf);
backToString.apply("123");     // "123"
```

+ `DoubleFunction<R>` — приймає примітив `double`, повертає об'єкт типу `R`;
+ `IntFunction<R>` — приймає примітив `int`, повертає об'єкт типу `R`;
+ `LongFunction<R>` — приймає примітив `long`, повертає об'єкт типу `R`.

> Ці спеціалізації працюють із примітивами, а не з обгортками `Double`/`Integer`/`Long`, — саме тому вони й існують: щоб уникати автобоксингу.

## Що таке `default`-методи інтерфейсу?
Java 8 дозволяє додавати в інтерфейс неабстрактні реалізації методів за допомогою ключового слова `default`. Головна мотивація — еволюція API: можна додати новий метод до інтерфейсу (наприклад, `Collection.stream()`), не зламавши всі наявні реалізації.

```java
interface Example {
    int process(int a);
    default void show() {
        System.out.println("default show()");
    }
}
```

+ Якщо клас реалізує інтерфейс, він може, але не зобов'язаний, перевизначати `default`-методи — за замовчуванням він успадковує їхню реалізацію.
+ Якщо клас реалізує кілька інтерфейсів з однаковим `default`-методом (конфлікт «успадкування ромбом»), клас зобов'язаний перевизначити цей метод самостійно. Аналогічно, якщо в одному інтерфейсі метод `default`, а в іншому той самий метод абстрактний — реалізація за замовчуванням не успадковується.
+ `default`-метод не може перевизначати методи класу `java.lang.Object` (наприклад, `toString`, `equals`).
+ Дозволяють розширювати інтерфейси без ризику зламати наявні класи.
+ Дозволяють уникати службових класів-утиліт, бо допоміжні методи можуть жити в самому інтерфейсі.
+ Однією з основних причин впровадження `default`-методів була можливість додати в Java 8 методи на кшталт `forEach`, `stream`, `removeIf` прямо в інтерфейси колекцій.

[до змісту](#java-8)


## Що таке `static`-метод інтерфейсу?
Статичні методи інтерфейсу (з Java 8) схожі на `default`-методи, але належать самому інтерфейсу і не успадковуються та не перевизначаються в класах-реалізаціях.

+ Статичний метод є частиною інтерфейсу; його не можна перевизначити для об'єктів класу-реалізації;
+ методи класу `java.lang.Object` не можна оголосити статичними в інтерфейсі;
+ статичні методи інтерфейсу зручні для допоміжних/фабричних операцій, наприклад перевірки на `null`, порівняння, створення компараторів (`Comparator.comparing(...)`) тощо.

## Що таке `Optional`?
Ментальна модель: `Optional<T>` — це коробка, яка містить рівно нуль або один елемент. Це контейнер, що явно сигналізує «значення може бути відсутнім», замість прихованого `null`. Такий підхід допомагає уникати `NullPointerException` і робить «можливу відсутність» частиною типу.

> Важливо: `Optional.of(value)` кидає `NullPointerException`, якщо передати `null`. Щоб обгорнути значення, яке може бути `null`, використовуйте `Optional.ofNullable(value)`, а порожній — `Optional.empty()`. Тобто твердження «контейнер, що може містити значення `null`» некоректне — у непорожньому `Optional` значення завжди не-`null`.

```java
Optional<String> optional = Optional.of("hello");

optional.isPresent();                               // true
optional.ifPresent(s -> System.out.println(s.length())); // надрукує 5
optional.get();                                     // "hello"
optional.orElse("ops...");                          // "hello"
```

Замість `isPresent()` + `get()` ідіоматичніше використовувати `map`, `filter`, `orElse`, `orElseGet`, `orElseThrow`. `Optional` призначений насамперед для типів повернення методів; зберігати його в полях чи передавати як параметр зазвичай не рекомендують.

[до змісту](#java-8)

## Що таке `Stream`?
Ментальна модель: `Stream` — це не структура даних, а конвеєр обробки. Він описує послідовність операцій над елементами джерела, не зберігаючи самі елементи.

`java.util.stream.Stream` представляє послідовність елементів, над якою можна виконувати різні операції. Операції бувають _проміжними (intermediate)_ або _термінальними/конечними (terminal)_. Проміжні операції повертають новий стрім (тож їх можна об'єднувати в ланцюжок), а термінальна — повертає результат або викликає побічний ефект і завершує конвеєр.

Стрім може мати скільки завгодно проміжних операцій і завершується однією термінальною. Усі проміжні операції _ліниві_: поки не викликано термінальну операцію, реальної обробки не відбувається (аналогія: створення `Thread`/`Runnable` без виклику `start()`).

Стріми створюються на основі джерел — наприклад, колекцій з `java.util.Collection` (метод `stream()`).

`Map` (`HashMap` тощо) не має методу `stream()` напряму, але стрімити можна її представлення: `map.entrySet().stream()`, `map.keySet().stream()`, `map.values().stream()`.

Операції над стрімами можуть виконуватися послідовно або паралельно.

Стрім не можна використати повторно: після виклику термінальної операції стрім вважається спожитим, і повторна спроба кидає `IllegalStateException`.

Крім універсальних об'єктних, існують спеціальні примітивні стріми для `int`, `long` і `double`: `IntStream`, `LongStream` і `DoubleStream`. Вони працюють так само, як об'єктні, але:

+ використовують спеціалізовані функціональні інтерфейси (`IntFunction`, `IntPredicate` тощо) замість `Function`/`Predicate` — без боксингу;
+ мають додаткові термінальні операції `sum()`, `average()`, `summaryStatistics()` та проміжну `mapToObj()` для повернення до об'єктного стріму.

[до змісту](#java-8)

## Якими способами можна створити стрім?
1. З колекції:
```java
Stream<String> fromCollection = Arrays.asList("x", "y", "z").stream();
```
2. З набору значень:
```java
Stream<String> fromValues = Stream.of("x", "y", "z");
```
3. З масиву:
```java
Stream<String> fromArray = Arrays.stream(new String[]{"x", "y", "z"});
```
4. З файлу (кожний рядок файлу — окремий елемент стріму):
```java
Stream<String> fromFile = Files.lines(Paths.get("input.txt"));
```
5. З рядка (стрім кодів символів):
```java
IntStream fromString = "0123456789".chars();
```
6. За допомогою `Stream.builder()`:
```java
Stream<String> fromBuilder = Stream.<String>builder().add("x").add("y").add("z").build();
```
7. За допомогою `Stream.iterate()` (нескінченний):
```java
Stream<Integer> fromIterate = Stream.iterate(1, n -> n + 1);
```
8. За допомогою `Stream.generate()` (нескінченний):
```java
Stream<String> fromGenerate = Stream.generate(() -> "0");
```

> Нескінченні стріми (`iterate`/`generate`) обов'язково треба обмежувати, наприклад `limit(n)`, інакше термінальна операція ніколи не завершиться.

## У чому різниця між `Collection` та `Stream`?
Колекція дозволяє працювати з елементами поодинці (додавати, видаляти, звертатися за індексом), а стрім — ні; натомість стрім надає змогу декларативно описати обробку даних як єдиного цілого.

Різниця і в самій суті понять: `Collection` — це передусім _структура даних_, сховище елементів (наприклад, `Set` реалізує ідею множини унікальних елементів). `Stream` — це абстракція _конвеєра обчислень_, тому результатом його роботи є вже якась структура даних або підсумок (число, прапорець, агрегат тощо).

| | `Collection` | `Stream` |
|---|---|---|
| Призначення | зберігати дані | обробляти дані |
| Доступ до елементів | поодинці, багаторазово | лише через конвеєр, одноразово |
| Обчислення | активне (eager) | ліниве (lazy) |
| Скінченність | завжди скінченна | може бути нескінченним |
| Зміна даних | модифікується | не змінює джерело |

[до змісту](#java-8)

## Для чого потрібен метод `collect()` у стрімах?
`collect()` — це термінальна операція _змінного згортання_ (mutable reduction): вона збирає елементи стріму в колекцію чи іншу структуру даних.

`collect()` приймає `Collector<ТипЕлемента, ТипАкумулятора, ТипРезультату>`, що складається з кількох частин: _supplier_ — створення порожнього акумулятора, _accumulator_ — додавання елемента до акумулятора, _combiner_ — об'єднання двох акумуляторів при паралельному виконанні, _finisher_ — необов'язкове фінальне перетворення акумулятора. У класі `Collectors` реалізовано чимало готових колекторів:

+ `toList()`, `toSet()`, `toCollection()` — збирають у список, множину чи задану колекцію;
+ `toMap()`, `toConcurrentMap()` — збирають у `Map`;
+ `averagingInt()`, `averagingDouble()`, `averagingLong()` — обчислюють середнє;
+ `summingInt()`, `summingDouble()`, `summingLong()` — обчислюють суму;
+ `summarizingInt()`, `summarizingDouble()`, `summarizingLong()` — повертають `SummaryStatistics` (count, sum, min, max, average);
+ `partitioningBy()` — ділить елементи на дві групи за умовою, повертає `Map<Boolean, List<T>>`;
+ `groupingBy()` — групує елементи за ключем, повертає `Map<K, List<T>>`;
+ `mapping()`, `joining()`, `reducing()` — додаткові перетворення для складених колекторів.

Можна створити власний колектор через `Collector.of()`:

```java
Collector<String, ?, List<String>> toList = Collector.of(
    ArrayList::new,
    List::add,
    (l1, l2) -> { l1.addAll(l2); return l1; }
);
```

## Для чого у стрімах призначений метод `limit()`?

## Для чого у стрімах призначений метод `sorted()`?
`sorted()` — проміжна (stateful) операція, що сортує елементи в натуральному порядку або за заданим `Comparator`.

Джерело лишається незмінним — `sorted()` лише створює відсортоване представлення в межах конвеєра. Зверніть увагу: щоб відсортувати, операція мусить спочатку зібрати всі елементи, тож для нескінченних стрімів без попереднього `limit()` вона не завершиться.

## Для чого у стрімах призначені методи `flatMap()`, `flatMapToInt()`, `flatMapToDouble()`, `flatMapToLong()`?
`flatMap()` схожий на `map()`, але кожний елемент перетворює на _стрім_ елементів, а потім «сплющує» всі ці підстріми в один. Тобто з одного елемента може вийти нуль, один або багато елементів. Найтиповіше застосування — розгортання вкладених колекцій (список списків → плаский стрім).

```java
Stream
    .of("H e l l o", "w o r l d !")
    .flatMap((p) -> Arrays.stream(p.split(" ")))
    .toArray(String[]::new); //["H", "e", "l", "l", "o", "w", "o", "r", "l", "d", "!"]
```

`flatMapToInt()`, `flatMapToDouble()`, `flatMapToLong()` — аналоги `flatMap()`, що повертають відповідний примітивний стрім.

## Розкажіть про паралельну обробку в Java 8.
Стріми бувають послідовними й паралельними. Послідовний стрім обробляється в одному потоці, паралельний — у кількох. Паралельні стріми за замовчуванням використовують спільний пул `ForkJoinPool.commonPool()`. Дані розбиваються на частини (spliterator), кожна частина обробляється окремо, після чого результати об'єднуються.

Створити паралельний стрім можна методом `parallelStream()` інтерфейсу `Collection` або викликом `parallel()` на наявному стрімі. `isParallel()` показує, чи стрім паралельний. Методами `parallel()` і `sequential()` можна перемикати режим; стан застосовується до всього конвеєра при термінальній операції (а не до окремих ділянок, як могло б здатися):

```java
collection
    .stream()
    .peek(...)
    .parallel()
    .map(...)
    .sequential()
    .reduce(...);
// Незалежно від розташування викликів, увесь конвеєр виконається в режимі останнього з них — тут sequential().
```

Зазвичай елементи зберігають _encounter order_ навіть у паралельних стрімах. Виняток — `forEach()`, що може видавати елементи в довільному порядку; щоб зберегти порядок, застосовують `forEachOrdered()`.

Критерії, що впливають на продуктивність паралельних стрімів:

+ **Обсяг даних.** На малих обсягах накладні витрати на розбиття/об'єднання переважають виграш — паралелізм лише шкодить.
+ **Кількість ядер.** На одноядерній машині паралельний стрім фактично виконується послідовно, тож сенсу не має.
+ **Структура джерела.** Дані з `ArrayList` чи масиву легко розбиваються на рівні частини, тому паралеляться добре. `LinkedList`, `TreeMap`-діапазони чи стрім із `iterate()` розбиваються погано.
+ **Примітиви vs об'єкти.** Операції над примітивними стрімами зазвичай швидші за об'єктні (немає боксингу).
+ **Блокувальні операції.** Дуже не рекомендується використовувати паралельні стріми для тривалих/блокувальних операцій (мережа, I/O): усі вони ділять спільний `commonPool`, тож одна довга задача може «заморозити» паралельні стріми по всій JVM через брак вільних потоків. Паралельні стріми доречні лише для коротких CPU-bound операцій.
+ **Збереження порядку** додає накладні витрати; якщо порядок неважливий, його можна вимкнути проміжною операцією `unordered()`, що може прискорити обробку.

```java
collection.parallelStream()
    .filter(...)
    .unordered()
    .collect(Collectors.toList());
```

## Які термінальні (конечні) методи роботи зі стрімами ви знаєте?
+ `findFirst()` — повертає `Optional` з першим елементом;
+ `findAny()` — повертає `Optional` з будь-яким елементом (зручно в паралельних стрімах);
+ `collect()` — збирає результат у колекцію чи іншу структуру;
+ `count()` — повертає кількість елементів;
+ `anyMatch()` — `true`, якщо умова виконується хоча б для одного елемента;
+ `noneMatch()` — `true`, якщо умова не виконується для жодного елемента;
+ `allMatch()` — `true`, якщо умова виконується для всіх елементів;
+ `min()` — повертає `Optional` з мінімальним елементом за `Comparator`;
+ `max()` — повертає `Optional` з максимальним елементом за `Comparator`;
+ `forEach()` — застосовує дію до кожного елемента (порядок у паралельному стрімі не гарантується);
+ `forEachOrdered()` — застосовує дію до кожного елемента зі збереженням порядку;
+ `toArray()` — повертає масив значень;
+ `reduce()` — згортає елементи в один результат заданою бінарною операцією.

Для примітивних стрімів додатково доступні:

+ `sum()` — сума всіх чисел;
+ `average()` — середнє арифметичне (повертає `OptionalDouble`);
+ `summaryStatistics()` — count/sum/min/max/average одним викликом.

## Які додаткові методи для роботи з асоціативними масивами (maps) з'явилися в Java 8?
+ `putIfAbsent()` — додає пару «ключ-значення», лише якщо ключа ще немає (або його значення `null`):

`map.putIfAbsent("a", "Aa");`

+ `forEach()` — приймає `BiConsumer` і виконує дію над кожною парою:

`map.forEach((k, v) -> System.out.println(v));`

+ `compute()` — обчислює нове значення на основі ключа та поточного значення (поточне може бути `null`):

`map.compute("a", (k, v) -> String.valueOf(k).concat(v)); //["a", "aAa"]`

+ `computeIfPresent()` — якщо ключ існує (з не-`null` значенням), оновлює значення на обчислене:

`map.computeIfPresent("a", (k, v) -> k.concat(v));`

+ `computeIfAbsent()` — якщо ключа немає (або значення `null`), створює значення за функцією від ключа:

`map.computeIfAbsent("a", k -> "A".concat(k)); //["a","Aa"]`

+ `getOrDefault()` — повертає значення за ключем або задане значення за замовчуванням, якщо ключа немає:

`map.getOrDefault("a", "not found");`

+ `merge()` — приймає ключ, значення й функцію злиття. Якщо значення під ключем відсутнє — записує передане; інакше об'єднує наявне й передане функцією. Якщо функція повертає `null`, запис видаляється:

`map.merge("a", "z", (oldValue, newValue) -> oldValue.concat(newValue)); //["a","Aaz"]`

+ `replaceAll()` — замінює всі значення результатом функції `BiFunction`;
+ `replace()` — замінює значення лише за наявного ключа.

## Що таке `LocalDateTime`?

## Як визначити повторювану анотацію?
Щоб зробити анотацію повторюваною (можна застосувати кілька разів до одного елемента), потрібно створити анотацію-контейнер зі значенням-масивом і позначити повторювану анотацію мета-анотацією `@Repeatable`, вказавши контейнер:

```java
@interface Schedulers {
    Scheduler[] value();
}

@Repeatable(Schedulers.class)
@interface Scheduler {
    String birthday() default "Jan 8 1935";
}
```

Тепер `@Scheduler` можна застосувати кілька разів; компілятор неявно загорне їх у `@Schedulers`.

## Що таке `Nashorn`?
__Nashorn__ — рушій JavaScript на Java від Oracle, що з'явився в Java 8. Дозволяв виконувати JavaScript усередині Java-застосунків. Порівняно з попереднім рушієм _Rhino_, Nashorn компілював JS у байт-код JVM безпосередньо в пам'яті, що давало значно вищу продуктивність; можливий був і виклик Java-коду з JavaScript.

> Застаріле: Nashorn оголошено deprecated у Java 11 (JEP 335) і повністю прибрано з JDK у Java 15 (JEP 372). У сучасних версіях для виконання JS використовують окремі рушії, наприклад GraalVM JavaScript.

[до змісту](#java-8)

## Що таке `jjs`?
`jjs` — утиліта командного рядка (з Java 8), що дозволяла виконувати програми JavaScript прямо в консолі за допомогою рушія Nashorn.

> Застаріле: `jjs` прибрана разом із Nashorn у Java 15.

[до змісту](#java-8)

## Який клас з'явився в Java 8 для кодування/декодування даних?
`java.util.Base64` — потокобезпечний клас, що реалізує кодувальник і декодувальник за схемою Base64 згідно з _RFC 4648_ та _RFC 2045_. (До Java 8 доводилося користуватися непублічними або сторонніми класами.)

Клас надає три пари фабричних методів:

+ `getEncoder()` / `getDecoder()` — стандартний Base64 за _RFC 4648_;
+ `getUrlEncoder()` / `getUrlDecoder()` — URL-safe варіант за _RFC 4648_ (символи `+` та `/` замінено на `-` та `_`);
+ `getMimeEncoder()` / `getMimeDecoder()` — MIME-варіант за _RFC 2045_ (розбиття на рядки).

[до змісту](#java-8)

## Як створити Base64-кодувальник і декодувальник?
```java
// Кодування
String b64 = Base64.getEncoder()
        .encodeToString("input".getBytes(StandardCharsets.UTF_8)); // aW5wdXQ=
// Декодування
new String(Base64.getDecoder().decode(b64), StandardCharsets.UTF_8); // input
```

> Виправлено: рядок `"input"` (5 байтів) у Base64 — це `aW5wdXQ=` з _одним_ знаком `=`, а не `aW5wdXQ==`. Також замість `getBytes("utf-8")` (кидає перевірюваний `UnsupportedEncodingException`) краще використовувати `StandardCharsets.UTF_8`.
