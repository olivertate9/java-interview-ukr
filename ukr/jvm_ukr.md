## За що відповідає _JVM_:

+ Завантаження, перевірка та виконання байткоду;
+ Надання середовища виконання для байткоду;
+ Керування пам'яттю та прибирання сміття (Garbage Collection).

Віртуальна машина Java (Java Virtual Machine) — це механізм, що надає середовище виконання для запуску Java-коду або застосунків. Віртуальна машина є незалежним прошарком виконання коду, завдяки якому можливий її запуск на будь-якій ОС, без впливу ОС на виконувану програму.

JVM працює з двома категоріями типів: примітивними типами (**primitive types**) та посилальними типами (**reference types**).

**Примітиви**

JVM працює з примітивними значеннями (цілими числами та числами з плаваючою комою). За базовою моделлю набір інструкцій JVM орієнтований на 32-бітні комірки. Типи `long` і `double`, які є 64-бітними, підтримуються безпосередньо, але займають дві комірки пам'яті в масиві локальних змінних фрейма (`frame's local variables`) або в стеку операндів, оскільки кожна комірка має розмір 32 біти.
Типи `boolean`, `byte`, `short` і `char` розширюються зі знаком (окрім `char`, який розширюється нулями) і обробляються як 32-розрядні цілі, так само як і тип `int`.
Менші типи мають лише кілька специфічних для типу інструкцій для завантаження, зберігання та перетворення типів.
Значення `boolean` працює як 8-бітне значення `byte`, де 0 представляє **false**, а 1 — **true**.

**Посилальні типи та значення**

Існує три різновиди посилальних типів: типи класів, типи масивів і типи інтерфейсів.
Їхні значення є посиланнями на динамічно створювані екземпляри класів, масиви або екземпляри класів, що реалізують відповідні інтерфейси.

## Classloader

Завантажувач класів (class loader) є частиною JRE, яка динамічно завантажує Java-класи в JVM. Зазвичай класи завантажуються лише на вимогу. Завдяки завантажувачу класів система виконання Java не повинна знати про файли та файлові системи. **Делегування є важливою концепцією**, яку виконує завантажувач. Завантажувач класів відповідає за пошук бібліотек, читання їхнього вмісту та завантаження класів, що містяться в бібліотеках. Це **завантаження** зазвичай виконується **«на вимогу»** (lazy), оскільки воно не відбувається доти, доки програма не звернеться до класу.
**Клас із заданим іменем може бути завантажений лише один раз цим конкретним завантажувачем класів.**

При запуску JVM використовуються три завантажувачі класів:
* Bootstrap class loader (завантажувач Bootstrap)
* Extensions class loader (завантажувач розширень)
* System class loader (системний завантажувач класів)

**Завантажувач Bootstrap** завантажує основні бібліотеки Java, розташовані в теці `<JAVA_HOME>/jre/lib`. Цей завантажувач є частиною ядра JVM і написаний нативним кодом.

**Завантажувач розширень** завантажує код із каталогів розширень (`<JAVA_HOME>/jre/lib/ext` або будь-якого іншого каталогу, заданого системною властивістю `java.ext.dirs`).

**Системний завантажувач** завантажує код, знайдений у `java.class.path`, що відповідає змінній середовища `CLASSPATH`. Він реалізований класом `sun.misc.Launcher$AppClassLoader`.

> **Примітка щодо сучасних версій Java.** Опис вище відповідає Java 8 і раніше. Починаючи з Java 9 (з появою модульної системи) ця ієрархія змінилася:
> + структура `<JAVA_HOME>/jre/...` зникла — JRE більше не виокремлюється в підтеку;
> + **завантажувач розширень** (extensions class loader) і механізм `java.ext.dirs` **прибрано**;
> + замість нього з'явився **platform class loader**, а внутрішній клас застосунків тепер `jdk.internal.loader.ClassLoaders$AppClassLoader` (а не `sun.misc.Launcher$AppClassLoader`);
> + актуальна ієрархія: **bootstrap → platform → application (system)**.

Завантажувач класів виконує три основні дії в суворому порядку:
* Завантаження (Loading): знаходить та імпортує двійкові дані для типу.
* Зв'язування (Linking): виконує перевірку, підготовку та (необов'язково) розв'язання.
    - Перевірка (Verification): гарантує коректність імпортованого типу.
    - Підготовка (Preparation): виділяє пам'ять для змінних класу й ініціалізує її значеннями за замовчуванням.
    - Розв'язання (Resolution): перетворює символічні посилання типу на прямі посилання.
* Ініціалізація (Initialization): виконує Java-код, що ініціалізує змінні класу їхніми правильними початковими значеннями.

**Користувацький завантажувач класів**

Завантажувач класів написаний на Java. Тому можна створити власний завантажувач класів, не вникаючи в тонкощі реалізації JVM. Кожен Java-завантажувач класів має батьківський завантажувач, визначений під час створення екземпляра нового завантажувача, або системний завантажувач класів за замовчуванням для віртуальної машини.

Це уможливлює таке:
* завантажувати або вивантажувати класи під час виконання (наприклад, динамічно завантажувати бібліотеки під час виконання, навіть з HTTP-ресурсу). Це важлива можливість для:
    - реалізації скриптових мов;
    - використання bean builders;
    - додавання користувацьких розширень;
    - забезпечення взаємодії кількох просторів імен. Наприклад, це одна з основ протоколів CORBA / RMI;
* змінити спосіб завантаження байткоду (наприклад, можна використовувати зашифрований байткод Java-класу);
* модифікувати завантажений байткод (наприклад, для вплетення аспектів під час завантаження при використанні аспектно-орієнтованого програмування).

## Області даних часу виконання

Коли JVM виконує програму, їй потрібна пам'ять для різних цілей: де зберігати об'єкти, де — змінні методів, де — завантажені класи. Для цього JVM ділить пам'ять на кілька **областей даних часу виконання** (Run-Time Data Areas).

Головне, що варто сказати на співбесіді: ці області діляться на дві групи.

+ **Спільні для всіх потоків** (створюються при старті JVM, живуть до її зупинки): **Heap** та **Metaspace**.
+ **Власні для кожного потоку** (створюються разом із потоком і знищуються разом із ним): **PC Register**, **JVM Stack** та **Native Method Stack**.

Така структура прямо пояснює, чому об'єкти (у спільній купі) видно з різних потоків, а локальні змінні методу (у стеку потоку) — ні.

### Спільні області (одна на всю JVM)

**Heap (купа)** — центральна область, де зберігаються **всі об'єкти та масиви**. Щойно ви робите `new`, об'єкт опиняється в купі. Купа спільна для всіх потоків, тому посилання на об'єкт можна передати з одного потоку в інший. Пам'ять у купі звільняється не вручну, а автоматично — **збирачем сміття** (garbage collector), коли на об'єкт більше немає досяжних посилань. Купа — це саме та область, нестача якої спричиняє `OutOfMemoryError: Java heap space`.

**Metaspace (метапростір)** — тут JVM зберігає **метадані класів**: структуру класу, його поля й методи, байткод методів, пул констант класу тощо. Простіше кажучи, коли завантажувач класів завантажує клас, результат лягає сюди. Metaspace розміщується в **нативній пам'яті ОС, поза купою**, і за замовчуванням росте автоматично. У специфікації ця концепція називається *Method Area* (область методів) — Metaspace є її конкретною реалізацією в сучасній HotSpot JVM.

> Раніше тут можна було почути слово «PermGen» — це стара реалізація області методів, яку **прибрали у Java 8** і замінили на Metaspace. Якщо запитають про різницю: PermGen був частиною купи з фіксованим лімітом (звідси типова `OutOfMemoryError: PermGen space`), а Metaspace лежить у нативній пам'яті й росте динамічно.

**Пул констант часу виконання (Run-Time Constant Pool)** — частина метаданих класу в Metaspace. Це таблиця констант конкретного класу: числові літерали, рядки, а також символічні посилання на інші класи, методи й поля, які JVM розв'язує (resolve) під час виконання. По суті, це «таблиця символів» класу.

### Області кожного потоку (свої на кожен потік)

**PC Register (лічильник команд)** — невеликий регістр, що зберігає адресу інструкції байткоду, яку потік виконує **зараз**. Оскільки потоків багато й вони виконуються незалежно, кожен має власний PC, щоб JVM знала, де саме спинився кожен потік. (Для нативних методів значення PC не визначене.)

**JVM Stack (стек викликів)** — стек, що зберігає **фрейми** (frames). Кожен виклик методу кладе на стек новий фрейм, кожне повернення — знімає його. Саме тут живуть локальні змінні методу та проміжні обчислення. Цей стек напряму відповідає тому, що ви бачите у stack trace. Якщо викликів стане забагато (наприклад, нескінченна рекурсія) — стек переповниться й ви отримаєте `StackOverflowError`.

**Native Method Stack** — окремий стек для нативних методів (написаних не на Java, а, наприклад, на C і викликаних через JNI).

## Frames (фрейми)

**Фрейм — це «робочий простір» одного виклику методу.** Щоразу, коли викликається метод, JVM створює новий фрейм і кладе його на стек поточного потоку. Коли метод завершується (нормально чи з винятком) — фрейм знімається зі стека й знищується. Тобто фреймів на стеку рівно стільки, скільки методів зараз «у процесі виконання» один в одному.

Активний фрейм лише один — той, що належить методу, який виконується просто зараз. Його називають **поточним фреймом** (current frame). Коли поточний метод викликає інший, поверх кладеться новий фрейм і стає поточним; коли той метод повертається — його фрейм знімається, а поточним знову стає попередній. Кожен фрейм належить своєму потоку, тож інші потоки до нього доступу не мають.

Кожен фрейм містить три складові: **масив локальних змінних**, **стек операндів** і **посилання на пул констант** класу поточного методу. Їхні розміри відомі ще на етапі компіляції (зашиті в class-файл), тому JVM може виділити фрейм за один крок при виклику методу.

**Локальні змінні**

Масив фіксованої довжини, де лежать параметри методу та його локальні змінні. Адресуються за індексом, починаючи з 0. Одна комірка вміщує значення типу `boolean`, `byte`, `char`, `short`, `int`, `float`, `reference` або `returnAddress`; значення `long` і `double` займають дві сусідні комірки.

Через цей масив передаються й параметри методу. Тонкий момент, який люблять на співбесіді: у **методі екземпляра** комірка 0 завжди зайнята посиланням `this`, а параметри йдуть з індексу 1; у **статичному методі** `this` немає, тож параметри починаються з індексу 0.

**Стек операндів (Operand Stack)**

Робочий стек (LIFO), на якому виконуються самі обчислення. Інструкції байткоду кладуть на нього значення (з констант, локальних змінних, полів), а арифметичні/логічні інструкції знімають операнди, обчислюють результат і кладуть його назад.

Наприклад, інструкція `iadd` (додавання двох `int`) працює так: знімає два верхні значення зі стека операндів, додає їх і кладе суму назад на стек. Через цей самий стек передаються аргументи у викликані методи й повертаються їхні результати.

**Динамічне зв'язування (Dynamic Linking)**

У байткоді виклики методів і звернення до полів записані як **символічні посилання** (за іменами: «клас такий-то, метод такий-то»), а не як конкретні адреси. Через посилання на пул констант фрейм виконує динамічне зв'язування — перетворює ці символічні посилання на конкретні (реальне розташування методу/поля в пам'яті), завантажуючи потрібні класи за необхідності. Це й дозволяє поліморфізм: який саме метод викликати, визначається під час виконання за фактичним типом об'єкта.

**Як метод завершується**

+ **Нормально** — метод доходить до інструкції повернення (`return`). Якщо є значення, що повертається, воно кладеться на стек операндів **попереднього** (викликаючого) фрейма, і виконання продовжується в ньому.
+ **Раптово (abrupt)** — у методі виникає виняток, який у ньому не обробляється. Тоді фрейм знімається без повернення значення, а виняток передається вище по стеку (поки не знайдеться `catch` або потік не завершиться).

## Execution Engine

Завантажувач класів кладе байткод у пам'ять, а виконує його **Execution Engine** (механізм виконання) — саме він перетворює інструкції байткоду на реальні дії процесора. Працює у зв'язці з двома підходами: інтерпретацією та JIT-компіляцією.

**Інтерпретатор**

Читає байткод інструкція за інструкцією й одразу виконує. Запускається миттєво (не треба нічого попередньо компілювати), але повільний на «гарячому» коді: якщо метод викликається тисячі разів, він щоразу інтерпретується наново.

**JIT-компілятор (Just-In-Time)**

Розв'язує проблему повільної інтерпретації. JVM **профілює** виконання й відстежує «гарячі точки» (hot spots) — методи та цикли, що виконуються часто. Такий код JIT компілює один раз у **нативний машинний код** і далі виконує його напряму, без інтерпретації — це значно швидше. Звідси й назва еталонної JVM — **HotSpot**.

На практиці HotSpot використовує **багаторівневу компіляцію (tiered compilation)**: спершу код інтерпретується, потім компілюється швидким компілятором C1, а найгарячіші ділянки — оптимізаційним компілятором C2, що дає максимальну продуктивність. Перевага такого підходу перед компіляцією наперед (як у C++) у тому, що JIT спирається на **реальну статистику виконання** і може робити оптимізації, недоступні статичному компілятору (вбудовування методів, усунення перевірок тощо).

**Garbage Collector (збирач сміття)**

Третя частина механізму виконання. Автоматично знаходить у купі об'єкти, на які більше немає досяжних посилань, і вивільняє зайняту ними пам'ять. Завдяки цьому в Java не потрібно звільняти пам'ять вручну (як `free` у C), що прибирає цілий клас помилок на кшталт витоків пам'яті та «висячих» вказівників.
