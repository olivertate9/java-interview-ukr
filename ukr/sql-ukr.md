## Що таке _«SQL»_?
SQL, Structured Query Language («мова структурованих запитів») — декларативна (непроцедурна) мова для створення, модифікації даних та керування ними в реляційній базі даних, якою керує відповідна система керування базами даних (СКБД).

Ключове слово тут — **декларативне**: ти описуєш, *що* хочеш отримати (множину рядків, що задовольняють умові), а не *як* це обчислити. Як саме виконати запит — за яким індексом пройтися, у якому порядку з'єднувати таблиці — вирішує оптимізатор СКБД, будуючи план виконання.

## Які бувають оператори SQL?
__Оператори визначення даних (Data Definition Language, DDL)__ — описують структуру:

+ `CREATE` — створює об'єкт БД (базу, таблицю, представлення, користувача тощо),
+ `ALTER` — змінює об'єкт,
+ `DROP` — видаляє об'єкт,
+ `TRUNCATE` — швидко очищає таблицю.

__Оператори маніпулювання даними (Data Manipulation Language, DML)__ — працюють із самими даними:

+ `SELECT` — вибирає дані, що задовольняють заданим умовам,
+ `INSERT` — додає нові дані,
+ `UPDATE` — змінює наявні дані,
+ `DELETE` — видаляє дані.

__Оператори керування доступом до даних (Data Control Language, DCL)__:

+ `GRANT` — надає користувачу (групі) дозволи на певні операції з об'єктом,
+ `REVOKE` — відкликає раніше видані дозволи,
+ `DENY` — задає заборону, що має пріоритет над дозволом (специфічне для Microsoft SQL Server).

__Оператори керування транзакціями (Transaction Control Language, TCL)__:

+ `COMMIT` — застосовує (фіксує) транзакцію,
+ `ROLLBACK` — відкочує всі зміни, зроблені в межах поточної транзакції,
+ `SAVEPOINT` — встановлює точку збереження всередині транзакції, до якої можна частково відкотитися.

## Що означає `NULL` у SQL?
`NULL` — спеціальне значення (псевдозначення), яке може бути записане в поле таблиці. `NULL` відповідає поняттю «порожнє поле», тобто «поле, що не містить жодного значення».

Ментальна модель: `NULL` — це не «нуль» і не «порожній рядок», а **відсутність відомостей** («значення невідоме»). За визначенням воно не належить до жодного типу даних. Тому `NULL` не дорівнює ні логічному `FALSE`, ні _порожньому рядку_, ні `0`.

Наслідок, який часто плутають на співбесіді: SQL використовує **тризначну логіку** (`TRUE` / `FALSE` / `UNKNOWN`). Порівняння `NULL` з будь-чим дає не `TRUE` і не `FALSE`, а `UNKNOWN`. Більше того, `NULL = NULL` теж дає `UNKNOWN`! Саме тому для перевірки на `NULL` використовують окремі предикати `IS NULL` / `IS NOT NULL`, а не `= NULL`. А оскільки `WHERE` пропускає лише рядки з результатом `TRUE`, рядки з `UNKNOWN` у вибірку не потраплять.

## Що таке _«тимчасова таблиця»_? Для чого вона використовується?
__Тимчасова таблиця__ — це об'єкт бази даних, який зберігається й управляється СКБД на тимчасовій основі (зазвичай у межах сесії або транзакції, після чого автоматично видаляється). Тимчасові таблиці бувають локальними (видимі лише поточній сесії) та глобальними (видимі всім сесіям).

Застосовують для збереження проміжних результатів виклику збереженої процедури, зменшення кількості рядків під час складних з'єднань, агрегування даних із різних джерел або як заміну курсорів.

[до змісту](#sql)

## Що таке _«представлення» (view)_ і для чого воно застосовується?
__Представлення__ (view) — віртуальна таблиця, що подає дані однієї або кількох таблиць у альтернативному вигляді.

Фактично представлення — це збережений у базі іменований запит `SELECT`. У запитах та операторах DML воно поводиться так само, як звичайна таблиця, але **власних даних не зберігає** — вони щоразу обчислюються із базових таблиць.

Навіщо це потрібно:
+ **інкапсуляція складності** — складне з'єднання кількох таблиць ховається за простим іменем;
+ **безпека** — можна надати доступ до частини стовпців/рядків таблиці, не відкриваючи всю таблицю;
+ **стабільний контракт** — структуру базових таблиць можна змінювати, а представлення лишити незмінним для застосунків.

Окремо варто згадати **матеріалізоване представлення** (materialized view) — на відміну від звичайного, воно зберігає результат фізично й періодично оновлюється, що пришвидшує читання ціною актуальності даних.

## Який загальний синтаксис оператора `SELECT`?
`SELECT` — оператор DML, що повертає набір даних (вибірку) з бази, який задовольняє заданій умові. Має таку структуру:

```sql
SELECT
       [DISTINCT | ALL]
       select_expression, ...
   FROM table_references
     [WHERE where_definition]
     [GROUP BY {unsigned_integer | column | formula}]
     [HAVING where_definition]
     [ORDER BY {unsigned_integer | column | formula} [ASC | DESC], ...]
```

Корисно пам'ятати **логічний порядок виконання** (він не збігається з порядком написання):
`FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY`.

Саме тому, наприклад, псевдонім стовпця, заданий у `SELECT`, зазвичай не можна використати у `WHERE` (воно виконується раніше), але можна в `ORDER BY` (виконується пізніше).

## Що таке `JOIN`?
__JOIN__ — оператор SQL, що реалізує операцію з'єднання реляційної алгебри. Призначений для вибірки даних із двох (або більше) таблиць та об'єднання їх в один результуючий набір.

Особливості операції з'єднання:

+ у схему таблиці-результату входять стовпці обох вихідних таблиць (таблиць-операндів), тобто схема результату є «зчепленням» схем операндів;
+ кожен рядок таблиці-результату є «зчепленням» рядка з однієї таблиці-операнда з рядком другої таблиці-операнда (за умовою з'єднання);
+ за потреби з'єднати не дві, а кілька таблиць, операція з'єднання застосовується послідовно кілька разів.

```sql
SELECT
  field_name [, ... n]
FROM
  Table1
  {INNER | {LEFT | RIGHT | FULL} OUTER | CROSS} JOIN
  Table2
    {ON <condition> | USING (field_name [, ... n])}
```

[до змісту](#sql)

## Які бувають типи `JOIN`?

| Тип | Що повертає | Симетричний? |
|---|---|---|
| `(INNER) JOIN` | лише рядки, для яких умова з'єднання виконується в обох таблицях | так |
| `LEFT (OUTER) JOIN` | усі рядки лівої таблиці + відповідні рядки правої; де відповідності немає — `NULL` | ні |
| `RIGHT (OUTER) JOIN` | усі рядки правої таблиці + відповідні рядки лівої; де немає — `NULL` | ні |
| `FULL (OUTER) JOIN` | усі рядки обох таблиць; відсутні відповідності заповнюються `NULL` | так |
| `CROSS JOIN` | декартів добуток — кожен рядок однієї таблиці з кожним рядком іншої | так |

__(INNER) JOIN__ — результатом є записи, спільні для лівої та правої таблиць. Порядок таблиць не важливий, бо оператор симетричний.

__LEFT (OUTER) JOIN__ — обирає всі записи першої (лівої) таблиці та відповідні їм записи другої. Якщо відповідних записів у правій таблиці немає, замість них підставляється `NULL`. Порядок важливий, бо оператор несиметричний.

__RIGHT (OUTER) JOIN__ — це `LEFT JOIN` з операндами в зворотному порядку. Порядок важливий.

__FULL (OUTER) JOIN__ — результатом є всі записи обох таблиць; для тих, що не мають пари, відсутні стовпці заповнюються `NULL`. Симетричний.

__CROSS JOIN (декартів добуток)__ — кожен рядок однієї таблиці поєднується з кожним рядком іншої, даючи всі можливі сполучення. Симетричний. На практиці застосовується рідко й здебільшого навмисно (наприклад, для генерації комбінацій); випадковий `CROSS JOIN` через забутий `ON` — типова причина «вибуху» кількості рядків.

[до змісту](#sql)

## Що краще використовувати — `JOIN` чи підзапити?
Зазвичай краще використовувати `JOIN`: у більшості випадків він зрозуміліший і його простіше оптимізувати СКБД (хоча 100% гарантії немає — на сучасних оптимізаторах багато корельованих підзапитів перетворюються на з'єднання внутрішньо). `JOIN` має помітну перевагу, коли в списку `SELECT` потрібні стовпці більш ніж з однієї таблиці.

Підзапити зручніші там, де треба обчислити агрегатне значення й порівняти з ним у зовнішньому запиті, а також у предикатах `EXISTS` / `IN`, коли сам факт наявності рядків важливіший за їхній вміст.

На практиці: пиши той варіант, що читабельніший для конкретної задачі, а різницю в продуктивності перевіряй за планом виконання (`EXPLAIN`), а не за здогадками.

## Для чого використовується оператор `HAVING`?
`HAVING` фільтрує результат `GROUP BY` за заданими логічними умовами — тобто відкидає цілі групи, які не задовольняють умові (зокрема за значеннями агрегатних функцій).

[до змісту](#sql)

## У чому різниця між операторами `HAVING` і `WHERE`?
Головна відмінність — **на якому етапі виконання** вони фільтрують.

`WHERE` працює **до** групування: спершу відбирає окремі рядки, і вже потім вони групуються та обчислюються агрегати. Тому `WHERE` не може містити агрегатних функцій — немає сенсу використовувати агрегат для відбору рядків, з яких цей агрегат лише обчислюватиметься.

`HAVING` працює **після** групування: відбирає вже сформовані групи за результатами агрегатних функцій. Тому `HAVING` зазвичай і містить агрегати.

| | `WHERE` | `HAVING` |
|---|---|---|
| Етап | до `GROUP BY` | після `GROUP BY` |
| Оперує | окремими рядками | групами |
| Агрегатні функції | не можна | можна (типово) |

Формально умову без агрегатів можна записати й у `HAVING`, але це майже завжди гірше: ту саму умову ефективніше застосувати раніше, у `WHERE`, щоб відсіяти зайві рядки ще до групування.

[до змісту](#sql)

## Для чого використовується оператор `ORDER BY`?
`ORDER BY` упорядковує вивід запиту за значеннями одного або кількох обраних стовпців. За кількох стовпців сортування є вкладеним (спочатку за першим, у межах рівних значень — за другим тощо). Для кожного стовпця можна задати зростання `ASC` або спадання `DESC`; за замовчуванням — зростання.

Важливо: без явного `ORDER BY` порядок рядків у результаті **не гарантований** — СКБД може повертати їх у будь-якому порядку.

[до змісту](#sql)

## Для чого використовується оператор `GROUP BY`?
`GROUP BY` групує рядки за значеннями заданих стовпців-атрибутів, щоб застосувати до кожної групи агрегатні функції (наприклад, порахувати суму чи кількість у межах групи).

[до змісту](#sql)

## Як `GROUP BY` обробляє значення `NULL`?
Під час групування всі значення `NULL` вважаються рівними між собою й потрапляють в одну спільну групу. Це окремий випадок, бо в інших контекстах `NULL` не дорівнює `NULL`.

[до змісту](#sql)

## У чому різниця між `GROUP BY` і `DISTINCT`?
`DISTINCT` лишає в результаті лише унікальні комбінації значень обраних стовпців (`NULL` рахується як окреме значення).

`GROUP BY` також формує групи за унікальними значеннями (включно з `NULL`), але його основне призначення — застосувати до кожної групи агрегатні функції.

Практичне правило: якщо потрібно лише прибрати дублікати — бери `DISTINCT`; якщо потрібно обчислити агрегат у межах групи — бери `GROUP BY`.

## Перелічіть основні агрегатні функції.
__Агрегатні функції__ — функції, що беруть групу значень і зводять її до одного значення.

| Функція | Що робить |
|---|---|
| `COUNT` | підраховує кількість записів, що задовольняють умові |
| `SUM` | обчислює суму всіх значень стовпця |
| `AVG` | обчислює середнє арифметичне значень |
| `MAX` | визначає найбільше зі значень |
| `MIN` | визначає найменше зі значень |

Важлива деталь: `SUM`, `AVG`, `MAX`, `MIN` та `COUNT({column})` **ігнорують `NULL`** (не враховують такі рядки). Виняток — `COUNT(*)`, що рахує всі рядки незалежно від `NULL`.

[до змісту](#sql)

## У чому різниця між `COUNT(*)` і `COUNT({column})`?
`COUNT(*)` підраховує кількість рядків у вибірці й **не ігнорує** `NULL`, оскільки оперує рядками, а не стовпцями.

`COUNT({column})` підраховує кількість **не-`NULL`** значень у заданому стовпці — рядки, де стовпець дорівнює `NULL`, не враховуються.

Окремо: `COUNT(DISTINCT {column})` рахує кількість унікальних не-`NULL` значень стовпця.

[до змісту](#sql)

## Що робить оператор `EXISTS`?
`EXISTS` бере підзапит як аргумент і повертає `TRUE`, якщо підзапит повертає хоча б один рядок, та `FALSE`, якщо не повертає жодного. Сам вміст рядків при цьому не важливий — важливий лише факт їх наявності, тому в підзапиті часто пишуть `SELECT 1`.

[до змісту](#sql)

## Для чого використовуються оператори `IN`, `BETWEEN`, `LIKE`?
`IN` — перевіряє належність значення до заданого набору.

```sql
SELECT * FROM Persons WHERE name IN ('Ivan', 'Petr', 'Pavel');
```

`BETWEEN` — задає діапазон значень (включно з обома межами). Межі треба вказувати від меншої до більшої: `BETWEEN 20 AND 25` працює, а `BETWEEN 25 AND 20` поверне порожній результат.

```sql
SELECT * FROM Persons WHERE age BETWEEN 20 AND 25;
```

`LIKE` — застосовується до рядкових типів (`CHAR`, `VARCHAR`) для пошуку за шаблоном. У шаблоні використовують спеціальні символи-підстановки (wildcards):

+ `_` — замінює рівно один будь-який символ. Наприклад, `'b_t'` відповідає `'bat'` чи `'bit'`, але не `'brat'`.
+ `%` — замінює послідовність із будь-якої кількості символів (зокрема нуля). Наприклад, `'%p%t'` відповідає `'put'`, `'posit'` чи `'opt'`, але не `'spite'`.

```sql
SELECT * FROM University WHERE name LIKE '%o';
```

[до змісту](#sql)

## Для чого застосовується ключове слово `UNION`?
`UNION` об'єднує результати двох (або більше) запитів в одну таблицю. Обидва запити мають повертати однакову кількість стовпців із сумісними типами даних у відповідних позиціях.

| | `UNION` | `UNION ALL` |
|---|---|---|
| Дублікати | прибирає (рядки унікальні) | лишає всі рядки |
| Продуктивність | повільніше (потрібна дедуплікація) | швидше |

Сам по собі `UNION` **не гарантує порядку** рядків: записи з другого запиту можуть опинитися на початку, в кінці або перемішатися з першими. Якщо потрібен певний порядок — додавай `ORDER BY` (один на весь об'єднаний результат, наприкінці).

[до змісту](#sql)

## Які обмеження цілісності даних існують у SQL?
`PRIMARY KEY` — набір полів (одне або кілька), значення яких утворюють унікальну комбінацію й однозначно ідентифікують рядок у таблиці. Для таблиці може бути лише одне таке обмеження; його стовпці не можуть бути `NULL`. Забезпечує **цілісність сутності**.

`UNIQUE` — гарантує відсутність дублікатів у стовпці (чи наборі стовпців). На відміну від `PRIMARY KEY`, таких обмежень у таблиці може бути кілька, і вони (зазвичай) допускають `NULL`.

`CHECK` — обмежує множину значень, які можна записати у стовпець (наприклад, `CHECK (age >= 0)`). Забезпечує **цілісність предметної області**.

`FOREIGN KEY` — забезпечує **посилальну цілісність**: захищає від дій, що порушили б зв'язки між таблицями. `FOREIGN KEY` в одній таблиці посилається на `PRIMARY KEY` (або `UNIQUE`) в іншій і не дозволяє існувати «осиротілим» записам, яким не відповідає жоден рядок батьківської таблиці.

Окремо часто згадують `NOT NULL` — обмеження на рівні стовпця, що забороняє в ньому `NULL`.

[до змісту](#sql)

## Які відмінності між обмеженнями `PRIMARY KEY` і `UNIQUE`?

| | `PRIMARY KEY` | `UNIQUE` |
|---|---|---|
| Кількість на таблицю | лише одне | може бути кілька |
| `NULL` | заборонено | дозволено (зазвичай) |
| Індекс за замовчуванням (у MS SQL Server) | кластерний | некластерний |

Щодо `NULL` у `UNIQUE` поведінка залежить від СКБД: стандарт SQL та більшість баз (PostgreSQL, Oracle, MS SQL Server) дозволяють кілька рядків із `NULL`; MS SQL Server при цьому допускає лише один `NULL`. Тип індексу (кластерний/некластерний) — це деталь реалізації MS SQL Server, у інших СКБД вона може відрізнятися.

[до змісту](#sql)

## Чи може значення в стовпці з обмеженням `FOREIGN KEY` дорівнювати `NULL`?
Так, може — якщо на цей стовпець не накладено окреме обмеження `NOT NULL`. `NULL` у зовнішньому ключі означає «зв'язку немає» і перевірку посилальної цілісності не порушує.

[до змісту](#sql)

## Як створити індекс?
Індекс можна створити явно оператором `CREATE INDEX`:

```sql
CREATE INDEX index_name ON table_name (column_name);
```

або неявно — оголосивши обмеження `UNIQUE` чи `PRIMARY KEY` в операторі `CREATE TABLE` (СКБД автоматично створює під них індекс).

[до змісту](#sql)

## Що робить оператор `MERGE`?
`MERGE` (його ще називають «upsert») зливає дані джерела з даними цільової таблиці за заданою умовою з'єднання: якщо відповідний рядок знайдено — виконується `UPDATE` (гілка `WHEN MATCHED`), якщо ні — `INSERT` (гілка `WHEN NOT MATCHED`). Стовпці, за якими йде зіставлення двох таблиць, у секції `UPDATE` змінювати не можна.

[до змісту](#sql)

## У чому відмінність між операторами `DELETE` і `TRUNCATE`?

| | `DELETE` | `TRUNCATE` |
|---|---|---|
| Категорія | DML | DDL |
| Умова `WHERE` | підтримує | ні (видаляє всі рядки) |
| Тригери | спрацьовують | не спрацьовують |
| Швидкість на повній очистці | повільніше (рядок за рядком) | швидше |
| Лічильник автоінкременту | зазвичай зберігає | зазвичай скидає |

`DELETE` — оператор DML: видаляє рядки, що задовольняють критерію `WHERE`, при цьому задіюються тригери, перевірки обмежень тощо.

`TRUNCATE` — оператор DDL: швидко очищає таблицю цілком (мінімально журналюючи операцію). Якщо на таблицю посилаються зовнішні ключі `FOREIGN KEY` або вона бере участь у реплікації, виконати `TRUNCATE` зазвичай не вдасться.

[до змісту](#sql)

## Що таке _«збережена процедура»_?
__Збережена процедура__ — об'єкт бази даних, що являє собою набір SQL-інструкцій, який зберігається й виконується на сервері. Збережені процедури схожі на звичайні процедури мов високого рівня: можуть мати вхідні й вихідні параметри та локальні змінні, виконувати обчислення й операції над даними, містити цикли та розгалуження (інструкції керування потоком виконання). У них доступні як DML-, так і DDL-операції.

Навіщо: підвищують продуктивність, розширюють можливості програмування на боці БД та допомагають із безпекою (доступ можна надати до процедури, не відкриваючи базові таблиці). У багатьох СКБД при першому запуску процедура компілюється (виконується синтаксичний аналіз і будується план доступу до даних), завдяки чому подальші виклики обробляються швидше.

[до змісту](#sql)

## Що таке _«тригер»_?
__Тригер (trigger)__ — це збережена процедура особливого типу, яку не викликають напряму: її виконання спричиняє подія модифікації даних — `INSERT`, `UPDATE` або `DELETE` у заданій таблиці. Тригери застосовують для забезпечення цілісності даних та реалізації складної бізнес-логіки.

Тригер запускається сервером автоматично, і всі його зміни виконуються в межах тієї самої транзакції, що спричинила спрацювання. Тому в разі помилки чи порушення цілісності цю транзакцію може бути відкочено цілком.

Момент запуску задають ключовими словами `BEFORE` (до події) або `AFTER` (після). Тригер `BEFORE` може змінити запис, що модифікується подією. Тригери можна прив'язувати не лише до таблиці, а й до представлення (`INSTEAD OF`-тригери), реалізуючи механізм «оновлюваного представлення» — у цьому разі замість самої події (вставки/оновлення/видалення) виконується тіло тригера.

[до змісту](#sql)

## Що таке _«курсор»_?
__Курсор__ — об'єкт бази даних, що дозволяє застосунку обробляти рядки результату **по одному**, а не всю множину одразу, як це роблять звичайні SQL-команди.

Порядок роботи з курсором:

+ оголосити курсор (`DECLARE`);
+ відкрити курсор (`OPEN`);
+ отримати рядок із курсора (`FETCH`);
+ обробити рядок...
+ закрити курсор (`CLOSE`);
+ звільнити курсор (`DEALLOCATE`) — коли видаляється останнє посилання, СКБД звільняє структури даних курсора.

Зауваження з практики: курсори зазвичай повільніші за множинні (set-based) операції, бо обробляють рядки поодинці. Тому до них варто вдаватися лише тоді, коли логіку справді не виразити одним SQL-запитом.

[до змісту](#sql)

## Опишіть різницю між типами даних `DATETIME` і `TIMESTAMP`.
Відповідь наведено для MySQL (у різних СКБД ці типи відрізняються).

| | `DATETIME` | `TIMESTAMP` |
|---|---|---|
| Що зберігає | дату й час «як є» | момент часу в UTC |
| Залежність від часового поясу | незалежний від поясу | конвертується в/з UTC за поясом сесії |
| Розмір | 8 байт | 4 байти |
| Діапазон | 1000–9999 роки | 1970 – 2038-01-19 (обмеження 32-бітної мітки) |

`DATETIME` зберігає значення дати й часу як є, незалежно від часового поясу сервера.

`TIMESTAMP` зберігає кількість секунд, що минули від півночі 1 січня 1970 року за UTC (епоха Unix). Під час читання значення відображається з урахуванням часового поясу сесії. Через 4-байтове подання класичний `TIMESTAMP` обмежений 2038 роком (відома «проблема 2038»).

[до змісту](#sql)

## Для яких числових типів неприпустимо застосовувати операції додавання/віднімання?
Як операнди додавання й віднімання не можна використовувати тип `BIT` (бітове поле). Його значення представляють набір бітів, а не число в арифметичному сенсі.

[до змісту](#sql)

## Яке призначення операторів `PIVOT` і `UNPIVOT` у Transact-SQL?
`PIVOT` і `UNPIVOT` — нестандартні реляційні оператори, що підтримуються Transact-SQL (MS SQL Server).

`PIVOT` «розвертає» таблицю з рядків у стовпці: унікальні значення одного стовпця перетворює на кілька вихідних стовпців і, за потреби, агрегує решту даних (типове застосування — побудова зведених/перехресних таблиць).

`UNPIVOT` виконує зворотну дію: «згортає» кілька стовпців назад у значення одного стовпця (рядки).

[до змісту](#sql)

## Розкажіть про основні функції ранжування у Transact-SQL.
Функції ранжування (window-функції) повертають значення для кожного рядка в межах групи (вікна) результуючого набору. На практиці їх застосовують для нумерації списку, побудови рейтингу, посторінкової навігації тощо. Вікно задають конструкцією `OVER(... [PARTITION BY ...] ORDER BY ...)`.

Припустімо, маємо такий набір даних:

`ROW_NUMBER` — повертає просто порядковий номер рядка у вікні (завжди унікальний, навіть за однакових значень).

Наприклад, запит:

```sql
SELECT Studentname,
       Subject,
       Marks,
       ROW_NUMBER() OVER(ORDER BY Marks) AS RowNumber
FROM ExamResult;
```

![ ](images/SQL/row_number-sql-rank-function.png)

```sql
SELECT Studentname,
       Subject,
       Marks,
```

Наприклад:

```sql
SELECT Studentname,
```

## Для чого використовуються оператори `INTERSECT`, `EXCEPT` у Transact-SQL?
Це оператори роботи з множинами рядків (повертають унікальні рядки):

+ `INTERSECT` — повертає рядки, що присутні **в обох** запитах (перетин).
+ `EXCEPT` — повертає рядки лівого запиту, яких **немає** у правому (різниця). Оператор несиметричний: `A EXCEPT B` ≠ `B EXCEPT A`.

(В Oracle аналогом `EXCEPT` є `MINUS`.)

## Напишіть запит...

```sql
CREATE TABLE my_table (
  id BIGINT NOT NULL AUTO_INCREMENT,
  created TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id)
);
```

Потрібно написати запит, що поверне максимальне значення `id` та відповідне йому `created`:

```sql
SELECT id, created FROM my_table WHERE id = (SELECT MAX(id) FROM my_table);
```

> Примітка: `table` — зарезервоване слово в SQL, тому таблицю названо `my_table`. У вихідному прикладі `DEFAULT 0` для `TIMESTAMP` теж було некоректним — замінено на `CURRENT_TIMESTAMP`.

---

```sql
CREATE TABLE track_downloads (
  download_id BIGINT NOT NULL AUTO_INCREMENT,
  track_id INT NOT NULL,
  user_id BIGINT NOT NULL,
  download_time TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (download_id)
);
```

Напишіть SQL-запит, що повертає всі пари `(download_count, user_count)`, де `user_count` — кількість користувачів, які зробили рівно `download_count` завантажень `19 листопада 2010 року` (потрібні лише групи з ненульовою кількістю користувачів):

```sql
SELECT download_count, COUNT(*) AS user_count
FROM (
    SELECT user_id, COUNT(*) AS download_count
    FROM track_downloads
    WHERE download_time >= '2010-11-19' AND download_time < '2010-11-20'
    GROUP BY user_id
) AS per_user
GROUP BY download_count;
```

> Примітка: порівняння `download_time = '2010-11-19'` з вихідного прикладу некоректне для типу `TIMESTAMP` — воно відсіє всі записи з ненульовим часом доби. Тому використано діапазон `>= '2010-11-19' AND < '2010-11-20'`, що охоплює всю добу. `DISTINCT` із зовнішнього `SELECT` прибрано як зайвий: `GROUP BY download_count` і так дає унікальні значення.
