## Що таке _«колекція»_?
__«Колекція»__ — це структура даних, набір якихось об'єктів. Даними (об'єктами в наборі) можуть бути числа, рядки, об'єкти користувацьких класів тощо.

[до змісту](#java-collections-framework)

## Назвіть основні інтерфейси JCF та їх реалізації.
На вершині ієрархії Java Collections Framework — два інтерфейси: `Collection` і `Map`. Вони ділять усі колекції за типом зберігання даних: прості послідовні набори елементів і набори пар «ключ-значення» відповідно.

Інтерфейс `Collection` розширюють:

+ `List` (список) — колекція, де **допустимі дублікати** й елементи мають **порядок та індекс**. Реалізації:
    + `ArrayList` — динамічний масив, що автоматично збільшується. Елементи доступні за індексом за O(1).
    + `LinkedList` — двозв'язний список вузлів, кожен з яких містить дані та посилання на сусідні вузли.
    + `Vector` — синхронізований динамічний масив (**legacy**, не рекомендований).
    + `Stack` — реалізація стека LIFO (**legacy**, замість нього — `ArrayDeque`).
+ `Set` (множина) — колекція **без дублікатів**. Реалізації:
    + `HashSet` — без гарантованого порядку; усередині використовує `HashMap`.
    + `LinkedHashSet` — зберігає **порядок додавання**.
    + `TreeSet` — зберігає елементи **відсортованими** (за natural ordering або `Comparator`).
+ `Queue` (черга) — для зберігання елементів із наперед визначеним способом вставки/витягу (зазвичай FIFO). Розширюється інтерфейсом `Deque` (двостороння черга). Реалізації:
    + `PriorityQueue` — черга з пріоритетом (порядок за natural ordering або `Comparator`).
    + `ArrayDeque` — реалізація `Deque` на масиві; ефективна і як черга (FIFO), і як стек (LIFO).
    + `LinkedList` — теж реалізує `Deque`.

Інтерфейс `Map` (пари «ключ-значення», **не** є `Collection`) реалізують:

+ `HashMap` — хеш-таблиця. Дозволяє один ключ `null` і будь-яку кількість значень `null`; порядок не гарантований.
+ `LinkedHashMap` — хеш-таблиця зі збереженням порядку (додавання або доступу).
+ `TreeMap` — реалізація на червоно-чорному дереві; відсортована (за natural ordering або `Comparator`).
+ `Hashtable` — синхронізована хеш-таблиця, не дозволяє `null` (**legacy**, замість неї — `HashMap` або `ConcurrentHashMap`).
+ `WeakHashMap` — хеш-таблиця зі **слабкими посиланнями** на ключі (запис видаляється GC, коли на ключ немає сильних посилань).

## Розташуйте у вигляді ієрархії інтерфейси: `List`, `Set`, `Map`, `SortedSet`, `SortedMap`, `Collection`, `Iterable`, `Iterator`, `NavigableSet`, `NavigableMap`.
```
Iterable
└── Collection
    ├── List
    ├── Set
    │   └── SortedSet
    │       └── NavigableSet
    └── Queue

└── SortedMap
    └── NavigableMap
```

Ключове: `Map` стоїть **окремо** від `Collection`, а `Iterator` — окремий інтерфейс (його повертає метод `iterator()` з `Iterable`).

[до змісту](#java-collections-framework)

## Чому `Map` — це не `Collection`, тоді як `List` і `Set` є `Collection`?
`Collection` — це сукупність **окремих елементів**, а `Map` — сукупність **пар «ключ-значення»**. Сигнатури методів принципово різні (`add(E)` проти `put(K, V)`), тому `Map` винесено в окрему ієрархію. Утім, із `Map` можна отримати «колекційні» подання: `keySet()`, `values()`, `entrySet()`.

[до змісту](#java-collections-framework)

## У чому різниця між класами `java.util.Collection` і `java.util.Collections`?
+ `java.util.Collection` — один з основних **інтерфейсів** JCF.
+ `java.util.Collections` — клас-утиліта зі **статичними методами** для роботи з колекціями (`sort`, `reverse`, `unmodifiableList`, `synchronizedList` тощо).

## Що таке «fail-fast поведінка»?
__fail-fast__ означає, що при виявленні стану, який може призвести до помилки, система **негайно припиняє роботу** й сповіщає про це, замість того щоб продовжувати в недетермінованому стані.

У JCF деякі ітератори поводяться як fail-fast і кидають `ConcurrentModificationException`, якщо після створення ітератора колекцію змінили **напряму** (додали/видалили елемент через саму колекцію, а не через методи ітератора).

Реалізується через лічильник модифікацій (`modCount`):
+ при зміні колекції лічильник змінюється;
+ при створенні ітератора він запам'ятовує поточне значення (`expectedModCount`);
+ при кожній операції ітератора значення звіряються; якщо не збігаються — кидається виняток.

> Важливо: fail-fast — це засіб **виявлення багів**, а не гарантія. Спрацьовування не гарантоване (особливо в багатопотоковому коді), тож покладатися на нього для логіки програми не можна.

[до змісту](#java-collections-framework)

## Яка різниця між fail-fast і fail-safe?
На противагу fail-fast, ітератори **fail-safe** не кидають винятків при зміні колекції під час перебору, бо працюють із **копією (знімком)** даних або з потокобезпечною структурою.

> Уточнення термінології: у документації Java термін «fail-safe» офіційно не вживається. Коректніше казати **«слабко узгоджений» (weakly consistent)** ітератор — він не кидає `ConcurrentModificationException`, але й не гарантує, що відобразить усі зміни, зроблені після його створення.

[до змісту](#java-collections-framework)

## Чим відрізняються `Enumeration` та `Iterator`?
Обидва призначені для обходу колекцій, але:
+ `Enumeration` не дозволяє видаляти елементи; `Iterator` має `remove()`;
+ `Iterator` має коротші, читабельніші назви методів (`hasNext()`/`next()` замість `hasMoreElements()`/`nextElement()`);
+ `Enumeration` лишився в legacy-класах (`Vector`, `Hashtable`), а `Iterator` є в усіх сучасних колекціях.

`Enumeration` вважається застарілим — у новому коді використовують `Iterator`.

[до змісту](#java-collections-framework)

## Як пов'язані `Iterable` та `Iterator`?
Інтерфейс `Iterable` має один основний метод — `iterator()`, який повертає `Iterator`. Тобто `Iterable` — це «те, що можна перебирати», а `Iterator` — «той, хто перебирає».

[до змісту](#java-collections-framework)

## Як пов'язані `Iterable`, `Iterator` і «for-each»?
Класи, що реалізують `Iterable`, можна використовувати в циклі `for-each` — компілятор «під капотом» викликає `iterator()` і перебирає елементи через `Iterator` (`hasNext()`/`next()`).

```java
for (String s : list) { ... }
// еквівалентно:
for (Iterator<String> it = list.iterator(); it.hasNext(); ) {
    String s = it.next();
    ...
}
```

[до змісту](#java-collections-framework)

## Порівняйте `Iterator` і `ListIterator`.
+ `ListIterator` **розширює** `Iterator`;
+ доступний лише для `List`;
+ дозволяє перебір у **обох напрямках** (`next()` / `previous()`), тоді як `Iterator` — лише вперед;
+ його позиція — **між** елементами (між тими, що повертають `previous()` і `next()`);
+ дозволяє **змінювати** список під час перебору: `add()`, `set()`, `remove()`. `Iterator` має лише `remove()`.

[до змісту](#java-collections-framework)

## Що станеться при виклику `Iterator.next()` без `Iterator.hasNext()`?
`hasNext()` не обов'язковий для виклику `next()`. Якщо наступний елемент є — він повернеться; якщо елементів більше немає — буде кинуто `NoSuchElementException`.

[до змісту](#java-collections-framework)

## Скільки елементів буде пропущено, якщо `next()` викликати після 10 викликів `hasNext()`?
Жодного. `hasNext()` лише **перевіряє** наявність наступного елемента й **не зсуває** позицію ітератора. Тож після будь-якої кількості `hasNext()` перший `next()` поверне той самий наступний елемент.

[до змісту](#java-collections-framework)

## Як поведеться колекція, якщо викликати `iterator.remove()`?
Якщо перед `iterator.remove()` було викликано `iterator.next()`, то видалиться елемент, який повернув `next()`. Інакше — `IllegalStateException`. Це **єдиний безпечний спосіб** видаляти елементи під час перебору (без `ConcurrentModificationException`).

[до змісту](#java-collections-framework)

## Як поведеться вже створений ітератор, якщо викликати `collection.remove()`?
При наступному виклику методів ітератора буде кинуто `ConcurrentModificationException` — бо колекцію змінили напряму, а не через ітератор (спрацьовує fail-fast).

[до змісту](#java-collections-framework)

## Як уникнути `ConcurrentModificationException` під час перебору колекції?
+ Видаляти елементи **через сам ітератор** (`iterator.remove()`) або через `removeIf(...)` (з Java 8) — найпростіший спосіб.
+ Використати потокобезпечні / слабко узгоджені колекції: `CopyOnWriteArrayList`, `ConcurrentHashMap`.
+ Перебирати копію колекції (а змінювати оригінал).
+ За багатопотоковості — синхронізувати доступ (`synchronized` чи блокування).

[до змісту](#java-collections-framework)

## Яка колекція реалізує дисципліну FIFO?
FIFO (First-In-First-Out, «першим прийшов — першим пішов») — за цим принципом працює `Queue` (черга).

[до змісту](#java-collections-framework)

## Яка колекція реалізує дисципліну FILO?
FILO/LIFO (First-In-Last-Out / Last-In-First-Out, «першим прийшов — останнім пішов») — за цим принципом працює стек. Класичний `Stack` — legacy; сучасна заміна — `ArrayDeque` (`push()`/`pop()`).

[до змісту](#java-collections-framework)

## Чим відрізняється `ArrayList` від `Vector`? Навіщо додали `ArrayList`?
+ Методи `Vector` **синхронізовані**, а `ArrayList` — ні (тому `ArrayList` швидший в однопотоковому коді).
+ При нестачі місця `Vector` зазвичай **подвоює** розмір, а `ArrayList` збільшує приблизно на **50%**.

`Vector` — **legacy**-клас, його використання не рекомендоване. Якщо потрібен потокобезпечний список — краще `CopyOnWriteArrayList` або `Collections.synchronizedList(...)`.

[до змісту](#java-collections-framework)

## Чим відрізняється `ArrayList` від `LinkedList`? Коли який краще?
`ArrayList` — список на основі **масиву**; `LinkedList` — класичний **двозв'язний список** із вузлів.

**ArrayList:**
+ доступ за індексом — O(1);
+ пошук за значенням — O(N);
+ додавання в кінець — амортизовано O(1);
+ вставка/видалення в середині — O(N) (елементи праворуч зсуваються);
+ мінімум накладних витрат пам'яті.

**LinkedList:**
+ доступ за індексом — O(N);
+ доступ до першого/останнього елемента — O(1);
+ додавання/видалення на початку чи в кінці — O(1);
+ вставка/видалення в середині — O(1) **за наявності ітератора на цій позиції**, але пошук позиції — O(N);
+ більше пам'яті на елемент (зберігаються ще два посилання).

> **Практичний висновок (важливо для співбесіди):** у переважній більшості випадків використовують **`ArrayList`** — він швидший за рахунок локальності даних у пам'яті та `System.arraycopy`, і економніший. `LinkedList` виправданий рідко: коли постійно додаєте/видаляєте з кінців (черга/дек) — і навіть тоді зазвичай кращий `ArrayDeque`.

[до змісту](#java-collections-framework)

## Що працює швидше — `ArrayList` чи `LinkedList`?
Залежить від операції. Для доступу за індексом і загалом на практиці — `ArrayList`. Для частих вставок/видалень із кінців — `LinkedList`/`ArrayDeque`. Деталі — у [попередньому питанні](#Чим-відрізняється-ArrayList-від-LinkedList-Коли-який-краще).

[до змісту](#java-collections-framework)

## Найгірший час `contains()` для `LinkedList`?
O(N) — пошук лінійний (перебір усіх елементів).

[до змісту](#java-collections-framework)

## Найгірший час `contains()` для `ArrayList`?
O(N) — пошук теж лінійний (перебір масиву).

[до змісту](#java-collections-framework)

## Найгірший час `add()` для `LinkedList`?
Додавання в кінець/початок — O(1). У контексті інтерфейсу `List` метод `add(index, element)` потребує O(N) на пошук позиції.

[до змісту](#java-collections-framework)

## Найгірший час `add()` для `ArrayList`?
Амортизовано O(1) для додавання в кінець, але **найгірший випадок — O(N)**: коли місткості масиву не вистачає, створюється новий більший масив і всі елементи копіюються.

[до змісту](#java-collections-framework)

## Треба додати 1 млн елементів — яку структуру оберете?
Залежить від сценарію: куди додаються елементи, що з ними робитимуть далі, чи є обмеження пам'яті/швидкості. Якщо додавання переважно в кінець і потрібен швидкий доступ — `ArrayList` (бажано з наперед заданою місткістю `new ArrayList<>(1_000_000)`, щоб уникнути перевиділень). Деталі — у [порівнянні `ArrayList`/`LinkedList`](#Чим-відрізняється-ArrayList-від-LinkedList-Коли-який-краще).

[до змісту](#java-collections-framework)

## Як відбувається видалення елементів з `ArrayList`? Як змінюється його розмір?
При видаленні елемента всі елементи праворуч зсуваються на одну позицію ліворуч (`System.arraycopy`), а `size` зменшується. **Місткість (capacity) масиву не зменшується автоматично** — механізм авторозширення є, а автостиснення немає. Стиснути можна вручну методом `trimToSize()`.

[до змісту](#java-collections-framework)

## Ефективний алгоритм видалення кількох суміжних елементів із середини `ArrayList`.
Замість того щоб викликати `remove()` `n` разів (щоразу зсуваючи «хвіст» — це O(n·N)), краще видалити весь діапазон за один прохід. Найпростіше й найшвидше — `subList(from, to).clear()`, який усередині використовує `System.arraycopy()`:

```java
list.subList(m, m + n).clear(); // видалити n елементів, починаючи з індексу m
```

Це один зсув «хвоста» — O(N) замість O(n·N).

[до змісту](#java-collections-framework)

## Скільки додаткової пам'яті потрібно при `ArrayList.add()`?
Якщо в масиві є місце — додаткова пам'ять не потрібна. Інакше створюється новий масив у **~1,5 раза** більший за поточний, і всі елементи копіюються в нього (старий потім збере GC).

[до змісту](#java-collections-framework)

## Скільки пам'яті виділяється при `LinkedList.add()`?
Створюється один новий об'єкт вкладеного класу `Node` (вузол із даними та двома посиланнями).

[до змісту](#java-collections-framework)

## Скільки пам'яті на зберігання одного `byte` у `LinkedList`?
Кожен елемент `LinkedList` — це вузол `Node` із трьома посиланнями (на дані, наступний і попередній вузли):
```java
private static class Node<E> {
    E item;
    Node<E> next;
    Node<E> prev;
}
```
Оцінка для **сучасної 64-бітної JVM зі стисненими посиланнями (compressed oops)** — типовий випадок:
+ вузол `Node`: заголовок об'єкта (12 байт) + 3 посилання × 4 байти = 24 байти (вирівняно до 8);
+ значення `byte` пакується в об'єкт `Byte`: заголовок (12) + 1 байт + вирівнювання = **16 байт** (хоча для значень `-128..127` `Byte` кешується, тож новий об'єкт може й не створюватися).

Разом ≈ **24 + 16 = 40 байт** на один елемент. (Без стиснених посилань, на дуже великих купах, цифри більші: ~40 + 24 = 64 байти.)

> Точні цифри залежать від JVM і налаштувань; на співбесіді достатньо ідеї: `LinkedList` дуже «дорогий» для дрібних значень через вузли й пакування.

[до змісту](#java-collections-framework)

## Скільки пам'яті на зберігання одного `byte` у `ArrayList`?
`ArrayList` зберігає лише **посилання** на об'єкти в масиві, а примітив пакується в `Byte`. На сучасній 64-бітній JVM зі стисненими посиланнями:
+ посилання в масиві — 4 байти;
+ запакований `Byte` — 16 байт.

Разом ≈ **20 байт** на елемент — суттєво менше за `LinkedList` (бо немає вузлів із додатковими посиланнями).

[до змісту](#java-collections-framework)

## Для `ArrayList` чи `LinkedList` додавання в середину повільніше?
`list.add(list.size()/2, newElement)`:

**ArrayList:**
+ перевірка місткості (за потреби — новий масив, O(N));
+ зсув правої половини на 1 позицію (`System.arraycopy`, O(N));
+ вставка (O(1)).

**LinkedList:**
+ пошук позиції — прохід до середини (O(N));
+ вставка (O(1)).

Обидва O(N), але на практиці `ArrayList` часто **швидший**, бо зсув виконує нативний `System.arraycopy`, тоді як `LinkedList` робить повільний послідовний обхід посилань до середини.

[до змісту](#java-collections-framework)

## Навіщо в `ArrayList` окреме поле `size`, якщо є `elementData.length`?
`elementData.length` — це **місткість (capacity)**, тобто розмір внутрішнього масиву, який зазвичай **більший** за реальну кількість елементів. `size` — це **фактична** кількість елементів. Масив навмисно тримають із запасом, щоб не перевиділяти його при кожному `add()`; тому два значення різні й `size` зберігається окремо.

[до змісту](#java-collections-framework)

## Порівняйте інтерфейси `Queue` і `Deque`. Хто кого розширює?
+ `Queue` — черга, зазвичай (але не обов'язково) FIFO: витяг із початку, вставка в кінець. Принцип може порушуватися, наприклад, у `PriorityQueue` (порядок за пріоритетом).
+ `Deque` (Double Ended Queue) — **розширює `Queue`**; це лінійна колекція зі вставкою/витягом з **обох кінців**. Дозволяє будувати і FIFO (черга), і LIFO (стек).

Отже, **`Deque` розширює `Queue`**. Реалізації зазвичай не перевизначають `equals()`/`hashCode()` (порівняння за посиланням).

[до змісту](#java-collections-framework)

## Чому `LinkedList` реалізує і `List`, і `Deque`?
`LinkedList` дозволяє додавати/видаляти елементи на початку й у кінці за O(1) — це природно лягає на інтерфейс `Deque`. Водночас він зберігає порядок та індексацію — це інтерфейс `List`. Тож клас реалізує обидва.

[до змісту](#java-collections-framework)

## `LinkedList` — однозв'язний, двозв'язний чи чотиризв'язний список?
**Двозв'язний**: кожен вузол зберігає посилання на попередній і наступний елементи.

[до змісту](#java-collections-framework)

## Як перебрати `LinkedList` у зворотному порядку без `get(index)`?
Використати зворотний ітератор: `descendingIterator()`. (`get(index)` для `LinkedList` повільний — O(N) на кожен виклик.)

[до змісту](#java-collections-framework)

## Що дозволяє `PriorityQueue`?
`PriorityQueue` — черга з пріоритетом: елементи витягуються не за порядком додавання, а за **пріоритетом**. За замовчуванням — natural ordering, або через переданий `Comparator`. Усередині — двійкова купа (binary heap). Не дозволяє `null`. Застосування: алгоритм Дейкстри, планувальники задач, «top-N» елементів тощо.

[до змісту](#java-collections-framework)

## `Stack` вважається застарілим. Чим його замінити й чому?
`Stack` доданий ще в Java 1.0 як LIFO-стек, але успадковує `Vector` — а це і повна синхронізація (втрата продуктивності), і порушення абстракції стека (через `Vector` можна звертатися до будь-якого елемента за індексом). Тому рекомендують реалізації `Deque` — передусім **`ArrayDeque`** (`push()`/`pop()`/`peek()`): швидший, без зайвої синхронізації, з чистою семантикою стека.

[до змісту](#java-collections-framework)

## Навіщо `HashMap`, якщо є `Hashtable`?
+ Методи `Hashtable` **синхронізовані** (повільніше), `HashMap` — ні;
+ `Hashtable` **не дозволяє** `null`, `HashMap` дозволяє один ключ `null` і будь-які значення `null`;
+ `HashMap` має fail-fast ітератор, `Hashtable` — застарілий `Enumeration`.

`Hashtable` — **legacy**-клас. Для потокобезпечності замість нього використовують **`ConcurrentHashMap`** (значно ефективніший за рахунок дрібнішого блокування).

[до змісту](#java-collections-framework)

## Різниця між `HashMap` і `IdentityHashMap`? Навіщо `IdentityHashMap`?
`IdentityHashMap` теж реалізує `Map`, але при порівнянні ключів використовує **порівняння посилань (`==`)**, а не `equals()`. Тобто два ключі рівні, лише якщо це **той самий об'єкт** (`k1 == k2`). Замість `hashCode()` використовує `System.identityHashCode()`.

Навіщо:
+ продуктивність, коли `equals()`/`hashCode()` дорогі (порівняння посилань дешеве);
+ незмінність ключа не потрібна (бо не використовуються `equals`/`hashCode`);
+ алгоритми обходу графів об'єктів (серіалізація, глибоке клонування), де треба відстежувати **саме об'єкти**, навіть якщо вони логічно «рівні».

```java
Map<CreditCard, String> normal = new HashMap<>();
Map<CreditCard, String> identity = new IdentityHashMap<>();
// якщо змінити поле ключа, що бере участь у hashCode:
// у HashMap елемент "загубиться", а в IdentityHashMap знайдеться (бо порівнюється посилання)
```

[до змісту](#java-collections-framework)

## Різниця між `HashMap` і `WeakHashMap`? Навіщо `WeakHashMap`?
У Java є 4 типи посилань: **сильні (strong)**, **м'які (soft)**, **слабкі (weak)** та **фантомні (phantom)**. Якщо до об'єкта можна дістатися лише через ланцюжок слабких посилань (сильних/м'яких немає), GC позначить його на видалення.

`WeakHashMap` зберігає **ключі через слабкі посилання**. Тож пара «ключ-значення» автоматично видаляється, щойно на об'єкт-ключ не лишається сильних посилань.

Застосування: розширення сторонніх об'єктів додатковими даними без зміни їхнього класу. Кладеш об'єкт ключем, а додаткову інформацію — значенням; поки об'єкт «живий» (є сильні посилання деінде), запис доступний, а щойно об'єкт зникає — запис автоматично прибирається. Типово використовується для кешів і метаданих.

[до змісту](#java-collections-framework)

## Чому б не зробити `SoftHashMap` на SoftReference?
Таку структуру цілком можна зробити (м'які посилання GC прибирає лише за нестачі пам'яті — це зручно для кешів). У стандартній бібліотеці її немає, але є в сторонніх (наприклад, Apache Commons), або реалізують самостійно на базі `SoftReference` + `ReferenceQueue`.

[до змісту](#java-collections-framework)

## Чому б не зробити `PhantomHashMap` на PhantomReference?
`PhantomReference.get()` **завжди повертає `null`** — дістати об'єкт через нього неможливо. Фантомні посилання потрібні для відстеження моменту фінального видалення об'єкта (заміна `finalize()`), а не для зберігання й доступу. Тому мапа на них не має сенсу.

[до змісту](#java-collections-framework)

## `LinkedHashMap` — що в ньому від `LinkedList`, а що від `HashMap`?
`LinkedHashMap` — це `HashMap` (та сама хеш-таблиця для O(1) доступу) **плюс** двозв'язний список поверх елементів, що задає **порядок перебору**.

+ За замовчуванням порядок — **порядок додавання** (insertion-order).
+ Якщо в конструкторі задати `accessOrder = true`, порядок стає **за останнім доступом** (access-order): при `get()`/`put()` елемент переміщується в кінець списку. Це основа для LRU-кешу.

Повторне додавання вже наявного ключа не змінює порядок перебору (для insertion-order).

[до змісту](#java-collections-framework)

## У чому проявляється «впорядкованість» `SortedMap`?
`SortedMap` (і `SortedSet`) тримає елементи **відсортованими за ключем**. Це проявляється не лише в `toString()`, а й при **ітерації** (елементи йдуть у відсортованому порядку), і дає додаткові методи: `first()`/`last()`, `headMap()`/`tailMap()`/`subMap()`, а `NavigableMap` — ще й `ceilingKey()`, `floorKey()`, `higherKey()`, `lowerKey()` тощо.

[до змісту](#java-collections-framework)

## Як влаштований `HashMap`?
Це ключове питання про колекції — варто розуміти його глибоко.

**Базова ідея.** `HashMap` зберігає пари «ключ-значення» в масиві **кошиків (buckets)**. Щоб знайти кошик для ключа, обчислюється його `hashCode()`, який перетворюється на індекс у масиві. У середньому це дає доступ за **O(1)**, бо ми одразу «стрибаємо» в потрібну комірку, а не перебираємо все.

**Покладання елемента (`put`):**
1. Обчислюється `hash` ключа (`HashMap` ще й «перемішує» біти: `h ^ (h >>> 16)`, щоб старші біти впливали на індекс).
2. За хешем визначається номер кошика: `index = hash & (capacity - 1)` (тому місткість завжди степінь двійки).
3. Якщо кошик порожній — туди кладеться новий вузол.
4. Якщо кошик зайнятий (**колізія**) — елемент додається в той самий кошик, де елементи зберігаються:
   + спершу як **зв'язний список** вузлів;
   + а коли список у кошику стає довгим (**8 елементів**) і загальна місткість ≥ 64, він перетворюється на **збалансоване червоно-чорне дерево** — це покращує найгірший випадок з O(N) до O(log N) (нововведення Java 8). При зменшенні до 6 елементів дерево знову стає списком.
5. Якщо в кошику знайдено вузол із **таким самим ключем** (рівний `hashCode()` **і** `equals()`) — значення **перезаписується**.

**Пошук (`get`):** обчислити хеш → знайти кошик → пройти список/дерево, порівнюючи ключі через `equals()`.

**Розширення (resize).** У `HashMap` є `loadFactor` (за замовчуванням **0.75**). Коли кількість елементів перевищує `capacity * loadFactor`, масив кошиків збільшується **вдвічі**, і всі елементи перерозподіляються (rehash) під нову місткість.

> **Сучасна деталь:** внутрішній клас вузла називається `Node<K,V>` (а не `Entry`, як у старих версіях до Java 8); для деревних кошиків — `TreeNode<K,V>`.

[до змісту](#java-collections-framework)

## Відкрита адресація чи метод ланцюжків — як реалізовано `HashMap` і чому?
`HashMap` реалізовано методом **ланцюжків (separate chaining)**: кожному кошику відповідає окремий список (або дерево), і при колізії новий елемент додається туди.

**Метод ланцюжків:**
+ коефіцієнт заповнення може бути > 1; продуктивність зі зростанням кількості елементів спадає поступово;
+ зручний, коли кількість елементів наперед невідома.

**Відкрита адресація** (лінійне/квадратичне пробування, подвійне хешування) — колізії розв'язуються пошуком іншої вільної комірки в самому масиві:
+ плюси: немає витрат на об'єкти-вузли; простіша серіалізація;
+ мінуси: кількість елементів обмежена розміром масиву; різке падіння продуктивності при високому заповненні; складніше видаляти елементи; проблема кластеризації.

Метод ланцюжків обрали через гнучкість (довільна кількість елементів) і простоту видалення.

[до змісту](#java-collections-framework)

## Як працює `HashMap` при однаковому `hashCode()`, але `equals() == false`?
За `hashCode()` обидва ключі потраплять в **один кошик**. Перед додаванням `HashMap` перевіряє наявні в кошику елементи: якщо знайдено елемент із таким самим хешем, але `equals()` повертає `false` — це **різні ключі**, тож новий елемент **додається** в кошик (у список або дерево). Якщо ж `equals()` повертає `true` — значення **перезаписується**.

[до змісту](#java-collections-framework)

## Яка початкова кількість кошиків у `HashMap`?
У конструкторі за замовчуванням — **16**. Через конструктори з параметрами можна задати власну початкову місткість (вона округлюється вгору до найближчого степеня двійки).

[до змісту](#java-collections-framework)

## Яка часова складність операцій `HashMap`? Чи гарантується вона?
У середньому додавання, пошук і видалення — **O(1)**.

Це не гарантується: усе залежить від якості хеш-функції. При багатьох колізіях у кошику складність зростає, але **з Java 8** довгі ланцюжки перетворюються на червоно-чорне дерево, тож навіть у найгіршому випадку буде не гірше за **O(log N)** (а не O(N), як було раніше). Якщо ж хеш-функція повертає завжди однакове значення, усі елементи опиняться в одному кошику.

[до змісту](#java-collections-framework)

## Чи може `HashMap` виродитися у список навіть з різними `hashCode()`?
Так — якщо різні хеш-коди після перетворення на індекс кошика (`hash & (capacity - 1)`) дають **однаковий номер кошика**. Тоді елементи з різними хешами все одно скупчаться в одному кошику.

[до змісту](#java-collections-framework)

## У якому випадку елемент може загубитися в `HashMap`?
Якщо ключ — **мутабельний** об'єкт, і після додавання в мапу змінити в нього поле, що бере участь у `hashCode()`. Тоді при пошуку за цим ключем обчислиться вже **інший хеш** → інший кошик → елемент не знайдеться. Навіть якщо випадково потрапити в той самий кошик, після наступного resize елемент із зміненим хешем майже напевно «поїде» в інший кошик і загубиться зовсім.

**Висновок:** ключі `HashMap` мають бути **незмінними** (immutable) за полями, що впливають на `hashCode`/`equals`. Тому `String` — ідеальний ключ.

[до змісту](#java-collections-framework)

## Чому не можна використовувати `byte[]` як ключ у `HashMap`?
Масиви **не перевизначають** `hashCode()` і `equals()` — вони успадковують реалізацію від `Object`: `hashCode()` базується на ідентичності об'єкта, а `equals()` порівнює **посилання**. Тож два різні масиви з однаковим вмістом вважатимуться **різними** ключами. Дістати збережений елемент можна лише за **тим самим посиланням** на масив, що й при збереженні, — це робить `byte[]` некорисним як ключ. Якщо потрібен ключ за вмістом — використовують `String`, `List<Byte>` або обгортку з коректними `equals`/`hashCode`.

[до змісту](#java-collections-framework)

## Яка роль `equals()` і `hashCode()` у `HashMap`?
+ `hashCode()` визначає **кошик** (грубий, швидкий пошук — куди дивитися);
+ `equals()` розрізняє ключі **всередині кошика** (точне порівняння серед колізій).

Обидва потрібні разом: спершу `HashMap` за хешем знаходить кошик, потім через `equals()` шукає в ньому потрібний ключ. Саме тому при перевизначенні `equals()` треба перевизначати й `hashCode()`.

[до змісту](#java-collections-framework)

## Яке максимальне число значень `hashCode()`?
Випливає із сигнатури `int hashCode()` — це діапазон `int`, тобто **2³²** значень.

[до змісту](#java-collections-framework)

## Найгірший час `get(key)` у `HashMap`?
До Java 8 — **O(N)** (виродження кошика у список). З Java 8 завдяки перетворенню довгих ланцюжків на червоно-чорне дерево найгірший випадок — **O(log N)**, навіть при поганій хеш-функції. Це стосується і ключа, що є в мапі, і відсутнього.

[до змісту](#java-collections-framework)

## Чому ключ у `HashMap` не зобов'язаний бути `Comparable`, а дерево все одно будується?
Червоно-чорне дерево потребує порівняння елементів. Зазвичай це робиться через `Comparable.compareTo()`, але вимагати `Comparable` від усіх ключів `HashMap` не стали. Натомість при побудові дерева застосовується такий порядок порівняння:
1. спершу порівнюються **хеші** ключів;
2. якщо хеші рівні й обидва ключі реалізують `Comparable` — викликається `compareTo()`;
3. якщо ні — використовується «запасний» порядок (`tieBreakOrder`): порівняння за іменами класів (`getClass().getName()`), а якщо й вони однакові — за `System.identityHashCode()`.

Так дерево вдається побудувати завжди, навіть без `Comparable`.

[до змісту](#java-collections-framework)

## Скільки переходів відбувається при `HashMap.get(key)`?
Концептуально: обчислити хеш ключа → визначити кошик → знайти елемент у кошику → повернути значення. Для ключа без колізій це фактично прямий доступ — O(1). (Конкретна реалізація змінювалася між версіями Java, тож точне «число кроків» — деталь реалізації, а не щось фіксоване.)

[до змісту](#java-collections-framework)

## Скільки об'єктів створюється при додаванні елемента в `HashMap`?
Один новий об'єкт вузла — `Node<K,V>` (або `TreeNode<K,V>`, якщо кошик уже деревний). Окремо може створитися новий масив кошиків, якщо `put` спричинив resize.

[до змісту](#java-collections-framework)

## Як і коли збільшується кількість кошиків у `HashMap`?
Крім `capacity`, у `HashMap` є `loadFactor` (за замовчуванням **0.75**), що задає поріг заповнення `capacity * loadFactor`. Щойно кількість елементів перевищує цей поріг, кількість кошиків **подвоюється**, і всі елементи перерозподіляються (rehash) з урахуванням нової місткості.

[до змісту](#java-collections-framework)

## Сенс параметрів `HashMap(int initialCapacity, float loadFactor)`.
+ `initialCapacity` — початкова кількість кошиків при створенні мапи. Задавши її наперед під відому кількість елементів, можна уникнути дорогих resize.
+ `loadFactor` — коефіцієнт заповнення, при перевищенні якого мапа розширюється. Це компроміс: менший `loadFactor` → менше колізій, але більше пам'яті; більший → економніше, але повільніше. Стандартні 0.75 — вдалий баланс.

[до змісту](#java-collections-framework)

## Чи працюватиме `HashMap`, якщо всі ключі мають однаковий `hashCode()`?
Так, працюватиме **коректно**, але неефективно: усі елементи потраплять в один кошик. До Java 8 це виродження у список з пошуком O(N); з Java 8 кошик стане деревом і пошук буде O(log N). У будь-якому разі переваги хеш-таблиці втрачаються.

[до змісту](#java-collections-framework)

## Як перебрати всі ключі / значення / пари `Map`?
+ ключі — `map.keySet()` повертає `Set<K>`;
+ значення — `map.values()` повертає `Collection<V>`;
+ пари — `map.entrySet()` повертає `Set<Map.Entry<K, V>>`.

```java
for (Map.Entry<K, V> e : map.entrySet()) {
    K key = e.getKey();
    V value = e.getValue();
}
// або (з Java 8):
map.forEach((key, value) -> { ... });
```
Перебір через `entrySet()` ефективніший за `keySet()` + `get(key)`, бо не робить повторного пошуку для кожного ключа.

[до змісту](#java-collections-framework)

## У чому відмінності `TreeSet` і `HashSet`?
+ `HashSet` — на основі `HashMap`; **без порядку**; операції в середньому **O(1)**.
+ `TreeSet` — на основі червоно-чорного дерева (`TreeMap`); зберігає елементи **відсортованими**; операції **O(log N)**; потребує `Comparable` або `Comparator`; має навігаційні методи (`first`, `last`, `ceiling`, `floor` тощо).

Коротко: потрібен швидкий набір без порядку — `HashSet`; потрібен відсортований набір — `TreeSet`.

[до змісту](#java-collections-framework)

## Що буде, якщо додавати в `TreeSet` за зростанням?
Нічого поганого. В основі `TreeSet` — **самобалансивне** червоно-чорне дерево, тож порядок додавання не має значення: дерево лишається збалансованим, а операції — O(log N). (На відміну від простого бінарного дерева пошуку, яке від відсортованого вводу виродилося б у список.)

[до змісту](#java-collections-framework)

## Чим `LinkedHashSet` відрізняється від `HashSet`?
`LinkedHashSet` усередині використовує `LinkedHashMap` замість `HashMap`. Завдяки цьому порядок перебору елементів **збігається з порядком додавання** (insertion-order). Повторне додавання наявного елемента порядок не змінює. У всьому іншому — як `HashSet`.

[до змісту](#java-collections-framework)

## Навіщо `EnumSet`, якщо є `HashSet`/`TreeSet`?
`EnumSet` — спеціальна реалізація `Set` для `enum`. Зберігає елементи одного типу `enum` за допомогою **бітового вектора** (один біт на можливе значення), що дає надзвичайну компактність і швидкість.

+ усі основні операції — O(1), зазвичай швидші за `HashSet`;
+ пакетні операції (`containsAll`, `retainAll`) — дуже швидкі (побітові операції);
+ перебір — у порядку оголошення констант enum;
+ зручні фабричні методи: `EnumSet.of(...)`, `EnumSet.allOf(...)`, `EnumSet.noneOf(...)`, `EnumSet.range(...)`.

Тобто для enum-ів `EnumSet` майже завжди кращий за `HashSet`/`TreeSet`.

[до змісту](#java-collections-framework)

## Які є способи перебирати елементи списку?
```java
// 1. for-each (найчитабельніший)
for (String e : list) { ... }

// 2. ітератор (дозволяє безпечне видалення через it.remove())
Iterator<String> it = list.iterator();
while (it.hasNext()) { String e = it.next(); ... }

// 3. індексний for
for (int i = 0; i < list.size(); i++) { String e = list.get(i); ... }

// 4. Stream / forEach (з Java 8)
list.forEach(e -> { ... });
list.stream().filter(...).forEach(...);
```
> Для `ArrayList` індексний `for` і for-each однаково швидкі. Для `LinkedList` індексний `get(i)` повільний (O(N) на елемент) — треба for-each/ітератор.

[до змісту](#java-collections-framework)

## Як отримати синхронізовані обгортки стандартних колекцій?
Статичними методами класу `Collections`: `synchronizedList()`, `synchronizedMap()`, `synchronizedSet()`. Вони повертають синхронізований декоратор переданої колекції. **Але** при ітерації потрібна **ручна синхронізація** на самій колекції:
```java
List<String> list = Collections.synchronizedList(new ArrayList<>());
synchronized (list) {            // обов'язково при переборі
    for (String s : list) { ... }
}
```
На практиці для багатопотоковості частіше використовують спеціалізовані колекції з пакета `java.util.concurrent`: `ConcurrentHashMap`, `CopyOnWriteArrayList`, `ConcurrentLinkedQueue` тощо — вони ефективніші за грубу синхронізацію через `synchronized*`.

[до змісту](#java-collections-framework)

## Як отримати колекцію лише для читання?
+ обгортки `Collections.unmodifiableList(list)` / `unmodifiableSet(set)` / `unmodifiableMap(map)` — повертають незмінне **подання** наявної колекції (спроба зміни → `UnsupportedOperationException`);
+ з Java 9 — фабрики незмінних колекцій: `List.of(...)`, `Set.of(...)`, `Map.of(...)`;
+ з Java 10 — `List.copyOf(coll)` / `Set.copyOf(...)` / `Map.copyOf(...)` для незмінної копії.

> Різниця: `unmodifiable*` — це лише обгортка над оригіналом (зміни оригіналу видно крізь неї), а `copyOf`/`of` — повністю незалежні незмінні колекції.

[до змісту](#java-collections-framework)

## Однопотокова програма, що кидає `ConcurrentModificationException`.
Зміна колекції напряму під час перебору через for-each (тобто через ітератор) спрацьовує fail-fast:
```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3));
for (Integer i : list) {
    list.remove(1);   // пряма зміна колекції під час перебору → ConcurrentModificationException
}
```
Правильно було б видаляти через `iterator.remove()` або `list.removeIf(...)`.

[до змісту](#java-collections-framework)

## Приклад, коли колекція кидає `UnsupportedOperationException`.
Спроба змінити незмінну колекцію:
```java
List<Integer> list = List.of(1, 2, 3); // незмінний список
list.add(0);                           // UnsupportedOperationException
```

## Симетрична різниця двох колекцій через методи `Collection`.
Симетрична різниця — елементи, що належать лише одній із колекцій (об'єднання мінус перетин):
```java
<T> Collection<T> symmetricDifference(Collection<T> a, Collection<T> b) {
    Collection<T> result = new ArrayList<>(a);
    result.addAll(b);                        // об'єднання (з дублями)

    Collection<T> intersection = new ArrayList<>(a);
    intersection.retainAll(b);               // перетин

    result.removeAll(intersection);          // прибираємо спільні елементи
    return result;
}
```

## Кеш з «invalidation policy» через `LinkedHashMap`.
Використати **LRU** (Least Recently Used) — `LinkedHashMap` з `accessOrder = true` (елемент при доступі переміщується в кінець, тож найменш уживані лишаються на початку) і перевизначити `removeEldestEntry()`, що автоматично видаляє найстаріший елемент при `put`, коли розмір перевищує ліміт:
```java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private static final int MAX_ENTRIES = 10;

    public LRUCache(int initialCapacity) {
        super(initialCapacity, 0.75f, true); // true = access-order
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > MAX_ENTRIES;
    }
}
```
