## Чим відрізняються JRE, JVM і JDK?
__JVM__, Java Virtual Machine (віртуальна машина Java) — основна частина середовища виконання Java (JRE). Віртуальна машина Java виконує байткод Java, попередньо створений із вихідного тексту Java-програми компілятором Java. JVM також може використовуватися для виконання програм, написаних іншими мовами програмування.

__JRE__, Java Runtime Environment (середовище виконання Java) — мінімально необхідна реалізація віртуальної машини для виконання Java-застосунків. Складається з JVM та стандартного набору бібліотек класів Java.

__JDK__, Java Development Kit (комплект розробки на Java) — це JRE плюс набір інструментів розробника застосунків мовою Java, що містить компілятор Java, стандартні бібліотеки класів, приклади, документацію та різні утиліти.

Коротко: __JDK__ — середовище для розробки програм на Java, що містить у собі __JRE__ — середовище для запуску Java-програм, яке, своєю чергою, містить __JVM__ — виконавця коду Java-програм.

## Які існують модифікатори доступу?
__private__ (приватний): члени класу доступні лише всередині класу. Позначається службовим словом `private`.

__default__, package-private, package level (доступ на рівні пакета): видимість класу/членів класу лише всередині пакета. Це модифікатор доступу за замовчуванням — спеціального позначення не потребує.

__protected__ (захищений): члени класу доступні всередині пакета та в нащадках. Позначається службовим словом `protected`.

__public__ (публічний): клас/члени класу доступні всім. Позначається службовим словом `public`.

Послідовність модифікаторів за зростанням рівня закритості: `public`, `protected`, `default`, `private`.

Під час успадкування можлива зміна модифікаторів доступу в бік більшої видимості (для дотримання _принципу підстановки Лісков_).

## Про що говорить ключове слово `final`?
Модифікатор `final` може застосовуватися до змінних, параметрів методів, полів і методів класу або до самих класів.

+ Клас не може мати нащадків;
+ Метод не може бути перевизначений у класах-нащадках;
+ Поле не може змінити своє значення після ініціалізації;
+ Параметри методів не можуть змінювати своє значення всередині методу;
+ Локальні змінні не можуть бути змінені після присвоєння їм значення.

## Якими значеннями ініціалізуються змінні за замовчуванням?
+ `byte` — `(byte)0`;
+ `short` — `(short)0`;
+ `int` — `0`;
+ `long` — `0L`;
+ `float` — `0f`;
+ `double` — `0d`;
+ `char` — ` `;
+ `boolean` — `false`;
+ Об'єкти (зокрема `String`) — `null`.

> Важливо: значення за замовчуванням отримують лише **поля** (статичні та поля екземпляра). **Локальні змінні** значення за замовчуванням не отримують — їх потрібно ініціалізувати явно, інакше буде помилка компіляції.

## Що ви знаєте про метод `main()`?

Метод `main()` — точка входу в програму. У застосунку може бути кілька таких методів (по одному в різних класах). Якщо метод відсутній, компіляція можлива, але під час запуску буде отримано помилку _`Error: Main method not found`_.

```java
public static void main(String[] args) {}
```

## Які логічні операції та оператори ви знаєте?
+ `&`: логічне _AND_ (І);
+ `&&`: скорочене _AND_ (з коротким замиканням);
+ `|`: логічне _OR_ (АБО);
+ `||`: скорочене _OR_ (з коротким замиканням);
+ `^`: логічне _XOR_ (виключне _OR_);
+ `!`: логічне унарне _NOT_ (НЕ);
+ `&=`: _AND_ із присвоєнням;
+ `|=`: _OR_ із присвоєнням;
+ `^=`: _XOR_ із присвоєнням;
+ `==`: дорівнює;
+ `!=`: не дорівнює;
+ `?:`: тернарний (трійковий) умовний оператор.

> Різниця між `&`/`|` та `&&`/`||`: скорочені оператори (`&&`, `||`) обчислюють правий операнд лише за потреби (короткe замикання), а `&`/`|` завжди обчислюють обидва операнди.

[до змісту](#java-core-частина-1-з-3)

## Що таке тернарний оператор вибору?
Тернарний умовний оператор `?:` — оператор, яким можна замінити деякі конструкції `if-then-else`.

Вираз записується у формі:
>умова ? вираз1 : вираз2

Якщо `умова` виконується, обчислюється `вираз1`, і його результат стає результатом усього оператора. Якщо ж `умова` дорівнює `false`, обчислюється `вираз2`, і його значення стає результатом роботи оператора. Обидва операнди — `вираз1` та `вираз2` — мають повертати значення однакового (або сумісного) типу.

## Які побітові операції ви знаєте?
+ `~`: побітовий унарний оператор NOT;
+ `&`: побітовий AND;
+ `&=`: побітовий AND із присвоєнням;
+ `|`: побітовий OR;
+ `|=`: побітовий OR із присвоєнням;
+ `^`: побітовий виключний XOR;
+ `^=`: побітовий виключний XOR із присвоєнням;
+ `>>`: зсув праворуч (ділення на 2 у степені зсуву);
+ `>>=`: зсув праворуч із присвоєнням;
+ `>>>`: зсув праворуч без урахування знака;
+ `>>>=`: зсув праворуч без урахування знака з присвоєнням;
+ `<<`: зсув ліворуч (множення на 2 у степені зсуву);
+ `<<=`: зсув ліворуч із присвоєнням.

## Де і для чого використовується модифікатор `abstract`?
Клас, позначений модифікатором `abstract`, називається абстрактним класом. Такі класи можуть бути лише предками для інших класів — створювати екземпляри самого абстрактного класу не дозволяється. При цьому нащадками абстрактного класу можуть бути як інші абстрактні класи, так і класи, що допускають створення об'єктів.

Метод, позначений ключовим словом `abstract`, — це абстрактний метод, тобто метод без реалізації. Якщо в класі є хоча б один абстрактний метод, увесь клас має бути оголошений абстрактним.

Використання абстрактних класів і методів дозволяє описати певний шаблон об'єкта, який має бути реалізований в інших класах. У самих же абстрактних класах описується лише деяка спільна для всіх нащадків поведінка.

## Дайте визначення поняттю _«інтерфейс»_. Які модифікатори за замовчуванням мають поля та методи інтерфейсів?
Ключове слово `interface` використовується для створення повністю абстрактних типів. Основне призначення інтерфейсу — визначати, яким чином можна використовувати клас, що його реалізує. Автор інтерфейсу визначає імена методів, списки аргументів і типи значень, що повертаються, але не реалізує їхньої поведінки. Усі методи неявно оголошуються як `public`.

Починаючи з Java 8, в інтерфейсах дозволено розміщувати реалізацію методів за замовчуванням (`default`) та статичних (`static`) методів. Починаючи з Java 9, дозволено також приватні (`private`) методи — як допоміжні для `default`-методів.

Інтерфейс також може містити поля. У цьому випадку вони автоматично є публічними (`public`), статичними (`static`) та незмінними (`final`), тобто фактично константами.

## Чим абстрактний клас відрізняється від інтерфейсу? У яких випадках слід використовувати абстрактний клас, а в яких інтерфейс?
+ У Java клас може одночасно реалізувати кілька інтерфейсів, але успадковуватися лише від одного класу.
+ Абстрактні класи використовуються тоді, коли є відношення типу «is a» (є). Інтерфейси можуть реалізовуватися класами, не пов'язаними один з одним.
+ Абстрактний клас — засіб, що дозволяє уникнути написання повторюваного коду, інструмент для часткової реалізації поведінки. Інтерфейс — це засіб вираження семантики класу, контракт, що описує можливості. Усі методи інтерфейсу неявно оголошуються як `public abstract` або (починаючи з Java 8) як `default`/`static` методи з реалізацією, а поля — як `public static final`.
+ Інтерфейси дозволяють створювати структури типів без ієрархії.
+ Успадковуючись від абстрактного класу, клас «розчиняє» власну індивідуальність. Реалізуючи інтерфейс, він розширює власну функціональність.

Абстрактні класи містять часткову реалізацію, яка доповнюється або розширюється в підкласах. При цьому всі підкласи схожі між собою в частині реалізації, успадкованої від абстрактного класу, і відрізняються лише власною реалізацією абстрактних методів предка. Тому абстрактні класи застосовують при побудові ієрархії однотипних, дуже схожих один на одного класів — успадкування від абстрактного класу, що реалізує поведінку за замовчуванням, дозволяє уникнути повторюваного коду. У решті випадків краще використовувати інтерфейси.

## Чому в деяких інтерфейсах взагалі не визначають методів?
Це так звані _маркерні інтерфейси_ (marker interfaces). Вони просто вказують, що клас належить до певного типу. Прикладом може бути інтерфейс `Cloneable`, який позначає, що клас підтримує механізм клонування, або `Serializable`, що позначає можливість серіалізації.

[до змісту](#java-core-частина-1-з-3)

## Чому не можна оголосити метод інтерфейсу з модифікатором `final`?
Для абстрактних методів інтерфейсу вказувати `final` немає сенсу: усі такі методи неявно абстрактні, тобто їх неможливо виконати, не реалізувавши десь в іншому місці, а цього не можна було б зробити, якби метод був `final` (бо `final` забороняє перевизначення).

> Зауваження: це стосується саме абстрактних методів. `static`-методи інтерфейсу теж не можуть бути `final`. А ось `default`-методи `final` бути не можуть за специфікацією — хоча технічно вони мають тіло, мова це забороняє.

## Чи може об'єкт отримати доступ до члена класу, оголошеного як `private`? Якщо так, то яким чином?
+ Усередині класу доступ до приватної змінної відкритий без обмежень;
+ Вкладений клас має повний доступ до всіх (зокрема приватних) членів класу, що його містить;
+ Доступ до приватних змінних ззовні можна організувати через відмінні від приватних методи, надані розробником класу. Наприклад: `getX()` та `setX()`;
+ Через механізм рефлексії (Reflection API):

```java
class Victim {
    private int field = 42;
}
//...
Victim victim = new Victim();
Field field = Victim.class.getDeclaredField("field");
field.setAccessible(true);
int fieldValue = (int) field.get(victim);
//...
```

## Який порядок виклику конструкторів і блоків ініціалізації з урахуванням ієрархії класів?
Спочатку викликаються всі статичні блоки — у порядку від першого статичного блоку кореневого предка й далі вниз по ланцюжку ієрархії до статичних блоків самого класу. Статичні блоки виконуються лише один раз — при першій ініціалізації класу.

Потім викликаються нестатичні блоки ініціалізації кореневого предка, конструктор кореневого предка і так далі — аж до нестатичних блоків і конструктора самого класу.

>Статичний(і) блок(и) предка → статичний(і) блок(и) нащадка → статичний(і) блок(и) онука
>
> → нестатичний(і) блок(и) предка → конструктор предка →
>
> → нестатичний(і) блок(и) нащадка → конструктор нащадка →
>
> → нестатичний(і) блок(и) онука → конструктор онука

Приклад 1:

```java
public class MainClass {

    public static void main(String args[]) {
        System.out.println(TestClass.v);
        new TestClass().a();
    }

}
```

```java
public class TestClass {

    public static String v = "Some val";

    {
        System.out.println("!!! Non-static initializer");
    }

    static {
        System.out.println("!!! Static initializer");
    }

    public void a() {
        System.out.println("!!! a() called");
    }

}
```

Результат виконання:

```
!!! Static initializer
Some val
!!! Non-static initializer
!!! a() called
```

Приклад 2:

```java
public class MainClass {

    public static void main(String args[]) {
        new TestClass().a();
    }

}
```


## Навіщо потрібні та які бувають блоки ініціалізації?
Блок ініціалізації — це код у фігурних дужках, розміщений усередині класу поза методами й конструкторами. Бувають **статичні** (`static { ... }`) та **нестатичні** (просто `{ ... }`).

### Навіщо вони потрібні
Здебільшого поле можна ініціалізувати прямо при оголошенні (`int x = 5;`) або в конструкторі. Блоки потрібні в ситуаціях, коли цього **недостатньо**:

+ **Ініціалізація потребує логіки, а не одного виразу** — циклу, `try/catch`, кількох кроків. Безпосередньо при оголошенні поля таке не напишеш:
  ```java
  static final Map<String, Integer> CODES = new HashMap<>();
  static {
      // наповнення вимагає кількох інструкцій — у виразі при оголошенні так не зробиш
      CODES.put("UA", 380);
      CODES.put("PL", 48);
  }
  ```
+ **Спільний код для кількох конструкторів** — якщо в класі є кілька конструкторів і всі мають виконати однакову підготовку, її можна винести в нестатичний блок, щоб не дублювати. Такий блок виконається перед **кожним** конструктором, тож код пишеться один раз. (Хоча частіше для цього все ж викликають один конструктор з іншого через `this(...)`.)
+ **Ініціалізація в анонімному класі**, де власного конструктора оголосити не можна взагалі — нестатичний блок є єдиним способом виконати код при створенні об'єкта:
  ```java
  List<String> list = new ArrayList<>() {{
      add("a"); // блок ініціалізації всередині анонімного класу
      add("b");
  }};
  ```
+ **Одноразове налаштування на рівні класу** — статичний блок виконує дороге чи складне налаштування один раз при завантаженні класу (наприклад, завантажити нативну бібліотеку, прочитати конфіг, побудувати кеш).

### Статичний vs нестатичний
+ **Статичний блок** належить класу, виконується **один раз** — коли клас ініціалізується завантажувачем класів (перед першим використанням класу). Має доступ лише до статичних членів.
+ **Нестатичний блок** належить об'єкту, виконується **щоразу** при створенні екземпляра — після конструктора суперкласу, але **перед** тілом конструктора самого класу. Має доступ і до полів екземпляра, і до статичних.

### Що варто пам'ятати
+ Кількох блоків може бути декілька — вони виконуються згори вниз у порядку запису в коді (упереміш із ініціалізаторами полів).
+ Нестатичний блок може кидати перевірюваний (checked) виняток лише якщо цей виняток оголошено в `throws` **усіх** конструкторів класу; статичний блок узагалі не може кидати checked-винятки.

> На практиці блоки ініціалізації — рідкісний інструмент. Зазвичай простішого й зрозумілішого результату досягають ініціалізацією при оголошенні або в конструкторі; блок беруть, коли цих способів справді не вистачає (логіка, анонімний клас, одноразове налаштування класу).

## Для чого в Java використовуються статичні блоки ініціалізації?
Статичні блоки ініціалізації використовуються для виконання коду, який має виконатися один раз при ініціалізації класу завантажувачем класів — у момент, що передує створенню об'єктів цього класу за допомогою конструктора. Такий блок (на відміну від нестатичних, що належать конкретному об'єкту класу) належить лише самому класу (об'єкту метакласу `Class`).

[до змісту](#java-core-частина-1-з-3)

## Що станеться, якщо в блоці ініціалізації виникне виняткова ситуація?
Для нестатичних блоків ініціалізації: якщо викидання винятку прописано явно, потрібно, щоб оголошення цих винятків були перелічені в `throws` усіх конструкторів класу. Інакше буде помилка компіляції. Для статичного блоку явне викидання перевірюваного (checked) винятку призводить до помилки компіляції.

У решті випадків взаємодія з винятками відбувається так само, як і в будь-якому іншому місці. Клас не буде ініціалізовано, якщо помилка виникає в статичному блоці, а об'єкт класу не буде створено, якщо помилка виникає в нестатичному блоці.

[до змісту](#java-core-частина-1-з-3)

## Який виняток викидається при виникненні помилки в блоці ініціалізації класу?
Якщо виняток, що виник, — нащадок `RuntimeException`:

+ для статичних блоків ініціалізації буде викинуто `java.lang.ExceptionInInitializerError`;
+ для нестатичних буде проброшено виняток-джерело.

Якщо виняток, що виник, — нащадок `Error`, то в обох випадках буде викинуто цей `Error`. Виняток становить `java.lang.ThreadDeath` (смерть потоку) — у цьому випадку жодного винятку не буде обгорнуто.

[до змісту](#java-core-частина-1-з-3)

## Чи може статичний метод бути перевизначений або перевантажений?
Перевантажений (overloaded) — так. Усе працює так само, як і зі звичайними методами: два статичні методи можуть мати однакове ім'я, якщо кількість їхніх параметрів або типи різняться.

Перевизначений (overridden) — ні. Вибір статичного методу відбувається при ранньому зв'язуванні (на етапі компіляції, а не виконання), і викликатиметься завжди метод того типу, через який звертаються. Якщо в підкласі оголосити статичний метод із такою самою сигнатурою, це **приховування методу (method hiding)**, а не перевизначення — синтаксично коректна конструкція, але поліморфізм на неї не поширюється.

Загалом до статичних полів і методів рекомендується звертатися через ім'я класу, а не через об'єкт.

[до змісту](#java-core-частина-1-з-3)

## Чи можуть нестатичні методи перевантажити статичні?
Так. У підсумку вийде два різні методи. Статичний належатиме класу й буде доступний через його ім'я, а нестатичний належатиме конкретному об'єкту й буде доступний через виклик методу цього об'єкта.

## Чи можна звузити рівень доступу/тип значення, що повертається, при перевизначенні методу?

+ При перевизначенні методу не можна звужувати модифікатор доступу (наприклад, з `public` у батьківському класі до `private` у класі-нащадку).
+ Тип значення, що повертається, при перевизначенні **можна** звузити, якщо типи сумісні (так звані _коваріантні типи повернення_) — замість батьківського типу повертати тип-нащадок.
+ Замінити тип повернення на несумісний не можна — буде помилка `attempting to use incompatible return type`.

Наприклад:

```java
public class Animal {

    public Animal eat() {
        System.out.println("animal eat");
        return null;
    }

    public Long calc() {
        return null;
    }

}
public class Dog extends Animal {

    public Dog eat() { // Dog — нащадок Animal, тож коваріантне звуження дозволене
        return new Dog();
    }
/* attempting to use incompatible return type — Integer не є нащадком Long
    public Integer calc() {
        return null;
    }
*/
}
```

## Чи можливо при перевизначенні методу змінити: модифікатор доступу, тип повернення, тип/кількість/імена/порядок аргументів, секцію `throws`?
При перевизначенні методу звужувати модифікатор доступу не дозволяється, бо це порушило б принцип підстановки Лісков. Розширення рівня доступу можливе.

Можна змінювати все, що не заважає компілятору зрозуміти, який саме метод батьківського класу мається на увазі:

+ Тип значення, що повертається, при перевизначенні можна змінювати лише в бік звуження типу (коваріантно: замість батьківського класу — нащадок).
+ При зміні типу, кількості чи порядку аргументів замість перевизначення відбудеться _перевантаження (overloading)_ методу.
+ Імена аргументів значення не мають — їх можна змінювати вільно.
+ Секцію `throws` можна не вказувати, але варто пам'ятати, що вона лишається чинною, якщо вже визначена в методі батьківського класу. Також можна додавати нові винятки, що є нащадками вже оголошених, або винятки типу `RuntimeException`. Порядок елементів у `throws` при перевизначенні значення не має.

## Чи можна оголосити метод абстрактним і статичним одночасно?
Ні. У такому разі компілятор видасть помилку: _"Illegal combination of modifiers: 'abstract' and 'static'"_. Модифікатор `abstract` каже, що метод буде реалізований в іншому класі (через перевизначення), а `static`, навпаки, вказує, що метод належить класу й не бере участі в поліморфізмі — ці вимоги суперечать одна одній.

[до змісту](#java-core-частина-1-з-3)

## У чому різниця між членом екземпляра класу та статичним членом класу?
Модифікатор `static` означає, що метод чи поле належать самому класу, і доступ до них можливий навіть без створення екземпляра класу. Поля, позначені `static`, ініціалізуються при ініціалізації класу. На методи, оголошені як `static`, накладається низка обмежень:

+ Вони можуть напряму викликати лише інші статичні методи.
+ Вони мають звертатися напряму лише до статичних змінних.
+ Вони не можуть посилатися на `this` чи `super`.

На відміну від статичних, поля екземпляра класу належать конкретному об'єкту й можуть мати різні значення для кожного. Виклик методу екземпляра можливий лише після попереднього створення об'єкта класу.

Приклад:
```java
public class MainClass {

	public static void main(String args[]) {
		System.out.println(TestClass.v);
		new TestClass().a();
		System.out.println(TestClass.v);
	}

}
```

```java
public class TestClass {

	public static String v = "Initial val";

	{
		System.out.println("!!! Non-static initializer");
		v = "Val from non-static";
	}

	static {
		System.out.println("!!! Static initializer");
		v = "Some val";
	}

	public void a() {
		System.out.println("!!! a() called");
	}

}
```

Результат:

```
!!! Static initializer
Some val
!!! Non-static initializer
!!! a() called
Val from non-static
```

+ Нестатичні поля можна ініціалізувати при оголошенні, у нестатичному блоці ініціалізації або в конструкторі.

## Які типи класів бувають у Java?
+ _Top level class_ (клас верхнього рівня):

## Розкажіть про вкладені класи. У яких випадках вони застосовуються?
Вкладений клас (_Nested class_) — це клас, оголошений усередині іншого класу. Існує **4 види**, які найпростіше розрізняти за двома питаннями: **чи прив'язаний клас до об'єкта зовнішнього класу** і **чи має він ім'я**.

| Вид | Прив'язаний до об'єкта зовнішнього класу? | Має ім'я? | Де оголошений |
|---|---|---|---|
| **Static nested** | ❌ ні | ✅ так | усередині класу |
| **Inner (member)** | ✅ так | ✅ так | усередині класу |
| **Local** | ✅ так | ✅ так | усередині методу |
| **Anonymous** | ✅ так | ❌ ні | у виразі, на місці |

Головна лінія поділу: **static nested стоїть осторонь** — він НЕ потребує екземпляра зовнішнього класу. Решта три (inner, local, anonymous) називають **внутрішніми (inner)** класами: кожен їхній об'єкт прив'язаний до об'єкта зовнішнього класу й має доступ до всіх його полів (зокрема приватних).

Загальне правило: вкладений клас створюють лише щоб обслуговувати зовнішній клас. Якщо він корисний деінде — має стати класом верхнього рівня.

> Стисле порівняння для пам'яті: **static nested не залежить від об'єкта; решта три — залежать. Local + anonymous сьогодні майже завжди замінюють лямбдами.**

[до змісту](#java-core-частина-1-з-3)

## Що таке _«статичний клас»_? (Static nested)
Звичайний клас, який поклали всередину іншого заради групування. Слово `static` означає, що йому **не потрібен об'єкт зовнішнього класу** — і доступ він має лише до статичних членів зовнішнього класу.

```java
class Outer {
    static class Node {       // не залежить від екземпляра Outer
        int value;
    }
}
Outer.Node n = new Outer.Node(); // створюється без об'єкта Outer
```

**На практиці: найчастіший із усіх.** Приклади — `Map.Entry`, патерн `Builder` (`Person.Builder`), допоміжні структури даних.

**Сказати на інтерв'ю:** *«Static nested — клас усередині класу, не прив'язаний до зовнішнього об'єкта. Використовую для допоміжних структур, як-от Builder».*

> До класів верхнього рівня модифікатор `static` незастосовний — лише до вкладених.

[до змісту](#java-core-частина-1-з-3)

## Inner (member) клас — прив'язаний до об'єкта
Оголошений без `static`. Кожен його об'єкт **належить конкретному об'єкту** зовнішнього класу й бачить усі його поля напряму. Тому й створюється лише через об'єкт зовнішнього класу.

```java
class Outer {
    private int x = 10;

    class Inner {
        void print() {
            System.out.println(x); // бачить приватне поле Outer
        }
    }
}
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner(); // потрібен об'єкт Outer
```

Якщо імена полів збігаються, до поля зовнішнього класу звертаються через `Outer.this.field`.

**На практиці: рідше.** Класичний приклад — ітератори всередині колекцій (`ArrayList` має внутрішній клас-ітератор, що бачить масив списку).

**Сказати на інтерв'ю:** *«Inner-клас прив'язаний до екземпляра зовнішнього класу й бачить його поля. Приклад — ітератор усередині колекції».*

> Звичайні (нестатичні) внутрішні класи зберігають приховане посилання на зовнішній об'єкт. До Java 16 вони не могли містити статичних членів; починаючи з Java 16 — можуть.

[до змісту](#java-core-частина-1-з-3)

## Що таке _«локальний клас»_? (Local)
Клас, оголошений прямо в тілі методу й видимий лише в ньому. По суті — як inner, але з областю видимості одного методу. Може використовувати локальні змінні методу, якщо вони `final` або _фактично фінальні_ (effectively final, з Java 8).

```java
void process() {
    class Validator {            // видно тільки всередині process()
        boolean isValid(String s) { return s != null; }
    }
    Validator v = new Validator();
}
```

**На практиці: майже не зустрічається** — у 99% випадків замінюється лямбдою.

**Сказати на інтерв'ю:** *«Локальний клас оголошується всередині методу й видимий лише в ньому. На практиці майже завжди замінюється лямбдою».*

[до змісту](#java-core-частина-1-з-3)

## Що таке _«анонімні класи»_? (Anonymous)
Реалізація інтерфейсу або розширення класу **без імені**, створена прямо на місці: оголошення й створення об'єкта відбуваються одночасно.

```java
Runnable r = new Runnable() {   // без імені — реалізуємо й створюємо одразу
    @Override
    public void run() {
        System.out.println("працюю");
    }
};
```

**На практиці: раніше — постійно, зараз — рідше.** До Java 8 це був головний спосіб передати «шматок поведінки» (слухачі подій, `Comparator`, `Runnable`). З **Java 8 лямбди замінили** анонімні класи в більшості випадків:

```java
Runnable r = () -> System.out.println("працюю"); // те саме лямбдою
```

Анонімний клас усе ще потрібен, коли треба реалізувати інтерфейс **з кількома методами** (лямбда — лише для функціональних інтерфейсів з одним абстрактним методом).

**Сказати на інтерв'ю:** *«Анонімний клас — реалізація без імені, створена на місці. Раніше для слухачів і Comparator, але з Java 8 здебільшого замінений лямбдами».*

[до змісту](#java-core-частина-1-з-3)

## Для чого використовується оператор `assert`?
__Assert__ (твердження) — це спеціальна конструкція, що дозволяє перевіряти припущення про значення довільних даних у довільному місці програми. Твердження може автоматично сигналізувати про виявлення некоректних даних, що зазвичай призводить до аварійного завершення програми з указанням місця виявлення.

Твердження істотно спрощують локалізацію помилок у коді. Навіть перевірка результатів виконання очевидного коду може стати корисною при подальшому рефакторингу, після якого код може стати не настільки очевидним і в нього може закрастися помилка.

Зазвичай твердження лишають увімкненими під час розробки й тестування програм, але вимикають у релізних версіях. (За замовчуванням перевірка тверджень **вимкнена** — її вмикають опцією JVM `-ea` або `-enableassertions`.)

Оскільки твердження можуть бути вимкнені, вони не повинні змінювати поведінку програми. Якщо внаслідок вимкнення твердження поведінка програми може змінитися — це явна ознака неправильного використання _assert_. Тому всередині _assert_ не можна викликати методи, що змінюють стан програми чи її зовнішнього оточення.

У Java перевірка тверджень реалізована оператором `assert`, який має форму:

`assert [вираз типу boolean];` або `assert [вираз типу boolean] : [вираз будь-якого типу, крім void];`

Під час виконання програми, якщо перевірку тверджень увімкнено, обчислюється значення булевого виразу, і якщо його результат `false`, генерується виняток `java.lang.AssertionError`. У разі використання другої форми оператора вираз після двокрапки задає детальне повідомлення про помилку (обчислений вираз буде перетворено на рядок і передано конструктору `AssertionError`).

> Важливо не плутати `assert` із валідацією вхідних даних. `assert` призначений для перевірки інваріантів під час розробки, а **не** для перевірки аргументів публічних методів чи вводу користувача — для цього використовують звичайні перевірки та винятки (наприклад, `IllegalArgumentException`), бо вони працюють завжди, незалежно від `-ea`.

## Що таке _Heap_ і _Stack_ пам'ять у Java? Яка між ними різниця?
Найпростіша ментальна модель:

> **Stack — це де *відбувається виконання*. Heap — це де *живуть об'єкти*.**

**Stack (стек)** обслуговує **виклики методів**. На кожен виклик методу JVM кладе на стек **фрейм**, у якому лежать локальні змінні методу, його параметри та проміжні обчислення. Коли метод завершується — фрейм просто знімається (LIFO). Тому стек ідеально лягає на природу викликів: метод A викликав B, B викликав C → фрейми A→B→C; повертаються вони в зворотному порядку.

**Heap (купа)** — це область, де лежать **усі об'єкти та масиви** (усе, що створене через `new`). Купою керує збирач сміття: об'єкт живе доти, доки на нього є досяжне посилання, і звільняється автоматично.

### Ключовий зв'язок між ними (це любить інтерв'юер)
Коли ти пишеш `Person p = new Person();`:
+ **сам об'єкт `Person`** створюється в **купі**;
+ **змінна-посилання `p`** (тобто «адреса» об'єкта) лежить у **стеку**, у фреймі поточного методу.

Тобто посилання й об'єкт — у різних місцях. І важливий нюанс: **поля об'єкта живуть у купі разом з об'єктом**, навіть якщо вони примітивні. Примітиви опиняються в стеку лише тоді, коли вони — **локальні змінні** методу.

### Головні відмінності

| | **Stack** | **Heap** |
|---|---|---|
| Що зберігає | фрейми: локальні змінні, параметри, посилання | об'єкти та масиви (`new`) |
| Спільність | **по одному стеку на потік** (приватний) | **один на всю JVM**, спільний для всіх потоків |
| Звільнення пам'яті | автоматичне й **детерміноване** — фрейм зникає на виході з методу | через **GC**, момент непередбачуваний |
| Час життя | прив'язаний до виклику методу | прив'язаний до досяжності об'єкта |
| Швидкість | дуже швидко (зсув вказівника) | повільніше (пошук місця, робота GC) |
| Розмір | малий | великий |
| Помилка при переповненні | `StackOverflowError` | `OutOfMemoryError: Java heap space` |

### Чому це справді важливо (а не просто «де що лежить»)
+ **Багатопотоковість.** Кожен потік має **власний** стек → локальні змінні автоматично потокобезпечні (інший потік їх не бачить). А купа **спільна** → саме за об'єкти в купі й точиться боротьба за синхронізацію.
+ **`StackOverflowError`** — це майже завжди **глибока або нескінченна рекурсія** (забагато вкладених фреймів), а не «велика структура даних».
+ **`OutOfMemoryError: Java heap space`** — купа переповнена: або реально багато живих об'єктів, або **витік пам'яті** (десь утримується посилання, і GC не може прибрати об'єкт).

### Приклад: що куди лягає
```java
void example() {
    int x = 5;                       // x — локальний примітив → СТЕК
    Person p = new Person("Іван");   // посилання p → СТЕК; сам об'єкт Person → КУПА
    int[] arr = new int[3];          // посилання arr → СТЕК; масив (і його елементи) → КУПА
}                                    // метод завершився → фрейм знятий зі стека,
                                     // об'єкти в купі стануть недосяжними → їх прибере GC
```

### Налаштування (можна згадати «на десерт»)
+ `-Xms` / `-Xmx` — початковий / максимальний розмір **купи**;
+ `-Xss` — розмір **стека** одного потоку.

> **Стисла відповідь для співбесіди:** *«Stack — пам'ять виконання: на кожен виклик методу кладеться фрейм з локальними змінними та посиланнями, по одному стеку на потік, звільняється детерміновано при виході з методу. Heap — спільна для всіх потоків область, де живуть усі об'єкти, керована GC. Посилання лежить у стеку, а сам об'єкт — у купі. Переповнення стека → StackOverflowError (рекурсія), переповнення купи → OutOfMemoryError».*

## Чи правильне твердження, що примітивні типи завжди зберігаються в стеку, а екземпляри посилальних типів — у купі?
Не зовсім. Примітивне **поле екземпляра класу** зберігається не в стеку, а в купі (разом з об'єктом, якому належить). У стеку лежать лише локальні змінні примітивних типів. Будь-який об'єкт (усе, що явно чи неявно створюється оператором `new`) зберігається в купі.

[до змісту](#java-core-частина-2-з-3)

## Яким чином змінні передаються в методи — за значенням чи за посиланням?
У Java параметри завжди передаються **лише за значенням** — тобто копіюється значення й передається копія. Для примітивів це копія самого значення. Для посилань — теж копія значення, тобто **копія посилання**. Через таку копію можна змінити внутрішні поля об'єкта, на який вона вказує, але **не можна** змінити саме оригінальне посилання (перенаправити його на інший об'єкт).

[до змісту](#java-core-частина-2-з-3)

## Для чого потрібен збирач сміття?
У мовах на кшталт C/C++ пам'ять звільняє програміст вручну (`free`/`delete`). Це джерело цілого класу багів: витоки пам'яті (забув звільнити), подвійне звільнення, звернення до вже звільненої пам'яті.

**Збирач сміття (Garbage Collector, GC) автоматизує це.** Він робить дві речі:

1. **знаходить об'єкти, які більше не потрібні** (на них ніхто не посилається);
2. **звільняє зайняту ними пам'ять у купі.**

Завдяки цьому розробник не керує пам'яттю вручну й позбувається згаданих помилок. Плата за зручність — GC споживає ресурси й може спричиняти короткі паузи в роботі застосунку.

[до змісту](#java-core-частина-2-з-3)

## Як працює збирач сміття?
В основі — поняття **досяжності (reachability)**. Об'єкт вважається живим, лише якщо до нього можна «дотягнутися» ланцюжком посилань від так званих **кореневих точок (GC Roots)**. Усе, до чого дотягнутися не вдалося, — сміття.

**GC Roots** — це «точки входу», що свідомо живі:
+ локальні змінні та параметри методів, що зараз виконуються (на стеку);
+ активні потоки;
+ статичні поля класів;
+ посилання з нативного коду (JNI).

GC починає з коренів і обходить граф об'єктів за посиланнями (mark — позначає досяжні). Усе непозначене — недосяжне й підлягає видаленню. Перевага підходу: **циклічні посилання не проблема** — якщо два об'єкти посилаються лише один на одного, але до них немає шляху від кореня, обидва вважаються сміттям. (Тому JVM використовує саме досяжність, а не простий підрахунок посилань.)

Після позначення пам'ять звільняють і зазвичай **ущільнюють (compact)** — зсувають живі об'єкти разом, щоб вільна пам'ять була неперервною. Це усуває фрагментацію й робить виділення нового об'єкта дуже швидким (просто зсув вказівника).

**Поколіннєва модель (generational).** GC спирається на емпіричне спостереження: *більшість об'єктів помирають молодими*. Тому купу ділять на:
+ **Young generation** — тут створюються нові об'єкти; збирається часто й швидко (**minor GC**);
+ **Old generation** — сюди «підвищують» об'єкти, що пережили кілька збирань у Young; збирається рідше (**major / full GC**).

Сенс: дешево й часто прибирати молоде покоління (де майже все одразу стає сміттям), не чіпаючи щоразу старе.

Важливо: момент запуску GC **непередбачуваний** — ним керує JVM. `System.gc()` лише *просить* запустити збирання, але не гарантує його.

## Які різновиди збирачів сміття реалізовані у віртуальній машині HotSpot?
Сучасна HotSpot пропонує кілька збирачів під різні задачі:

+ **G1 (Garbage-First)** — **за замовчуванням** (з Java 9). Універсальний баланс пропускної здатності та коротких пауз. Підходить для більшості застосунків. `-XX:+UseG1GC`.
+ **Parallel GC** — орієнтований на **максимальну пропускну здатність** (throughput) за рахунок довших пауз. Добрий для пакетної обробки, де паузи не критичні. `-XX:+UseParallelGC`.
+ **Serial GC** — однопотоковий, найпростіший. Для малих застосунків і контейнерів з обмеженими ресурсами. `-XX:+UseSerialGC`.
+ **ZGC** — збирач із **наднизькими паузами** (зазвичай < 1 мс) навіть на купах у терабайти. Майже вся робота — паралельно із застосунком. `-XX:+UseZGC`.
+ **Shenandoah** — теж націлений на наднизькі паузи, незалежні від розміру купи. `-XX:+UseShenandoahGC`.
+ **Epsilon** — «нульовий» GC: виділяє пам'ять, але нічого не звільняє. Лише для тестів продуктивності та діагностики. `-XX:+UseEpsilonGC`.

Як вибирати коротко: **G1** — за замовчуванням і для більшості; **Parallel** — коли важлива пропускна здатність, а не паузи; **ZGC/Shenandoah** — коли критичні мінімальні паузи на великих купах.

[до змісту](#java-core-частина-2-з-3)

## Опишіть алгоритм роботи якогось збирача сміття в HotSpot.
Опишемо **G1 (Garbage-First)** — сучасний збирач за замовчуванням.

**Ключова ідея — поділ купи на регіони.** Замість двох великих суцільних областей (Young/Old) G1 ділить купу на безліч однакових **регіонів** (зазвичай ~2048 штук). Поділ на покоління лишається, але **логічним**: кожен регіон у конкретний момент відіграє роль Eden, Survivor або Old (а дуже великі об'єкти займають кілька суміжних регіонів — *humongous*).

Як це працює:

1. **Нові об'єкти** створюються в регіонах Eden.
2. Коли Eden заповнюється — запускається **young collection** (stop-the-world): живі об'єкти з Eden копіюються (евакуюються) у Survivor-регіони, а ті, що пережили достатньо збирань, — у Old. Копіювання попутно **ущільнює** пам'ять.
3. Для старого покоління G1 виконує **позначення паралельно із застосунком** (concurrent marking) — більшість роботи з пошуку живих об'єктів іде без зупинки програми.
4. Далі G1 збирає насамперед ті регіони, де **найбільше сміття** (звідси назва «Garbage-First») — так за коротку паузу звільняється максимум пам'яті.

**Головна перевага** G1 — **прогнозовані паузи**: можна задати цільову тривалість паузи (`-XX:MaxGCPauseMillis=200`), і G1 підбирає, скільки регіонів зібрати за раз, щоб у неї вкластися. Тому він добре масштабується на великі купи без довгих «stop-the-world».

> Для розуміння контрасту: найпростіший **Serial GC** робить те саме (копіювання в молодому поколінні + mark-sweep-compact у старому), але **в один потік і з повною зупинкою застосунку** на весь час збирання. G1 же розбиває роботу на регіони й виконує велику її частину паралельно, тримаючи паузи короткими.

[до змісту](#java-core-частина-2-з-3)

## Що таке «пул рядків»?
__Пул рядків (String pool)__ — це набір рядків, що зберігається в _Heap_.

+ Пул рядків можливий завдяки незмінності рядків у Java та реалізації ідеї інтернування рядків;
+ Пул допомагає економити пам'ять, але з цієї ж причини створення рядка займає трохи більше часу;
+ Коли для створення рядка використовують лапки `"..."`, спочатку шукається рядок у пулі з таким самим значенням: якщо знайдено — повертається посилання, інакше створюється новий рядок у пулі й повертається посилання на нього;
+ При використанні оператора `new` створюється новий об'єкт `String` (у купі, поза пулом). Потім методом `intern()` цей рядок можна помістити в пул або отримати з пулу посилання на інший об'єкт `String` з таким самим значенням;
+ Пул рядків є прикладом патерна _«Пристосуванець» (Flyweight)_.

## Що таке `finalize()`? Навіщо він потрібен?
Через виклик методу `finalize()` (успадкованого від `java.lang.Object`) JVM реалізовувала функціональність, аналогічну деструкторам у C++. Метод викликався при знищенні об'єкта збирачем сміття, і, перевизначивши `finalize()`, можна було запрограмувати дії для коректного звільнення ресурсів — наприклад, закриття мережевих з'єднань, з'єднань з БД, зняття блокувань файлів тощо.

Після виконання цього методу об'єкт мав бути зібраний GC повторно (і це вважається серйозною проблемою `finalize()`, бо він заважає GC звільняти пам'ять). Виклик методу **не гарантується**, бо застосунок може завершитися до того, як запуститься збирання.

Об'єкт не обов'язково стає доступним для збирання одразу — `finalize()` може зберегти кудись посилання на об'єкт. Така ситуація називається «воскресінням» (resurrection) об'єкта й вважається антипатерном. Головна проблема цього трюку — «воскресити» об'єкт можна лише один раз.

> **Важливо (актуальність).** Метод `finalize()` **застарілий**: позначений `@Deprecated` з Java 9, а з Java 18 — deprecated for removal. Його **не можна** використовувати в сучасному коді через непередбачуваність (немає гарантій виклику, часу й порядку), вплив на продуктивність GC та ризики безпеки. Для звільнення ресурсів використовують **try-with-resources** (`AutoCloseable`), а для підстраховки — `java.lang.ref.Cleaner` чи `PhantomReference`.

Приклад (історичний):
```java
public class MainClass {

	public static void main(String args[]) {
		TestClass a = new TestClass();
		a.a();
		a = null;
		a = new TestClass();
		a.a();
		System.out.println("!!! done");
	}
}
```
```java
public class TestClass {

	public void a() {
		System.out.println("!!! a() called");
	}

	@Override
	protected void finalize() throws Throwable {
		System.out.println("!!! finalize() called");
		super.finalize();
	}
}
```
Оскільки тут GC може й не бути викликаний (через простоту застосунку), результат з великою ймовірністю буде:
```
!!! a() called
!!! a() called
!!! done
```
Якщо додати примусовий виклик `System.gc()`, результат може різнитися від запуску до запуску — наприклад, рядок `!!! finalize() called` з'явиться до або після `!!! done`. Це ще раз ілюструє: момент роботи GC (а отже й `finalize()`) непередбачуваний.

## Що станеться зі збирачем сміття, якщо `finalize()` виконується довго або кидає виняток?
Виклик `finalize()` відбувається в окремому потоці _Finalizer_ (`java.lang.ref.Finalizer.FinalizerThread`), який створюється при запуску віртуальної машини. Методи `finalize()` викликаються послідовно в порядку, у якому об'єкти були додані до списку збирачем. Відповідно, якщо якийсь `finalize()` зависне, він підвісить потік _Finalizer_, але **не сам GC**. Це означає, що об'єкти без `finalize()` справно видалятимуться, а от ті, що мають його, накопичуватимуться в черзі, поки потік _Finalizer_ не звільниться, не завершиться застосунок або не скінчиться пам'ять (що призводить до `OutOfMemoryError`).

Те саме стосується винятків, кинутих у `finalize()`: метод, що виконує фіналізацію, **ігнорує всі винятки**, кинуті під час `finalize()`. Тож виникнення винятку ніяк не позначиться на роботі збирача.

## Чим відрізняються `final`, `finally` і `finalize()`?
Модифікатор `final`:

+ Клас не може мати нащадків;
+ Метод не може бути перевизначений у класах-нащадках;
+ Поле не може змінити значення після ініціалізації;
+ Локальні змінні не можуть бути змінені після присвоєння;
+ Параметри методів не можуть змінювати значення всередині методу.

Оператор (блок) `finally` гарантує, що визначений у ньому код буде виконано незалежно від того, які винятки були згенеровані й перехоплені в блоці `try-catch`.

Метод `finalize()` викликається перед тим, як GC видалить об'єкт (застарілий — див. вище).

> Стисло: `final` — модифікатор (незмінність/заборона перевизначення); `finally` — блок керування потоком виконання; `finalize()` — метод життєвого циклу об'єкта. Спільне в них лише схожі назви.

## Розкажіть про приведення типів. Що таке звуження та розширення типу?
Java — строго типізована мова: кожен вираз і кожна змінна мають чітко визначений тип уже на етапі компіляції. Проте існує механізм _приведення типів (casting)_ — спосіб перетворення значення одного типу на значення іншого.

Розрізняють кілька різновидів приведення:

+ __Тотожне (identity)__. Перетворення виразу до точно такого самого типу — завжди допустиме й відбувається автоматично.
+ __Розширення (підвищення, upcasting) примітивного типу (widening primitive)__. Перехід від менш місткого типу до більш місткого (наприклад, `byte` → `int`). Безпечне: новий тип гарантовано вміщує всі дані, тож втрати немає. Відбувається автоматично.
+ __Звуження (пониження, downcasting) примітивного типу (narrowing primitive)__. Перехід від більш місткого типу до менш місткого. Є ризик втратити дані (наприклад, при `int` → `byte` старші біти просто відкидаються). Має виконуватися явно.
+ __Розширення об'єктного типу (widening reference)__. Неявне висхідне приведення — від конкретнішого типу до менш конкретного (від нащадка до предка). Дозволене завжди й автоматичне.
+ __Звуження об'єктного типу (narrowing reference)__. Низхідне приведення — від предка до нащадка. Можливе, лише якщо фактичний об'єкт справді є екземпляром цільового типу. При невідповідності типів під час виконання кидається `ClassCastException`. Потребує явного приведення.
+ __Перетворення на рядок (to String)__. Будь-який тип можна привести до рядка.
+ __Заборонені перетворення (forbidden)__. Не всі приведення допустимі — наприклад, між посилальним і примітивним типом (крім перетворення на рядок), або між класами на різних гілках дерева успадкування.

При приведенні посилальних типів із самим об'єктом нічого не відбувається — змінюється лише тип посилання, через яке звертаються до об'єкта.

Для перевірки можливості приведення використовують оператор `instanceof`:

```java
Parent parent = new Child();
if (parent instanceof Child) {
    Child child = (Child) parent;
}
```

> У сучасній Java (з Java 16) є **pattern matching для `instanceof`**, що поєднує перевірку й приведення: `if (parent instanceof Child child) { /* використовуємо child */ }`.

## Коли може бути викинуто `ClassCastException`?
`ClassCastException` (нащадок `RuntimeException`) — виняток, що виникає при помилці приведення посилального типу (спроба привести об'єкт до типу, екземпляром якого він не є).

## Що таке літерали?
__Літерали__ — це явно задані значення в коді програми, тобто константи певного типу, записані безпосередньо в коді.
```java
class Test {
   int a = 0b1101010110;
   public static void main(String[] args) {
       System.out.println("Hello world!");
   }
}
```
Тут `"Hello world!"` — рядковий літерал, а `0b1101010110` — двійковий цілочисельний літерал (значення, присвоюване змінній `a`).

> Зауваження: літералом є саме **значення** `0b1101010110`, а не змінна `a` (в оригінальному джерелі тут була неточність). `a` — це змінна, яка ініціалізується літералом.

Літерали бувають різних типів, що визначаються їх призначенням і способом запису (цілочисельні, з рухомою комою, символьні, рядкові, булеві, `null`).

## Що таке _autoboxing («автопакування»)_ і які правила пакування примітивів у класи-обгортки?
__Автопакування (autoboxing)__ — це механізм неявного створення об'єктів класів-обгорток (`Byte`, `Short`, `Integer`, `Long`, `Float`, `Double`, `Character`, `Boolean`) зі значень відповідних примітивних типів (`byte`, `short`, `int`...), без явного виклику конструктора. Зворотний процес — __розпакування (unboxing)__.

+ Автопакування відбувається при прямому присвоєнні примітива обгортці (через `=`) або при передачі примітива в параметр методу (типу обгортки).
+ Автопакуванню можуть підлягати як змінні примітивних типів, так і константи часу компіляції (літерали та `final`-примітиви).
+ Автопакування змінних потребує **точної відповідності** типу примітива типу обгортки. Наприклад, спроба запакувати `byte` у `Short` без попереднього явного приведення `byte` → `short` спричинить помилку компіляції.
+ Автопакування констант допускає ширші межі: компілятор може попередньо неявно розширити/звузити тип примітива до потрібного, а потім запакувати.

Додаткова важлива особливість: цілочисельні обгортки, створені автопакуванням у діапазоні `-128 ... +127`, **кешуються JVM** (Integer cache). Тому обгортки з однаковими значеннями в цьому діапазоні будуть посиланнями на **один об'єкт**:

```java
Integer a = 100, b = 100;
System.out.println(a == b); // true  — обидва з кешу, той самий об'єкт
Integer c = 200, d = 200;
System.out.println(c == d); // false — поза кешем, різні об'єкти
```
Це класична пастка на співбесіді: обгортки треба порівнювати через `equals()`, а не `==`.

[до змісту](#java-core-частина-2-з-3)

## Які особливості має клас `String`?
+ Це незмінний (immutable) і фінальний (`final`) тип даних;
+ Рядки-літерали та інтерновані об'єкти `String` JVM зберігає в пулі рядків;
+ Об'єкт `String` можна отримати, використовуючи подвійні лапки;
+ Для конкатенації рядків можна використовувати оператор `+`;
+ Починаючи з Java 7, рядки можна використовувати в конструкції `switch`.

[до змісту](#java-core-частина-2-з-3)

## Чому `String` — незмінний і фінальний клас?
Незмінність рядків дає кілька переваг:

+ **Пул рядків можливий лише завдяки незмінності** — різні змінні можуть указувати на той самий об'єкт у пулі, що економить пам'ять. Якби рядок був змінним, інтернування було б неможливим, бо зміна значення через одну змінну зачепила б усі інші.
+ **Безпека.** Рядки часто передають як облікові дані (ім'я користувача/пароль БД, хост і порт у сокетах). Незмінність гарантує, що значення не змінять «з-під ніг» після перевірки.
+ **Безпека потоків.** Незмінні об'єкти безпечні для багатопотоковості без синхронізації — один екземпляр можна спільно використовувати з різних потоків.
+ **Завантаження класів.** Рядки використовує classloader (імена класів), і незмінність забезпечує коректність завантаження.
+ **Кешування `hashCode()`.** Оскільки рядок незмінний, його `hashCode()` обчислюється один раз при створенні та кешується. Це робить рядок чудовим кандидатом на ключ у `HashMap` — обробка швидша.

Клас `final`, щоб ніхто не міг успадкуванням порушити гарантії незмінності.

[до змісту](#java-core-частина-2-з-3)

## Чому `char[]` кращий за `String` для зберігання пароля?
Рядок із моменту створення лишається в пам'яті (у пулі чи купі), доки його не прибере GC, — і момент цей непередбачуваний. Тому навіть після завершення використання пароль ще деякий час лишається в пам'яті у вигляді тексту, і запобігти цьому неможливо. Це ризик безпеки: будь-хто з доступом до дампа пам'яті може знайти пароль.

У разі масиву символів (`char[]`) його можна **явно очистити** (перезаписати) одразу після використання, усунувши цей ризик.

[до змісту](#java-core-частина-2-з-3)

## Чому рядок — популярний ключ у `HashMap`?
Оскільки рядки незмінні, їхній хеш-код обчислюється й кешується при створенні та не потребує повторного обрахунку при подальшому використанні. Тому як ключ `HashMap` рядки обробляються швидше. До того ж незмінність гарантує, що хеш-код ключа не зміниться після додавання в мапу (інакше елемент «загубився» б у неправильному кошику).

[до змісту](#java-core-частина-2-з-3)

## Що робить метод `intern()` у класі `String`?
Метод `intern()` поміщає рядок у пул рядків або повертає посилання, якщо такий рядок уже є в пулі. Дозволяє звести логічно однакові рядки до одного спільного об'єкта в пулі.

[до змісту](#java-core-частина-2-з-3)

## Чи можна використовувати рядки в конструкції `switch`?
Так, починаючи з Java 7 (раніші версії цього не підтримували). При цьому:

+ рядки чутливі до регістру;
+ для порівняння зі значеннями `case` використовується `equals()`, тому щоб уникнути `NullPointerException`, варто передбачити перевірку на `null` (передача `null` у `switch` кидає NPE);
+ компілятор формує для рядків у `switch` ефективніший байткод, ніж для ланцюжка `if-else` (внутрішньо використовує `hashCode()` + `equals()`).

[до змісту](#java-core-частина-2-з-3)

## Яка основна різниця між `String`, `StringBuffer`, `StringBuilder`?
+ `String` — **незмінний** (immutable): модифікувати об'єкт не можна, кожна «зміна» створює новий екземпляр. Часта конкатенація в циклі — неефективна.
+ `StringBuffer` — **змінний** (mutable) і **потокобезпечний** (методи `synchronized`). Використовують, коли потрібно часто модифікувати вміст у багатопотоковому середовищі.
+ `StringBuilder` — доданий у Java 5, в усьому ідентичний `StringBuffer`, але **не синхронізований**, тож методи виконуються значно швидше. Це варіант за замовчуванням для однопотокового коду.

[до змісту](#java-core-частина-2-з-3)

## Що таке клас `Object`? Які в ньому є методи?
`Object` — базовий клас для всіх інших об'єктів у Java. Будь-який клас успадковує `Object` і, відповідно, його методи:

+ `public boolean equals(Object obj)` — порівнює об'єкти за значенням;
+ `public int hashCode()` — повертає хеш-код об'єкта;
+ `public String toString()` — повертає рядкове представлення об'єкта;
+ `public final Class<?> getClass()` — повертає клас об'єкта під час виконання;
+ `protected Object clone()` — створює й повертає копію об'єкта;
+ `public final void notify()` — пробуджує один потік, що очікує на моніторі об'єкта;
+ `public final void notifyAll()` — пробуджує всі потоки, що очікують на моніторі;
+ `public final void wait()` — призупиняє потік, що викликав метод, доки інший потік не викличе `notify()`/`notifyAll()` для цього об'єкта;
+ `public final void wait(long timeout)` / `wait(long timeout, int nanos)` — те саме, але з тайм-аутом;
+ `protected void finalize()` — міг викликатися GC при видаленні об'єкта (застарілий).

[до змісту](#java-core-частина-2-з-3)

## Дайте визначення поняттю «конструктор».
__Конструктор__ — це спеціальний метод, що не має типу повернення й має те саме ім'я, що й клас, у якому оголошений. Конструктор викликається при створенні нового об'єкта класу й визначає дії, потрібні для його ініціалізації.

[до змісту](#java-core-частина-2-з-3)

## Що таке _«конструктор за замовчуванням»_?
Якщо в класі не визначено жодного конструктора, компілятор згенерує конструктор без аргументів — так званий __«конструктор за замовчуванням»__.

```java
public ClassName() {}
```

Якщо в класі вже визначено будь-який конструктор, конструктор за замовчуванням створений **не буде**, і за потреби його треба описати явно.

[до змісту](#java-core-частина-2-з-3)

## Чим відрізняються конструктор за замовчуванням, конструктор копіювання та конструктор з параметрами?
+ **Конструктор за замовчуванням** не має аргументів.
+ **Конструктор копіювання** приймає вже наявний об'єкт того самого класу, щоб створити його копію.
+ **Конструктор з параметрами** має у своїй сигнатурі аргументи (зазвичай потрібні для ініціалізації полів класу).

[до змісту](#java-core-частина-2-з-3)

## Де і як можна використати приватний конструктор?
Приватний (`private`) конструктор може використовуватися:

+ публічним статичним фабричним методом для створення об'єктів цього класу (патерни «фабричний метод», Singleton);
+ для класів-утиліт зі статичними методами, щоб заборонити створення екземплярів;
+ вкладеними класами для власних потреб;
+ у патерні Builder (об'єкт створюється лише через білдер).

## Розкажіть про завантажувачі класів і динамічне завантаження класів.
Основа роботи з класами в Java — завантажувачі класів (class loaders): звичайні Java-об'єкти, що надають інтерфейс для пошуку та створення об'єкта класу за його іменем під час роботи застосунку.

На початку роботи програми створюється 3 основні завантажувачі:

+ __базовий завантажувач (bootstrap)__. Завантажує основні системні та внутрішні класи JDK (Core API — пакети `java.*`). Є частиною JVM, тому його не можна створити в коді програми.
+ __завантажувач розширень (extension)__. Завантажує пакети розширень із `<JAVA_HOME>/lib/ext` або іншого каталогу з `java.ext.dirs`. Реалізований класом `sun.misc.Launcher$ExtClassLoader`.
+ __системний завантажувач (system/application)__. Завантажує класи зі шляхів у `CLASSPATH` чи вказаних після `-classpath`/`-cp`. Реалізований класом `sun.misc.Launcher$AppClassLoader`.

> **Актуальність (Java 9+).** Цей опис відповідає Java 8 і раніше. У сучасній Java (через модульну систему): структура `<JAVA_HOME>/lib/ext`, `java.ext.dirs`, файли `rt.jar`/`i18n.jar` та **завантажувач розширень прибрані**; його замінив **platform class loader**, а клас застосунків тепер `jdk.internal.loader.ClassLoaders$AppClassLoader`. Актуальна ієрархія: **bootstrap → platform → application (system)**.

Завантажувачі ієрархічні: кожен (крім базового) має батьківський завантажувач і зазвичай **спершу делегує** запит батьку, перш ніж завантажувати клас самостійно (модель delegation). Таке делегування означає, що пошук класів відбувається в порядку довіри: спершу Core API, потім розширення/платформа, потім локальні файли `CLASSPATH`. Це захищає від підміни системних класів.

Процес завантаження класу складається з трьох частин:

+ _Loading_ — пошук і фізичне завантаження файлу класу. Створюється базове представлення класу в пам'яті.
+ _Linking_ — поділяється на:
    + _Bytecode verification_ — перевірка байткоду на відповідність специфікації JVM.
    + _Preparation_ — створення та ініціалізація структур для полів, методів, реалізованих інтерфейсів тощо (полям присвоюються значення за замовчуванням).
    + _Resolution_ — розв'язання символічних посилань на інші класи.
+ _Initialization_ — виклик статичних блоків ініціалізації та присвоєння статичним полям заданих значень.

Особливості динамічного завантаження:

+ _відкладене (lazy) завантаження_ — класи завантажуються лише за потреби;
+ _перевірка коректності коду (type safeness)_ — контроль типів під час завантаження;
+ _програмоване завантаження_ — користувацький завантажувач сам контролює процес (шукати байткод сам чи делегувати), що дозволяє працювати з кодом із ненадійних джерел;
+ _множинні простори імен_ — кожен завантажувач має власний простір імен, тож класи, завантажені різними завантажувачами з того самого байткоду, у системі вважатимуться різними.

Завантаження класу можна ініціювати:

+ явно: `ClassLoader.loadClass()` або `Class.forName()`;
+ неявно: коли застосунку вперше потрібен раніше не використаний клас, JVM ініціює його завантаження.

[до змісту](#java-core-частина-2-з-3)

## Що таке _Reflection_?
__Рефлексія (Reflection)__ — це механізм отримання даних про програму та маніпулювання нею під час виконання (runtime). У Java реалізується через _Java Reflection API_ — класи пакетів `java.lang` і `java.lang.reflect`.

Можливості Java Reflection API:

+ Визначення класу об'єкта;
+ Отримання інформації про модифікатори, поля, методи, конструктори та суперкласи;
+ Визначення інтерфейсів, які реалізує клас;
+ Створення екземпляра класу;
+ Отримання й встановлення значень полів об'єкта (зокрема приватних);
+ Виклик методів об'єкта;
+ Створення нового масиву.

> На практиці рефлексію широко використовують фреймворки (Spring, Hibernate, JUnit) для впровадження залежностей, мапінгу, виклику тестів тощо. Мінуси: повільніша за звичайні виклики, обходить перевірки доступу й типобезпеку, ускладнює рефакторинг.

## Навіщо потрібен `equals()`? Чим він відрізняється від `==`?
Метод `equals()` визначає відношення **еквівалентності об'єктів** (рівність за вмістом).

+ `==` для об'єктів порівнює лише **посилання** — чи вказують обидві змінні на один і той самий об'єкт у пам'яті.
+ `equals()` (перевизначений розробником) порівнює об'єкти за їхнім **внутрішнім станом** (значеннями полів).

Тобто два різні об'єкти з однаковим вмістом: `a == b` → `false`, але `a.equals(b)` → `true` (якщо `equals` перевизначено).

> Для примітивів `==` порівнює значення — це єдиний коректний спосіб їх порівняти.

[до змісту](#java-core-частина-3-з-3)

## Якими властивостями володіє відношення еквівалентності, породжене `equals()`?
Перевизначаючи `equals()`, треба дотримати **контракту** — п'яти властивостей:

+ **Рефлексивність**: для будь-якого `x` → `x.equals(x)` повертає `true`.
+ **Симетричність**: `x.equals(y)` повертає `true` тоді й лише тоді, коли `y.equals(x)` повертає `true`.
+ **Транзитивність**: якщо `x.equals(y)` і `y.equals(z)` істинні, то й `x.equals(z)` істинне.
+ **Несуперечливість (consistency)**: повторні виклики `x.equals(y)` дають той самий результат, якщо дані, що використовуються в порівнянні, не змінилися.
+ Для будь-якого ненульового `x` → `x.equals(null)` повертає `false`.

[до змісту](#java-core-частина-3-з-3)

## Правила перевизначення методу `Object.equals()`.
Канонічна реалізація:

1. Перевірити через `==`, чи аргумент — це те саме посилання (оптимізація). Якщо так — `true`. Якщо аргумент `null` — `false`.
2. Перевірити тип аргументу: через `instanceof MyClass` або `getClass() == that.getClass()` (про різницю — нижче). Якщо тип не той — `false`.
3. Привести аргумент до потрібного типу.
4. Порівняти всі значущі поля. Якщо всі збігаються — `true`, інакше — `false`.

Після написання `equals()` варто перевірити, чи дотримано всіх п'яти властивостей контракту (рефлексивність, симетричність, транзитивність, несуперечливість, поведінка з `null`).

> На практиці `equals()` і `hashCode()` майже ніколи не пишуть вручну. Використовують:
> + допоміжні методи `java.util.Objects`: `Objects.equals(a, b)` (безпечний до `null`) і `Objects.hash(field1, field2, ...)`;
> + автогенерацію IDE або Lombok (`@EqualsAndHashCode`);
> + **`record`** (з Java 16) — генерує коректні `equals`/`hashCode`/`toString` автоматично.

[до змісту](#java-core-частина-3-з-3)

## Який зв'язок між `hashCode()` і `equals()`?
Головне правило контракту: **рівні об'єкти повинні мати однаковий хеш-код**. Тому **перевизначаючи `equals()`, обов'язково треба перевизначати й `hashCode()`** (і навпаки).

Формально:
+ якщо `a.equals(b)` → `true`, то `a.hashCode() == b.hashCode()` **обов'язково**;
+ зворотне не вимагається: однаковий хеш-код **не** гарантує рівності об'єктів (можлива колізія).

[до змісту](#java-core-частина-3-з-3)

## Що буде, якщо перевизначити `equals()`, не перевизначивши `hashCode()`?
Класи, що покладаються на цей контракт (передусім хеш-колекції — `HashMap`, `HashSet`), працюватимуть некоректно. Типовий приклад: ви кладете пару «ключ-значення» в `HashMap`, а потім шукаєте за **новим екземпляром** логічно того самого ключа — і не знаходите.

Чому: `HashMap` спочатку за `hashCode()` визначає кошик (bucket). Якщо хеш-коди двох «рівних» ключів різні, вони потраплять у різні кошики, і пошук навіть не дійде до порівняння через `equals()`. Об'єкт фактично «загубиться».

[до змісту](#java-core-частина-3-з-3)

## Як реалізовані `hashCode()` і `equals()` у класі `Object`?
`Object.equals()` зводиться до порівняння посилань:
```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

`Object.hashCode()` оголошений як `native` (реалізований не Java-кодом і залежить від реалізації JVM):
```java
public native int hashCode();
```
У сучасній HotSpot хеш-код за замовчуванням — це псевдовипадкове значення, що генерується при першому виклику й кешується в заголовку об'єкта (а не виводиться з адреси в пам'яті, як часто помилково кажуть, — інакше воно б ламалося при переміщенні об'єкта збирачем сміття).

[до змісту](#java-core-частина-3-з-3)

## Для чого потрібен метод `hashCode()`?
`hashCode()` повертає **хеш-код** об'єкта — ціле число (`int`), отримане з його стану. Використовується хеш-колекціями для швидкого розміщення й пошуку елементів. Властивості:
+ для одного й того самого об'єкта (незмінного стану) хеш-код завжди однаковий;
+ множина хеш-кодів обмежена діапазоном `int`, а об'єктів — нескінченна, тож **колізії неминучі**.

Звідси:
+ якщо хеш-коди **різні** → об'єкти **гарантовано різні**;
+ якщо хеш-коди **рівні** → об'єкти **не обов'язково** рівні (треба дозрівняти через `equals()`).

[до змісту](#java-core-частина-3-з-3)

## Які поля варто використовувати при обчисленні `hashCode()`?
Загальна порада: брати поля, що з високою ймовірністю **різнитимуться** між об'єктами (унікальні, бажано примітивні — наприклад, `id`, `uuid`).

Головне правило узгодженості: **набір полів для `hashCode()` має збігатися з набором полів для `equals()`**. Якщо поле бере участь у `equals()`, воно має брати участь і в `hashCode()` — інакше порушується контракт.

[до змісту](#java-core-частина-3-з-3)

## Чи можуть різні об'єкти мати однаковий `hashCode()`?
Так. `hashCode()` не гарантує унікальності. Ситуація, коли різні об'єкти мають однаковий хеш-код, називається **колізією**. Її ймовірність залежить від якості хеш-функції. Колізії — нормальне явище; хеш-колекції їх обробляють (зберігають елементи з однаковим хешем в одному кошику й розрізняють через `equals()`).

[до змісту](#java-core-частина-3-з-3)

## Чи коректно працюватимуть точки `Point` у `HashSet`, якщо `hashCode()` повертає лише `x`?
Йдеться про клас `Point{int x, y;}` з коректним `equals()` (порівнює `x` і `y`), але `hashCode()` у вигляді `return x;`.

**Так, працюватимуть коректно, але повільно.** Контракт не порушено: рівні об'єкти (однакові `x` і `y`) дадуть однаковий хеш-код. Тож додавання та витяг із `HashSet` відбуватимуться правильно.

Проблема — у **продуктивності**. Усі точки з однаковим `x` (але різним `y`) потраплять в один кошик, і хеш-таблиця **виродиться у список**. Пошук замість O(1) стане O(n). Тобто це коректна, але погана хеш-функція.

[до змісту](#java-core-частина-3-з-3)

## Чи можуть різні об'єкти `(ref0 != ref1)` мати `ref0.equals(ref1) == true`?
Так — саме для цього й перевизначають `equals()` (порівняння за вмістом). Якщо ж використовується стандартний `Object.equals()`, то `true` буде лише тоді, коли обидва посилання вказують на один об'єкт (тобто `ref0 == ref1`).

[до змісту](#java-core-частина-3-з-3)

## Чи можуть різні посилання на один об'єкт `(ref0 == ref1)` мати `ref0.equals(ref1) == false`?
У нормі — ні (це порушило б рефлексивність). Таке можливе лише за **некоректної** реалізації `equals()`, що не дотримує властивості рефлексивності (`x.equals(x)` має завжди повертати `true`). Окремий відомий приклад — `Double.NaN`: `Double.compare` і поведінка з `NaN` мають свої тонкощі, але для звичайних об'єктів `ref0 == ref1` має давати `equals == true`.

[до змісту](#java-core-частина-3-з-3)

## Чи можна реалізувати `equals` через `hashCode`?
Маємо на увазі `equals(Object that) { return this.hashCode() == that.hashCode(); }`.

Строго кажучи — **ні**: `hashCode()` не гарантує унікальності, тож два **різні** об'єкти з однаковим хешем (колізія) хибно вважатимуться рівними. Це порушує сенс `equals()`. (Лише для «голого» `Object` такий код випадково спрацював би, бо там хеш практично унікальний, але покладатися на це не можна.)

[до змісту](#java-core-частина-3-з-3)

## Різниця між `this.getClass() == that.getClass()` і `that instanceof MyClass` в `equals()`?
+ `instanceof` повертає `true` і для екземплярів **підкласів** (та реалізацій інтерфейсу). Тобто підклас вважатиметься сумісним типом.
+ `getClass() == that.getClass()` вимагає **точного збігу** класів — підклас не пройде перевірку.

Різниця принципова через **симетричність** контракту: якщо використати `instanceof`, об'єкт батьківського класу й об'єкт підкласу можуть несиметрично «вважати» одне одного рівними. Тому для строгого дотримання контракту `equals()` зазвичай рекомендують `getClass()`. (Виняток — підхід, де підкласи не додають значущих полів; тоді `instanceof` теж допустимий.)

[до змісту](#java-core-частина-3-з-3)

## Чи перевизначає `equals(MyClass that)` метод `Object.equals()`?
**Коротко:** ні — це поширена пастка. `equals` перевизначати потрібно, але **тільки з параметром `Object`**. Якщо параметр інший — це вже не перевизначення.

**Чому.** Метод у класі `Object` має точну сигнатуру `public boolean equals(Object obj)`. Перевизначення (override) спрацьовує лише при **точному збігу** сигнатури. Якщо написати параметр `MyClass` замість `Object`, Java сприйме це як **окремий метод** — перевантаження (overload). У класі стане два `equals`:
+ твій `equals(MyClass)` — новий метод;
+ успадкований `equals(Object)` — досі порівнює лише посилання (`==`).

**Проблема:** хеш-колекції (`HashMap`, `HashSet`) і весь стандартний код викликають саме `equals(Object)`, тож твій `equals(MyClass)` вони просто не побачать.

```java
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    // ПАСТКА: параметр Point → це перевантаження, а не перевизначення
    public boolean equals(Point that) {
        return this.x == that.x && this.y == that.y;
    }
}

Point a = new Point(1, 2), b = new Point(1, 2);
a.equals(b);                  // true  — викликався твій equals(Point)
a.equals((Object) b);        // false — викликався Object.equals(Object) → порівняння посилань

Set<Point> set = new HashSet<>();
set.add(a);
set.contains(b);             // false ❌ — колекція викликає equals(Object), а не твій метод
```

**Правильно** — параметр `Object`, перевірка типу й приведення всередині:
```java
@Override                                  // компілятор перевірить, що це справді override
public boolean equals(Object obj) {
    if (this == obj) return true;
    if (obj == null || getClass() != obj.getClass()) return false;
    Point that = (Point) obj;
    return this.x == that.x && this.y == that.y;
}
```

**Висновок:** завжди став `@Override`. Якщо додати її до «пасткового» `equals(Point)`, отримаєш помилку компіляції *«method does not override...»* — баг спіймається одразу, а не як загадкова поведінка `HashMap`.

[до змісту](#java-core-частина-3-з-3)

## Чому хеш-код `31 * x + y` кращий за `x + y`?
Множник робить хеш залежним від **порядку** полів. Для `x + y` точки `(1, 2)` і `(2, 1)` дали б однаковий хеш (колізія), а `31 * x + y` — різний. Це зменшує колізії й покращує розподіл.

Чому саме `31`: це непарне просте число, а множення на нього JVM оптимізує як `(x << 5) - x` (зсув і віднімання). Саме цей множник використовує стандартний `String.hashCode()`.

## Розкажіть про клонування об'єктів.
Оператор присвоєння не створює нового об'єкта — лише копіює посилання (обидва вказують на той самий об'єкт). Щоб отримати **новий об'єкт із таким самим станом**, використовують клонування.

Вбудований механізм:
+ Клас `Object` має `protected` метод `clone()`, що виконує побітове копіювання об'єкта.
+ Щоб ним скористатися, клас має реалізувати маркерний інтерфейс **`Cloneable`** (інакше `clone()` кине `CloneNotSupportedException`) і перевизначити `clone()` як `public`, викликавши всередині `super.clone()`.
+ Об'єкт при цьому створюється **без виклику конструктора**.

Стандартний `clone()` робить **поверхневу** копію — цього достатньо лише для полів примітивних типів і незмінних об'єктів. Якщо є **мутабельні посилальні поля**, їх треба клонувати окремо (інакше оригінал і копія ділитимуть один вкладений об'єкт). Також `final` посилальні поля заважають такому клонуванню, бо їх не можна перепризначити.

Альтернативи вбудованому `clone()`:

**Конструктор копіювання** — приймає об'єкт того ж класу й копіює його поля:
```java
class Person {
    String name;
    List<String> hobbies;

    Person(Person other) {
        this.name = other.name;
        this.hobbies = new ArrayList<>(other.hobbies); // глибоко — новий список
    }
}
Person copy = new Person(original);
```

**Фабричний метод** — статичний метод, що повертає новий екземпляр (по суті, обгортка над конструктором копіювання):
```java
class Person {
    // ...
    static Person copyOf(Person other) {
        return new Person(other); // делегує конструктору копіювання
    }
}
Person copy = Person.copyOf(original);
```

**Серіалізація** — об'єкт записують у потік байтів і відновлюють назад; дає глибоку копію автоматично, але повільно (клас має реалізувати `Serializable`):
```java
ByteArrayOutputStream baos = new ByteArrayOutputStream();
try (ObjectOutputStream oos = new ObjectOutputStream(baos)) {
    oos.writeObject(original);          // серіалізація в байти
}
Person copy;
try (ObjectInputStream ois =
         new ObjectInputStream(new ByteArrayInputStream(baos.toByteArray()))) {
    copy = (Person) ois.readObject();   // десеріалізація → глибока копія
}
```

[до змісту](#java-core-частина-3-з-3)

## У чому різниця між _поверхневим_ і _глибоким_ клонуванням?
+ **Поверхневе (shallow)** — копіюються значення полів «як є». Примітиви копіюються за значенням, а для посилальних полів копіюється **саме посилання** → оригінал і копія вказують на ті самі вкладені об'єкти. Це поведінка `Object.clone()` за замовчуванням.
+ **Глибоке (deep)** — копіюються не лише поля, а й **усі вкладені об'єкти** рекурсивно → копія повністю незалежна від оригіналу. Кожне мутабельне посилальне поле теж має клонуватися.

Простими словами: після поверхневої копії зміна вкладеного об'єкта в копії зачепить і оригінал; після глибокої — ні.

[до змісту](#java-core-частина-3-з-3)

## Який спосіб клонування кращий?
Найбезпечніший і рекомендований — **конструктор копіювання** (або статичний фабричний метод), а не `Cloneable`/`clone()`:

+ немає проблем зі спадкуванням (не треба боятися, що в нащадка з'являться нові поля, які `clone()` не скопіює);
+ поля для копіювання задаються явно;
+ можна копіювати навіть `final`-поля;
+ не потрібен `Cloneable`, немає `CloneNotSupportedException`, повертається правильний тип.

Механізм `Cloneable`/`clone()` вважається невдалим рішенням мови (про це писав Джошуа Блох в «Effective Java»), і його уникають у новому коді.

[до змісту](#java-core-частина-3-з-3)

## Чому `clone()` оголошено в `Object`, а не в `Cloneable`?
`clone()` оголошено в `Object` як `native`, щоб надати доступ до стандартного механізму побітового копіювання (його неможливо реалізувати звичайним Java-кодом). Водночас він `protected`, щоб його не можна було викликати в об'єктів, які його не перевизначили. Інтерфейс `Cloneable` — **маркерний** (без методів) і лише позначає сам факт, що об'єкт готовий до клонування. Виклик `clone()` для об'єкта, що не реалізує `Cloneable`, спричиняє `CloneNotSupportedException`.

[до змісту](#java-core-частина-3-з-3)

## Опишіть ієрархію винятків.
Усі винятки мають спільного предка — клас **`Throwable`**. Його прямі нащадки:

+ **`Error`** — серйозні проблеми рівня JVM, які, за специфікацією, не варто обробляти у власному коді (наприклад, скінчилася пам'ять). Це unchecked.
+ **`Exception`** — проблеми в програмі, що в принципі розв'язні й передбачувані (наприклад, файл не знайдено). Серед них окремо стоїть:
    + **`RuntimeException`** — unchecked-винятки (наприклад, `NullPointerException`);
    + решта нащадків `Exception` — checked-винятки (наприклад, `IOException`).

```
Throwable
├── Error            (unchecked)
└── Exception        (checked)
    └── RuntimeException  (unchecked)
```

[до змісту](#java-core-частина-3-з-3)

## Що таке _checked_ і _unchecked_ винятки?
+ **Checked (перевірювані)** — мають бути або оброблені блоком `catch`, або оголошені в заголовку методу через `throws`. Це **перевіряється на етапі компіляції**. Нащадки `Exception` (окрім `RuntimeException`). Приклад: `IOException`.
+ **Unchecked (неперевірювані)** — не вимагають обов'язкової обробки чи оголошення. Це `RuntimeException` та його нащадки (`NullPointerException`, `IllegalArgumentException` тощо), а також `Error`. Перевіряються лише під час виконання.

Ідея: checked — для відновлюваних ситуацій, які викликач має свідомо обробити; unchecked — здебільшого програмні помилки (баги), яких не мало б бути.

[до змісту](#java-core-частина-3-з-3)

## Який оператор дозволяє примусово викинути виняток?
Оператор `throw`:
```java
throw new Exception();
```

[до змісту](#java-core-частина-3-з-3)

## Про що говорить ключове слово `throws`?
Модифікатор `throws` у заголовку методу вказує, що метод потенційно може викинути виняток зазначеного типу. Це частина контракту методу для checked-винятків — викликач зобов'язаний їх обробити або теж оголосити.

```java
void readFile() throws IOException { ... }
```

[до змісту](#java-core-частина-3-з-3)

## Як написати власний виняток?
Успадкуватися від базового класу потрібного типу — `Exception` (для checked) або `RuntimeException` (для unchecked):

```java
class CustomException extends Exception {
    public CustomException() {
        super();
    }

    public CustomException(final String message) {
        super(message);
    }

    public CustomException(final Throwable cause) {
        super(cause);
    }
}
```

Гарна практика — передбачити конструктор із `cause` (першопричиною), щоб зберегти ланцюжок винятків.

[до змісту](#java-core-частина-3-з-3)

## Які бувають _unchecked_ винятки?
Найпоширеніші: `ArithmeticException`, `ClassCastException`, `ConcurrentModificationException`, `IllegalArgumentException`, `IllegalStateException`, `IndexOutOfBoundsException`, `NoSuchElementException`, `NullPointerException`, `UnsupportedOperationException`.

[до змісту](#java-core-частина-3-з-3)

## Що являють собою помилки класу `Error`?
`Error` — найсерйозніші проблеми рівня JVM (наприклад, скінчилася пам'ять — `OutOfMemoryError`, або переповнення стека — `StackOverflowError`). Обробляти їх не заборонено, але не рекомендується, бо застосунок зазвичай не може коректно з них відновитися.

[до змісту](#java-core-частина-3-з-3)

## Що ви знаєте про `OutOfMemoryError`?
`OutOfMemoryError` викидається, коли JVM не може розмістити об'єкт через брак пам'яті, а збирач сміття не може звільнити достатньо місця. Тип помилки залежить від того, у якій області забракло місця:

+ `java.lang.OutOfMemoryError: Java heap space` — бракує місця в **купі** (де живуть створювані об'єкти). Зазвичай причина — витік пам'яті. Розмір задається `-Xms`/`-Xmx`.
+ `java.lang.OutOfMemoryError: Metaspace` — бракує місця в **Metaspace** (метадані класів). Розмір — `-XX:MaxMetaspaceSize`. *(До Java 8 це була помилка `PermGen space` з параметрами `-XX:PermSize`/`-XX:MaxPermSize`; PermGen прибрано у Java 8.)*
+ `java.lang.OutOfMemoryError: GC overhead limit exceeded` — GC працює майже безперервно, але звільняє надто мало пам'яті.
+ `java.lang.OutOfMemoryError: unable to create new native thread` — неможливо створити новий потік (вичерпано ресурси ОС).

[до змісту](#java-core-частина-3-з-3)

## Опишіть роботу блоку _try-catch-finally_.
+ `try` — позначає блок коду, що потенційно може спричинити виняток.
+ `catch` — блок для перехоплення та обробки винятку певного типу (якщо він виник).
+ `finally` — необов'язковий блок після всіх `catch`; його код виконується **завжди**, незалежно від того, був виняток чи ні (зазвичай для звільнення ресурсів).

```java
try {
    // код, що потенційно кидає виняток
} catch (SomeException e) {
    // обробка винятку
} finally {
    // необов'язковий блок, що виконується завжди
}
```

[до змісту](#java-core-частина-3-з-3)

## Що таке механізм _try-with-resources_?
Конструкція (з Java 7), що дозволяє використовувати `try`, не турбуючись про закриття ресурсів. Ресурси оголошуються в дужках одразу після `try`, а компілятор сам неявно генерує `finally`, де ресурси закриваються. Ресурс — це будь-який об'єкт, що реалізує інтерфейс **`AutoCloseable`** (його метод `close()` викликається автоматично).

```java
try (var reader = new BufferedReader(new FileReader("file.txt"))) {
    // робота з reader
} catch (IOException ex) {
    // ...
}
// reader закриється автоматично — навіть якщо виник виняток
```

Важливо: ресурси закриваються **до** виконання блоків `catch` і явного `finally`, і в порядку, зворотному до оголошення. Це чистіша й безпечніша альтернатива ручному закриттю у `finally`.

[до змісту](#java-core-частина-3-з-3)

## Чи можливе використання блоку _try-finally_ (без `catch`)?
Так, такий запис допустимий (наприклад, коли треба гарантовано звільнити ресурс, але виняток обробляється вище за стеком). Проте якщо виняток очікуваний і його можна обробити тут — краще мати блок `catch`.

[до змісту](#java-core-частина-3-з-3)

## Чи може один блок `catch` ловити кілька винятків?
Так, з Java 7 є **multi-catch** — кілька типів через `|`:

```java
try {
    // ...
} catch (IOException | SQLException ex) {
    // ...
}
```
Типи в multi-catch не повинні перебувати у відношенні «предок-нащадок» (інакше це надлишково). Змінна `ex` тут фактично `final`.

[до змісту](#java-core-частина-3-з-3)

## Чи завжди виконується блок `finally`?
Майже завжди — `finally` виконується незалежно від того, був виняток чи ні, і навіть якщо в `try`/`catch` є `return`.

[до змісту](#java-core-частина-3-з-3)

## Чи бувають ситуації, коли блок `finally` не виконається?
Так:
+ примусовий вихід із JVM — `System.exit(...)` у блоці `try`;
+ аварійне завершення JVM (фатальна помилка, `kill` процесу);
+ нескінченний цикл чи блокування потоку в `try` (керування ніколи не дійде до `finally`);
+ зупинка потоку-демона при завершенні застосунку.

```java
try {
    System.exit(0); // JVM завершується — finally недосяжний
} finally {
    // не виконається
}
```

[до змісту](#java-core-частина-3-з-3)

## Чи може `main()` викинути виняток назовні, і де він оброблятиметься?
Так, може. Якщо виняток не оброблено в `main()`, він передається у **JVM**, яка завершує потік, друкує stack trace у `System.err` і (для непійманого винятку в останньому ненульовому потоці) завершує застосунок. Можна перехопити такі винятки через `Thread.setDefaultUncaughtExceptionHandler(...)`.

[до змісту](#java-core-частина-3-з-3)

## У якій послідовності мають іти блоки `catch` для `IOException` і `FileNotFoundException`?
Загальне правило: обробляти винятки **від конкретнішого до загальнішого**. Не можна першим ставити `catch (Exception ex)` — він перехопить усе, і подальші блоки стануть недосяжними (помилка компіляції).

Оскільки `FileNotFoundException extends IOException`, спершу — `FileNotFoundException`, потім — `IOException`. Виконається **лише один** блок `catch` — перший відповідний.

```java
try {
    // ...
} catch (FileNotFoundException ex) {
    // спочатку конкретніший
} catch (IOException ex) {
    // потім загальніший
}
```

[до змісту](#java-core-частина-3-з-3)

## Що таке _generics_?
Головна ідея, з якої все випливає:

> **Generics — це параметр, але не для значення, а для типу.**

Порівняй: метод приймає **параметр-значення** (`void add(String item)`), а generic-клас приймає **параметр-тип** (`List<String>` — «скажи мені, з яким типом я працюю»). У дужках `<...>` ти ніби передаєш аргумент, тільки це **тип**, а не значення.

Аналогія: `List` — це коробка, а `List<String>` — коробка з наклейкою «тільки рядки». Компілятор стежить, щоб у неї не поклали нічого іншого.

### Навіщо: «було / стало»
**Було** (до Java 5) — колекції зберігали `Object`:
```java
List list = new ArrayList();
list.add("привіт");
list.add(42);                       // ніхто не заборонив покласти Integer
String s = (String) list.get(1);    // 💥 ClassCastException у РАНТАЙМІ + ручне приведення
```
**Стало** (з generics):
```java
List<String> list = new ArrayList<>();
list.add("привіт");
list.add(42);                       // ❌ помилка КОМПІЛЯЦІЇ — не зібереться
String s = list.get(0);             // приведення не потрібне
```

Звідси дві головні переваги:
+ **типобезпека на етапі компіляції** — помилку видно одразу, а не як `ClassCastException` у проді;
+ **без ручного приведення** — компілятор знає тип елементів.

### Три форми, які варто впізнавати
**Generic-клас** — `<T>` після імені класу:
```java
class Box<T> {
    private T value;
    void set(T value) { this.value = value; }
    T get() { return value; }
}
Box<String> b = new Box<>();   // тут T = String
```
**Generic-метод** — `<T>` перед типом повернення:
```java
static <T> T first(List<T> list) {
    return list.get(0);
}
```
**Обмеження (bounds)** — `extends` звужує дозволені типи:
```java
static <T extends Number> double sum(List<T> nums) { /* можна .doubleValue() */ }
```
> Літери — лише домовленість: `T` (Type), `E` (Element), `K`/`V` (Key/Value), `N` (Number).

### Стирання типів (type erasure)
> **Generics існують лише для компілятора. У байткоді вони стираються.**

Компілятор перевіряє типи, а потім **викидає** інформацію про `<...>`. У рантаймі `List<String>` і `List<Integer>` — це просто `List`. Зроблено так заради **сумісності назад** (generics додали в Java 5, і новий код мав працювати зі старими бібліотеками без generics).

Аналогія: generics — як **риштування** на будівництві. Поки будують (компіляція) — воно не дає помилитися; будинок готовий (байткод) — риштування прибрали.

Наслідок: оскільки в рантаймі типу вже немає, **не можна зробити того, що потребує типу під час виконання**:
+ `new T()` — ні;
+ `new T[]` — ні;
+ `obj instanceof List<String>` — ні (рантайм бачить лише `List`);
+ не можна мати два методи, що відрізняються лише параметром generic.

> **Стисло для співбесіди:** *«Generics — це параметризація типом: я кажу класу чи методу, з яким типом він працює (`List<String>`). Дає типобезпеку на компіляції та прибирає ручне приведення. Реалізовані через стирання типів — у рантаймі параметр зникає, тому не можна `new T()` чи `instanceof List<String>`».*

## Що таке _«інтернаціоналізація»_, _«локалізація»_?
+ __Інтернаціоналізація (i18n)__ — проєктування застосунку так, щоб його можна було легко адаптувати під різні мови та регіони **без зміни коду** (винесення текстів, форматів дат/чисел, валют тощо).
+ __Локалізація (l10n)__ — власне адаптація під конкретну мову/регіон: переклад текстів, формати дат, чисел, валют для певної `Locale`.

Спрощено: інтернаціоналізація — це підготовка застосунку до перекладу; локалізація — сам переклад під конкретну аудиторію. У Java для цього використовують `Locale`, `ResourceBundle`, `MessageFormat`, `NumberFormat`/`DateTimeFormatter`.
